🔑 认证接口 CLIENT
Authentication: Login / Logout / Refresh — Opaque Token + Redis Session
👤 用户在客户端输入账号密码登录，服务端验证凭证并检查 MFA 状态。认证通过后生成不透明 Token（Opaque Token），在 Redis 中创建 Session 并返回用户基本信息。Token 不含任何敏感信息，所有权限数据仅存储在服务端 Session 中。
POST /api/v1/auth/login 用户登录
📤 Request Body
字段类型必填说明
usernameString必填用户名
passwordString必填密码（加密传输）
mfaCodeString选填MFA 动态验证码（启用 MFA 时必填）
deviceIdString必填设备指纹标识
clientIpString选填客户端 IP（网关透传）
✅ Response (成功)
JSON { "code": 0, "data": { "token": "opaque_token_abc123def456", "expiresIn": 28800, // 8小时，单位秒 "userInfo": { "userId": 100001, "entityId": 1001, // 法律实体ID（牌照） "username": "zhangsan", "realName": "张三", "orgId": 2001, // 所属机构 "deptId": 3001, // 所属部门 "activeRoles": [1001, 1002], // 当前激活角色 "mfaVerified": true } } }
📌 Opaque Token：Token 本身不含任何数据，仅作为 Redis Session Key 的引用。服务端通过 Token 查找 Session 获取权限上下文，避免 JWT 膨胀和吊销困难问题。
⚠️ MFA 校验：若用户已启用 MFA 但未传 mfaCode，返回 HTTP 200 + code=MFA_REQUIRED，客户端需引导用户输入动态码后重新提交。
▼ ▼ ▼
POST /api/v1/auth/logout 用户登出
删除 Redis Session，Token 立即失效。客户端清除本地缓存的 Token。
POST /api/v1/auth/refresh 刷新 Token
📌 携带旧 Token 请求，服务端验证 Session 有效性后生成新 Token，旧 Token 立即失效（Token Rotation）。延长 Session 有效期。

🔍 权限校验 PDP INTERNAL
Permission Decision Point — Core RBAC+ABAC Engine, P99 < 20ms
🏦 交易引擎在处理用户下单请求前，通过内部 mTLS 通道调用 PDP（Policy Decision Point）进行权限校验。PDP 引擎在 20ms 内完成 RBAC 角色匹配 + ABAC 策略评估，返回 ALLOW/DENY 决策以及数据范围过滤规则和字段脱敏规则。所有决策记录异步写入审计日志。
POST /internal/v1/pdp/check 权限校验（核心）
📋 Request Headers
Header类型必填说明
X-Entity-IdLong必填法律实体ID
X-Trace-IdString必填全链路追踪ID
AuthorizationString必填mTLS 服务身份认证
📤 Request Body
JSON { "requestId": "req_20260419_trade_001", "subject": { "userId": 100001 }, "resource": { "appCode": "TRADE_SYSTEM", "resourceCode": "ORDER:CREATE", "dataContext": { "accountId": "ACC_001", "market": "HKEX" } }, "action": "EXECUTE", "environment": { "clientIp": "192.168.1.100", "deviceId": "iphone_abc123", "time": "2026-04-19T10:30:00Z" } }
✅ Response (ALLOW)
ALLOW { "decision": "ALLOW", "decisionReason": "RBAC_ROLE_MATCH + ABAC_POLICY_PASS", "hitPolicies": ["POL_TRADE_001", "POL_MARKET_HK"], "riskScore": 15, "dataScope": { "type": "DEPARTMENT", "filterExpr": "dept_id IN (3001, 3002)" }, "maskingRules": [ { "field": "idCard", "rule": "MASK_MIDDLE_6" }, { "field": "phone", "rule": "MASK_MIDDLE_4" } ], "expiresAt": "2026-04-19T10:31:00Z", "latencyMs": 8 }
🚫 Response (DENY)
DENY { "decision": "DENY", "decisionReason": "SOD_CONFLICT", "hitPolicies": ["SOD_RULE_005"], "denyMessage": "职责分离冲突：交易员不可同时审批", "riskScore": 92 }
🔒 性能要求：PDP 核心链路 P99 < 20ms。采用本地缓存(Caffeine) + Redis 二级缓存 + 预计算位图实现。降级策略：缓存穿透时返回 DENY + 告警。

📌 前端登录后一次性拉取所有菜单/按钮权限，缓存在客户端，减少逐项校验的网络开销。批量接口支持最多 200 个资源码。

JSON { "userId": 100001, "appCode": "TRADE_SYSTEM", "roles": [ { "roleId": 1001, "roleCode": "TRADER", "active": true }, { "roleId": 1002, "roleCode": "RISK_VIEWER", "active": true } ], "permissions": ["ORDER:CREATE", "ORDER:VIEW", "RISK:VIEW"], "abacPolicies": ["POL_TRADE_001"], "dataScope": { "type": "DEPARTMENT", "scopeExpr": "dept_id IN (3001, 3002)" }, "snapshotAt": "2026-04-19T10:30:00Z" }
⚠️ 快照用于调试和审计，不应用于实时权限判断。实时判断请使用 /pdp/check 接口。

🙋 用户自服务 CLIENT
Self-Service: My Permissions / Activate Roles / Request Access
👤 交易员张三登录后查看自己的权限列表。需要临时审批权限时，通过自服务界面提交权限申请，系统自动触发审批流。张三也可以动态激活/去激活不同角色组合，系统实时检查 SoD（职责分离）约束，防止冲突角色同时激活。
GET /api/v1/me/permissions?appCode={appCode} 查询我的权限
✅ Response
JSON { "code": 0, "data": { "userId": 100001, "activeRoles": [ { "roleId": 1001, "roleName": "交易员", "roleCode": "TRADER" }, { "roleId": 1002, "roleName": "风控查看", "roleCode": "RISK_VIEWER" } ], "permissions": [ { "resourceCode": "ORDER:CREATE", "actions": ["EXECUTE"] }, { "resourceCode": "ORDER:VIEW", "actions": ["VIEW"] }, { "resourceCode": "RISK:VIEW", "actions": ["VIEW"] } ], "dataScope": { "type": "DEPARTMENT" } } }
▼ ▼ ▼
POST /api/v1/me/activate-roles 激活角色（动态 SoD）
📤 Request Body
JSON { "roleIds": [1001, 1002] }
✅ Response
JSON { "code": 0, "data": { "activatedRoles": [1001, 1002], "sodCheckPassed": true, "sessionUpdated": true } }
🔒 SoD 实时校验：若请求激活的角色组合触发 SoD 冲突规则（如交易员+审批员），返回 code=SOD_CONFLICT 并拒绝激活，附带冲突规则说明。

👥 用户与角色管理 ADMIN
User CRUD / Role Management / Grant & Revoke / Inheritance Tree
⚙️ 管理员在权限管理后台创建用户、配置角色、绑定权限资源。角色支持层级继承（如"高级交易员"继承"交易员"的所有权限），管理员可随时查看继承树。通过 Grant/Revoke 精确控制权限分配，支持用户禁用、离职冻结、强制踢出等生命周期管理操作。
POST /admin/v1/users 创建用户
📤 Request Body
JSON { "entityId": 1001, "username": "zhangsan", "realName": "张三", "email": "zhangsan@company.com", "phone": "138****1234", "orgId": 2001, "deptId": 3001, "initialRoleIds": [1001] }
▼ ▼ ▼
GET /admin/v1/users/{userId} 用户详情
返回用户基本信息、所属机构/部门、已分配角色列表、账户状态、最后登录时间等。
POST /admin/v1/users/{userId}/disable 禁用用户
⚠️ 禁用后用户无法登录，已有 Session 在下次校验时失效。不删除数据，可恢复启用。
▼ ▼ ▼
POST /admin/v1/users/{userId}/offboard 离职冻结
🔒 离职冻结：1) 立即清除所有 Session 2) 回收全部角色 3) 标记 OFFBOARDED 4) 保留审计数据。不可恢复，需重新走入职流程。
POST /admin/v1/users/{userId}/kick 强制踢出所有会话
⚠️ 清除用户在 Redis 中所有 Session，强制所有设备重新登录。用于安全事件应急响应。

📜 策略与合规管理 ADMIN
ABAC Policy Lifecycle / SoD Rules / Zombie Grant Cleanup
📋 合规团队需要创建一条 ABAC 策略：限制某分支机构仅在工作日 9:00-17:00 可执行大额交易。策略经过草稿 → 审批 → 发布的完整生命周期管理。同时，系统定期扫描 SoD 冲突和僵尸权限（90天未使用），生成合规报告并支持批量回收。
POST /admin/v1/policies 创建 ABAC 策略（草稿）
📤 Request Body
JSON { "entityId": 1001, "policyCode": "POL_TRADE_TIME_LIMIT", "policyName": "交易时间限制策略", "effect": "DENY", // ALLOW / DENY "priority": 100, "conditions": { "allOf": [ { "attr": "environment.time.hour", "op": "NOT_BETWEEN", "value": [9, 17] }, { "attr": "resource.appCode", "op": "EQ", "value": "TRADE_SYSTEM" }, { "attr": "resource.resourceCode", "op": "EQ", "value": "ORDER:CREATE" } ] }, "description": "非工作时间(9-17点以外)禁止创建交易订单" }
📌 策略创建后状态为 DRAFT，不会立即生效。需经过审批 → 发布流程后方可在 PDP 引擎中参与决策。
▼ ▼ ▼
POST /admin/v1/policies/{id}/submit-approval 提交审批
⚠️ 策略变更推送至外部审批系统。高风险策略（涉及交易/资金）需双人审批。
POST /admin/v1/policies/{id}/publish 发布策略（支持定时）
📤 Request Body
JSON { "scheduledAt": "2026-04-20T09:00:00Z", // 可选，定时发布 "publishNote": "审批通过，定时次日生效" }
▼ ▼ ▼
POST /admin/v1/policies/{id}/rollback 回滚策略
🔒 回滚到上一个 PUBLISHED 版本。当前版本标记为 ROLLED_BACK。PDP 缓存立即刷新。
POST /admin/v1/policies/test Dry-run 测试

🔗 外部联动 INTERNAL
HR Events (Onboard/Offboard/Transfer) + Approval Callback
🏢 HR 系统在员工入职、离职、调动时自动推送事件到权限系统。入职事件触发用户创建+默认角色分配；离职事件触发权限冻结+Session 清除；调动事件触发组织归属变更+数据范围重算。外部审批系统审批完成后通过回调写回结果，权限系统据此执行授权或拒绝。
POST /internal/v1/events/hr/onboard HR 入职事件
📤 Request Body
JSON { "eventId": "HR_ONBOARD_20260419_001", "entityId": 1001, "employeeId": "EMP_20260419", "realName": "新员工A", "orgId": 2001, "deptId": 3001, "position": "交易员", "effectiveDate": "2026-04-20" }
入职事件处理：1) 创建用户账号 2) 根据岗位自动分配默认角色 3) 设置初始密码（需首次登录修改）4) 发送欢迎通知。
▼ ▼ ▼
POST /internal/v1/events/hr/offboard HR 离职事件
🔒 离职处理链：1) 冻结账号 2) 清除所有 Session 3) 回收全部角色 4) 触发数据交接流程 5) 标记 OFFBOARDED。
POST /internal/v1/events/hr/transfer HR 调动事件
⚠️ 调动处理：1) 更新 orgId/deptId 2) 重新计算数据范围 3) 保留通用角色，回收原部门专属角色 4) 分配新部门默认角色。
▼ ▼ ▼
POST /internal/v1/callbacks/approval 审批结果回调
📤 Request Body
JSON { "approvalNo": "APR_20260419_001", "approvalType": "ROLE_GRANT", // ROLE_GRANT / POLICY_CHANGE / ACCESS_REQUEST "result": "APPROVED", // APPROVED / REJECTED "targetUserId": 100001, "targetRoleId": 1003, "payload": { "orgScopeType": "BRANCH", "durationHours": 72 }, "approvedBy": [100001, 100002], // 审批人列表 "approvedAt": "2026-04-19T11:30:00Z" }
✅ Response
JSON { "code": 0, "data": { "processed": true, "grantId": 9010, "effectiveFrom": "2026-04-19T11:30:00Z", "effectiveTo": "2026-04-22T11:30:00Z" } }
📌 审批通过后自动执行 Grant 操作。REJECTED 时通知申请人并记录拒绝原因。所有回调必须验签 + 幂等处理。

📊 审计查询 INTERNAL
Audit Log Query — Full Traceability for Compliance
🔍 合规审计人员需要查询某用户在过去一个月内的所有权限变更和访问记录。审计日志包含登录/登出、权限校验决策、角色变更、策略变更等全部操作，支持按时间范围、用户、操作类型多维度检索。所有日志不可篡改，写入 WORM 存储。
