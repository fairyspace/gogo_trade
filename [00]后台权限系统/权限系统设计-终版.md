# 统一权限管理系统架构设计文档（V6.0）

> **设计说明**：基于 NIST RBAC/ABAC 标准，结合金融行业 IAM 工程实践，形成可直接落地的完整架构设计。采用渐进式复杂度原则，核心表结构 14 张，支持按需扩展（信息屏障、PAM 特权管理、Merkle 防篡改、区块链存证等模块均预留接入点，无需改动核心模型）。

---

## 一、系统定位与边界

### 1.1 系统定位

统一权限管理系统（PMS, Permission Management System）是集团的**统一身份治理与访问控制中台**，职责边界如下：

**负责：**
- 员工/外包/服务账号的统一身份生命周期管理（入职、转岗、离职、冻结）
- 多应用（CRM、柜台、风控、清算、运营后台等）SSO 单点登录
- 会话管理（Redis 集中管控，支持踢人、并发控制）
- RBAC + ABAC + PBAC 三层权限模型的统一维护与决策
- 角色体系、角色继承（闭包表）、SoD 职责分离互斥规则
- 行级数据范围控制 + 列级脱敏规则
- 审计日志采集、HMAC 防篡改、合规归档
- 风险评分、异常行为检测、动态策略决策
- 多法律实体（多牌照）与多司法辖区差异化合规

**不负责（由外部系统承担，按需扩展）：**
- 审批工作流 → 独立通用审批流服务，PMS 通过 API 提交+回调
- 特权凭据托管 → 配置中心 / KMS
- 信息屏障、区块链存证、PAM 特权管理 → 预留扩展点，Phase 5+ 按需叠加

### 1.2 核心设计原则

| 序号 | 原则 | 说明 |
|------|------|------|
| 1 | **Zero Trust 永不信任** | 所有请求无论内外网都需实时验证，不因 IP 内网而信任 |
| 2 | **最小权限 Least Privilege** | 默认拒绝，仅授予完成工作所必需的最小权限，临时权限强制过期 |
| 3 | **职责分离 SoD** | 四眼原则系统强制，交易录入≠审批、开发≠运维、管理≠审计 |
| 4 | **策略与内容分离** | 策略配置化（ABAC 策略 JSON + 管理后台维护），代码层不硬编码业务规则 |
| 5 | **DENY 优先 + 白名单模式** | DENY 策略优先级最高、直接拦截；未显式 ALLOW 即拒绝 |
| 6 | **可降级** | PDP 故障三级降级，高危操作降级必拒绝，宁可业务中断不可权限失控 |
| 7 | **合规内建** | 硬性合规（日志写入、离职冻结）代码层强制，不可绕过 |
| 8 | **审计不可篡改** | HMAC 签名 + WORM 冷存储，关键审计长期不可删改 |
| 9 | **多租户多辖区** | `entity_id` 贯穿全表，支持集团下多法律实体（多牌照）、多司法辖区差异化策略 |
| 10 | **性能优先** | 分层缓存（L1 Caffeine + L2 Redis + L3 DB），PDP P99 决策延迟 < 20ms |
| 11 | **渐进式复杂度** | 先交付 RBAC，再叠加 ABAC，最后 AI 风控，避免过度设计与实施烂尾 |
| 12 | **可观测与可追溯** | 每次决策带 `trace_id` + `decision_reason`，全链路可回溯 |

## 二、通用实践与落地方式

| 实践 | 本设计落地 |
| --- | --- |
| 联邦式 IAM 中台 | HR 主数据 → IAM 中台 → 各业务系统 SSO |
| SSO + MFA | Opaque Token + Redis Session + TOTP/FIDO2 |
| RBAC + ABAC + PBAC 三层混合 | 三层融合引擎，DENY 优先、ALLOW 白名单 |
| 策略即代码（Policy as Code） | 结构化 JSON 策略 + 版本化 + 定时生效 |
| SoD 职责分离 | 静态互斥授权前拦截，动态互斥会话激活时拦截 |
| 不可篡改审计 | HMAC 单条签名 + WORM 冷存储 |
| 数据分级 + 列级脱敏 | 数据分级字典 + 按角色动态脱敏 |
| 三层对象模型 | Subject → Grant → Permission 三表分离 |
| 多法律实体隔离 | `entity_id` 贯穿全表 |
| 审批工作流 | 独立通用审批流服务，PMS 通过 API 对接 |

### 3.2 认证方案：Opaque Token + Redis Session

不使用 JWT，采用服务端会话：Token 仅是随机字符串，会话状态存 Redis，踢人/冻结即时生效。

```
登录  → 生成随机 Token → Redis SET token:{value} = session（TTL 8h）→ 返回 Token
请求  → 网关 GET token:{value} → 未命中 401 / 命中 → 刷新 TTL + 鉴权
踢人  → DEL token:{value}（立即生效）
并发  → user_sessions:{userId} 有序集合，超限踢最早 session
```

会话数据完全存 Redis，不落 MySQL，审计日志记录登录/登出事件满足追溯。

### 3.3 两条控制链路

#### A. 在线鉴权链路（同步、低延迟）

适用于业务应用的每次 API/菜单/按钮访问鉴权。

**特点**：强一致、低延迟（P99 < 20ms）、高并发（峰值 50K QPS）、必须可降级。

#### B. 授权治理链路（异步、强合规）

适用于授权申请、审批、权限变更、SoD 扫描、僵尸权限清理。

**特点**：强流程、强审计、全留痕。审批环节由**独立审批流服务**承担，PMS 负责提交审批请求和接收审批结果回调。

> 两条链路**逻辑隔离**：鉴权链路独立部署，治理链路故障不影响在线鉴权。

### 3.4 三级降级策略

```
权限校验降级三级模式（PMS 故障时业务不失控）：

┌─────────────────────────────────────────────────────────────────┐
│  正常模式（PDP在线）                                              │
│  → L2 Redis 读取权限快照，命中率 > 95%                           │
│  → 未命中走数据库查询，实时执行 RBAC+ABAC+SoD 全套决策            │
│  → 延迟要求：P99 < 20ms                                         │
└───────────────────────────┬─────────────────────────────────────┘
                            │ PDP 错误率 > 5% 或 P99 > 100ms 持续 30s
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│  降级模式一：L1 本地缓存模式                                       │
│  → 使用网关/应用本地 Caffeine 缓存（TTL 30s）                    │
│  → 仅校验 RBAC 基础权限，跳过动态 ABAC/风险评分                  │
│  → 允许继续工作，但打上 decision=DEGRADED_ALLOW 标签              │
│  → 记录降级日志，事后 T+1 合规补偿审查                            │
└───────────────────────────┬─────────────────────────────────────┘
                            │ L1 缓存也失效或连续 1 分钟未刷新
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│  降级模式二：保守白名单模式                                        │
│  → 只放行预定义的"必须可用操作集合"（登录、查看自己的订单、查看持仓） │
│  → 所有写操作、大额、敏感数据全部拒绝                              │
│  → 触发 P0 告警 + 值班人员紧急干预                                │
│  → 降级期间所有请求记录原始上下文，恢复后自动重放审计              │
└─────────────────────────────────────────────────────────────────┘

关键规则：
• 降级期间，高风险操作（出金、大额交易、敏感数据导出、权限变更）
  必须全部拒绝，宁可业务中断，不可权限失控
• 降级状态由 PDP 健康检查自动判定，支持人工强制切换
• 每级降级均触发 P0 告警，SRE 值班必须 5 分钟内响应
```

## 四、权限模型与领域字典

### 4.1 三层权限模型

| 层级 | 模型 | 决策依据 | 适用场景 | 示例 |
|------|------|---------|---------|------|
| **基础层** | **RBAC** | 角色-权限点绑定 | 菜单、按钮、API、功能点 | 客户经理可访问"客户列表"菜单 |
| **控制层** | **ABAC** | 主体/资源/环境属性 | 数据范围、时间窗口、IP 约束 | 只能查看**本机构**客户数据 |
| **策略层** | **PBAC** | 业务规则 + 合规策略 | 动态业务规则、合规条件 | 大额出金需**双人审批**+**交易时段** |

### 4.5 SoD 职责分离互斥矩阵

| 角色 A | 角色 B | 互斥类型 | 违规处置 | 合规依据 |
|-------|-------|---------|---------|---------|
| `COUNTER_CLERK` 交易录入员 | `COUNTER_SUPERVISOR` 交易审批员 | **静态互斥** | 直接拒绝 | SOX、四眼原则 |
| `WITHDRAWAL_OPERATOR` 出金操作员 | `WITHDRAWAL_APPROVER` 出金审批员 | **静态互斥** | 直接拒绝 | 反洗钱双人控制 |
| `CRM_*` 客户经理类 | `COMPLIANCE_REVIEWER` 合规审查员 | **静态互斥** | 直接拒绝 | 利益冲突 |
| `ACCOUNT_OPENER` 开户员 | `ACCOUNT_CLOSER` 注销员 | **动态互斥** | 同会话拒绝 | 账户生命周期管控 |
| `GLOBAL_SYSADMIN` 系统管理员 | `GLOBAL_AUDITOR` 系统审计员 | **静态互斥** | 直接拒绝 | 管理与审计分离 |
| `RISK_MANAGER` 风控建模 | `RISK_DIRECTOR` 风控审批 | **静态互斥** | 触发审批豁免流程 | 模型开发与审批分离 |
| `TRADER` 交易员 | `SETTLEMENT` 清算员 | **静态互斥** | 直接拒绝 | 前中后台分离 |
| `IB_PARTNER` 经纪人 | `CRM_STAFF` 客户经理 | **静态互斥** | 直接拒绝 | 利益冲突 |

> **互斥类型说明**：
> - **静态互斥（Static SoD）**：授权时拦截，用户永远不能同时持有两个角色。
> - **动态互斥（Dynamic SoD）**：用户可以同时持有两个角色，但同一会话不能同时激活。

### 4.6 数据权限分级

| 范围类型 | 代码 | 说明 | 典型适用角色 |

| 自定义 | `CUSTOM` | 自定义机构 ID 列表或 SQL 条件 | 跨区域专员、特殊项目组 |

## 五、核心对象模型（三层对象模型）

权限系统采用 **Subject（主体） → Grant（授权关系） → Permission（权限点）** 三层对象模型。

| 层级 | 对象 | 说明 |
|------|------|------|
| 第一层 | **Subject（主体）** | 用户、服务账号、应用等可被授权的实体 |
| 第二层 | **Grant（授权关系）** | 主体与角色的绑定关系，含范围、有效期、授权人等元数据 |
| 第三层 | **Permission（权限点）** | 角色所绑定的具体资源操作权限（含 ALLOW/DENY） |

## 六、状态机

### 6.1 授权关系（User-Role Grant）状态机

```
（外部审批流服务审批通过后回调 PMS）
    │
    ▼
ACTIVE（正式生效，可鉴权）
    │
    ├──→ SUSPENDED（暂停，管理员手动或异常行为触发）
    │         │
    │         └──→ ACTIVE（解除暂停）
    │
    ├──→ EXPIRED（过期，临时授权到期自动流转）
    │
    └──→ REVOKED（回收，人工或离职联动回收，终态）
```

> 申请中（APPLYING）、待审批（PENDING_APPROVAL）、审批拒绝（REJECTED）等状态由外部审批流服务管理，PMS 仅在审批通过后创建 ACTIVE 状态的授权记录。

### 6.2 ABAC 策略版本状态机

```
DRAFT（草稿，运营/安全人员编辑）
    │
    ▼
PENDING_APPROVAL（待审批，提交到外部审批流服务）
    │
    ├──→ REJECTED（审批拒绝，回到 DRAFT 修改）
    │
    ▼
APPROVED（审批通过，待生效）
    │
    ▼（到达 effective_at）
ACTIVE（生效中，同 policy_code 旧版本自动进入 DEPRECATED）
    │
    ├──→ DEPRECATED（废弃，被新版本替代，保留 90 天供回滚）
    │
    └──→ ROLLED_BACK（被回滚，重新激活上一版本后进入此态）
```

## 七、核心流程

### 7.1 鉴权完整生命周期

```

         │
         │ Opaque Token + 资源 + 操作 + 上下文
         ▼
┌──────────────────────────────────────────────────────────────┐
│                    零信任网关 PEP                              │
│                                                              │

│  ┌────────────────────────────────────────────────┐         │
│  │ Redis GET token:{tokenValue}                    │         │

│  └────────────────────────────────────────────────┘         │
└──────────────────────────────┬───────────────────────────────┘
                               │
                               ▼
┌──────────────────────────────────────────────────────────────┐
│                    PDP 决策中心                                │
│                                                              │
│  Step 3: L2 Redis 缓存查询                                    │
│  ┌────────────────────────────────────────────────┐         │
│  │ 命中 PermissionSnapshot → 回填 L1，进入 Step 5  │         │
│  │ 未命中 → 进入 Step 4                            │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  Step 4: 数据库权限快照构建（首次加载 + 缓存穿透保护）          │

│  │ + 数据范围 + 脱敏规则                             │         │
│  │ 构建 PermissionSnapshot，写 L2 Redis（TTL 5min） │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  Step 5: 风险评分（异步并行计算，500ms 内完成）                 │
│  ┌────────────────────────────────────────────────┐         │
│  │ 评分维度：                                      │         │
│  │   - 设备信任度（新设备 +30）                     │         │
│  │   - 地理位置（异地 +20、异国 +40）               │         │
│  │   - 时间（非工作时间 +15）                       │         │
│  │   - 资源敏感度（敏感等级 4 +20）                 │         │
│  │   - 历史行为（近 7 天异常 +10-30）               │         │
│  │ score >= 80 → 直接 DENY                        │         │
│  │ 60 <= score < 80 → STEP_UP_REQUIRED            │         │
│  │ score < 60 → 继续                              │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  Step 6: DENY 优先策略检查（硬性拦截，最高优先级）              │
│  ┌────────────────────────────────────────────────┐         │
│  │ 6.1 全局 DENY（离职账号、冻结账号）               │         │
│  │ 6.2 SoD 动态互斥（会话激活的角色冲突）            │         │
│  │ 6.3 ABAC DENY 策略（优先级 1-50，按顺序执行）     │         │
│  │ 任一命中 → 立即 DENY，不检查 ALLOW              │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  Step 7: RBAC 基础权限检查（从快照读取，O(1) 查询）            │
│  ┌────────────────────────────────────────────────┐         │
│  │ resourceCode 在 allowedResources 集合中？       │         │

│  │   是 → 继续                                    │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  Step 8: ABAC 动态策略检查（按优先级顺序）                     │
│  ┌────────────────────────────────────────────────┐         │
│  │ 8.1 数据范围检查（资源 org_id 是否在用户范围内）  │         │
│  │ 8.2 环境约束（时间、IP、设备合规）                │         │
│  │ 8.3 业务规则（金额阈值、账户状态）                │         │
│  │ 全部 ALLOW → 允许                              │         │
│  │ 任一不满足 → DENY                              │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  Step 9: 返回决策 + 数据权限上下文                             │
│  ┌────────────────────────────────────────────────┐         │

│  │ expiresAt（决策缓存有效期）                      │         │
│  └────────────────────────────────────────────────┘         │
└──────────────────────────────┬───────────────────────────────┘
                               │

┌──────────────────────────────────────────────────────────────┐

│                                                              │
│  Step 10: 写审计日志（MQ 异步写入，不影响响应）                 │
│  Step 11: 高风险操作实时告警（score >= 60 触发 SIEM）          │
│  Step 12: 行为特征入库（供 UEBA 分析）                         │
└──────────────────────────────────────────────────────────────┘
```

## 十、优先级、SLA 与缓存策略

### 10.1 鉴权请求分级 SLA

| 优先级 | 典型场景 | 延迟 SLA | 降级策略 | 缓存策略 |
|--------|---------|---------|---------|---------|
| **P0 关键** | 交易下单前鉴权、出金审批 | P99 < 20ms | 降级必拒绝 | L1 本地缓存必命中 |
| **P1 重要** | 业务 API 鉴权、客户数据访问 | P99 < 30ms | 降级允许 RBAC | L1+L2 双级缓存 |
| **P2 普通** | 菜单/按钮可见性、报表查询 | P99 < 50ms | 降级允许 | L2 Redis 缓存 |
| **P3 低优** | 管理后台操作、配置变更 | P99 < 200ms | 降级直通 | 直读 DB |

### 10.2 缓存分层策略

```
┌──────────────────────────────────────────────────────┐
│  L1: 应用/网关本地 Caffeine                           │
│  • 容量：每节点 10 万条                                │
│  • TTL：30 秒                                        │
│  • 目标命中率：> 85%                                  │
│  • 失效：MQ Fanout 主动失效 + TTL 兜底                │
└──────────────────────────────────────────────────────┘
                      │ 未命中
                      ▼
┌──────────────────────────────────────────────────────┐
│  L2: Redis 集群                                       │
│  • TTL：5 分钟                                       │
│  • 缓存 KEY：pms:snapshot:{entityId}:{userId}:{app}   │
│  • 目标命中率：> 95%                                  │
│  • 失效：权限变更事件主动 DEL + TTL 兜底               │
│  • 防穿透：空对象缓存 + 布隆过滤器                     │
└──────────────────────────────────────────────────────┘
                      │ 未命中
                      ▼
┌──────────────────────────────────────────────────────┐
│  L3: MySQL 权威数据源                                 │
│  • 读写分离：写主库，读从库                            │
│  • 权限快照构建：单用户 < 50ms                        │
│  • 分库分表：按 entity_id 分库，用户表按 user_id 取模  │
└──────────────────────────────────────────────────────┘
```

- 单个用户最多授予角色数：**20** 个
- 角色继承深度上限：**3** 层
- 单个角色最多权限点：**500** 个
- 权限快照大小上限：**64KB**（超过拆分多个快照）

## 十一、高并发与高可用关键机制

### 11.1 PDP 水平扩展

```
PDP 决策中心：
• 无状态设计，所有状态在 Redis / DB
• 按 CPU/内存自动扩容，单实例 QPS 承载 1-2 万
• 多可用区部署，单可用区故障自动切换
```

### 11.2 风险评分异步化

```
风险评分流程不阻塞主鉴权链路：
• 低延迟核心：RBAC + ABAC 决策路径必须 < 20ms
• 风险评分：并行执行，500ms 内完成
  - 若评分在主决策返回前完成：合并到响应
  - 若评分超时：主决策先返回，评分结果异步入审计库
• 高风险行为：通过 SIEM 实时告警，人工二次确认
```

### 11.3 缓存雪崩防护

```
防护措施：
• 随机 TTL：基础 TTL 5 分钟 + 随机 0-60 秒，避免集中失效
• 热点 key 预热：调度器每 4 分钟刷新 Top 1000 活跃用户快照
• 布隆过滤器：过滤不存在用户的穿透请求
• 限流熔断：L3 DB 单机 QPS > 2000 自动熔断走降级
• 单飞（SingleFlight）：同一 key 的并发重建仅 1 次查询 DB
```

### 11.4 审计日志高性能写入

```
写入路径：
• 同步写入 MQ（Kafka audit-log Topic）：RT < 5ms
• 消费者批量写入 MySQL（1000 条或 1 秒触发）
• 详情表异步写入，主表不等待
• ES 索引：Kafka → Logstash → ES，实时检索

吞吐指标：
• 峰值写入：10 万 TPS
• 查询延迟：ES 全文检索 P99 < 500ms
• 冷数据：OSS + WORM，按需查询
```

### 11.5 PDP 熔断与降级

```
熔断策略：
• 健康检查：每 10s 探测 PDP 响应时间和错误率
• 连续 3 次错误率 > 5% → 触发降级模式一（L1 快照）
• 连续 5 次错误率 > 20% → 触发降级模式二（保守白名单）
• 降级期间每 30s 探测恢复，连续 3 次正常 → 恢复主模式
• 降级状态同步：通过 Redis 广播到所有 PEP 节点
```

## 十三、安全与合规设计

### 13.1 传输与存储安全

- **传输加密**：全链路 TLS 1.3，服务间 mTLS 双向认证
- **存储加密**：MySQL 透明加密（TDE）+ 列级加密（手机号、身份证）
- **凭据管理**：数据库密码等凭据通过配置中心加密存储，不落明文
- **敏感信息脱敏**：日志、审计、异常堆栈中自动识别并脱敏手机号/身份证/卡号
- **数据分级**：L1 公开 / L2 内部 / L3 机密 / L4 高度机密 / L5 绝密

### 13.2 身份认证安全

- **密码策略**：12 位以上、复杂度要求、90 天强制修改、历史 5 次不可重用
- **MFA 强制**：所有敏感操作、高风险设备必须 MFA（TOTP/FIDO2）
- **异常登录检测**：异地/异设备/异时间 → 强制 Step-Up 认证
- **会话管理**：Redis Session，8 小时无操作自动过期、风险评分过高强制重新认证、支持即时踢人
- **暴力破解防护**：连续 5 次失败锁定 30 分钟，告警通知用户

### 13.3 合规留痕矩阵

| 留痕内容 | 保留期限 | 存储介质 | 合规依据 |
|---------|---------|---------|---------|
| 登录/登出日志 | 5 年 | MySQL + ES + OSS | 等保 2.0 |
| 权限决策日志 | 5 年 | MySQL + ES + OSS（高危 WORM） | SOX |
| 授权变更日志 | 10 年 | MySQL + OSS WORM | SOX |
| 策略变更日志 | 10 年 | MySQL + OSS WORM | 内控要求 |
| 数据访问日志（明文） | 5 年 | MySQL + ES | 个人信息保护法 |

### 13.4 多司法辖区差异化

| 辖区 | 关键要求 | 本设计适配 |
|------|---------|-----------|
| **CN** | 等保 2.0、个保法 | 内网边界防护 + 数据不出境 + 数据本地化 |
| **HK** | SFC、PDPO | 客户数据本地存储 + 同意管理 |
| **SG** | MAS TRM、PDPA | 加密+审计+数据主权 |
| **US** | SOX、PCI-DSS | 四眼原则+审计+PCI |
| **EU** | GDPR、MiFID II | 数据主权+被遗忘权+跨境传输同意 |

## 十七、核心架构决策总结

| 决策点 | 本方案选择 | 理由 |
|--------|----------|------|
| **认证方案** | Opaque Token + Redis Session | 即时踢人/冻结，无 JWT 失效延迟问题，架构更简单 |
| **会话存储** | Redis（不落 MySQL） | 高性能、即时失效，审计日志覆盖追溯需求 |
| **权限模型** | RBAC + ABAC + PBAC 三层融合 | NIST 标准 + 动态策略双重需求 |
| **角色继承** | 闭包表，最大 3 层 | O(1) 查询，避免递归；超过 3 层复杂度爆炸 |
| **DENY 策略** | 默认不继承；全局 DENY 显式标记继承 | 避免意外权限剥夺 |
| **策略存储** | 结构化 JSON + 字段 schema 约束 | 可查询、可验证、可索引、可测试 |
| **策略更新** | 版本化 + 审批 + 定时生效 + 一键回滚 | 生产策略变更高风险 |
| **审批工作流** | **独立通用审批流服务** | 审批是通用能力，不应耦合在权限系统中 |
| **凭据管理** | 配置中心加密存储 | 够用、简单，无需 Vault 级别复杂度 |
| **审计写入** | 异步批量 + 主详表分离 + HMAC | 不影响 RT + 防篡改 |
| **缓存策略** | L1 Caffeine + L2 Redis + L3 DB 三级 + 主动失效 | 鉴权是热路径 |
| **降级策略** | 三级降级 + 高危降级必拒绝 | PMS 故障时业务不失控 |
| **多法律实体** | `entity_id` 贯穿全表 | 多牌照合规举证清晰 |
| **SoD 实施** | 静态+动态双重互斥 + 豁免审批流 | 四眼原则刚需 |
| **僵尸权限治理** | 定时任务扫描 + 通知，无独立表 | 轻量实现，避免过度设计 |
| **实施节奏** | 2 个月出 Phase 1，分 5 期迭代 | 业务等不起，避免烂尾 |
