# gogo-trade

零售 / 专业客户多资产 CFD·FX 经纪平台的**系统设计文档库**（非运行时代码）。覆盖 16 个业务域 + 平台跨域契约，方向设计不锁定实现语言。

---

## 架构总览

按业务域分层，一域一类权威数据；跨域只走事件 / RPC，禁止直连对方库表。

```text
L0 平台基座     权限 · 通知 · 跨域契约
L1 客户与合规   用户 · 准入/KYC · 营销 · 内容/社区
L2 账户与资金   账户 · 资金
L3 市场与产品   品种 · 交易配置 · 行情
L4 交易执行核   交易引擎 · 风控
L5 程序化生态   量化 · 社交交易(跟单/排行)
L6 渠道         代理 IB/Affiliate
L7 经营只读     结单 / BI / 监管报送
```

| 层 | 域 | 权威数据 |
|----|-----|----------|
| L0 | [00] 权限 · [03] 通知 · 跨域契约 | RBAC、送达、错误码/事件/ID |
| L1 | [04] 用户 · [01] 准入 · [14] 营销 · [15] 内容 | 身份、KYC、活动、资讯/IM |
| L2 | [05] 账户 · [09] 资金 | 账户主体、ledger/出入金 |
| L3 | [06] 品种 · [02] 交易配置 · [08] 行情 | 合约规格、路由/时段/杠杆、报价 |
| L4 | [10] 交易 · [11] 风控 | Order/Deal/Position、PreTrade/强平 |
| L5–L6 | [12] 量化 · [13] 社交 · [07] 代理 | API/回测、跟单、佣金账本 |

核心原则：成交只经交易引擎；余额变更只经引擎资金 RPC；跨域契约统一见《平台跨域契约补充卷》。

## 核心数据流

### 1. 开户链路

```text
注册(用户) → KYC/AML(准入) → 开户 RPC(账户) → 入金(资金)
         → 事件通知(通知) / 归因(代理)
```

### 2. 交易热路径

```text
下单 SubmitOrder
  → 预交易风控(内存) + 时段/杠杆/路由 RPC
  → 行情 Core SymbolBook 定价
  → 引擎原子提交：Deal + Position + ledger 意图 + Outbox
  → 事件 fan-out：通知 / 代理 / 社交 / 风控
```

### 3. 生态旁路

```text
量化信号 → 社交跟单放大 → SubmitOrder(source=SIGNAL)
营销发奖意图 → 资金 AdjustCredit
路由网关 → revenue.realized.v1 → 代理佣金计提
```

关键越界：IM 不得自动下单；营销不得改写归因；Deal 不可变（纠错用冲正）。

## 使用指南

1. **先读总览**：`[总览]经纪平台域地图.md` — 域边界、分层、主事件清单。
2. **再读契约**：`[平台]跨域契约补充卷.md` — 错误码 / 事件 / 角色 / ID / API / 容灾基线（跨域唯一权威）。
3. **按需深入**：进入 `[00]`–`[15]` 对应目录的 `*设计*.md`；部分域附有 `architecture.html` / `api.html` 可视化。

| 你想了解… | 从这里开始 |
|-----------|------------|
| 全平台边界与依赖 | `[总览]经纪平台域地图.md` |
| 事件名 / 错误码 / RPC 骨架 | `[平台]跨域契约补充卷.md` |
| 下单 / 成交 / 持仓 | `[10]交易系统/` |
| 出入金与账本 | `[09]资金体系/` |
| 报价与 Core Book | `[08]行情体系/` |
| 跟单与排行 | `[13]跟单与排行系统/` |

修订域文档时：跨域内容删除本地重复定义，统一引用补充卷对应章节。
