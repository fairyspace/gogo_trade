🪪 用户体系架构流程图

User Service · 9 Tables · 4 Auth Scenarios · Multi-Entity Isolation
💡 架构设计理念 (5 大原则)
用户体系作为交易系统的 身份承载、认证授权与权限映射 核心基座，不承担 KYC 审核等重业务。
1
多监管实体隔离
用户归属某 entity_id,决定其杠杆上限、负余额保护等规则;同一自然人可在不同实体(牌照品牌)下各注册独立账号。
2
适当性快照
KYC 在 Onboarding 完成,交易系统单向缓存适当性结果,避免高频交易时的跨服务 RPC。
3
IB 溯源解耦
主表固化 IB ID 作为分润基准,代理树与分润计算交由独立代理系统。
4
多态凭证与 OAuth 分离
密码 / 2FA / API Key 抽象为统一凭证表;OAuth 独立建表,遵循"授权"语义。
5
软硬风控分离
硬风控交由独立风控引擎,用户表仅保留"单日止损"等软风控偏好。

📥 多渠道注册入口
用户 通过以下任一渠道发起注册,网关按品牌/域名/App 标识确定 entity_id(多实体隔离)
📧 邮箱注册
📱 手机注册
🔵 Google OAuth
🍎 Apple OAuth
💬 微信/Line
🛡️ API Key 创建
👤 IB 邀请链接
📌 关键决策点:由 register_channel 字段(WEB/IOS/ANDROID/API/ADMIN/OAUTH)记录来源,触发不同的凭证初始化路径
📄 t_user (CREATE) 📄 t_user_preference (CREATE)
按 register_channel 分流到不同凭证初始化路径 🔀
2
🪪 凭证 & 适当性初始化
系统 根据渠道执行不同的初始化逻辑
2.1
邮箱/手机注册
1. 写入 t_user(status=0 待激活)
2. 创建 PASSWORD 凭证(bcrypt)
3. 异步发送激活链接/短信
2.2
OAuth 首次登录
1. 查 t_user_oauth 未命中 → 创建 user
2. status=0(未激活),待补充手机/邮箱
3. 写 OAuth 绑定记录
2.3
API Key 创建
1. 用户必须已存在(主账号)
2. 创建 API_KEY 凭证
3. 返回 AccessKey/Secret(仅一次)
2.4
KYC 同步(MQ)
Onboarding 通过后 MQ 推 UserApprovedEvent
触发 Upsert t_user_profile_snapshot
🔐 凭证密文:PASSWORD=bcrypt; TOTP=种子(AES加密); API_KEY=SecretKey(AES加密)
🔑 cred_identifier:PASSWORD=user_uid; API_KEY=AccessKey; TOTP/PASSKEY=credential_id
📄 t_user_credential (CREATE/UPSERT) 📄 t_user_oauth (CREATE) 📄 t_user_profile_snapshot (UPSERT,via MQ)
激活后进入 4 种登录鉴权场景 🔑
3
🔑 场景 A · 账号密码 / 手机验证码登录
📌 链路
  1. 用户输入 Email + Password;网关按品牌/域名/App 标识确定 entity_id
  2. 用 (entity_id, email) 查 t_user 解析 user_id(多实体隔离)
  3. 按 user_id + cred_type='PASSWORD' 查 t_user_credential,校验 bcrypt,检查 fail_count/locked_until
  4. 查 t_user_device 校验设备指纹;is_trusted=0 或 mfa_enabled=1 时触发二次验证
  5. 签发的 Token 内嵌当前 security_stamp;异步更新 last_login_at、last_active_at,写 t_user_login_log
✅
密码正确
校验通过
重置 fail_count=0
签发 Token
❌
密码错误
fail_count + 1
≥5次 → locked_until + 30min
写 login_log(fail_reason)
🛡️
需 2FA
设备未受信
或 mfa_enabled=1
跳 TOTP/Passkey 校验
📖 t_user (READ) 📖 t_user_credential (READ, PASSWORD) 📖 t_user_device (READ) 📄 t_user_login_log (CREATE)
或:场景 B OAuth / 场景 C API Key / 场景 D 令牌校验 ↘
4
🔵 场景 B · Google / Apple / 微信 第三方登录
📌 链路
  1. 客户端拉起第三方 SDK,得到 provider、open_id;网关确定 entity_id
  2. 查 t_user_oauth (entity_id, provider, open_id)
  3. 命中:取 user_id,发 Token 登录(能查到即有效绑定)
  4. 未命中:创建 t_user(status=0, register_channel=OAUTH) → 插入绑定记录 → 引导补充手机/邮箱(合规要求) → status=1
🔓 解绑流程
DELETE FROM t_user_oauth WHERE id=?
+ 写 t_user_operation_log(module=OAUTH, action=UNBIND)留痕
⚠️ 解绑前需校验至少保留一种可登录凭证
🤔 设计要点
✏️ OAuth 本质是"授权",仅存 OpenID
✏️ 纯登录场景不持久化第三方 Token
✏️ 按 entity_id 隔离唯一
📖 t_user_oauth (READ/CREATE/DELETE) 📄 t_user (CREATE, 未命中时) 📄 t_user_operation_log (CREATE,解绑时)
或:程序化交易登录 ⬇
5
🤖 场景 C · API Key 程序化交易登录
📌 链路
  1. 量化客户用 API Key/Secret 连接撮合网关
  2. 查 t_user_credential (cred_type='API_KEY'),校验 Secret 与 api_ip_whitelist
  3. 校验 api_permissions 含 TRADE → 放行
  4. 异步更新 last_used_at,写 t_user_login_log (event_type=API_KEY_CREATED/LOGIN)
📋 权限粒度 (api_permissions):READ(只读) · TRADE(下单/撤单) · WITHDRAW(出金)
🌐 IP 白名单 (api_ip_whitelist):逗号分隔,缺失即不限
⏰ 过期 (expired_at):NULL=永不过期;风控策略可批量下架老 Key
📖 t_user_credential (READ, API_KEY) 📄 t_user_credential (UPDATE last_used_at)
每次请求进入令牌校验 ⬇
6
🛡️ 场景 D · 令牌校验与全局下线
📌 链路
  1. 每次请求验签后,比对 Token 内 security_stamp 与 t_user.security_stamp(Redis 缓存),不一致即失效
  2. 改密码 / 重置 2FA / 全局冻结 / "退出所有设备"时刷新 security_stamp,历史 Token 立即全部失效
  3. status=2 (全局冻结) / status=4 (已注销) 用户在验签前直接拦截
✅
stamp 匹配
Token 有效
放行请求
业务系统继续
❌
stamp 不一致
强制重新登录
code = TOKEN_STALE
401 Unauthorized
⛔
用户被冻结/注销
status ∈ {2, 4}
验签前直接拒绝
code = USER_DISABLED
🎯 设计精髓:security_stamp 是"反向心跳"——无需遍历 Session,改一个字段就让所有历史 Token 失效
🔴 Redis (security_stamp 缓存) 📖 t_user (READ status/security_stamp)
登录后进入用户运营管理 ↘
7
📱 设备管理与信任机制
系统 每次登录自动 upsert t_user_device;受信设备有有效期,防止僵尸设备长期留存
设备字段 数据来源 业务规则
device_fingerprint 客户端 SDK 生成 UNIQUE (user_id, device_fingerprint)
is_trusted 用户主动标记 / 2FA 后自动 未受信设备登录需二次验证
expire_at 授信时计算(默认 90 天) 过期需重新验证
last_active_at 请求异步上报 长期不活跃可被自动清理
🛡️ "踢出所有设备":DELETE /users/{id}/devices → 删 t_user_device → 刷 security_stamp → 历史 Token 全部失效
📄 t_user_device (CREATE/UPSERT/DELETE) 📄 t_user (UPDATE security_stamp)

⏳ 注销冷静期 (status = 3)
用户 主动注销,进入冷静期(可撤回,默认 7-15 天)
📌 注销链路
  1. 用户提交注销 → status=3 (注销冷静期)
  2. 刷 security_stamp,所有在线 Token 立即失效
  3. 通知系统发送"撤回链接"到注册邮箱/手机
  4. 冷静期内可一键恢复 status=1
  5. 冷静期结束(定时任务)→ status=4 (已注销)
  6. 将 email/phone 改写为匿名占位值(释放复用,符合 GDPR "被遗忘权")
📄 t_user (UPDATE status, 匿名化 email/phone) 📄 t_user_operation_log (CREATE) 📨 MQ Notification (发送撤回链接)

🌐 系统边界交互
用户体系是交易系统的身份基座,与下列系统存在明确边界:
下游 / 上游系统 交互方式 数据走向
Onboarding
准入系统		 KYC 通过后经 MQ 发 UserApprovedEvent,交易系统创建/更新 t_user 并 Upsert t_user_profile_snapshot ⬇ 单向同步
(不反向查询)
Account
账号体系		 开户时账号体系按 t_user.entity_id 过滤可用 t_account_group,确保牌照隔离 ↔ 同步读取
IB
代理系统		 仅注册时落库 introducing_broker_id;代理系统按 IB 拉取用户列表 + 账单流水自行结算分润,用户体系不参与 ⬇ 写时落库
读时同步
Risk Control
风控引擎		 硬风控(制裁名单/刷单)直接拦截交易请求;软风控(daily_loss_limit)同步至风控 Redis,网关本地校验 ⬇ 软风控同步
Notification
通知服务		 用户体系不维护通知偏好;仅投递 {user_id, event_type} 到 MQ,通知系统自查偏好决定渠道 ⬇ 事件投递
🎯 关键设计要点 & 常见陷阱
1
凭证定位符设计
cred_identifier 用与用户绑定的全局唯一值:PASSWORD=user_uid、API_KEY=AccessKey、TOTP/PASSKEY=credential_id。先经 t_user 解析 user_id 再按 user_id 加载,避免"凭证明文 → user_id"反向解析。
2
注销匿名化
注销时 email/phone 改写为匿名占位值(例如 deleted_user_20260626_a1b2@anonymized.local),释放 (entity_id, email) 唯一槽位,允许他人重新注册,同时满足 GDPR "被遗忘权"。
3
适当性乱序防护
t_user_profile_snapshot.source_version 字段防 MQ 乱序/重复回写:仅当入参 version 更大时才覆盖,避免旧事件覆盖新事件。
4
OAuth 解绑物理删除
OAuth 解绑走 DELETE(释放 (entity_id, provider, open_id) 唯一槽位),解绑动作写 t_user_operation_log 留痕;解绑前需校验用户至少保留一种可登录凭证。
5
设备受信有效期
设备授信设置 expire_at(默认 90 天),过期需重新验证,防止僵尸设备长期留存;长期不活跃(last_active_at)可被自动清理。
6
日志分区滚动
登录日志与操作日志按季度 RANGE 分区;定时任务每季度滚动新增并 DROP 超出保留期的旧分区,避免表无限膨胀。日志不可修改(只 INSERT,合规审计要求)。
