# 资金系统设计 — 终版 v2.0

> 本文是资金域（Fund Service）的权威设计。对标 MetaQuotes / Spotware / Devexperts / Apex / DriveWealth / TT / Broadridge / Talos / 恒生电子等全球经纪与清算科技实践，并结合本平台已落地的账户体系、交易引擎契约与代理体系契约。
> **余额权威、原路退回粒度、幂等、拒付、客户资金隔离、事件契约以本文为准**。方向设计描述职责分层，**不锁定实现语言**。

---

## 一、系统定位与对标决策

### 1.1 系统定位

Fund Service 是经纪平台的**支付编排、客户资金生命周期、出款执行、公司账簿分录与外部对账中心**，支持：

- 法币入金 / 出金（PSP、银行电汇、电子钱包、本地支付）
- 同用户多交易账户内部转账（含跨币种换汇）
- 出金冻结、风控审核、原路退回（AML）
- 拒付（Chargeback）回拨与客户负债处理
- Partner 钱包出款执行（佣金账本仍属代理域）
- 客户 ledger 与公司双分录 journal 的日切对账

以下概念**不属于**本系统：

- 交易盈亏、佣金、Swap、分红的计算（交易/结算域写入 ledger）
- KYC/AML 身份审核本身（准入域；本系统只消费结果）
- 代理佣金计提与 Partner 余额账本（代理域）
- 撮合引擎内存保证金实时计算（引擎域；本系统只调用预检/资金 RPC）
- 美股清算托管、期货 SPAN、加密 Prime 全栈

### 1.2 行业参考的正确使用方式

| # | 厂商 | 可借鉴能力 | 本设计采纳 / 取舍 |
|---|------|-----------|------------------|
| 1 | **MetaQuotes** | Deal/Ledger 联动；Credit 不可提；Manager 资金操作接口形态 | 采纳 Credit 不可提、ledger 类型与交易联动；**不把 Manager 当完整支付/合规系统** |
| 2 | **Spotware (cTrader)** | 身份与多账户；内部转账 | 采纳同用户多账户转账；**不做多币种 Asset 钱包**（与账户体系决策一致） |
| 3 | **Devexperts** | 风险维度可提校验（含保证金水平） | 出金前 `PreviewWithdraw`；不做完整组合保证金账本 |
| 4 | **Apex Fintech** | API-first 出入金状态；拒付/回拨；账户类型约束 | 采纳 Chargeback、状态机、API 幂等 |
| 5 | **DriveWealth** | 资金指令编排、账实分离 | 采纳 Payout Instruction 统一出款；不做美股清算托管全栈 |
| 6 | **TradingView** | — | 不在资金域 |
| 7 | **Trading Technologies** | 账户级出金前风控阈值 | 阈值下沉到账户组/资金风控规则；不做 SPAN |

### 1.3 产品定位（拍板）

零售 / 专业客户多资产 CFD/FX 经纪平台资金域：

- **法币优先（P0）**，加密链上为 **P2**
- 单账户单结算币种；跨币种 = 同用户多账户 + 内部转账换汇
- 与《账户体系设计》《代理系统设计》《交易系统设计》引擎资金契约兼容

## 二、十二条不可破坏决策

1. **余额唯一变更路径**：任何改变交易账户 `balance / credit / frozen_balance` 的操作，必须经交易引擎 RPC：`CreditAccount / DebitAccount / FreezeFunds / UnfreezeFunds / AdjustCredit / TransferBetweenAccounts`。**Fund 编排层禁止直写** `t_trading_account` 资金字段。
2. **客户账本权威是 `t_account_ledger`**：链式 `balance_before → balance_after` 为客户侧真相；账户表资金字段仅为引擎同步快照。
3. **券商账簿双分录独立**：`t_journal_entry` + `t_journal_line` 记录客户资金负债、PSP 应收应付、手续费收入、汇兑损益；**不替代**客户 ledger。
4. **单账户单币种不变**：跨币种需求由内部转账 + 汇率快照满足。
5. **`entity_id` 贯穿**：所有资金聚合与流水带实体；禁止跨实体转账 / 出款 / 调账。
6. **幂等优先**：外部回调、引擎资金 RPC、审核动作均带 `idempotency_key`；重复请求返回首次结果。
7. **冻结先于扣款**：出金 / 争议先 `FreezeFunds`，成功再 `DebitAccount` 并释放冻结；拒绝则 `UnfreezeFunds`。禁止“先扣后补”。
8. **原路退回按入金批次 FIFO**：权威在 `t_deposit_lot` + `t_deposit_lot_consumption`；盈利部分才允许非原路通道；拒付优先消耗对应批次。
9. **Credit 永不计入可提**：可提金额以引擎实时态为准（含浮动盈亏与保证金缓冲）。
10. **事件经 Outbox**：至少发布 `deposit.created.v1`、`deposit.completed.v1`、`deposit.chargeback.v1`、`withdrawal.completed.v1`、`withdrawal.failed.v1`、`transfer.completed.v1`、`partner.payout.completed.v1`、`partner.payout.failed.v1`。
11. **四眼原则**：人工审核通过、人工调账、大额出金满足 SoD（权限系统 RBAC）；申请人 ≠ 审批人。
12. **冲正只追加不改历史**：拒付、失败回滚、纠错一律新增 ledger / journal 反向分录，禁止改历史金额。

## 三、边界与跨系统契约

### 3.1 领域边界

| 系统 | 权威数据 | Fund Service 的使用方式 |
|------|---------|------------------------|
| Onboarding / 准入 | KYC 等级、AML 状态、协议签署 | 消费结果驱动限额与通道可见性 |
| User | `user_id`、`entity_id` | 引用主体；不复制身份资料 |
| Account | 交易账户、账户组、`can_withdraw`、结算币种 | 读状态与实体；写余额只经引擎 |
| Trading Engine | 实时 balance/equity/margin、ledger 写入 | 资金 RPC + PreviewWithdraw |
| Partner | 佣金账本、结算单、Partner 可用余额 | Fund 仅执行 `PARTNER_WALLET` 出款 |
| Notification | 模板与送达 | Fund 只发标准业务事件 |
| PMS / 权限 | RBAC/ABAC/SoD | 审核与调账权限校验 |

### 3.2 引擎资金 RPC 契约（供 Fund 编排层调用）

所有写接口必须携带 `request_id`（幂等键）。重复 `request_id` 返回首次成功结果，不二次变更余额。

| RPC | 语义 | 典型调用方 |
|-----|------|-----------|
| `CreditAccount` | 增加 balance（或指定 ledger_type），写 ledger，刷新内存 | 入金入账、调账入、内部转入、模拟注资 |
| `DebitAccount` | 扣减 balance（需满足可扣约束），写 ledger | 出金完成、拒付扣回、调账出、内部转出 |
| `FreezeFunds` | 增加 frozen，不改 balance；写冻结明细关联 | 出金创建、争议冻结 |
| `UnfreezeFunds` | 释放冻结 | 出金拒绝/失败/取消 |
| `AdjustCredit` | 发放/回收 credit（不可提） | 运营赠金 |
| `PreviewWithdraw` | 只读：返回 withdrawable 与模拟扣款后 margin_level | 出金前置校验 |
| `TransferBetweenAccounts` | 同请求号原子转出+转入（可含汇率后金额） | 内部转账 |
| `GetAccountState` | 只读实时态 | 查询可提、运营看板 |

```protobuf
// 示意：engine/fund.proto
message CreditAccountRequest {
  int64  account_id     = 1;
  string amount         = 2;  // 十进制字符串，法币 4 位小数
  string currency       = 3;
  uint32 ledger_type    = 4;
  int32  direction      = 5;  // 恒为 +1
  string reference_no   = 6;
  int64  deposit_id     = 7;
  string request_id     = 8;  // 幂等键，必填
  int64  entity_id      = 9;
  string remark         = 10;
}

message FundMutationResponse {
  bool   success        = 1;
  string reject_reason  = 2;
  int64  ledger_id      = 3;
  string balance_after  = 4;
  string frozen_after   = 5;
  string equity_after   = 6;
}

message PreviewWithdrawRequest {
  int64  account_id     = 1;
  string amount         = 2;  // 拟提金额；空则只返回上限
  string request_id     = 3;
}

message PreviewWithdrawResponse {
  bool   allowed              = 1;
  string reject_reason        = 2;
  string withdrawable         = 3;
  string balance              = 4;
  string equity               = 5;
  string margin_used          = 6;
  string margin_level_after   = 7;  // 模拟扣款后
  string frozen_balance       = 8;
  string credit               = 9;
}
```

以上信封字段是引擎与本域协作时的业务语义细化；对外/跨域的规范信封以 `FundService` canonical proto 为准（见十三、13.1）。

### 3.3 关键事件（Outbox）

所有事件必须包含：`event_id`、`event_type`、`occurred_at`、`entity_id`、`aggregate_id`、`aggregate_version`、`correlation_id`、`payload`。生产方写库与写 Outbox 同事务；禁止“写库成功后尽力发 MQ”。

| 事件 | 生产方 | 主要消费方 | 平台事件表登记情况 |
|------|--------|-----------|------|
| `deposit.created.v1` | Fund | Notification | 已登记（跨域契约补充卷 §4.3） |
| `deposit.completed.v1` | Fund | Partner（首存 CPA）、Notification、Account 激活 | 已登记 |
| `deposit.chargeback.v1` | Fund | Partner（CPA 回拨）、Risk、Notification | 已登记 |
| `withdrawal.completed.v1` | Fund | Notification、报表 | 已登记 |
| `withdrawal.failed.v1` | Fund | Notification、运营 | 已登记 |
| `transfer.completed.v1` | Fund | Notification | 域内事件，尚未登记平台事件表，跨域订阅前需补充登记 |
| `partner.payout.completed.v1` | Fund | Partner | 已登记（由 Fund 执行出款后产出，供代理域消费出款状态） |
| `partner.payout.failed.v1` | Fund | Partner | 已登记（补充卷 §4.3，Fund 出款失败回调） |

## 五、金额模型与可提公式

### 5.1 精度

- 法币账户金额：`DECIMAL(18,4)`，与引擎定点数（×10000）对齐
- 汇率：`DECIMAL(18,8)`；加点前后均落快照
- 禁止在业务层使用 IEEE 浮点做资金运算

### 5.2 账户资金语义（与账户体系一致）

| 字段 | 含义 |
|------|------|
| `balance` | 已结算资金（不含浮动盈亏） |
| `credit` | 券商赠送、可交易、**不可提取** |
| `equity` | `balance + credit + floating_pnl` |
| `margin_used` | 已用保证金 |
| `frozen_balance` | 出金/争议等冻结合计（明细见 `t_account_fund_freeze`） |

### 5.3 可提公式（引擎权威）

```
equity       = balance + credit + floating_pnl
available    = equity - margin_used - frozen_balance

# credit 不可提：可提不得超过「非信用已结算资金」
cash_equity  = balance - frozen_balance   # balance 本身不含 credit

withdrawable = min(
  max(cash_equity, 0),
  max(available - margin_call_buffer, 0)
)

# 出金预检：模拟扣款 amount 后
#   margin_level_after = (equity - amount) / margin_used * 100
#   必须仍 >= account_group.margin_call_level（或更严的资金风控阈值）
# 无持仓(margin_used=0)时跳过 margin_level 检查，仅校验 withdrawable
```

`margin_call_buffer` 的权威来源是跨域契约事件 `margin.call.buffer.resolved.v1`（风控域在 PreTrade 前解析并发布）：

```
margin_call_buffer = max(
  账户组 margin_call_level 推导缓冲,      // = (1 - margin_call_level/100) × equity
  t_fund_risk_rule 更严值,                // 资金风控规则
  账户级 margin_call_level_override       // 账户覆盖（见账户体系 §9.2）
)
```

资金域**不自行推算** `margin_call_buffer`，只消费风控域发布的结果；`PreviewWithdraw` 内部引用该值。

出金创建前必须调用 `PreviewWithdraw`；Redis `fund:available:{account_id}` 仅为展示投影，**不得**作为放行依据。

### 5.4 原路退回与盈利

```
某支付方式可退余额 = Σ lot.remaining_amount（同 user + channel + payment_method + currency）
用户总可退本金     = Σ 所有 lot.remaining_amount（同 user，按账户币种折算后）
盈利可自选部分     = max(withdrawable - 总可退本金折算, 0)
```

出金分配规则：

1. 优先按 FIFO 消耗可退 lot（原路通道 + 原支付方式）
2. 超出本金的部分记为 `PROFIT_PORTION`，允许自选已验证通道
3. 监管要求强制原路时，盈利部分仍可限制为白名单通道

## 六、核心状态机

### 6.1 入金 `t_deposit_order`

```
CREATED → PENDING_PAY → PAID → CONFIRMING → CREDITING → CREDITED → COMPLETED
                 │         │         │           │
                 │         │         │           └→ RECONCILE（引擎已入账但lot/journal未闭环, 见八、8.1）
                 │         │         └→ REJECTED（风控拒绝，走退回/不入账, 见八、8.3）
                 │         └→ FAILED
                 └→ EXPIRED / CANCELLED
```

| 状态 | 含义 |
|------|------|
| CREATED | 单已建 |
| PENDING_PAY | 等待用户在 PSP/链上完成支付 |
| PAID | 外部已确认收款（金额/币种已核对） |
| CONFIRMING | 资金风控审核中 |
| CREDITING | 已调用引擎入账，等待确认（防超时重试） |
| CREDITED | 引擎入账成功，lot 已生成 |
| COMPLETED | journal/outbox 收尾完成 |
| RECONCILE | 引擎入账成功但 lot/journal 持续写入失败，转人工核对（见八、8.1） |
| FAILED / EXPIRED / CANCELLED / REJECTED | 终态 |

### 6.2 出金 `t_withdrawal_order`

```
CREATED → FROZEN → PENDING_REVIEW → APPROVED → PAYOUT_SUBMITTED ──┬→ COMPLETED
             │           │                          │             │
             └→ REJECTED └→ CANCELLED                │             ├→ FAILED（可重试或转REJECTED解冻）
                                                      │             └→ PAYOUT_DONE_DEBIT_PENDING（外部已出款,引擎Debit待重试, 见八、8.2）
                                                      │                    └→ COMPLETED / RECONCILE（重试耗尽转人工）
```

| 状态 | 含义 |
|------|------|
| CREATED | 校验通过，待冻结 |
| FROZEN | 引擎冻结成功，lot 消耗预占已写入 |
| PENDING_REVIEW | 待自动/人工审核 |
| APPROVED | 审核通过，待出款 |
| PAYOUT_SUBMITTED | 已提交 PSP/银行/链上 |
| PAYOUT_DONE_DEBIT_PENDING | 外部支付已成功，但引擎 DebitAccount 尚未确认成功，进入受控重试（见八、8.2） |
| COMPLETED | 外部确认成功 + 引擎 Debit 成功 |
| RECONCILE | `PAYOUT_DONE_DEBIT_PENDING` 重试耗尽，转人工核对账实差异 |
| REJECTED / CANCELLED / FAILED | 终态（失败路径必须 Unfreeze + 释放 lot 预占） |

### 6.3 内部转账 `t_internal_transfer`

```
CREATED → PROCESSING → COMPLETED
                └→ FAILED
```

同事务语义由引擎 `TransferBetweenAccounts` 保证；Fund 侧状态为编排投影。

### 6.4 拒付 `t_chargeback`

```
OPEN → RECOVERING → FUNDS_RECOVERED → CLOSED
                 └→ DEFICIT_CREATED → CLOSED
```

## 七、核心业务流程

### 7.1 入金（Deposit）

```
用户发起入金
  → 校验账户状态 / entity / 通道 / KYC / 限额
  → 创建 t_deposit_order (PENDING_PAY) + 幂等键
  → 法币: 调 PSP 获取支付链接
    加密(P2): 分配充值地址
  → PSP/链上回调 (idempotency = provider+psp_order_id)
      → 验签、金额币种核对、汇率快照
      → status=PAID → 风控
          PASS → CREDITING
               → Engine.CreditAccount(request_id=deposit_no)
               → 写 t_deposit_lot
               → 写公司 journal（借: PSP应收/银行  贷: 客户资金负债）
               → Outbox deposit.completed.v1
               → CREDITED → COMPLETED
          HOLD → CONFIRMING → 人工四眼 → 同上或 REJECTED
          REJECT → 不入账，原路退回/标记失败（见八、8.3）
```

**关键约束：** 仅引擎入账成功后才存在客户可见余额；PAID 但未 CREDITED 不计可交易资金。

### 7.2 出金（Withdrawal）

```
用户发起出金
  → Engine.PreviewWithdraw
  → 原路 FIFO 规划（t_deposit_lot）+ 盈利部分通道校验
  → 冷静期 / 限额 / 密码修改锁 / 账户 can_withdraw
  → 创建订单 CREATED
  → Engine.FreezeFunds(request_id=withdrawal_no:freeze)
  → 写 t_account_fund_freeze + lot consumption(RESERVED)
  → FROZEN → 风控路由
       自动通过 / 人工四眼 → APPROVED
       拒绝 → Unfreeze + 释放 RESERVED → REJECTED
  → 创建 t_payout_instruction → 适配器出款 → PAYOUT_SUBMITTED
  → 外部成功回调
       → Engine.DebitAccount(request_id=withdrawal_no:debit)
       → freeze 状态→DEBITED，lot consumption→CONSUMED
       → journal（借: 客户资金负债  贷: PSP应付/银行）
       → Outbox withdrawal.completed.v1
       → 若 Debit 失败见八、8.2
  → 外部失败
       → Unfreeze + 释放 RESERVED
       → Outbox withdrawal.failed.v1
```

### 7.3 内部转账

```
校验: 同 user OWNER、同 entity、源账户可转、币种/汇率
  → Engine.TransferBetweenAccounts(request_id=transfer_no)
       写 TRANSFER_OUT + TRANSFER_IN 两条 ledger
  → 跨币种: 落 t_exchange_rate_snapshot + journal 汇兑损益
  → Outbox transfer.completed.v1
```

### 7.4 拒付 Chargeback

```
PSP 拒付通知 (幂等)
  → 创建 t_chargeback OPEN
  → 定位原 deposit + lots
  → Engine.DebitAccount 或不足额时部分扣 + 账户限制
  → lot 标记 chargeback 消耗
  → journal 冲回客户负债 / 记损失
  → Outbox deposit.chargeback.v1（供 Partner CPA 回拨）
  → FUNDS_RECOVERED 或 DEFICIT_CREATED → CLOSED
```

### 7.5 AML 深度检测

资金域在入金/出金流程中叠加以下检测，命中后写 `t_fund_risk_check_log` 并可转人工：

- **快进快出检测**：入金后 24h 内出金 &gt; 80% 且期间无交易 → 标记为可疑（`check_result=4`），转人工审核。
- **Structuring 检测**：单日多笔入金金额恰好低于监管申报阈值 → 标记可疑。
- **制裁/AML 命中后的账户联动**：资金域**消费**用户/风控域发布的 `user.frozen.v1`（而非生产该事件），收到后将关联账户 `can_withdraw` 回写为 0，阻断后续出金申请；制裁判定本身由准入域 `ONB_STATE_AML_SANCTION_HIT` 触发，两个动作通过事件解耦，资金域不越权判定制裁。
- **SAR/STR 报送**：合规官（`PMS_COMPLIANCE_OFFICER`）审批后报送监管，留痕 `t_fund_compliance_log`。

## 八、异常处理与补偿 Saga

分布式支付编排里，以下三类中间态是真实会发生且必须显式设计补偿路径的场景，而不能依赖人工偶然发现。

### 8.1 CREDITING 中间态（引擎已入账，Fund 后续写入失败）

```
入金 PAID → 引擎 CreditAccount 成功 → Fund 写 lot/journal 失败
  检测：定时扫描 status=CREDITING 且停留超过 60s（监控指标 deposit_stuck_in_crediting，见十四、14.2）
  Saga：
    1. 重试 Fund 写 lot/journal（幂等，同 request_id）
    2. 重试 N 次仍失败 → status=RECONCILE，告警人工
    3. 人工介入：核对引擎余额 vs ledger，手动补 lot/journal 或按审计流程处理
  SLO：由本场景产生的 RECONCILE 数量目标 = 0（见十四、14.1）
```

### 8.2 PAYOUT_SUBMITTED 中间态（PSP 已出款，Debit 失败）

```
出金 PAYOUT_SUBMITTED → PSP 回调成功 → Fund DebitAccount 失败
  status = PAYOUT_DONE_DEBIT_PENDING
  Saga：
    1. 锁定账户 can_withdraw=0，防止重复出金
    2. 重试 DebitAccount（幂等，同 request_id）
    3. 重试失败 → status=RECONCILE，告警 P1（监控指标 withdraw_payout_done_debit_pending）
    4. 人工：核对 PSP 流水 vs ledger，手动补扣或挂账待追回
  SLO：由本场景产生的 RECONCILE 数量目标 = 0
```

### 8.3 PAID 后风控拒绝需原路退回

```
入金 PAID → 风控/AML 拒绝（发生在 CONFIRMING 阶段，尚未调用 CreditAccount）
  → status = REJECTED
  → 调用 PSP Refund（原路）
  → PSP 退款成功 → 公司 journal 冲正：借 SUSPENSE / 贷 PSP_PAYABLE
     （客户从未被 CreditAccount，不产生客户 ledger 分录，只有公司账簿动作）
  → 退款失败 → 人工工单，资金暂挂 journal SUSPENSE 科目待追踪
  SLA：拒绝后 24h 内发起退款
```

## 十、Ledger 全链路联动

### 10.1 写入方

| ledger_type | 写入方 | 触发时机 |
|-------------|--------|----------|
| DEPOSIT / WITHDRAWAL / TRANSFER_* / CHARGEBACK / ADJUSTMENT / CREDIT_* / FEE(资金侧) | 引擎（由 Fund 发起 RPC） | 出入金/转账/拒付/调账 |
| TRADE_PROFIT / COMMISSION / SWAP / DIVIDEND / FEE(交易侧) | 引擎（交易路径） | 成交/日切/公司行动 |

**任何路径最终都是引擎写 ledger**；Fund 与交易子系统都不直连改 balance。

### 10.2 入金入账链路

```
t_deposit_order: PAID → 风控通过 → CREDITING
  → Engine.CreditAccount(ledger_type=DEPOSIT, request_id=deposit_no)
  → t_account_ledger 新增一行, balance_after 更新
  → 引擎内存 balance 更新
  → Fund: 写 t_deposit_lot, journal, outbox, status=CREDITED/COMPLETED
```

### 10.3 出金完成链路

```
t_withdrawal_order: PAYOUT_SUBMITTED + 外部成功
  → Engine.DebitAccount(ledger_type=WITHDRAWAL, request_id=withdrawal_no:debit)
  → 同时将对应 freeze 标记为已扣减(引擎侧 frozen 减少)
  → Fund: lot CONSUMED, journal, outbox, COMPLETED
```

### 10.4 出金拒绝（无 ledger）

```
REJECTED → Engine.UnfreezeFunds → freeze RELEASED → lot RESERVED 释放
balance 不变
```

### 10.5 与交易 ledger 的一致性约束

任意时刻：

```
最后一条 ledger.balance_after == 引擎内存 balance == t_trading_account.balance（最终一致快照）
```

日切任务校验链式连续性与账户余额（见十二、对账与审计校验）。

### 10.6 分区与归档

`t_account_ledger` 按月 RANGE 分区，支撑长期高吞吐写入与归档：

```sql
ALTER TABLE t_account_ledger PARTITION BY RANGE (TO_DAYS(created_at)) (
  PARTITION p202607 VALUES LESS THAN (TO_DAYS('2026-08-01')),
  PARTITION p202608 VALUES LESS THAN (TO_DAYS('2026-09-01')),
  ...
  PARTITION pmax VALUES LESS THAN MAXVALUE
);
```

- 月度分区按季度批量预建，任务自动新增下季度分区。
- 冷归档：超过 2 年的数据迁移至冷存储（S3 + 索引），保留审计查询能力。
- 覆盖索引 `(account_id, id)` 支持按账户顺序扫描。

## 十二、对账与审计校验

### 12.1 日切对账作业

1. 拉取 PSP Settlement / 银行对账文件
2. 与当日 `COMPLETED` 入金/出金逐笔匹配 → `t_fund_reconciliation(_detail)`
3. 校验 journal 借贷平衡
4. 校验 `Σ CLIENT_MONEY_LIABILITY` 与客户 ledger 净入金口径（按实体）
5. **三方比对**（T+1 08:00 前完成）：

```
1. 引擎内存余额快照（dump 至 t_account_snapshot_eod）
2. ledger Σ balance_after 按 account_id 聚合
3. 账户快照 t_trading_account.balance
4. 比对：
   - 引擎 vs ledger：不一致 → 自动修复（以 ledger 为准，引擎重放）
   - ledger vs 账户快照：不一致 → 告警人工（账户快照为投影，可能 lag）
5. 阈值：自动修复 < 1 USD，超阈值人工工单
```

6. 差异工单人工处理，禁止直接改历史流水

监控指标：`account_ledger_mismatch`（见十四、14.2）。

### 12.2 审计 SQL

```sql
-- 校验1: 账户余额 vs 最后一条 ledger
SELECT a.id, a.account_no, a.balance AS account_balance, l.balance_after AS ledger_balance,
       CASE WHEN a.balance = l.balance_after THEN 'OK' ELSE 'MISMATCH' END AS check_result
FROM t_trading_account a
LEFT JOIN (
    SELECT account_id, balance_after,
           ROW_NUMBER() OVER (PARTITION BY account_id ORDER BY id DESC) AS rn
    FROM t_account_ledger
) l ON l.account_id = a.id AND l.rn = 1
WHERE a.status IN (0, 1, 2);

-- 校验2: ledger 链式连续
SELECT curr.id, curr.account_id, curr.balance_before, prev.balance_after,
       CASE WHEN curr.balance_before = prev.balance_after THEN 'OK' ELSE 'CHAIN_BROKEN' END AS check_result
FROM t_account_ledger curr
JOIN (
    SELECT id, account_id, balance_after,
           LEAD(id) OVER (PARTITION BY account_id ORDER BY id) AS next_id
    FROM t_account_ledger
) prev ON prev.next_id = curr.id AND prev.account_id = curr.account_id
HAVING check_result = 'CHAIN_BROKEN';

-- 校验3: 分录借贷平衡
SELECT e.id, e.entry_no,
       SUM(CASE WHEN l.dc_flag='D' THEN l.amount ELSE 0 END) AS debit_sum,
       SUM(CASE WHEN l.dc_flag='C' THEN l.amount ELSE 0 END) AS credit_sum
FROM t_journal_entry e
JOIN t_journal_line l ON l.entry_id = e.id
WHERE e.status = 1
GROUP BY e.id, e.entry_no
HAVING debit_sum <> credit_sum;

-- 校验4: lot 守恒
SELECT id, lot_no, original_amount, remaining_amount, reserved_amount, consumed_amount,
       CASE WHEN original_amount = remaining_amount + reserved_amount + consumed_amount
            THEN 'OK' ELSE 'LOT_BROKEN' END AS check_result
FROM t_deposit_lot;
```

| `/api/v1/callbacks/{pspCode}/chargeback` | POST | PSP 拒付通知 |

### 13.3 Webhook 验签规范

- Header `X-PSP-Signature: <algo>=<hex>`，HMAC-SHA256(payload, shared_secret)。
- 时间戳防重放：`X-PSP-Timestamp`，±5min 窗口。
- 幂等：`X-PSP-Event-Id` 作为 `idempotency_key`。
- 验签失败返回 `FUND_EXTERNAL_PSP_WEBHOOK_INVALID`（403）。

## 十六、分阶段落地与验收

### P0 — 法币可上线

**范围：** 通道、收付款方式、入金、出金、冻结、引擎 Credit/Debit/Freeze、客户 ledger、幂等、基础风控、人工审核、最小 journal、Outbox（deposit/withdrawal completed）。

**验收：**

1. 重复 PSP 回调不重复入账（幂等）
2. Fund 进程无任何 SQL 更新 `t_trading_account.balance`
3. 出金拒绝后余额与冻结恢复正确
4. 入金完成后 ledger 链式连续且与引擎一致
5. 发出 `deposit.completed.v1` / `withdrawal.completed.v1` 且可被 Inbox 消费一次

### P1 — 合规与内部资金

**范围：** 内部转账换汇、lot FIFO 原路退回、日切对账、Chargeback、四眼审批完善、PreviewWithdraw 有持仓场景、异常处理 Saga（见八）。

**验收：**

1. 本金出金强制原路；盈利部分可自选
2. 拒付触发 `deposit.chargeback.v1`，Partner 可回拨
3. 对账差异可定位到单笔并闭环
4. 审批人不能等于提交人
5. `CREDITING_STUCK`/`PAYOUT_DONE_DEBIT_PENDING` 均能按 Saga 自动重试并在耗尽后转 `RECONCILE`

### P2 — 加密与监管报表

**范围：** 地址池、链上监听、批量出款、客户资金监管报表（CLIENT_MONEY_LIABILITY vs 银行/PSP 余额）。

**验收：**

1. 确认数达标才入账；重组/双花有拒绝路径
2. 监管报表按 `entity_id` 日切可导出

## 十八、跨域契约对齐

本域遵循《跨域契约补充卷》的平台级约定：

### 18.1 错误码

遵循补充卷 §三命名规则 `<DOMAIN>_<CATEGORY>_<DETAIL>`，使用 `FUND_*` 前缀（完整列表见十四、14.3）。

### 18.2 事件

遵循补充卷 §四 CloudEvents envelope。`deposit.created/completed/chargeback.v1`、`withdrawal.completed/failed.v1`、`partner.payout.completed/failed.v1` 已登记；`transfer.completed.v1` 仍为域内事件，若下游需要跨域订阅应按 MINOR 版本补充登记（见三、3.3）。

### 18.3 角色

出金操作与审批遵循补充卷 §五：`PMS_FUND_PAYOUT_OPERATOR` 与 `PMS_FUND_PAYOUT_APPROVER` 互斥（四眼原则，见二、决策 11）。

### 18.4 ID 与幂等

`request_id` 由调用方生成（补充卷 §六）；`reward_id`（营销发奖幂等）由营销域生成，资金域仅在 `AdjustCredit` 场景接收。

### 18.5 API

内部 gRPC 与对外 HTTP/Webhook 见十三；均遵循补充卷 §九 `FundService` 信封。

### 18.6 容灾 / 容量 / 监控

遵循补充卷 §十/§十一/§十二：ledger RPO=0、RTO &lt; 5min、同城双活；容量假设与 SLO 见十四、14.1。

### 18.7 entity_id

遵循补充卷 §六/§七：所有资金聚合与流水贯穿 `entity_id`（二、决策 5）；禁止跨实体转账/出款/调账。

## 结语

本资金体系以三句话收口：

1. **Fund 编排支付与合规，Trading Engine 独占客户余额写入。**
2. **客户单边 ledger 保交易真相，公司复式 journal 保资金隔离与报表。**
3. **批次级原路退回 + 幂等 + Outbox + 四眼，构成可持牌运营的最小完备集。**

与账户体系的契约保持不变：单账户单币种；`frozen_balance` 快照对应 `t_account_fund_freeze` 明细；所有余额变动仍只经 `t_account_ledger`（由引擎写入）。与代理体系的契约：Fund 执行 `PARTNER_WALLET` 出款并发布 `partner.payout.*` / `deposit.chargeback.v1`，不拥有佣金账本。
