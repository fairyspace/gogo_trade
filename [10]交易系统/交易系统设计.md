# 交易系统设计（权威版）

> **版本**：v1.0 Final  
> **定位**：零售 / 专业客户多资产 CFD·FX 经纪平台的**订单—成交—持仓生命周期权威源**。  
> **运行时分层**：Trading Engine（热路径权威）+ Trade API（接入编排 / 查询）+ MySQL（持久化投影）+ Kafka Outbox（跨域事件）。**方向设计不锁定实现语言。**  
> **对标**：MetaQuotes / Spotware(cTrader) / Devexperts / Apex / DriveWealth / TradingView / TT / Broadridge / Talos / 恒生电子。  
> **与原稿关系**：原稿中 Order/Deal/Position 三元模型、A/B/Mixed 路由、Hedging/Netting、活跃/历史分表、FOK·IOC·Return 予以吸收；**运行时权威、幂等、成交原子性、Core Book 定价、事件契约、ledger 边界以本文为准**。

## 〇、相对原稿的变更说明

| 保留 | 修改 / 删除 |
|------|-------------|
| Order → Deal → Position 三元模型（MT5 语义） | 删除「每步状态都同步写库」的流程观 → **内存原子提交 + 异步持久化** |
| A-Book / B-Book / Mixed | B-Book 成交价**必须**取 Core SymbolBook，禁止吃 Client Quote |
| Hedging / Netting | 权威落在账户组 `allow_hedging`；Netting 反向超量 → Deal.entry=`INOUT` |
| 活跃订单 / 历史订单分表 | 热路径挂单触发**禁止扫库**，只扫引擎 `SymbolBook` 内存索引 |
| FOK / IOC / Return | 补齐与 A-Book 部分成交、拒单、超时的完整语义 |
| 市价 / 挂单 / 平仓主流程 | 补：改单、SL/TP 修改、强平、日切 Swap、冲正、Dealer 审批 |
| `t_order` / `t_deal` / `t_position*` | 增加 `entity_id`、`client_order_id`、幂等、执行态、Outbox；**不重新定义** `t_account_ledger`（资金域权威） |
| 订单状态机 | 去掉易误导的 VALIDATED 持久态；增加 `WORKING` / `PENDING_EXEC` / `DEALER_HOLD` |

## 一、系统定位与对标决策

### 1.1 系统定位

Trading Domain 是经纪平台的：

1. **客户交易指令权威**（Order）  
2. **成交事实权威**（Deal，不可变）  
3. **持仓生命周期权威**（Position / Position History）  
4. **交易侧资金变动写入方**（经引擎写 `t_account_ledger` 的 `TRADE_PROFIT / COMMISSION / SWAP / DIVIDEND / FEE`）  
5. **跨域交易事件生产方**（`order.*` / `deal.*` / `position.*` / `margin.*`）

以下概念**不属于**本系统：

| 概念 | 归属 |
|------|------|
| LP 原始行情接入、Core/Client 报价 | 行情体系 |
| 品种规格、合约、时段配置权威 | 品种 / 时间配置 |
| 路由规则、LP 网关、markup profile 配置权威 | 路由网关 |
| 出入金、冻结、拒付、公司双分录 | 资金体系 |
| 预交易规则配置、相关性敞口、熔断策略权威 | 风控系统（规则由引擎内存执行） |
| IB/Affiliate 佣金计提账本 | 代理体系（只消费 `deal.*`） |
| 美股 DTC 清算、期货 SPAN 全栈、加密 Prime | 明确不做 |

### 1.2 行业参考的正确使用方式

| # | 厂商 | 可借鉴 | 本设计采纳 / 取舍 |
|---|------|--------|------------------|
| 1 | **MetaQuotes** | Order/Deal/Position；Hedging/Netting；SL/TP；StopOut；Group 参数 | **主语义基线**；不照搬 Manager 单体与内嵌服务器一切包办 |
| 2 | **Spotware (cTrader)** | Netting 清晰；REST/WS API；Fill 策略 | 采纳 API 幂等与 Fill 语义；不做 cTrader 多 Asset 钱包 |
| 3 | **Devexperts** | 多资产订单模型；风险维度校验嵌入交易路径 | 预交易检查嵌入引擎；不做完整组合保证金账本（账户组预留枚举） |
| 4 | **Apex Fintech** | API-first；订单状态可查询；冲正可审计 | 采纳 `client_order_id` 幂等、状态可查询、冲正只追加 |
| 5 | **DriveWealth** | 指令编排与执行分离；账实可对 | 采纳「指令→执行报告→成交事实」分层；不做美股托管清算 |
| 6 | **TradingView** | Broker API 下单契约；图表与交易解耦 | 采纳外部入口适配层；**不把 TV 当内部 OMS** |
| 7 | **TT** | 盘前风控阈值、执行透明度、多场所意识 | 阈值在账户组/风控规则；多场所以路由网关表达 |
| 8 | **Broadridge** | Books & Records；实体隔离；审计 | `entity_id` 贯穿；Deal 不可篡改；四眼仅运营干预路径 |
| 9 | **Talos** | 多 LP 执行、SOR 思想、机构透明度 | 路由规则 + A-Book 执行；不做加密 Prime 全栈 |
| 10 | **恒生电子** | 成交回报闭环、日终对账意识 | 日切 Swap/对账事件；**不照搬证券中央对手方撮合**（CFD 是 STP/内部簿记） |

### 1.3 产品定位（拍板）

- **P0**：FX / 贵金属 / 指数 / 商品 CFD 的市价、限价、Stop、Stop-Limit；Hedging 默认；A/B/Mixed 路由；SL/TP；强平；Swap。  
- **P1**：Dealer 人工审批；部分成交完整化；订单/持仓冲正；外部 Broker API（含 TradingView）。  
- **P2**：Stop-Limit 完整、OCO、Trailing Stop、Netting 账户组规模化、DMA 深度吃单。

## 二、十四条不可破坏决策

1. **运行时权威在引擎内存**：`AccountState` / `SymbolBook` / `OrderState` / `PositionState` 是成交瞬间的真相；MySQL 是持久化投影与冷启动恢复源，不是热路径决策源。  
2. **成交原子性**：一笔逻辑成交必须在同一引擎事务中完成——生成 Deal、更新 Position、调整 margin/equity/balance、写 ledger 意图、发 Outbox——**要么全成，要么全不成**。  
3. **Deal 不可变**：成交事实只追加；纠错用 `deal.reversed.v1` + 反向 Deal / 反向 ledger，禁止 UPDATE 金额字段。  
4. **定价分层**：展示用 Client Quote；**估值、保证金、B-Book 成交、强平、挂单触发**一律用 Core SymbolBook（与行情体系对齐）。  
5. **STALE 禁成交**：Core Book 标记 STALE / 断流时，拒绝新开仓与依赖报价的触发；仅允许管理员强制干预（审计）。  
6. **余额变更只经引擎**：交易路径写 ledger 类型 `TRADE_PROFIT/COMMISSION/SWAP/DIVIDEND/FEE`；Trade API / 接入编排层 **禁止**直写 `t_trading_account` 资金字段与 `t_account_ledger`。  
7. **`t_account_ledger` 结构权威在资金域**：本系统只约定写入哪些 `ledger_type` 与关联票号字段，不另起账本表。  
8. **幂等优先**：客户端 `client_order_id`（账户内唯一）+ 引擎 `request_id`；重复提交返回首次结果，不二次成交。  
9. **`entity_id` 贯穿**：Order/Deal/Position/Outbox 均带实体；禁止跨实体持仓合并或资金串账。  
10. **事件经 Outbox**：引擎状态提交与 Outbox 同行持久化意图；禁止“写库成功后尽力发 MQ”。  
11. **A-Book 客户可见成交 ≠ LP 原始成交可延迟对账**：对客 Deal 先按执行回报落定；LP 费用/差异进运营对账，不静默改客户 Deal。  
12. **挂单触发不扫库**：Tick 路径只访问内存订单簿索引；活跃表仅为恢复与运维查询。  
13. **强平是系统订单**：`source=SYSTEM`，`close_reason=STOP_OUT`，走与平仓相同的成交原子路径（可跳过部分预交易限制，不可跳过账本完整性）。  
14. **配置热更新原子切换**：品种/账户组/路由/风控配置经版本号切换；进行中的订单使用「接受时快照」或「明确的下一 Tick 生效」策略，禁止半新半旧参数混算。

## 三、边界与跨系统契约

### 3.1 领域边界

| 系统 | 权威数据 | 交易系统的使用方式 |
|------|---------|-------------------|
| 行情 | Core SymbolBook、`seq`、STALE | 只读订阅；触发/成交/估值 |
| 品种 | digits、contract_size、手数、订单类型开关 | 启动加载 + 配置版本刷新 |
| 时间配置 | 交易时段、日历、DST | 判断可交易 / Day 单过期 |
| 路由网关 | `t_route_rule`、gateway、mapping | 执行路径选择与 LP 下单 |
| 账户 | 账户、账户组、杠杆、`allow_hedging` | 加载 AccountState；不拥有开户 |
| 资金 | 出入金、冻结、ledger 结构 | 交易只写交易类 ledger；出金预检调 `PreviewWithdraw` |
| 风控 | 规则定义、熔断开关 | 引擎内嵌执行；异步记 `t_risk_event_log` |
| 代理 | 佣金账本 | 只消费 `deal.created/reversed` |
| 通知 | 模板送达 | 只发业务事件 |
| 权限 | RBAC/SoD | Dealer 审批、人工强平、冲正 |

### 3.2 引擎交易 RPC（供 Trade API / Gateway 调用）

所有写接口必须带 `request_id`（幂等）。重复返回首次结果。

| RPC | 语义 |
|-----|------|
| `SubmitOrder` | 下单（市价/挂单/平仓） |
| `CancelOrder` | 撤挂单 |
| `ModifyOrder` | 改挂单价格/量/SL/TP/过期 |
| `ModifyPosition` | 改持仓 SL/TP（不产生成交） |
| `ClosePosition` | 市价平仓（全平/部分） |
| `DealerApprove` / `DealerReject` | 人工审批路径 |
| `ReverseDeal` | 运营冲正（四眼） |
| `GetAccountState` | 实时资金/持仓/挂单 |
| `GetOrder` / `GetPosition` | 票号查询 |
| 资金 RPC | 见资金体系：`CreditAccount` 等（同一引擎进程） |

```protobuf
// 示意：engine/trade.proto
message SubmitOrderRequest {
  int64  account_id         = 1;
  int64  entity_id          = 2;
  string symbol_key         = 3;
  uint32 order_type         = 4;   // BUY/SELL/LIMIT/STOP/...
  uint32 side               = 5;
  string volume             = 6;   // 十进制字符串
  string price              = 7;   // 挂单价；市价可空
  string price_trigger      = 8;   // Stop/StopLimit
  string price_sl           = 9;
  string price_tp           = 10;
  uint32 filling_mode       = 11;  // FOK/IOC/RETURN
  uint32 expiration_type    = 12;  // GTC/DAY/SPECIFIED
  int64  expire_at_ms       = 13;
  uint32 max_deviation_pts  = 14;
  int64  close_position_id  = 15;  // 平仓时必填（Hedging）
  string client_order_id    = 16;  // 账户内幂等键，必填
  string request_id         = 17;  // 传输层幂等，必填
  string source             = 18;  // CLIENT/EA/API/SYSTEM/DEALER
  string comment            = 19;
  string client_ip          = 20;
}

message SubmitOrderResponse {
  bool   success            = 1;
  string reject_reason      = 2;
  int64  order_ticket       = 3;
  uint32 order_status       = 4;
  // 若瞬时成交，可附带首笔 deal 摘要
  int64  deal_ticket        = 5;
  string fill_price         = 6;
  string fill_volume        = 7;
  int64  position_ticket    = 8;
}
```

### 3.3 标准事件（Outbox → Kafka）

所有事件必备：`event_id`、`event_type`、`occurred_at`、`entity_id`、`aggregate_id`、`aggregate_version`、`correlation_id`、`payload`。

| 事件 | 生产时机 | 主要消费方 |
|------|---------|-----------|
| `order.accepted.v1` | 预交易通过，订单进入 WORKING/PLACED | Notification、审计 |
| `order.rejected.v1` | 预交易/执行拒绝 | Notification、风控 |
| `order.cancelled.v1` | 客户/系统/过期撤单 | Notification |
| `order.filled.v1` | 全部成交 | 查询投影、审计 |
| `order.partially_filled.v1` | 部分成交 | 查询投影 |
| `deal.created.v1` | 每笔 Deal 落定 | **Partner**、Notification、报表 |
| `deal.reversed.v1` | 冲正 | Partner 回拨、资金对账 |
| `position.opened.v1` | 新开持仓 | 查询投影 |
| `position.updated.v1` | 加仓/部分平/改 SLTP | 查询投影 |
| `position.closed.v1` | 全平归档 | 查询投影、报表 |
| `margin.call.v1` | 触发追保 | Notification、风控 |
| `margin.stopout.v1` | 强平发生 | Notification、风控、运营 |

## 四、领域模型

### 4.1 聚合与不变式

```
Account (交易账户)
  ├── Orders[]      活跃指令
  ├── Positions[]   活跃持仓
  └── Ledger*       由引擎写入（结构属资金域）

Order  1 ──* Deal
Deal   * ──1 Position   （开仓 Deal 创建/更新；平仓 Deal 减少/关闭）
Position 全平 ──► PositionHistory
```

**不变式（必须在引擎内强制）：**

1. `volume_remaining + volume_filled == volume_requested`（拒单/取消后剩余归零并归档）。  
2. 同一 `account_id + client_order_id` 至多一个业务订单。  
3. Hedging：平仓必须指定 `close_position_id`，且 side 相反。  
4. Netting：同品种单一净头寸；反向成交先平后开（INOUT）。  
5. Position.volume ≥ 0；归零则移入 history，禁止残留 OPEN 零仓。  
6. 平仓 Deal.profit + 相关 commission/swap 与 ledger 金额可勾稽。  
7. 任何 balance 变化必有 ledger；任何 ledger 必有引擎幂等键。

### 4.2 订单类型

| 码 | 类型 | 行为摘要 |
|----|------|---------|
| 1 | BUY | 市价买（开多或平空，取决于模式/参数） |
| 2 | SELL | 市价卖 |
| 3 | BUY_LIMIT | Ask ≤ price 触发后按限价逻辑执行 |
| 4 | SELL_LIMIT | Bid ≥ price |
| 5 | BUY_STOP | Ask ≥ trigger |
| 6 | SELL_STOP | Bid ≤ trigger |
| 7 | BUY_STOP_LIMIT | Ask≥trigger 后挂 BUY_LIMIT@price |
| 8 | SELL_STOP_LIMIT | Bid≤trigger 后挂 SELL_LIMIT@price |

> 开平语义：CFD/FX 零售常见「显式平仓」（`close_position_id`）与「反向开仓」并存。  
> **拍板**：Hedging 账户**必须**显式平仓字段才能减少指定持仓；裸反向单创建对冲持仓。Netting 账户反向单自动冲减净头寸。

### 4.3 填充模式（Filling Mode）

| 模式 | 语义 | 典型用途 |
|------|------|---------|
| FOK | 必须全部成交，否则整单拒绝 | 默认市价（B-Book / 要求全额） |
| IOC | 能成交多少算多少，剩余取消 | A-Book 深度不足 |
| RETURN | 部分成交后剩余转挂单继续 | 部分 LP / 特殊账户组 |

A-Book 超时：按账户组策略 `REJECT_REMAINING`（默认）或 `CANCEL_REMAINING`；已成交部分**不回滚**（除非整单 FOK 且未达量）。

### 4.4 持仓模式

| 模式 | 来源 | 行为 |
|------|------|------|
| Hedging | `t_account_group.allow_hedging=1` | 每笔开仓新 Position；可同品种多空并存 |
| Netting | `allow_hedging=0` | 每账户每品种一个净 Position；加权均价；反向冲减 |

## 五、状态机

### 5.1 订单状态（持久化）

```
                    ┌─────────────┐
         提交失败    │  REJECTED   │◄── 预交易失败 / 执行失败(无成交)
                    └─────────────┘
                           ▲
                           │
  Submit ──► ┌─────────────┴──────────────┐
             │         WORKING            │  已接受，路由/等待执行中
             └───────┬───────────┬────────┘
                     │           │
           挂单确认   │           │ 需人工
                     ▼           ▼
              ┌──────────┐  ┌─────────────┐
              │  PLACED  │  │ DEALER_HOLD │
              └────┬─────┘  └──────┬──────┘
                   │               │ approve
        触发/改单   │               ▼
                   │         ┌──────────┐
                   └────────►│ PENDING_ │  已发 LP，等回报（A-Book）
                             │  EXEC    │
                             └────┬─────┘
                                  │
                    ┌─────────────┼─────────────┐
                    ▼             ▼             ▼
           PARTIALLY_FILLED   FILLED      CANCELLED / EXPIRED
                    │
                    └──► (剩余继续 WORKING/PLACED 或按 filling 取消)
```

**说明：**

- `SUBMITTED` / `VALIDATED` **不落库**（或仅出现在访问日志）。对客与 DB 可见的最小接受态是 `WORKING`（市价）或 `PLACED`（挂单）。  
- 终态：`FILLED` / `CANCELLED` / `REJECTED` / `EXPIRED` → 移入 `t_order_history`。  
- `PARTIALLY_FILLED` 仍在活跃表，直到剩余量处理完毕。

### 5.2 持仓状态

```
OPEN ──部分平仓──► OPEN(volume↓) ──全平──► (移入 t_position_history)
```

活跃表不保留 `CLOSED` 行，避免零仓污染强平扫描。

### 5.3 Deal 类型与 entry

| deal_type | entry | 含义 |
|-----------|-------|------|
| BUY / SELL | IN | 开仓 |
| BUY_CLOSE / SELL_CLOSE | OUT | 平仓 |
| BUY / SELL | INOUT | Netting 反向超量：先平尽再开反向 |

## 六、运行时架构

### 6.1 逻辑视图

```
 Client / EA / TV / FIX API
            │
            ▼
   Trade API / Client Gateway
            │  SubmitOrder (RPC / RingBuffer)
            ▼
 ┌──────────────────────────────────────────────┐
 │           Trading Engine                     │
 │  单写线程 EventLoop（账户分片可多线程，见下）    │
 │                                              │
 │  Tick → 挂单触发 → SL/TP → 浮盈 → 强平扫描     │
 │  Order → PreTrade → Route → Exec → Commit    │
 │                                              │
 │  内存: SymbolBook / AccountState / Route/Risk │
 └───────┬─────────────┬─────────────┬──────────┘
         │             │             │
         ▼             ▼             ▼
   Async MySQL    Redis 投影    Outbox→Kafka
   Writer         (equity等)    (跨域事件)
         │
         ▼
   FIX/REST LP Gateway  ◄── A-Book 执行（异步回报注入 EventLoop）
```

### 6.2 并发与分片（落地拍板）

- **默认 MVP**：单引擎主线程处理全部订单与 Tick 副作用（确定性最强，延迟最优）。  
- **扩展**：按 `account_id` 分片多线程；品种 Tick 广播到相关分片；**禁止跨分片无锁改同一账户**。  
- LP 回报线程**不得**直接改账户；一律投递到主循环队列。

### 6.3 持久化模型（关键）

```
引擎 Commit 顺序（单账户）:
  1. 内存状态应用（Order/Deal/Position/margin/balance）
  2. 追加 WAL / 持久化队列记录（含 Outbox 意图）  ← 对客 ACK 的前置条件
  3. 异步批量写 MySQL（活跃表 + deal + ledger + outbox）
  4. Outbox Relay 发 Kafka
  5. 查询侧 Redis 投影更新

崩溃恢复:
  WAL 重放 → 与 MySQL 对账 → 未完成 A-Book 订单按 external_order_id 查询 LP 状态机收敛
```

> **拍板**：对客「已成交」ACK 不得早于 WAL 落盘（或等价的同步复制日志）。MySQL 可异步，但必须可追平。

### 6.4 Trade API 职责边界

| 允许 | 禁止 |
|------|------|
| 鉴权、限流、参数粗检、转发引擎 | 自算保证金并写库 |
| 读历史（order_history/deal/position_history） | 直连改 balance |
| 组装客户可见 DTO（可拼 Client Quote 展示） | 用展示价当成交价 |
| 管理端冲正/强平入口（再调引擎） | 绕过四眼直接改 Deal |

## 七、核心流程

### 7.1 市价开仓（B-Book 快路径）

```
SubmitOrder(BUY 1 lot EURUSD)
  │
  ├─ 幂等: account+client_order_id 已存在 → 返回首次结果
  │
  ├─ PreTrade（纯内存）
  │    账户/交易开关/EA 开关
  │    品种状态与时段（时间配置 + 品种）
  │    账户组品种准入 / CLOSE_ONLY
  │    手数步进与上下限
  │    持仓数 / 挂单数 / 单品种上限
  │    保证金: calcMargin(Core Ask, 杠杆/阶梯/品种覆盖/对冲折扣)
  │    margin_free 与「下单后不跌破 margin_call」检查
  │    自定义风控规则（REJECT / DEALER_HOLD / ALERT）
  │    Core Book 非 STALE
  │
  ├─ 接受订单 → status=WORKING → order.accepted
  │
  ├─ Route: 匹配 t_route_rule
  │    B-Book → fill = Core Ask(买)/Bid(卖)
  │    应用 max_deviation；失败则 REJECT（无 Deal）
  │
  ├─ Atomic Commit
  │    Deal(IN) + Position(OPEN) + commission ledger
  │    margin_used↑ equity 重算
  │    Order → FILLED → 归档意图
  │    deal.created + position.opened + order.filled
  │
  └─ ACK 客户（含 deal/position 票号）
```

### 7.2 市价开仓（A-Book）

```
PreTrade 通过 → WORKING
  → 发送 LP NewOrderSingle（记录 external_order_id）→ PENDING_EXEC
  → 等待 ExecutionReport
       全成 → 同 Atomic Commit（价格用 LP 成交价，经滑点保护）
       部成 → PARTIALLY_FILLED + 按 filling_mode 处理剩余
       拒绝/超时 → 无成交则 REJECTED；有成交则保留已成部分
```

**滑点保护拍板：**

- 对客户不利滑点超过 `max_deviation` / 路由规则上限 → 拒收该回报（A-Book 需向 LP 发取消/冲正策略，按网关能力）；  
- 价格改善默认接受。

### 7.3 挂单

```
PreTrade（保证金用挂单价与 margin_rate_limit）
  → PLACED，进入 SymbolBook 对应有序集合
  → Tick 路径: 价格触及 → 转为执行（同 7.1/7.2）
  → Cancel / Day收盘 / expire_at → CANCELLED/EXPIRED 归档并释放预占保证金
```

**预占保证金拍板：** 挂单在 PLACED 时预占（或按账户组配置「触发时再检」）。  
**默认**：PLACED 预占，防止无限挂单击穿保证金；修改挂单时重算预占差额。

### 7.4 平仓

```
ClosePosition / SubmitOrder(close_position_id)
  → 校验持仓归属、可平量、时段（CLOSE_ONLY 允许）
  → 路由执行（通常与开仓同规则，或 close_route 覆盖）
  → Atomic Commit:
       Deal(OUT) + profit/commission/swap 结算
       Position.volume↓ 或移除并写 position_history
       balance += profit + swap + commission(负)
       margin_used↓
```

PnL（账户币种）：

```
BUY 仓:  profit = (close - open) * volume * contract_size * fx_to_account
SELL 仓: profit = (open - close) * volume * contract_size * fx_to_account
```

汇率路径使用引擎内存中的转换链（品种体系定义），禁止临时读库。

### 7.5 SL / TP

- 存在 Position 上；Tick 用 Core Bid/Ask 检查。  
- 触发后生成 `source=SYSTEM` 的平仓订单，`close_reason=SL|TP`。  
- 修改 SL/TP 走 `ModifyPosition`，不产生 Deal。  
- `freeze_level` 内禁止修改（品种交易配置）。

### 7.6 强平（Stop Out）

```
Tick/定时: equity / margin_used * 100 <= stop_out_level
  → margin.stopout 事件
  → 按账户组策略排序持仓（默认浮亏从大到小）
  → 逐笔系统市价平仓，直到 margin_level 恢复到安全阈值或无仓
  → 允许负余额时记 NEGATIVE_BALANCE_CORRECTION（资金域类型，引擎写）
```

### 7.7 日切 Swap

```
结算引擎日切:
  对每笔 OPEN 持仓按品种 swap 配置计息
  → position.swap 累加
  → ledger SWAP
  → 可选立即影响 balance/equity（拍板：立即入账，与 MT5 一致）
```

### 7.8 冲正

```
Dealer 四眼 → ReverseDeal(deal_ticket)
  → 生成反向 Deal（关联 orig_deal_ticket）
  → 回滚持仓或生成补偿持仓（按是否已平）
  → 反向 ledger
  → deal.reversed.v1（Partner 回拨佣金）
```

## 八、保证金与费用（交易侧契约）

### 8.1 保证金

完整公式见风控系统设计；交易系统保证：

- 开仓/挂单预占使用同一 `calcMargin`；  
- 对冲折扣读取 `margin_hedged`；  
- 浮动杠杆按名义敞口阶梯；  
- `margin_used` 为账户聚合快照，持仓级 `position.margin` 可重算对齐。

### 8.2 佣金

- 规则权威：`t_account_group_commission`；  
- 开仓 Deal、平仓 Deal 可分别计；  
- 金额写入 Deal.commission，并写 ledger `COMMISSION`；  
- **代理佣金不在本系统计算**。

### 8.3 与资金域的协作

| 场景 | 调用 |
|------|------|
| 客户出金预检 | Fund → `PreviewWithdraw`（含浮动盈亏） |
| 出金冻结 | Fund → `FreezeFunds`（交易持仓仍由引擎估值） |
| 入金 | Fund → `CreditAccount` |
| 交易盈亏 | 引擎直接写 ledger，Fund 只读对账 |

## 十、验收 SLO 与对账

### 10.1 SLO（同机房）

| 指标 | 目标 |
|------|------|
| B-Book 市价：Submit→Deal ACK P99 | &lt; 20ms |
| 挂单触发（Tick 已在引擎内）P99 | &lt; 1ms |
| A-Book：Submit→发往 LP P99 | &lt; 5ms（不含 LP 往返） |
| Tick→浮盈/SLTP 扫描（单品种）P99 | &lt; 1ms |
| 资金账实：日切 ledger vs balance | 差异 = 0 |
| Deal 与 Position 勾稽 | 日检 0 差异 |

### 10.2 日检 SQL（示意）

```sql
-- 活跃持仓量 = 开仓 Deal 入 - 平仓 Deal 出（按 position_ticket）
-- 账户 balance 链: 昨日末 + 今日 ledger 净额 = 今日末快照
-- Outbox 无 status=0 且 created_at < now()-5m 的长期积压
```

### 10.3 可观测性

每笔订单强制 `correlation_id`（可等于 `request_id`）贯穿：API → 引擎 → LP → Deal → Outbox → Partner。  
指标：拒单原因计数、A-Book 超时、滑点分布、强平次数、WAL 落后、MySQL writer lag。

## 十一、落地分期

| 阶段 | 范围 | 完成标准 |
|------|------|---------|
| **M1** | B-Book 市价开平、Hedging、保证金、ledger、Core Book 定价、Outbox `deal.created` | 模拟盘可交易；账实日检为 0 |
| **M2** | 限价/Stop 挂单、SL/TP、预占保证金、Day 过期 | Tick 触发不扫库；恢复后挂单一致 |
| **M3** | A-Book + `t_execution_link`、IOC/FOK、滑点保护 | LP 超时可收敛；对客 Deal 可审计 |
| **M4** | Stop Out、Swap 日切、MarginCall 事件 | 强平可回放；负余额策略可配置 |
| **M5** | Dealer、冲正、TV/Broker API、Netting 账户组 | 四眼冲正；Partner 回拨正确 |

## 十二、原稿价值评估（结论）

| 原稿点 | 结论 |
|--------|------|
| Order/Deal/Position 三元 | **高价值**，保留为语义基线 |
| A/B/Mixed 路由叙事 | **有价值**，需补执行关联表与超时态 |

| 自建 ledger 归属交易系统 | **不采纳**；结构归资金域，引擎写交易类型 |

### 14.3 WAL Entry Schema（与撮合引擎 §6.2 一致）

```
WAL Entry（二进制，按 seq 单调）:
  - seq: int64          # 引擎内单调递增
  - account_id: int64
  - payload_type: enum # ORDER/DEAL/POSITION/MARGIN
  - payload_crc: uint32
  - payload: bytes     # 序列化的变更
  - outbox_intents: []  # 关联的 Outbox 事件
  - timestamp: int64
```

- WAL 前置 ACK，RPO = 0；同步复制到备机（同城双活）。
- 崩溃恢复：重放 WAL → 与 MySQL 投影对账 → 修复不一致。

### 14.4 Mixed 路由时序（与撮合引擎 §5.2C 一致）

```
客户单 SubmitOrder
  → RouteResolver.ResolveRoute 返回 allocations[{gateway_id, ratio, ...}]
  → 引擎按 allocation 拆子单
    ├── 子单1 → B-Book（内部簿记）→ 子Deal1
    └── 子单2 → A-Book（LP执行）→ 子Deal2
  → 客户侧单 Deal = mergeDeals(子Deal1, 子Deal2)
    - price = 加权均价（按各子单实际成交量）
    - commission = sum(子.commission)
    - 子单在 t_execution_link 关联
  → commitTransaction(clientDeal)
  → 返回 SubmitOrderResponse(FILLED, deal_id=clientDeal.id)
```

客户只看到一笔聚合 Deal；子执行链路在 `t_execution_link` 留痕，供对账与代理计提。

### 14.5 Outbox 死信与重试

`t_engine_outbox` 字段补充：

| 字段 | 说明 |
|------|------|
| `retry_count` | 已重试次数 |
| `next_retry_at` | 下次重试时间（指数退避） |
| `status` | PENDING/PUBLISHED/DEAD |
| `dlq_topic` | 死信 topic（`dlq.trading.deal` 等） |

- 重试 N 次（默认 5）入死信，告警 + 人工。
- Kafka 发布成功后 `status=PUBLISHED`，定期清理。
