# 交易系统设计 — 终版 v2.0

> **定位**：零售 / 专业客户多资产 CFD·FX 经纪平台的**订单—成交—持仓生命周期权威源**，同时是 **Trading Engine 运行时实现说明**——域决策（WHAT）与运行时算法（HOW）合于一稿，避免跨文档割裂。
> **运行时分层**：Trading Engine（热路径权威，内存优先）+ Trade API（接入编排 / 查询）+ MySQL（持久化投影）+ Kafka Outbox（跨域事件）。**方向设计描述职责与算法，不锁定实现语言。**
> **对标**：MetaQuotes / Spotware(cTrader) / Devexperts / Apex / DriveWealth / TradingView / TT / Broadridge / Talos / 恒生电子。
> **术语**：本平台「撮合」= 零售 CFD/FX 的**簿记执行引擎**（STP / B-Book），**不是**交易所多边中央对手方 Match Engine。
> **跨域契约**：遵循《平台跨域契约补充卷》——错误码（§三）、事件（§四）、ID 命名（§六）、API（§九）、容灾（§十）、监控（§十二）。

## 一、系统定位与对标决策

### 1.1 系统定位

Trading Domain 是经纪平台的：

1. **客户交易指令权威**（Order）
2. **成交事实权威**（Deal，不可变）
3. **持仓生命周期权威**（Position / Position History）
4. **交易侧资金变动写入方**（经引擎写 `t_account_ledger` 的 `TRADE_PROFIT / COMMISSION / SWAP / DIVIDEND / FEE`）
5. **跨域交易事件生产方**（`order.*` / `deal.*` / `position.*` / `margin.*`）

以下概念**不属于**本系统：

| 概念 | 归属 |
|------|------|
| LP 原始行情接入、Core/Client 报价 | 行情体系 |
| 品种规格、合约、时段配置权威 | 品种 / 时间配置 |
| 路由规则、LP 网关、markup profile 配置权威 | 路由网关 |
| 出入金、冻结、拒付、公司双分录 | 资金体系 |
| 预交易规则配置、相关性敞口、熔断策略权威 | 风控系统（规则由引擎内存执行） |
| IB/Affiliate 佣金计提账本 | 代理体系（只消费 `deal.*`） |
| 美股 DTC 清算、期货 SPAN 全栈、加密 Prime | 明确不做 |

### 1.2 行业参考的正确使用方式

| # | 厂商 | 可借鉴 | 本设计采纳 / 取舍 |
|---|------|--------|------------------|
| 1 | **MetaQuotes** | Order/Deal/Position；Hedging/Netting；SL/TP；StopOut；Group 参数 | **主语义基线**；不照搬 Manager 单体与内嵌服务器一切包办 |
| 2 | **Spotware (cTrader)** | Netting 清晰；REST/WS API；Fill 策略 | 采纳 API 幂等与 Fill 语义；不做 cTrader 多 Asset 钱包 |
| 3 | **Devexperts** | 多资产订单模型；风险维度校验嵌入交易路径 | 预交易检查嵌入引擎；不做完整组合保证金账本（账户组预留枚举） |
| 4 | **Apex Fintech** | API-first；订单状态可查询；冲正可审计 | 采纳 `client_order_id` 幂等、状态可查询、冲正只追加 |
| 5 | **DriveWealth** | 指令编排与执行分离；账实可对 | 采纳「指令→执行报告→成交事实」分层；不做美股托管清算 |
| 6 | **TradingView** | Broker API 下单契约；图表与交易解耦 | 采纳外部入口适配层；**不把 TV 当内部 OMS** |
| 7 | **TT** | 盘前风控阈值、执行透明度、多场所意识 | 阈值在账户组/风控规则；多场所以路由网关表达 |
| 8 | **Broadridge** | Books & Records；实体隔离；审计 | `entity_id` 贯穿；Deal 不可篡改；四眼仅运营干预路径 |
| 9 | **Talos** | 多 LP 执行、SOR 思想、机构透明度 | 路由规则 + A-Book 执行；不做加密 Prime 全栈 |
| 10 | **恒生电子** | 成交回报闭环、日终对账意识 | 日切 Swap/对账事件；**不照搬证券中央对手方撮合**（CFD 是 STP/内部簿记） |

### 1.3 产品定位（拍板）

- **P0**：FX / 贵金属 / 指数 / 商品 CFD 的市价、限价、Stop、Stop-Limit；Hedging 默认；A/B/Mixed 路由；SL/TP；强平；Swap。
- **P1**：Dealer 人工审批；部分成交完整化；订单/持仓冲正；外部 Broker API（含 TradingView）。
- **P2**：Stop-Limit 完整、OCO、Trailing Stop、Netting 账户组规模化、DMA 深度吃单。

## 二、十四条不可破坏决策

1. **运行时权威在引擎内存**：`AccountState` / `SymbolBook` / `OrderState` / `PositionState` 是成交瞬间的真相；MySQL 是持久化投影与冷启动恢复源，不是热路径决策源。
2. **成交原子性**：一笔逻辑成交必须在同一引擎事务中完成——生成 Deal、更新 Position、调整 margin/equity/balance、写 ledger 意图、发 Outbox——**要么全成，要么全不成**。
3. **Deal 不可变**：成交事实只追加；纠错用 `deal.reversed.v1` + 反向 Deal / 反向 ledger，禁止 UPDATE 金额字段。
4. **定价分层**：展示用 Client Quote；**估值、保证金、B-Book 成交、强平、挂单触发**一律用 Core SymbolBook（与行情体系对齐）。
5. **STALE 禁成交**：Core Book 标记 STALE / 断流时，拒绝新开仓与依赖报价的触发；仅允许管理员强制干预（审计）。
6. **余额变更只经引擎**：交易路径写 ledger 类型 `TRADE_PROFIT/COMMISSION/SWAP/DIVIDEND/FEE`；Trade API / 接入编排层 **禁止**直写 `t_trading_account` 资金字段与 `t_account_ledger`。
7. **`t_account_ledger` 结构权威在资金域**：本系统只约定写入哪些 `ledger_type` 与关联票号字段，不另起账本表。
8. **幂等优先**：客户端 `client_order_id`（账户内唯一）+ 引擎 `request_id`；重复提交返回首次结果，不二次成交。
9. **`entity_id` 贯穿**：Order/Deal/Position/Outbox 均带实体；禁止跨实体持仓合并或资金串账。
10. **事件经 Outbox**：引擎状态提交与 Outbox 同行持久化意图；禁止"写库成功后尽力发 MQ"。
11. **A-Book 客户可见成交 ≠ LP 原始成交可延迟对账**：对客 Deal 先按执行回报落定；LP 费用/差异进运营对账，不静默改客户 Deal。
12. **挂单触发不扫库**：Tick 路径只访问内存订单簿索引；活跃表仅为恢复与运维查询。
13. **强平是系统订单**：`source=SYSTEM`，`close_reason=STOP_OUT`，走与平仓相同的成交原子路径（可跳过部分预交易限制，不可跳过账本完整性）。
14. **配置热更新原子切换**：品种/账户组/路由/风控配置经版本号切换；进行中的订单使用「接受时快照」或「明确的下一 Tick 生效」策略，禁止半新半旧参数混算。

## 三、边界与跨系统契约

### 3.1 领域边界

| 系统 | 权威数据 | 交易系统的使用方式 |
|------|---------|-------------------|
| 行情 | Core SymbolBook、`seq`、STALE | 只读订阅；触发/成交/估值 |
| 品种 | digits、contract_size、手数、订单类型开关 | 启动加载 + 配置版本刷新 |
| 时间配置 | 交易时段、日历、DST | 判断可交易 / Day 单过期 |
| 路由网关 | `t_route_rule`、gateway、mapping | 执行路径选择与 LP 下单 |
| 账户 | 账户、账户组、杠杆、`allow_hedging` | 加载 AccountState；不拥有开户 |
| 资金 | 出入金、冻结、ledger 结构 | 交易只写交易类 ledger；出金预检调 `PreviewWithdraw` |
| 风控 | 规则定义、熔断开关 | 引擎内嵌执行；异步记 `t_risk_event_log` |
| 代理 | 佣金账本 | 只消费 `deal.created/reversed` |
| 通知 | 模板送达 | 只发业务事件 |
| 权限 | RBAC/SoD | Dealer 审批、人工强平、冲正 |

### 3.2 运行时跨域依赖方式

3.1 描述了业务边界层面的权威归属；下表是引擎运行时实际消费这些权威数据的机制（RPC / 共享内存 / 本地缓存），与六、6.2 引擎启动加载顺序一一对应：

| 依赖 | 方式 | 提供方 | 缓存策略 |
|------|------|--------|----------|
| Core SymbolBook | 共享内存（SHM SPSC） | 行情体系 | 实时订阅，`seq` 单调 |
| 交易时段 | gRPC `SessionQueryService` | 时间配置域 | 本地缓存 + 订阅 `t_session_override` Pub/Sub 失效 |
| 杠杆解析 | gRPC `LeverageResolver` | 杠杆配置域 | Caffeine + Redis，按 `config_version` 失效 |
| 路由解析 | gRPC `RouteResolver` | 路由网关 | 本地缓存 + 配置广播失效 |
| 资金写 | gRPC `FundService` | 资金域 | 不缓存，事务内调用 |
| 风控规则 | 引擎内嵌内存 | 风控域 | 配置热更 shadow → swap |

### 3.3 引擎交易 RPC（供 Trade API / Gateway 调用）

所有写接口必须带 `request_id`（幂等）。重复返回首次结果。

| RPC | 语义 |
|-----|------|
| `SubmitOrder` | 下单（市价/挂单/平仓） |
| `CancelOrder` | 撤挂单 |
| `ModifyOrder` | 改挂单价格/量/SL/TP/过期 |
| `ModifyPosition` | 改持仓 SL/TP（不产生成交） |
| `ClosePosition` | 市价平仓（全平/部分） |
| `DealerApprove` / `DealerReject` | 人工审批路径 |
| `ReverseDeal` | 运营冲正（四眼） |
| `GetAccountState` | 实时资金/持仓/挂单 |
| `GetOrder` / `GetPosition` | 票号查询 |
| 资金 RPC | 见资金体系：`CreditAccount` 等（同一引擎进程） |

```protobuf
// 示意：engine/trade.proto
message SubmitOrderRequest {
  int64  account_id         = 1;
  int64  entity_id          = 2;
  string symbol_key         = 3;
  uint32 order_type         = 4;   // BUY/SELL/LIMIT/STOP/...
  uint32 side               = 5;
  string volume             = 6;   // 十进制字符串
  string price              = 7;   // 挂单价；市价可空
  string price_trigger      = 8;   // Stop/StopLimit
  string price_sl           = 9;
  string price_tp           = 10;
  uint32 filling_mode       = 11;  // FOK/IOC/RETURN
  uint32 expiration_type    = 12;  // GTC/DAY/SPECIFIED
  int64  expire_at_ms       = 13;
  uint32 max_deviation_pts  = 14;
  int64  close_position_id  = 15;  // 平仓时必填（Hedging）
  string client_order_id    = 16;  // 账户内幂等键，必填
  string request_id         = 17;  // 传输层幂等，必填
  string source              = 18;  // CLIENT/EA/API/SYSTEM/DEALER/ADMIN（与 t_order.source 一致，见 §9.2）
  string comment             = 19;
  string client_ip           = 20;
  // int64 magic_number      = 21;  // 预留：SIGNAL 来源与策略/订阅归因，量化域与社交交易域协同推进中的目标态字段，尚未落地（见 [12]/[13] 对应章节），落地前不接受 source=SIGNAL
}

message SubmitOrderResponse {
  bool   success            = 1;
  string reject_reason      = 2;
  int64  order_ticket       = 3;
  uint32 order_status       = 4;
  // 若瞬时成交，可附带首笔 deal 摘要
  int64  deal_ticket        = 5;
  string fill_price         = 6;
  string fill_volume        = 7;
  int64  position_ticket    = 8;
}
```

### 3.4 标准事件（Outbox → Kafka）

所有事件必备：`event_id`、`event_type`、`occurred_at`、`entity_id`、`aggregate_id`、`aggregate_version`、`correlation_id`、`payload`。

| 事件 | 生产时机 | 主要消费方 |
|------|---------|-----------|
| `order.created.v1` | 预交易通过，订单进入 WORKING/PLACED | Notification、审计 |
| `order.rejected.v1` | 预交易/执行拒绝 | Notification、风控 |
| `order.cancelled.v1` | 客户/系统/过期撤单 | Notification |
| `order.filled.v1` | 全部成交 | 查询投影、审计 |
| `order.partially_filled.v1` | 部分成交 | 查询投影 |
| `deal.created.v1` | 每笔 Deal 落定 | **Partner**、Notification、报表 |
| `deal.reversed.v1` | 冲正 | Partner 回拨、资金对账 |
| `position.opened.v1` | 新开持仓 | 查询投影 |
| `position.updated.v1` | 加仓/部分平/改 SLTP | 查询投影 |
| `position.closed.v1` | 全平归档 | 查询投影、报表 |
| `margin.call.v1` | 触发追保 | Notification、风控 |
| `margin.stopout.v1` | 强平发生 | Notification、风控、运营 |

## 四、领域模型

### 4.1 聚合与不变式

```
Account (交易账户)
  ├── Orders[]      活跃指令
  ├── Positions[]   活跃持仓
  └── Ledger*       由引擎写入（结构属资金域）

Order  1 ──* Deal
Deal   * ──1 Position   （开仓 Deal 创建/更新；平仓 Deal 减少/关闭）
Position 全平 ──► PositionHistory
```

**不变式（必须在引擎内强制）：**

1. `volume_remaining + volume_filled == volume_requested`（拒单/取消后剩余归零并归档）。
2. 同一 `account_id + client_order_id` 至多一个业务订单。
3. Hedging：平仓必须指定 `close_position_id`，且 side 相反。
4. Netting：同品种单一净头寸；反向成交先平后开（INOUT）。
5. Position.volume ≥ 0；归零则移入 history，禁止残留 OPEN 零仓。
6. 平仓 Deal.profit + 相关 commission/swap 与 ledger 金额可勾稽。
7. 任何 balance 变化必有 ledger；任何 ledger 必有引擎幂等键。

### 4.2 订单类型

| 码 | 类型 | 行为摘要 |
|----|------|---------|
| 1 | BUY | 市价买（开多或平空，取决于模式/参数） |
| 2 | SELL | 市价卖 |
| 3 | BUY_LIMIT | Ask ≤ price 触发后按限价逻辑执行 |
| 4 | SELL_LIMIT | Bid ≥ price |
| 5 | BUY_STOP | Ask ≥ trigger |
| 6 | SELL_STOP | Bid ≤ trigger |
| 7 | BUY_STOP_LIMIT | Ask≥trigger 后挂 BUY_LIMIT@price |
| 8 | SELL_STOP_LIMIT | Bid≤trigger 后挂 SELL_LIMIT@price |

> 开平语义：CFD/FX 零售常见「显式平仓」（`close_position_id`）与「反向开仓」并存。
> **拍板**：Hedging 账户**必须**显式平仓字段才能减少指定持仓；裸反向单创建对冲持仓。Netting 账户反向单自动冲减净头寸。

### 4.3 填充模式（Filling Mode）

| 模式 | 语义 | 典型用途 |
|------|------|---------|
| FOK | 必须全部成交，否则整单拒绝 | 默认市价（B-Book / 要求全额） |
| IOC | 能成交多少算多少，剩余取消 | A-Book 深度不足 |
| RETURN | 部分成交后剩余转挂单继续 | 部分 LP / 特殊账户组 |

A-Book 超时：按账户组策略 `REJECT_REMAINING`（默认）或 `CANCEL_REMAINING`；已成交部分**不回滚**（除非整单 FOK 且未达量）。

### 4.4 持仓模式

| 模式 | 来源 | 行为 |
|------|------|------|
| Hedging | `t_account_group.allow_hedging=1` | 每笔开仓新 Position；可同品种多空并存 |
| Netting | `allow_hedging=0` | 每账户每品种一个净 Position；加权均价；反向冲减 |

## 五、状态机

### 5.1 订单状态（持久化）

```
                    ┌─────────────┐
         提交失败    │  REJECTED   │◄── 预交易失败 / 执行失败(无成交)
                    └─────────────┘
                           ▲
                           │
  Submit ──► ┌─────────────┴──────────────┐
             │         WORKING            │  已接受，路由/等待执行中
             └───────┬───────────┬────────┘
                     │           │
           挂单确认   │           │ 需人工
                     ▼           ▼
              ┌──────────┐  ┌─────────────┐
              │  PLACED  │  │ DEALER_HOLD │
              └────┬─────┘  └──────┬──────┘
                   │               │ approve
        触发/改单   │               ▼
                   │         ┌──────────┐
                   └────────►│ PENDING_ │  已发 LP，等回报（A-Book）
                             │  EXEC    │
                             └────┬─────┘
                                  │
                    ┌─────────────┼─────────────┐
                    ▼             ▼             ▼
           PARTIALLY_FILLED   FILLED      CANCELLED / EXPIRED
                    │
                    └──► (剩余继续 WORKING/PLACED 或按 filling 取消)
```

**说明：**

- `SUBMITTED` / `VALIDATED` **不落库**（或仅出现在访问日志）。对客与 DB 可见的最小接受态是 `WORKING`（市价）或 `PLACED`（挂单）。
- 终态：`FILLED` / `CANCELLED` / `REJECTED` / `EXPIRED` → 移入 `t_order_history`。
- `PARTIALLY_FILLED` 仍在活跃表，直到剩余量处理完毕。

> **枚举编号权威**：活跃表 `t_order.status`（见九、9.2）与历史表 `t_order_history.status`（见九、9.3）**不共享同一枚举空间**——活跃态：`WORKING=1`、`PLACED=2`、`PENDING_EXEC=3`、`PARTIALLY_FILLED=4`、`DEALER_HOLD=5`；终态迁入历史表后重新编号：`FILLED=10`、`CANCELLED=11`、`REJECTED=12`、`EXPIRED=13`。任何引擎内部实现描述引用这些状态码时必须以此为准。

### 5.2 持仓状态

```
OPEN ──部分平仓──► OPEN(volume↓) ──全平──► (移入 t_position_history)
```

活跃表不保留 `CLOSED` 行，避免零仓污染强平扫描。

### 5.3 Deal 类型与 entry

| deal_type | entry | 含义 |
|-----------|-------|------|
| BUY / SELL | IN | 开仓 |
| BUY_CLOSE / SELL_CLOSE | OUT | 平仓 |
| BUY / SELL | INOUT | Netting 反向超量：先平尽再开反向 |

## 六、运行时架构

### 6.1 逻辑视图

```
 Client / EA / TV / FIX API
            │
            ▼
   Trade API / Client Gateway
            │  SubmitOrder (RPC / RingBuffer)
            ▼
 ┌──────────────────────────────────────────────┐
 │           Trading Engine                     │
 │  单写线程 EventLoop（账户分片可多线程，见下）    │
 │                                              │
 │  Tick → 挂单触发 → SL/TP → 浮盈 → 强平扫描     │
 │  Order → PreTrade → Route → Exec → Commit    │
 │                                              │
 │  内存: SymbolBook / AccountState / Route/Risk │
 └───────┬─────────────┬─────────────┬──────────┘
         │             │             │
         ▼             ▼             ▼
   Async MySQL    Redis 投影    Outbox→Kafka
   Writer         (equity等)    (跨域事件)
         │
         ▼
   FIX/REST LP Gateway  ◄── A-Book 执行（异步回报注入 EventLoop）
```

引擎内存的四个功能子引擎——报价引擎（QuoteEng）、订单引擎（OrderEng）、风控引擎（RiskEng）、结算引擎（SettleEng）——通过进程内 EventBus 协作（`CORE_BOOK_DELTA`/`ORDER_ACCEPTED`/`DEAL_CREATED`/`POSITION_OPENED`/`MARGIN_CALL`/`STOP_OUT` 等内部事件），最终收敛到统一的 Commit 与 Outbox 出口（见 6.4）。

### 6.2 引擎启动加载顺序

引擎重启/冷启动必须按顺序完成以下步骤，任一步失败即拒绝进入 `ACTIVE`，告警人工介入：

```
Step 1: 加载基础配置
    ├── t_system_config → SystemConfig{}
    │     服务器时区、全局开关、默认参数
    └── 订阅 SessionQueryService（时间配置域）
          初始化 SessionCache{}：拉取当日 + 次日 t_daily_session 快照
          订阅 t_session_override Pub/Sub，实时失效本地缓存

Step 2: 加载品种体系
    ├── t_symbol_category → Map<categoryId, Category>
    ├── t_symbol_group → Map<groupId, SymbolGroup>
    ├── t_symbol → Map<symbolKey, SymbolConfig>
    │     合约规格、计价模式、报价过滤参数；品种级 NULL → 从 SymbolGroup 继承
    ├── t_symbol_trading_config → 合并到 SymbolConfig
    ├── t_symbol_swap_config → 合并到 SymbolConfig（Swap BASE，加点在路由）
    └── t_symbol_spread_schedule → 合并到 SymbolConfig

Step 3: 加载账户体系
    ├── t_account_group → Map<groupId, AccountGroupConfig>
    │     强平阈值、追保阈值、负余额保护、持仓限制（杠杆值不在此加载，由 LeverageResolver 实时解析）
    ├── t_trading_account → Map<accountId, AccountState>
    │     余额、信用、冻结、净值、保证金占用（引擎快照）
    └── t_user_account_relation → 关系索引

Step 4: 加载持仓与挂单
    ├── t_position（活跃）→ Map<accountId, List<Position>>
    │     含 spec_snapshot_id（品种规格快照引用）
    ├── t_order WHERE status IN (WORKING, PLACED, PENDING_EXEC, DEALER_HOLD, PARTIALLY_FILLED)
    │   → 重建 SymbolBook 挂单簿（按品种分组，价格排序）
    └── t_execution_link WHERE status = PENDING → 重建 A-Book 在途

Step 5: 加载风控规则
    ├── t_risk_rule → RiskRuleSet{}（按 priority 降序，scope 多级）
    ├── t_risk_config_version → 版本戳
    └── 订阅风控配置热更广播（shadow → 原子 swap）

Step 6: 加载路由规则（缓存层）
    ├── 调用 RouteResolver.PullSnapshot() → RouteCache{}（含 match_conditions 求值器）
    └── 订阅路由配置广播，config_version 变更时失效

Step 7: 加载杠杆解析（缓存层）
    ├── 调用 LeverageResolver.PullSnapshot() → LeverageCache{}
    └── 订阅杠杆配置广播，config_version 变更时失效

Step 8: 连接行情与网关
    ├── 订阅 Core SymbolBook SHM 流（行情体系）→ 初始化 SymbolBook{}：bid/ask/seq/stale/gap_level
    ├── 连接路由网关 Gateway Client（A-Book 执行）
    └── 连接资金域 FundService Client（ledger 写）

Step 9: 启动 WAL 恢复
    ├── 读取本地 WAL 日志，重放未 Commit 的事务
    ├── 与 MySQL 投影对账，修复不一致
    └── 进入 ACTIVE 状态，开始接受订单
```

**启动幂等与重入：**

- 引擎重启用 `instance_id + boot_seq` 标识，WAL 重放按 `seq` 单调去重。
- 多实例分片启动时，按 `account_id shard` 亲和，避免跨分片持仓。

### 6.3 并发、分片与内存容量

**并发与分片（落地拍板）：**

- **默认 MVP**：单引擎主线程处理全部订单与 Tick 副作用（确定性最强，延迟最优）。
- **扩展**：按 `account_id` 分片多线程；品种 Tick 广播到相关分片；**禁止跨分片无锁改同一账户**；单分片 ≤ 10 万账户。
- LP 回报线程**不得**直接改账户；一律投递到主循环队列。
- 跨分片持仓对冲敞口：由风控域 `HouseExposureBook` 全局聚合，不在引擎分片间直接同步。

**内存估算（基线，10 万账户）：**

| 项 | 单位 | 估算 |
|----|------|------|
| AccountState | ~1KB/账户 | 100MB |
| SymbolBook（含挂单簿） | ~50KB/品种 | 25MB（500 品种） |
| Position | ~200B/持仓 | 200MB（人均 10 持仓） |
| 风控/路由/杠杆缓存 | — | 50MB |
| WAL 缓冲 | — | 100MB |
| **合计** | — | **~500MB–1GB** |

### 6.4 持久化模型：WAL + Commit

> **拍板**：对客「已成交」ACK 不得早于 WAL 落盘（或等价的同步复制日志）。MySQL 可异步，但必须可追平。

**WAL 设计原则：**

- **Write-Ahead**：变更先写本地 WAL（同步落盘 + 同步复制备机，RPO=0），再应用到内存、写 Outbox、最终 ACK 客户端。
- **崩溃恢复**：重启时重放 WAL，与 MySQL 投影对账，修复不一致；未完成 A-Book 订单按 `external_order_id` 查询 LP 状态机收敛。

```
WAL Entry（二进制，按 seq 单调）:
  - seq: int64          # 引擎内单调递增
  - account_id: int64
  - payload_type: enum # ORDER/DEAL/POSITION/MARGIN
  - payload_crc: uint32
  - payload: bytes     # 序列化的变更
  - outbox_intents: []  # 关联的 Outbox 事件
  - timestamp: int64
```

**Commit 流程（`commitTransaction`，单账户）：**

```
function commitTransaction(change):
    # 1. 写 WAL（同步落盘 + 复制备机）— write-ahead，保证崩溃后可重放
    walSeq = wal.append(change)
    # 2. 应用到内存（Order/Deal/Position/margin/balance 状态生效）
    applyToMemory(change)
    # 3. 写 Outbox（与 WAL 同一持久化单元，携带 walSeq 关联）
    for event in change.outboxEvents:
        outbox.append(event, walSeq)
    # 4. ACK 客户端（不得早于步骤 1 WAL 落盘）
    ack(change)
    # 5. 异步：MySQL Writer 批量刷（活跃表+deal+ledger+outbox）+ Kafka 发布 Outbox
    # 6. 查询侧 Redis 投影更新（equity 等展示字段）
```

**Outbox 死信与重试：**`t_engine_outbox`（见九、9.8）含 `retry_count`、`next_retry_at`、`dlq_topic` 字段；重试 N 次（默认 5）后入死信 topic（如 `dlq.trading.deal`），告警 + 人工；Kafka 发布成功后标记 `status=PUBLISHED`，定期清理。

### 6.5 Trade API 职责边界

| 允许 | 禁止 |
|------|------|
| 鉴权、限流、参数粗检、转发引擎 | 自算保证金并写库 |
| 读历史（order_history/deal/position_history） | 直连改 balance |

## 七、核心流程

### 7.1 PreTrade 校验链

所有下单入口（市价/挂单/平仓）共用同一条校验链（Step 0–10，共 11 步），**顺序不可重排**：

```
function submitOrder(req):
    # 幂等：先查 (account_id, client_order_id)
    existing = idempotentCache.get(req.account_id, req.client_order_id)
    if existing: return existing

    acct = AccountState[req.account_id]
    sym = SymbolConfig[req.symbol_id]

    # Step 0: 自成交防护
    if isSelfTrade(acct, req):
        return reject(TRD_VALIDATION_SELF_TRADE)

    # Step 1: 账户状态
    if acct.status not in [ACTIVE]: return reject(ACC_STATE_ACCOUNT_FROZEN)
    if not acct.can_trade: return reject(ACC_STATE_ACCOUNT_FROZEN)

    # Step 2: Kill Switch（风控域广播）
    if riskKillSwitchActive(acct.entity_id, sym.symbol_id):
        return reject(RISK_STATE_KILL_SWITCH_ON)

    # Step 3: 品种可访问性 + 时段
    sess = SessionQueryService.IsTradable(sym.symbol_id, now, acct.entity_id)
    if not sess.tradable:
        if sess.runtime_state == CLOSE_ONLY and req.is_open:
            return reject(SESS_STATE_CLOSE_ONLY)
        return reject(SESS_STATE_MARKET_CLOSED)

    # Step 4: 订单参数（手数步进/上下限等）
    if not validateOrderParams(req, sym): return reject(TRD_VALIDATION_VOLUME_INVALID)

    # Step 5: 持仓/挂单数限额
    if exceedsPositionLimit(acct, sym, req): return reject(TRD_LIMIT_POSITION_EXCEEDED)

    # Step 6: 保证金（调用 LeverageResolver）
    lev = LeverageResolver.Resolve(acct.id, sym.id, notional, equity, config_version)
    margin = calcMargin(req, lev, sym)
    if acct.free_margin < margin: return reject(TRD_LIMIT_MARGIN_INSUFFICIENT)

    # Step 7: 日损限额
    if exceedsDailyLoss(acct): return reject(TRD_LIMIT_DAILY_LOSS_EXCEEDED)

    # Step 8: 自定义风控规则（风控域 RiskRuleSet）
    if not evalCustomRules(acct, sym, req): return reject(RISK_LIMIT_*)

    # Step 9: Core Book 健康
    if SymbolBook[sym.id].stale and req.is_open:
        return reject(TRD_DEGRADED_STALE_QUOTE)

    # Step 10: 公司敞口预检
    if not houseExposurePrecheck(acct, sym, req): return reject(RISK_LIMIT_HOUSE_EXPOSURE_EXCEEDED)

    # 通过 → 路由
    route = RouteResolver.ResolveRoute(sym.id, acct.id, sess.session, req.volume, req.side, config_version)
    if route.reject_reason: return reject(route.reject_reason)

    return executeByRoute(req, route, lev)
```

> `ACC_*`/`RISK_*`/`SESS_*` 前缀的错误码归属账户/风控/时间配置域的错误码命名空间（见十三、13.2 各域自有前缀），本域只登记 `TRD_*`；引擎在实现层引用其它域的错误码是允许的跨域消费，不代表本域拥有其定义权。各步骤与风控域细分子原因（`details.sub_reason`）的映射见 `[11]风控系统/风控系统设计.md` §4.2。

### 7.2 市价开仓（B-Book 快路径）

```
SubmitOrder(BUY 1 lot EURUSD)
  ├─ 幂等: account+client_order_id 已存在 → 返回首次结果
  ├─ PreTrade 通过（见七、7.1） → status=WORKING → order.created
  ├─ Route: 匹配 t_route_rule → B-Book
  │    fill = Core Ask(买)/Bid(卖)；应用 max_deviation；失败则 REJECT（无 Deal）
  ├─ Atomic Commit（见六、6.4）
  │    Deal(IN) + Position(OPEN) + commission ledger
  │    margin_used↑ equity 重算
  │    Order → FILLED → 归档意图
  │    deal.created + position.opened + order.filled
  └─ ACK 客户（含 deal/position 票号）
```

引擎内部执行伪代码：

```
function executeBBook(req, route, lev):
    book = SymbolBook[req.symbol_id]
    if book.stale: return reject(TRD_DEGRADED_STALE_QUOTE)

    price = (req.side == BUY) ? book.ask : book.bid
    if req.slippage_points and priceDeviation(price, req) > req.slippage_points:
        return reject(TRD_LIMIT_SLIPPAGE_EXCEEDED)

    deal = createDeal(req, price, lev)
    commitTransaction(deal)   # WAL + 内存 + Outbox，见六、6.4
    # 异步：FundService.CreditAccount/DebitAccount 写 ledger
    # 异步：发 deal.created.v1（Outbox）
    return SubmitOrderResponse(order_id, FILLED, deal_id=deal.id)
```

### 7.3 市价开仓（A-Book）

```
PreTrade 通过 → WORKING
  → 发送 LP NewOrderSingle（记录 external_order_id）→ PENDING_EXEC
  → 等待 ExecutionReport
       全成 → 同 Atomic Commit（价格用 LP 成交价，经滑点保护）
       部成 → PARTIALLY_FILLED + 按 filling_mode 处理剩余
       拒绝/超时 → 无成交则 REJECTED；有成交则保留已成部分
```

**滑点保护拍板：** 对客户不利滑点超过 `max_deviation` / 路由规则上限 → 拒收该回报（A-Book 需向 LP 发取消/冲正策略，按网关能力）；价格改善默认接受。

引擎内部执行伪代码：

```
function executeABook(req, route, lev):
    order.status = PENDING_EXEC
    writeWAL(order)
    # 调用网关 Gateway.SubmitToLP(route.gateway_id, req)
    link = createExecutionLink(req, route)
    return SubmitOrderResponse(order_id, PENDING_EXEC)
    # LP 回报经 onLpReport 回调处理

function onLpReport(link, report):
    if report.timeout:
        # 延迟队列 + 下一 Tick 重定价（不用固定 sleep）
        scheduleReprice(link, nextTickOrDelay(route.execution_delay_ms))
        if link.retry_count >= max_retry:
            cancelOrder(link.order_id, reason=TRD_EXTERNAL_LP_TIMEOUT)
        return
    if report.rejected:
        order.status = REJECTED
        order.reject_reason = TRD_EXTERNAL_LP_REJECTED
        commitTransaction(order)
        return
    deal = createDealFromReport(link, report)
    commitTransaction(deal)
```

### 7.4 Mixed 路由拆单

客户单按 `t_route_rule` 的 `allocations`（`gateway_id`、`ratio`、`execution_profile_id`）拆成多个子单，分别走 B-Book/A-Book 执行，客户只看到一笔聚合 Deal：

```
客户单 SubmitOrder
  → RouteResolver.ResolveRoute 返回 allocations[{gateway_id, ratio, ...}]
  → 引擎按 allocation 拆子单
    ├── 子单1 → B-Book（内部簿记）→ 子Deal1
    └── 子单2 → A-Book（LP执行）→ 子Deal2
  → 客户侧单 Deal = mergeDeals(子Deal1, 子Deal2)
    - price = 加权均价（按各子单实际成交量）
    - commission = sum(子.commission)
    - 子单在 t_execution_link 关联
  → commitTransaction(clientDeal)
  → 返回 SubmitOrderResponse(FILLED, deal_id=clientDeal.id)
```

引擎内部执行伪代码：

```
function executeMixed(req, route, lev):
    subDeals = []
    for alloc in route.allocations:
        sub = splitOrder(req, alloc.ratio)
        if alloc.gateway_id == INTERNAL:
            subDeal = executeBBook(sub, route, lev)
        else:
            subDeal = executeABook(sub, route, lev)
        subDeals.append(subDeal)

    clientDeal = mergeDeals(subDeals)
    clientDeal.price = weightedAvg(subDeals, byVolume)
    clientDeal.commission_amount = sum(sub.commission for sub in subDeals)
    commitTransaction(clientDeal)
    return SubmitOrderResponse(order_id, FILLED, deal_id=clientDeal.id)
```

客户只看到一笔聚合 Deal；子执行链路在 `t_execution_link` 留痕，供对账与代理计提。

### 7.5 挂单

```
PreTrade（保证金用挂单价与 margin_rate_limit）
  → PLACED，进入 SymbolBook 对应有序集合
  → Tick 路径: 价格触及 → 转为执行（同 7.2/7.3）
  → Cancel / Day收盘 / expire_at → CANCELLED/EXPIRED 归档并释放预占保证金
```

**预占保证金拍板：** 挂单在 PLACED 时预占（或按账户组配置「触发时再检」）。**默认**：PLACED 预占，防止无限挂单击穿保证金；修改挂单时重算预占差额。

**Tick 入口（Core Book Delta，驱动挂单触发/SL-TP/预警/强平检查的统一分发点）：**

```
function onCoreBookDelta(delta):
    book = SymbolBook[delta.symbol_id]
    if delta.seq <= book.seq:
        return  # 乱序丢弃，行情侧保证 seq 单调
    book.bid = delta.bid
    book.ask = delta.ask
    book.seq = delta.seq
    book.stale = delta.stale
    book.gap_level = delta.gap_level
    EventBus.emit(CORE_BOOK_DELTA, book)

    if not book.stale:
        checkPendingOrders(book)      # 挂单触发（本节）
        checkSLTP(book)               # 止损止盈（见七、7.7）
        checkPriceAlert(book)         # 价格预警
    checkMarginCallAndStopOut(book)   # 强平（见七、7.8，含 stale 处理）
```

**挂单触发（只扫内存）：**

```
function checkPendingOrders(book):
    pendings = book.pendingOrdersByPrice()  # 内存有序集合
    for order in pendings:
        if order.type == LIMIT and priceCrossed(order, book):
            fillOrder(order, book)
        elif order.type == STOP and priceCrossed(order, book):
            promoteToMarket(order, book)
    # 禁止热路径扫 t_order，只扫 SymbolBook 内存索引
```

**穿越判定（防漏触发）：**

- LIMIT BUY：`book.ask <= order.price`
- LIMIT SELL：`book.bid >= order.price`
- STOP BUY：`book.ask >= order.price`
- STOP SELL：`book.bid <= order.price`
- 跳空（`gap_level > 0`）：按穿越后第一档价成交，记录滑点到 deal。

**STALE 处理：** `book.stale = true` 时：**禁开新仓**（返回 `TRD_DEGRADED_STALE_QUOTE`），允许平仓与挂单触发（用最后已知价）；强平在 STALE 期间**跳过**，避免用坏价打客户，恢复后立即补扫。

### 7.6 平仓

```
ClosePosition / SubmitOrder(close_position_id)
  → 校验持仓归属、可平量、时段（CLOSE_ONLY 允许）
  → 路由执行（通常与开仓同规则，或 close_route 覆盖）
  → Atomic Commit:
       Deal(OUT) + profit/commission/swap 结算
       Position.volume↓ 或移除并写 position_history
       balance += profit + swap + commission(负)
       margin_used↓
```

PnL（账户币种）：

```
BUY 仓:  profit = (close - open) * volume * contract_size * fx_to_account
SELL 仓: profit = (open - close) * volume * contract_size * fx_to_account
```

汇率路径使用引擎内存中的转换链（品种体系定义），禁止临时读库。

### 7.7 SL / TP

- 存在 Position 上；Tick 用 Core Bid/Ask 检查（由 7.5 `onCoreBookDelta` 的 `checkSLTP(book)` 调用）。
- 触发后生成 `source=SYSTEM` 的平仓订单，`close_reason=SL|TP`。
- 修改 SL/TP 走 `ModifyPosition`，不产生 Deal。
- `freeze_level` 内禁止修改（品种交易配置）。

### 7.8 强平（Stop Out）

```
Tick/定时: equity / margin_used * 100 <= stop_out_level
  → margin.stopout 事件
  → 按账户组策略排序持仓（默认浮亏从大到小）
  → 逐笔系统市价平仓，直到 margin_level 恢复到安全阈值或无仓
  → 允许负余额时记 NEGATIVE_BALANCE_CORRECTION（资金域类型，引擎写）
```

**Margin Call 与 Stop Out 触发（由 7.5 `onCoreBookDelta` 的 `checkMarginCallAndStopOut(book)` 调用）：**

```
function checkMarginCallAndStopOut(book):
    for acct in AccountState.values():
        ml = marginLevel(acct, book)
        if ml <= acct.stop_out_level:
            stopOut(acct, book)
        elif ml <= acct.margin_call_level:
            if not acct.in_margin_call:
                acct.in_margin_call = true
                EventBus.emit(MARGIN_CALL, acct)
                # 发 margin.call.v1（Outbox）
```

**Stop Out 执行与三模式：**

```
function stopOut(acct, book):
    mode = acct.stop_out_mode
    positions = sortedPositions(acct, by=mode)
    while marginLevel(acct, book) <= acct.stop_out_level and positions:
        pos = positions.pop(0)
        deal = closePosition(pos, book, reason=STOP_OUT)
        commitTransaction(deal)
        # 释放保证金后重新计算 ml
    if acct.balance < 0 and acct.negative_balance_protection:
        FundService.CreditAccount(acct, -acct.balance, ledger_type=NEGATIVE_BALANCE_CORRECTION)
```

| 模式 | 排序 |
|------|------|
| 1 亏损最大优先 | 按 floating_pnl 升序 |
| 2 保证金占用最大 | 按 margin_used 降序 |
| 3 Smart 部分平仓 | 按释放量/损失比降序，部分平仓至 ml > stop_out |

**强平失败升级：** 无报价/STALE 时跳过，恢复后补扫；连续失败 N 次触发 Dealer 介入 + 品种 HALT + 通知运营（`risk.kill_switch.toggled.v1`）；最大滑点/重试次数由账户组配置。

### 7.9 日切 Swap 与公司行动

```
结算引擎日切（00:00 UTC 触发）:
  对每笔 OPEN 持仓按品种 swap 配置计息（三倍 Swap 日、伊斯兰账户豁免特殊处理）
  → position.swap 累加
  → FundService.DebitAccount(ledger_type=SWAP)
  → 可选立即影响 balance/equity（拍板：立即入账，与 MT5 一致）
  → 生成日切对账报告；与资金域 ledger 三方对账（见十、10.2）
```

**公司行动执行：** 消费 `symbol.corporate_action_scheduled.v1` 事件。`SPLIT`/`REVERSE_SPLIT`：调整持仓量与价格，写新 `spec_snapshot_id`；`CASH_DIVIDEND`：`FundService.CreditAccount`；`DELISTING`：强制平仓 + 通知。

### 7.10 Dealer 审批

```
function dealerHold(req):
    order.status = DEALER_HOLD
    writeWAL(order)
    # 发 dealer.review.requested 事件（通知域）
    # Dealer 经 RiskOps API 审批（DealerApprove/DealerReject）
    # 超时策略：默认 60s → 自动 REJECTED（reason=TRD_STATE_ORDER_REJECTED）
    scheduleDealerTimeout(order.id, 60s)
```

### 7.11 冲正

```
Dealer 四眼 → ReverseDeal(deal_ticket)
  → 生成反向 Deal（关联 orig_deal_ticket）
  → 回滚持仓或生成补偿持仓（按是否已平）
  → 反向 ledger
  → deal.reversed.v1（Partner 回拨佣金）
```

## 八、保证金与费用（交易侧契约）

### 8.1 保证金

完整公式见风控系统设计；交易系统保证：

- 开仓/挂单预占使用同一 `calcMargin`；
- 对冲折扣读取 `margin_hedged`；
- 浮动杠杆按名义敞口阶梯（经 `LeverageResolver` 八级解析链，见三、3.2）；
- `margin_used` 为账户聚合快照，持仓级 `position.margin` 可重算对齐。

### 8.2 佣金

- 规则权威：`t_account_group_commission`；
- 开仓 Deal、平仓 Deal 可分别计；
- 金额写入 Deal.commission，并写 ledger `COMMISSION`；
- **代理佣金不在本系统计算**（代理域只消费 `deal.created.v1`，见十三、13.5 佣金单一权威）。

### 8.3 与资金域的协作

| 场景 | 调用 |
|------|------|
| 客户出金预检 | Fund → `PreviewWithdraw`（含浮动盈亏） |
| 出金冻结 | Fund → `FreezeFunds`（交易持仓仍由引擎估值） |
| 入金 | Fund → `CreditAccount` |
| 交易盈亏 | 引擎直接写 ledger，Fund 只读对账 |

## 十、验收 SLO、容灾与对账

### 10.1 SLO

| 指标 | 范围 | SLO |
|------|------|-----|
| Submit → Deal ACK（B-Book 市价，端到端） | 客户请求到确认成交，含 PreTrade+Route+Atomic Commit | P99 &lt; 20ms |
| Route 后 Atomic Commit（B-Book，引擎内部） | `executeBBook`：定价+WAL+内存应用+Outbox 写入 | P99 &lt; 5ms |
| Submit → 发往 LP（A-Book，端到端） | 客户请求到网关入队完成，不含 LP 往返 | P99 &lt; 20ms |
| Tick → SymbolBook 更新 | Core Book Delta 到内存写入完成 | P99 &lt; 1ms |
| Tick → 挂单触发/SL-TP/浮盈扫描（单品种） | Book 更新到检查完成 | P99 &lt; 1ms |
| WAL 写入 | 单笔变更落盘 + 同步复制备机 | P99 &lt; 1ms |
| Outbox lag | 事件产生到 Kafka 发布确认 | &lt; 5s |
| 资金账实：日切 ledger vs balance | 日切对账差异 | = 0 |
| Deal 与 Position 勾稽 | 日检差异 | = 0 |

> `Submit→发往LP` 与 `Route 后 Commit` 曾在拆分的两份文档中各给出一套未注明范围的数字（如 5ms/20ms 互换），合并时按"端到端"与"引擎内部"两个清晰边界统一口径，并取更保守的上界作为 SLO。

### 10.2 日检 SQL（示意）

```sql
-- 活跃持仓量 = 开仓 Deal 入 - 平仓 Deal 出（按 position_ticket）
-- 账户 balance 链: 昨日末 + 今日 ledger 净额 = 今日末快照
-- Outbox 无 status=0 且 created_at < now()-5m 的长期积压
```

### 10.3 可观测性

每笔订单强制 `correlation_id`（可等于 `request_id`）贯穿：API → 引擎 → LP → Deal → Outbox → Partner。
指标：拒单原因计数、A-Book 超时、滑点分布、强平次数、WAL 落后、MySQL writer lag。

### 10.4 容灾与降级

| 场景 | 降级 |
|------|------|
| 主引擎故障 | WAL 重放，备机接管 &lt; 30s；切换期间**只平不开** |
| Core Book STALE | 禁开仓，允许平仓 |
| SessionQueryService 不可用 | 本地缓存兜底（最近快照），告警 |
| LeverageResolver 不可用 | Caffeine + Redis 兜底，超时用最近解析值 |
| RouteResolver 不可用 | 本地 RouteCache 兜底，信用检查降级为拒绝 |
| FundService 不可用 | 拒绝新成交（避免账实不一致），允许平仓挂账待恢复 |
| Kafka 不可用 | Outbox 积压本地，恢复后补发 |

## 结语

本交易域设计以下述几点收口：

1. **运行时权威在引擎内存**，MySQL 是持久化投影，不是热路径决策源。
2. **成交原子性 + Deal 不可变 + ledger 结构归资金域**，构成账实一致的最小完备集。
3. **引擎是运行时契约消费方**：时段/杠杆/路由/资金/行情的配置真相分别由对应域服务提供（`SessionQueryService`/`LeverageResolver`/`RouteResolver`/`FundService`/Core Book SHM），引擎只持有运行时内存与 WAL，不重复持有配置表。

本文即交易域权威设计，涵盖域决策与运行时实现；评审与实现均以本文十四条决策为准。
