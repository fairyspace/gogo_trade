🔗 二、表间关系
-- 网关层 t_gateway_group (1)──<(N) t_gateway_group_member (N)>──(1) t_gateway t_liquidity_pool (1)──<(N) t_liquidity_pool_member (N)>──(1) t_gateway │ 映射 t_gateway_symbol_mapping -- 路由规则层 t_route_rule (1)──<(N) t_route_rule_allocation ──> GATEWAY(t_gateway) / POOL(t_liquidity_pool) / INTERNAL │ 引用 ├──▶ t_execution_profile -- 执行参数 └──▶ t_markup_profile -- 加价/费用 -- 运行时 t_gateway_credit_limit (1)──(1) t_gateway_credit_usage -- 配置/状态分离 -- 配置发布 配置变更 ──> t_config_change_log -- 前后镜像留痕 t_config_version ──> 各节点内存快照原子刷新
📌 两类表区分：配置表（t_gateway / t_route_rule / Profile 等）全量入内存，运行时不查库；运行时表（credit_usage / audit_log / metric）高频读写，不入内存。   |   交易时段/可否交易 由时间配置模块提供，路由层只消费结论，不重复建模。
🗺️ 三、路由决策主流程（读路径）
1
📥 接收订单上下文
订单 携带账户、品种、方向、量、客户属性、持仓、请求环境，以及来自时间配置模块的当前时段与可交易状态。

全部为内存对象，不查数据库。配置快照已由上一次发布预加载至内存。
📌 核心架构：配置全量入内存 → 运行时内存匹配 → 数据库不参与实时查询
📖 内存配置快照（由 t_config_version 驱动加载）
按 priority 降序遍历规则
2
⚙️ 规则匹配
路由引擎 从内存快照取 status=1 且在生效期（effective_start/end）的规则，按 priority DESC 逐条评估 match_conditions 条件树：
📐 条件树结构
AND/OR/NOT 逻辑节点无限嵌套，叶子节点为 {dimension, field, operator, value}
📊 可匹配维度
ORDER · SYMBOL · ACCOUNT · CLIENT · POSITION · REQUEST（含 session/weekday）
🔣 运算符
EQ / NEQ / GT / GTE / LT / LTE / IN / NOT_IN / BETWEEN / CONTAINS / REGEX
🎯 命中逻辑
第一条命中即停；全部未中则用 is_default=1 兜底规则
📖 t_route_rule（内存）
规则命中 → 读取分配列表
3
🎯 确定路由目标与分配方式
读 execution_mode（A/B/HYBRID）与 allocation_mode，从 t_route_rule_allocation 取该规则所有启用分配，按模式选目标：
🔢 PRIORITY（故障转移）
取 allocation_priority 最大且可用的目标；拒单/不可用时按 reroute_to_allocation_id 链式降级
⚖️ RATIO（按比例）
按 allocation_ratio（之和=100）加权随机选目标；整体流量逼近配置比例
✂️ SPLIT（拆单）
volume > splitThresholdVolume 时拆为 ≤ maxSplitCount 笔，每笔 ≥ minSplitVolume
🎯 目标三选一
GATEWAY（单 LP 网关）
POOL（流动性池聚合）
INTERNAL（B-Book 内部撮合引擎）
📖 t_route_rule_allocation（内存）
目标为 POOL → 聚合最优价；目标为 GATEWAY → 映射转换
4
📊 报价聚合 & 品种映射
A. 流动性池聚合（目标为 POOL 时）
从池内各 LP 成员拉取实时报价，按以下顺序处理：
① 过滤：丢弃过期报价（报价龄 > stale_quote_threshold_ms） 丢弃价格偏离过大报价（|价−中位数| > price_deviation_threshold_bps 基点） 有效源数 < min_active_sources → 池降级 ② 聚合（按 aggregation_mode）： BEST_PRICE → bid = max(各有效bid)，ask = min(各有效ask) // 点差最优 VWAP → 按量加权平均 DEPTH_MERGE → 合并各 LP 订单簿深度
B. 品种映射转换（与具体网关交互时，读 t_gateway_symbol_mapping）
代码转换：remote_symbol ↔ 本系统 symbol_id 价格转换：本系统价 = 上游价 × price_scale_factor （NULL=1） 手数转换：上游手数 = 本系统手数 × volume_scale_factor （NULL=1） 报价过滤：点差 = ask − bid；> max_spread_points 则丢弃
📖 t_liquidity_pool（内存） 📖 t_liquidity_pool_member（内存） 📖 t_gateway_symbol_mapping（内存）
信用检查（credit_check_mode ≠ NONE）
5
🔐 信用额度检查
读静态配置（内存）+ 实时占用（Redis/内存）：
可用额度 = limit_amount − used_amount − reserved_amount 本单占用（按 credit_type）： EXPOSURE → 名义敞口金额 VOLUME → 手数 NETTING → 净头寸增量 若 本单占用 > 可用额度 → 按 exceed_action 处理： SKIP → 跳过此网关，路由下一个 REJECT → 直接拒单 ALERT → 仅告警，仍然路由 REROUTE → 触发重路由
📌 热路径性能：实时占用（used/reserved）热路径走内存/Redis（原子 INCRBY），DB 为异步落库副本，严禁在订单主链路同步 UPDATE。
  并发安全：UPDATE ... SET used_amount=used_amount+:v, version=version+1 WHERE id=:id AND used_amount+reserved_amount+:v<=:limit AND version=:ver
📖 t_gateway_credit_limit（内存） ⚡ t_gateway_credit_usage（实时）
信用通过 → 计算客户最终价格
6
💰 定价计算（两层加点 + 佣金）
加点分两层，叠加应用：
── 第一层：接入层加点（t_gateway_symbol_mapping，与客户无关）── bid1 = 上游bid + bid_markup_points ask1 = 上游ask + ask_markup_points ── 第二层：客户分层加点（t_markup_profile，按规则引用）── 覆盖优先级：symbol_overrides > category_overrides > default 最终bid = bid1 + profile_bid_markup + session_overrides.additionalBidMarkup 最终ask = ask1 + profile_ask_markup + session_overrides.additionalAskMarkup ── 佣金（按 commission_type）── PER_LOT → 佣金 = commission_per_lot × volume PERCENT → 佣金 = 名义金额 × commission_percent PER_DEAL → 固定金额 最终佣金 = max(佣金, min_commission)
📖 t_markup_profile（内存）
校验执行能力 → 下发给执行器
7
🚀 执行下发
读 t_execution_profile（内存），校验 t_gateway.capabilities，生成执行指令后下发：
✅ 成交方式
fill_policy：MARKET / IOC / FOK / RETURN
📉 滑点控制
max_slippage_points，与时段倍率（来自时间模块）叠加
⏱️ 延迟执行
delay_type（NONE/TIME/TICK）+ delay_value
👤 人工审核
auto_execution=0 或满足 manual_review_conditions → 转人工
🔌 断线处理
REJECT / QUEUE / INTERNAL / REROUTE
🤖 算法执行
VWAP / TWAP / ICEBERG / SNIPER，携带 algo_config
执行目标边界：
A-Book → LP 网关适配器（外部 LP 成交，结果异步回传）
B-Book → 撮合/Dealing 引擎（内部成交，订单簿/持仓/盈亏由撮合引擎负责）
本模块到"下发执行指令"为止，不涉及撮合逻辑。
📖 t_execution_profile（内存） 📖 t_gateway.capabilities（内存）
异步落库全过程决策
8
📝 审计记录
路由决策完成后，异步批量写入审计日志，不阻塞路由主链路：
match_detail
命中规则 ID/Code、评估规则数、匹配耗时（微秒，单调时钟）
sor_detail
各 LP 报价、最优 Bid/Ask、场所选择原因
credit_check_result
各网关限额使用情况、是否通过
reroute_history
重路由来源→原因→目标链路
markup_detail
原始价 / 各层加点 / 最终价
execute_result
0-失败 / 1-成功 / 2-超时 / 3-拒绝 / 4-部分成交
✍️ t_route_audit_log（异步）
✍️ 四、状态写路径
触发时机运算写入表
路由成功提交 reserved_amount += 本单占用（乐观锁原子 UPDATE，version 校验） t_gateway_credit_usage
成交回报 reserved -= 本单占用; used += 实际成交占用 t_gateway_credit_usage
网关拒单 按组 reroute_config + 成员 can_reroute + 分配链选下一目标，追加重路由记录 t_route_audit_log.reroute_history
定时聚合（每 10s） 统计窗口内延迟/成交率/拒单率/滑点 t_gateway_metric
信用周期重置 按 reset_cycle（DAILY/WEEKLY/MONTHLY）清零占用 t_gateway_credit_usage
配置发布 写版本号 + 快照 checksum；节点拉快照 → 原子指针切换 t_config_version
配置变更 与业务变更同事务写入前后镜像（append-only，禁 UPDATE/DELETE） t_config_change_log
