1
🏛️ 平台每周交易时段配置
运营 维护平台周一到周日的开闭市基线时段，所有品种时段必须落在此范围内

系统 校验 close_min > open_min、close_min ≤ 2880（允许跨日）、同 weekday 内不重叠
📌 session_type 仅允许静态时段类型：PRE_OPEN / OPEN / PRE_CLOSE / CLOSE_ONLY / AUCTION / POST_CLOSE
⛔ HALT / MAINTENANCE / QUOTE_ONLY 属于运行时状态，由 Step 8 覆盖表表达
📄 t_platform_session (CRUD) 📝 t_config_audit (INSERT)
平台基线确定 → 配置品种时段 📈
2
📈 品种每周时段 + DST Profile 标记
运营 为每个品种配置周时段，标记 dst_profile（USA/EUROPE/AUSTRALIA/CHILE/NZ/NONE）

系统 校验：①不超出平台时段；②同 (symbol_id, week_day) 不重叠；③跨日时段单行表达 (close_min ∈ [0,2880])
🔑 关键设计 - 跨日时段：周日 22:00 → 周一 05:00 单行表达
    week_day=7, open_min=1320 (22:00), close_min=1740 (次日05:00)
🔑 关键设计 - 版本管理：同 (symbol_id, week_day, seq_no) 多版本通过 effective_from/to 隔离
📄 t_symbol_session (CRUD) 📖 t_platform_session (READ-校验)
品种配置完成 → 接入外部假日数据 🌐
3
🌐 交易所日历数据接入（多源 + 冲突仲裁）
系统 定期从外部数据源同步交易所假日数据，多源数据共存于 t_exchange_holiday，按 confidence 仲裁
📡 外部数据源
第三方 主要来源:
  • JPX J-Quants API (东京)
  • HKEX iCal (香港)
  • EODHD API (70+交易所)
  • Nasdaq Data Link EFPH
  • pandas_market_calendars
⚖️ 冲突仲裁
运营 处理冲突:
  1. 自动按 confidence DESC 取主
  2. 字段冲突 → 告警
  3. 运营裁定 → status=1
  4. 其他版本置 superseded_by
  5. 转特殊日期实例
🔑 关键设计：交易所日历仅作参考数据，不直接参与运行时计算。运营确认后转化为 t_special_session 实例 (source=EXCHANGE_SYNC)
📄 t_exchange_holiday (CREATE) ⚖️ resolve_conflict 📄 t_special_session (INSERT)
外部数据 → 内部规则定义 📜
4
📜 特殊日期规则定义 + 年度递推
运营 一次定义、自动递推 — 支持 9 种 recur_type，覆盖全球主要节日体系

🗓️ recur_type 全类型
📅 FIXED — 公历固定日期（元旦）
📆 WEEKDAY_OBSERVED — 周末顺延（独立日）
🔢 NTH_WEEKDAY — 第N个星期几（感恩节）
🔚 LAST_WEEKDAY — 最后周几（阵亡纪念日）
🌙 LUNAR — 农历（春节，CN/JP/KR/VN）
🕌 ISLAMIC — 伊斯兰历（开斋节）
✡️ JEWISH — 希伯来历（赎罪日）
🧮 EXPRESSION — DSL（easter+1d）
✋ MANUAL — 不自动递推，临时手动
🔑 实务指引：99% 场景使用 stack_policy=OVERRIDE（默认值）
📐 INTERSECT 仅在"收窄原时段"时用; UNION 仅在"周日加开"等扩展场景使用
🔮 提供 /dry-run 接口预览未来 5 年生成实例
📄 t_special_session_rule (CRUD) 🧠 规则引擎递推 📄 t_special_session (BATCH INSERT)
规则定义完成 → DST 半自动接管 🌅
5
🌅 DST 半自动版本生成
系统 每年 12 月触发，为下一年所有 dst_profile != 'NONE' 的品种自动生成春令时/冬令时版本

🤖 自动生成步骤
  1. 查 t_dst_schedule(profile, year+1) → 获取春进/秋退日期
  2. 扫描所有 t_symbol_session 中 effective_to IS NULL 的记录
  3. 春进日：原版本 effective_to=spring_forward_date；新版本 open/close -60min
  4. 秋退日：再生成版本 open/close +60min 复原（UTC 视角看时段提前/延后）
  5. source='DST_AUTO', version+=1，写 t_snapshot_run_log
💡 关键价值：1万品种 × 每年 2 次切换 = 2万条手动版本 → 全部自动化
🛡️ 合规：所有自动生成均写入 t_config_audit 审计，change_source=DST_AUTO
📖 t_dst_schedule (READ) 📄 t_symbol_session (INSERT versions) 📝 t_config_audit (BATCH) 📊 t_snapshot_run_log
配置就绪 → 每日预计算 ⏰

系统 每日 00:00 UTC 触发调度任务，合并平台 ∩ 品种 ∩ 特殊日期，通过三阶段管道生成最终时段

🔑 三阶段管道处理顺序固定 (避免歧义)：
  1️⃣ INTERSECT 在前：先施加约束，避免 UNION 扩展又被截掉
  2️⃣ UNION 在中：受限范围内扩展（"受限开放"）
  3️⃣ OVERRIDE 在后：最高优先级直接拍板

🎯 核心机制专题
🧠
三阶段管道 — 多规则叠加确定性
阶段1 INTERSECT
逐条取交集
open=max, close=min
→
阶段2 UNION
区间合并
多时段展开多行
→
阶段3 OVERRIDE
最高优先级替换
priority+created_at
示例：常规 09:30-16:00，假期 OVERRIDE 缩短到 09:30-13:00 + 盘前 UNION 加开 08:00-09:30
→ 阶段1透传 [570,960] → 阶段2 UNION 后无变化（OVERRIDE 优先级更高）→ 阶段3 OVERRIDE 替换为 [570,780]
注：实际混合规则按各 stack_policy 进入对应阶段，OVERRIDE 优先级 > UNION/INTERSECT

SUPERSEDED 替换链：同 target 出现更严格覆盖时，旧记录置 SUPERSEDED + superseded_by 关联新记录，保留完整事件链
PHASED 链路：每个阶段产生新 override，通过 previous_phase_id 串联，任一阶段可手动 LIFT 跳过后续
🏗️
层级覆盖 — 取最严格状态
品种实际状态 = 最严格 (PLATFORM + EXCHANGE + CATEGORY + PRODUCT 各自的 ACTIVE override)
严格度排序：HALT > CLOSE_ONLY > MAINTENANCE > QUOTE_ONLY > PRE_OPEN > AUCTION
🔑 规则：
① 高层级生效自动级联到低层级
② 同 (target_kind, target_value) 同时只能有一条 ACTIVE (UNIQUE KEY)
③ 低层级覆盖不覆盖高层级 (PLATFORM HALT 时 PRODUCT 写入被拒绝)
④ 高层级解除时，低层级品种恢复各自原状态
