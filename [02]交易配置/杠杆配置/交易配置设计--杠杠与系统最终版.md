# 杠杆系统设计 — 最终版 v2.0

> **版本**：v2.0 Final（与项目品种 / 账户体系完成对齐）  
> **范围**：FX / 贵金属 / 能源 / 股指 / CFD / 加密货币 / 美股融资 / A 股两融 / 期货 / 期权（全球多监管）  
> **依赖**：项目已有的 `t_account_group`、`t_trading_account`、`t_symbol_category`、`t_symbol_group`、`t_symbol`、`t_account_group_symbol_config`  
> **核心**：14 张 `t_` 前缀的杠杆专有表，所有命名、字段类型、注释风格、软删除模式与项目其它模块完全统一。

## 一、设计哲学

杠杆系统是金融系统的"风险阀门"，一切设计取舍都围绕**正确、可解释、可审计、可演进**四个目标展开。

### 1.1 七条设计原则

1. **账户组（`t_account_group`）是后台运营的中心管理单元**——延续 MT5 Group / cTrader Group 思路，所有交易行为以"账户挂在哪个账户组下"为起点。
2. **监管杠杆是绝对天花板**——任何配置、任何运营调整、任何客户级覆盖，最终生效杠杆都不得突破对应监管区的法定上限。
3. **主数据与业务规则物理分离**——`t_symbol`、`t_account_group` 等主体表不耦合杠杆字段；杠杆规则集中在本模块 14 张表中独立演进。
4. **拒绝多态关联**——每个杠杆配置表都以明确的外键关联具体主体（账户组 / 客户 / 账户 / 品种 / 品种组），不使用 `target_type + target_id` 这种类型擦除模式。
5. **统一的 LeverageBlock 字段约定**——所有"杠杆与保证金"配置表共享一组标准字段（`default_leverage`、`max_leverage`、`min_leverage`、`leverage_step`、`allowed_leverage_values`、`margin_rate`、`maintenance_margin_rate`、`hedged_margin_rate`），减少认知负担与维护成本。
6. **缓存不落库**——`ResolvedLeverage` 的解析结果只存在于 Caffeine（JVM 进程内）+ Redis（集群共享）两级缓存，不持久化到 MySQL，避免一致性陷阱。
7. **全链路审计**——所有杠杆配置变更经应用层 AOP 统一拦截写入 `t_leverage_change_log`，并通过 Outbox + CDC 通知缓存失效，满足 MiFID II / SEC Rule 17a-4 / CSRC 审计要求。

## 二、术语与模块边界

### 2.1 关键术语

| 术语 | 含义 | 对应表 |
|---|---|---|
| 用户（Client） | 系统注册用户，KYC/AML 的主体 | `t_user`（用户体系，外部模块） |
| 交易账户（Trading Account） | 真正承载交易行为与资金的账户实体 | `t_trading_account`（账户体系） |
| 账户组（Account Group） | 后台运营核心管理单元，决定一组账户共享的杠杆 / 风控 / 手续费 / 可交易品种 | `t_account_group`（账户体系） |
| 品种（Symbol） | 可交易标的，如 EURUSD / XAUUSD / AAPL | `t_symbol`（品种体系） |
| 品种组（Symbol Group） | 同类品种逻辑分组，如 FOREX_MAJOR / CRYPTO_MAJOR | `t_symbol_group`（品种体系） |
| 监管区（Jurisdiction） | 一个法定监管辖区下的运营实体，如 ESMA / FCA / ASIC / CYSEC / FSCA / SFC / FSA_JP / CFTC / SEC / CSRC | **`t_jurisdiction`（本模块）** |
| 杠杆模板（Leverage Template） | 预定义的"杠杆参数包"，挂在账户组的 `leverage_template_id` 上批量生效 | **`t_leverage_template`（本模块）** |
| 动态分层（Leverage Tier） | 根据持仓名义价值 / 手数 / 账户净值，对杠杆做阶梯式收紧 | **`t_leverage_tier`（本模块）** |
| 解析上下文（ResolvedLeverage） | 一次性返回给撮合 / 风控的"最终生效杠杆 + 解析来源" | 值对象（缓存中） |

### 2.2 杠杆模块的职责边界

| 模块 | 职责 | 不做 |
|---|---|---|
| 杠杆模块（本文档） | 杠杆 / 保证金率 / 动态分层 / 监管上限 / 客户与账户级覆盖 / 假期事件 / 担保品折算 / 集中度 / 变更审计 | 不重复定义账户、账户组、品种、品种组、隔夜利息、手续费 |
| 账户体系（`t_account_group` / `t_trading_account`） | 账户的开户 / 杠杆当前值 / 状态 / 资金快照 / 强平等级 / 负余额保护 | 不计算"应当生效杠杆"，由本模块解析 |
| 品种体系（`t_symbol` / `t_symbol_group` / `t_symbol_swap_config`） | 品种主数据、合约规格、隔夜利息计算公式 | 不存杠杆值 |
| 资金 / 撮合引擎 | 实时净值、保证金占用、强平触发 | 不存配置 |

### 2.3 对 `t_account_group` 的依赖字段

杠杆模块运行依赖账户体系的 `t_account_group` 已经提供（或后续补齐）以下字段——本文档**不修改**账户体系，仅在此处声明依赖：

## 四、八级杠杆解析链

### 4.1 解析顺序（高优先级 → 低优先级）

| 优先级 | 层级 | 数据来源 | 触发条件 |
|---|---|---|---|
| **L8（绝对上限）** | 监管法定上限 | `t_jurisdiction_leverage_limit` | 必查，永远生效 |
| **L7（运行时叠加）** | 假期 / 事件 | `t_margin_event_config` | 命中时间窗口时叠加 |
| **L6（运行时叠加）** | 动态分层 | `t_leverage_tier` | 持仓名义价值 / 手数突破阈值时叠加 |
| **L5（客户主动选择）** | 账户级覆盖 | `t_account_leverage_override` | 该账户在该品种 / 品种组有显式记录 |
| **L4（运营或合规调整）** | 客户级覆盖 | `t_client_leverage_override` | 该客户在该品种 / 品种组有显式记录 |
| **L3（账户组运营管理）** | 账户组×品种覆盖 | `t_account_group_leverage_rule` | **核心层**：同品种不同账户组不同杠杆 |
| **L2（品种维度默认）** | 品种维度默认 | `t_symbol_leverage_default` | 按 监管区×品种×账户类型 维度查询 |
| **L1（品种组维度默认）** | 品种组维度默认 | `t_symbol_group_leverage_default` | 按 监管区×品种组×账户类型 维度查询 |
| **L0（兜底）** | 账户组默认 / 模板基线 | `t_account_group.default_leverage` 或 `t_leverage_template.base_leverage` | 上述均未命中 |

### 4.2 解析伪代码

```text
function resolveLeverage(account_id, symbol_id, now):
    ctx     ← loadContext(account_id, symbol_id)   // 一次 JOIN 拿到账户/账户组/品种/品种组/监管区
    cap     ← lookupRegulatoryCap(ctx.jurisdiction_id, ctx.symbol_group_id, ctx.client_category)

    // L5 → L1：自下而上找第一个命中的覆盖 / 默认
    resolved ← findAccountOverride(ctx.account_id, ctx.symbol_id, ctx.symbol_group_id, now)
           ?? findClientOverride(ctx.client_id, ctx.symbol_id, ctx.symbol_group_id, now)
           ?? findAccountGroupRule(ctx.account_group_id, ctx.symbol_id, ctx.symbol_group_id, now)
           ?? findSymbolDefault(ctx.symbol_id, ctx.jurisdiction_id, ctx.account_group_code, now)
           ?? findSymbolGroupDefault(ctx.symbol_group_id, ctx.jurisdiction_id, ctx.account_group_code, now)

    // L0：兜底
    resolved ← resolved ?? ctx.account_group.default_leverage
                       ?? template.base_leverage
                       ?? SYSTEM_FALLBACK  // 100:1 或者拒绝交易

    // L6：动态分层
    tier ← matchLeverageTier(ctx.account_group.leverage_template_id, ctx.symbol_group_id, ctx.notional_value)
    if tier exists:
        resolved ← MIN(resolved, tier.leverage_value)

    // L7：假期 / 事件
    event ← matchMarginEvent(ctx.symbol_id, ctx.symbol_group_id, ctx.account_group_id, now)
    if event exists:
        if event.absolute_leverage exists:
            resolved ← event.absolute_leverage
        else:
            resolved ← resolved * event.leverage_factor

    // L8：监管上限封顶
    if cap exists:
        resolved ← MIN(resolved, cap.max_leverage)

    return resolved
```

## 六、完整数据库设计

### 6.1 通用约定

- **存储引擎**：InnoDB
- **字符集**：`utf8mb4 / utf8mb4_unicode_ci`
- **主键**：`id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '自增主键'`
- **时间戳**：`DATETIME(3)` 含毫秒，统一 `DEFAULT CURRENT_TIMESTAMP(3)` + `ON UPDATE`
- **软删除**：`deleted_at BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '删除时间戳: 0-正常, 非0-删除时的Unix毫秒时间戳'`
- **状态字段**：`status TINYINT UNSIGNED NOT NULL DEFAULT 1 COMMENT '状态: 0-停用 1-启用'`
- **枚举字段**：使用 `VARCHAR(N) + COMMENT` 描述取值，**不使用 ENUM 类型**（项目惯例：扩展性优先）
- **金额 / 杠杆 / 比率类型**：
  - 杠杆：`DECIMAL(10,2)`（如 500.00 表示 1:500）
  - 比率：`DECIMAL(10,6)`（如 0.033333 表示 3.33%）
  - 金额阈值：`DECIMAL(20,2)`
  - 货币金额：`DECIMAL(18,4)`
- **唯一键命名**：`uk_<语义>`，复合 `deleted_at` 以支持软删除后再插同 code
- **普通索引命名**：`idx_<字段组合>`
- **外键命名**：`fk_<本表缩写>_<引用表缩写>`
- **审计字段**：`created_by` / `updated_by` 在审计敏感表中保留

### 6.2 监管层

#### 表 1：`t_jurisdiction` — 监管管辖区

```sql
-- ----------------------------------------------------------------
-- 监管管辖区
--
-- 全球各监管辖区的主数据, 含杠杆基线和负余额保护要求
-- 每个监管区对应一个运营实体, 决定该区客户适用哪套监管规则
--
-- 典型监管区:
--   ESMA / FCA / ASIC / CYSEC / FSCA / SFC / FSA_JP / CFTC / SEC / CSRC
-- ----------------------------------------------------------------
CREATE TABLE `t_jurisdiction` (
    `id`                            BIGINT UNSIGNED     NOT NULL AUTO_INCREMENT COMMENT '自增主键',
    `jurisdiction_code`             VARCHAR(32)         NOT NULL COMMENT '监管区编码: ESMA/FCA/ASIC/CYSEC/FSCA/SFC/FSA_JP/CFTC/SEC/CSRC',
    `jurisdiction_name`             VARCHAR(128)        NOT NULL COMMENT '监管区中文名称',
    `country_code`                  CHAR(2)             DEFAULT NULL COMMENT 'ISO 3166-1 alpha-2 国家代码',
    `region`                        VARCHAR(32)         DEFAULT NULL COMMENT '所属区域: EU/EMEA/APAC/AMERICAS',
    `legal_entity`                  VARCHAR(128)        DEFAULT NULL COMMENT '本公司在该监管区运营的法律实体名称',

    -- ==================== 监管基线 ====================
    `max_retail_leverage`           DECIMAL(10,2)       NOT NULL DEFAULT 100.00 COMMENT '散户最大杠杆基线',
    `max_professional_leverage`     DECIMAL(10,2)       NOT NULL DEFAULT 500.00 COMMENT '专业客户最大杠杆基线',
    `max_institutional_leverage`    DECIMAL(10,2)       NOT NULL DEFAULT 1000.00 COMMENT '机构客户最大杠杆基线',
    `leverage_reduction_rule`       VARCHAR(32)         NOT NULL DEFAULT 'NOTIONAL_TIERED'
        COMMENT '杠杆递减规则: FIXED-固定 / VOLUME_TIERED-按手数 / NOTIONAL_TIERED-按名义价值 / SPAN-按SPAN / PORTFOLIO_MARGIN-组合保证金',

    -- ==================== 生效区间 ====================
    `effective_date`                DATE                DEFAULT NULL COMMENT '生效日期',
    `expiry_date`                   DATE                DEFAULT NULL COMMENT '失效日期, NULL=长期',

    `description`                   VARCHAR(512)        DEFAULT NULL COMMENT '监管规则描述',

    -- ==================== 状态与审计 ====================
    `status`                        TINYINT UNSIGNED    NOT NULL DEFAULT 1 COMMENT '状态: 0-停用 1-启用',
    `created_at`                    DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    `updated_at`                    DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    `deleted_at`                    BIGINT UNSIGNED     NOT NULL DEFAULT 0
        COMMENT '删除时间戳: 0-正常, 非0-删除时的Unix毫秒时间戳',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_jurisdiction_code` (`jurisdiction_code`, `deleted_at`),
```

```sql
-- ----------------------------------------------------------------
```

```sql
    `margin_currency`               VARCHAR(8)          NOT NULL DEFAULT 'USD' COMMENT '保证金计算币种',

    `calc_method`                   VARCHAR(16)         NOT NULL DEFAULT 'TIERED'
        COMMENT '阶梯算法: TIERED-阶梯累加 / FLAT-按当前档统一',
    `priority`                      INT                 NOT NULL DEFAULT 0 COMMENT '同阈值冲突优先级',

    -- ==================== 生效区间与状态 ====================

    `effective_to`                  DATETIME(3)         DEFAULT NULL COMMENT '生效截止',

    `created_at`                    DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    `updated_at`                    DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    `deleted_at`                    BIGINT UNSIGNED     NOT NULL DEFAULT 0 COMMENT '删除时间戳',

    KEY `idx_template_status` (`template_id`, `status`),

    KEY `idx_effective` (`effective_from`, `effective_to`),

    CONSTRAINT `chk_lt_tier_level` CHECK (`tier_level` > 0),
    CONSTRAINT `chk_lt_threshold` CHECK (
        `threshold_from` >= 0
        AND (`threshold_to` IS NULL OR `threshold_to` > `threshold_from`)

    CONSTRAINT `chk_lt_leverage_positive` CHECK (`leverage_value` > 0)
```

```sql
        SUM(pos.`market_value` * chr.`haircut_rate`) AS pledged_value,
        MIN(chr.`maintenance_collateral_ratio`)     AS maintenance_ratio

        AND chr.`deleted_at` = 0
        AND chr.`is_long_eligible` = 1
    WHERE pos.`account_id` = :account_id
)
SELECT
    cv.`pledged_value`,
    cv.`pledged_value` * 100 / cv.`maintenance_ratio` AS available_financing_amount
FROM collateral_value cv;
```

## 八、合规校验与审计

### 8.1 双层校验

```
写入路径:
  [应用层] LeverageConfigService.save()
       │
       ├─► [Step 1] RegulatoryComplianceValidator.validate()
       │       └─► 查询 t_jurisdiction_leverage_limit, 对比 max_leverage / default_leverage
       │           不满足直接抛出 RegulatoryViolationException
       │
       ├─► [Step 2] DB CHECK 约束 (兜底)
       │       └─► leverage > 0, threshold_from < threshold_to, effective_to > effective_from 等
       │
       └─► [Step 3] @Transactional 内:
              ├─► INSERT/UPDATE 业务表
              ├─► INSERT t_leverage_change_log (AOP 自动)
              └─► INSERT outbox 表 (CDC 推送缓存失效)
```

### 8.2 审计写入：应用层 AOP，不再使用数据库触发器

```java
@Aspect
@Component
public class LeverageAuditAspect {

    @Around("@annotation(LeverageAudit) && execution(* save*(..))")
    public Object aroundSave(ProceedingJoinPoint pjp) throws Throwable {
        // 1. 抓取调用上下文: operator / IP / trace / reason
        ChangeContext ctx = ChangeContextHolder.get();

        // 2. 调用前快照
        Object before = readBefore(pjp);

        // 3. 执行业务方法
        Object result = pjp.proceed();

        // 4. 调用后快照 → 计算 diff
        Object after = readAfter(pjp);
        ChangeDiff diff = ChangeDiffCalculator.diff(before, after);

        // 5. 写入 t_leverage_change_log + outbox
        leverageChangeLogService.record(ctx, diff);
        outboxService.publish(new LeverageConfigChangedEvent(diff));

        return result;
    }
}
```

> **为什么不用数据库触发器？**
> 1. 触发器无法拿到应用层上下文（操作人、IP、Trace ID）。
> 2. 14 张配置表的触发器维护成本极高，DDL 变更需要同步改触发器。
> 3. 触发器不利于回放、降级、单元测试。
> 4. 应用层 AOP + Outbox 既能拿到完整上下文，又能保证与缓存失效的最终一致性。

## 十、缓存与一致性方案

### 10.1 二级缓存架构

```
┌─────────────────┐  miss   ┌──────────────┐  miss   ┌──────────┐
│ JVM (Caffeine)  │────────►│ Redis Cluster│────────►│  MySQL  │
│  10万 entry     │         │ 100万 entry  │         │  主库    │
│  TTL 30s        │         │ 无 TTL       │         └──────────┘
└─────────────────┘         └──────────────┘
        ▲                          ▲
        │                          │
        └─────── invalidate ───────┘
                       ▲
                       │
              Kafka leverage.config.changed (主题)
                       ▲
                       │ consume
                       │
                Debezium CDC (抓取 outbox 表新增记录)
                       ▲
                       │ Outbox 模式
                       │
              LeverageConfigService 写业务表 + outbox 表 (同一事务)
```

### 10.2 缓存 Key 设计

```
leverage:resolved:{account_id}:{symbol_id}                          ResolvedLeverage 完整对象
leverage:template:{template_id}                                     模板配置
leverage:account_group:{account_group_id}                           账户组主数据
leverage:regulatory:{jurisdiction_id}:{symbol_group_id}:{cat}       监管上限
leverage:tier:{template_id}:{threshold_type}                        动态分层规则列表 (JSON)
```

### 10.3 失效粒度

| 变更对象 | 失效范围 |
|---|---|
| `t_jurisdiction_leverage_limit` | 该监管区下所有账户 |
| `t_account_group` / `t_account_group_leverage_rule` | 该账户组下所有账户 |
| `t_symbol_group_leverage_default` | 该品种组相关账户 |
| `t_symbol_leverage_default` | 该品种相关账户 |
| `t_client_leverage_override` | 该客户所有账户 |
| `t_account_leverage_override` | 该账户 |
| `t_leverage_template` / `t_leverage_tier` | 引用该模板的所有账户组下所有账户 |
| `t_margin_event_config` | 该品种 / 品种组 / 账户组相关账户 |
| 持仓变化（账户内部事件） | 该账户 × 该品种（强制重算动态分层） |

### 10.4 一致性保证

- **写入时**：业务表 + outbox 表在同一本地事务，保证原子性。
- **CDC 推送**：Debezium 监听 outbox，毫秒级延迟，至少一次投递。
- **消费者幂等**：每条事件带 `change_id`，消费者去重。
- **降级**：Redis 不可用时直接回源 MySQL，可用性优先于性能。

### 10.5 持仓变化触发动态分层重算

- 撮合引擎 / 风控引擎每次开仓 / 平仓，发布 `PositionUpdated(accountId, symbolId, newNotional)`。
- `LeverageCacheManager` 订阅该事件，重新解析 + 写回 Redis。
- 保证"持仓刚突破 50 万美元，杠杆从 500:1 立即降到 200:1"，下一笔下单时风控立即生效。

| 业务模式 | 关键表 / 字段 | calc_mode | 杠杆表达 | 典型示例 |
|---|---|---|---|---|
| **外汇 / 贵金属 / 能源 / 股指 CFD** | `t_leverage_tier` 按名义价值分层 | FIXED_RATE | 显式 leverage，margin = notional / leverage | EURUSD 离岸 500:1 |
| **加密货币** | `t_leverage_tier` 更激进分层 | FIXED_RATE | 显式 leverage | BTCUSD 离岸 100:1，ESMA 2:1 |

| **期货 CFD** | `t_leverage_tier` + 名义价值分层 | FIXED_RATE | 显式 leverage | WTI CFD 离岸 100:1 |
| **期权** | `t_margin_requirement_rule.calc_mode = TICK_VOLATILITY` 或 `SPAN` | TICK_VOLATILITY / SPAN | 基于希腊字母与波动率 | TSLA 200 Call 期权 |

### 11.1 美股 Reg T 示例

```
配置:
- t_symbol_leverage_default (AAPL, jurisdiction=SEC, account_group_code_pattern='LIVE_US_%')
    intraday_leverage  = 4.00
    overnight_leverage = 2.00

运行时:
- 09:30-15:55 ET → 取 intraday_leverage
- 15:55-16:00 ET → 系统自动平仓超出 overnight_leverage 的仓位
- 16:00 ET 之后 → 取 overnight_leverage

PDT 触发:
- 风控引擎检测到 5 个交易日内 4 次日内交易, 写入 t_account_leverage_override
  (override_type=PDT_RESTRICTION, trigger_event=PDT_TRIGGERED), 强制将 effective_leverage 全程降到 2:1
```

### 11.2 A 股融资融券示例

```
账户净资产 = 现金 + 持仓担保品价值(按 haircut_rate 折算)
          = 100,000 现金 + 200,000 持仓 × 0.70 (haircut)
          = 100,000 + 140,000 = 240,000

可融资额 = 担保品价值 ÷ 维持担保比例 × 100 − 已融资额
         = 240,000 ÷ 130 × 100 = 184,615

可买入金额 = 现金 + 可融资额 = 100,000 + 184,615 = 284,615
最大杠杆等效 ≈ 2.85:1
```

### 11.3 期货 SPAN 示例

```
ETL 任务每日收盘后下载 CME SPAN 文件, 导入 t_margin_requirement_rule:
    symbol_id          = ES2509 (E-mini S&P 500)
    calc_mode          = 'SPAN'
    span_margin_value  = 13,500.00  (USD per contract)
    span_file_version  = 'CME_2026Q2_v3'

下单时:
    initial_margin = volume × span_margin_value
                   = 5 × 13,500 = 67,500 USD
```

### C.2 `LeverageResolver` Protobuf 契约

```protobuf
service LeverageResolver {
  rpc Resolve(ResolveLeverageRequest) returns (ResolveLeverageResponse);
  rpc PullSnapshot(PullRequest) returns (PullSnapshotResponse);
}

message ResolveLeverageRequest {
  int64 account_id = 1;
  int64 symbol_id = 2;
  string entity_id = 3;
  decimal notional_value = 4;
  decimal equity = 5;
  string config_version = 6;
}
message ResolveLeverageResponse {
  LeverageBlock block = 1;     // default/max/min/step/allowed/margin_rate/...
  string resolved_from = 2;    // L0..L8
  string config_version = 3;
  string reject_reason = 4;    // 引用补充卷 §三 LEV_*
}
```

撮合引擎 PreTrade Step 6 调用此 RPC，**禁止**引擎内嵌杠杆表直算（旧稿 `t_leverage_template_tier` 表名错误且未走八级链，已作废）。

### C.3 TIERED vs FLAT 数值例题

**例题**：账户净值 55,000 USD，品种 EURUSD，TIERED 阶梯：

| Tier | 名义区间（USD） | margin_rate | 名义分段 | 保证金 |
|------|----------------|-------------|----------|--------|
| 1 | 0 – 20,000 | 1% (1:100) | 20,000 | 200 |
| 2 | 20,001 – 50,000 | 2% (1:50) | 30,000 | 600 |
| 3 | 50,001 – 100,000 | 3% (1:33) | 5,000 | 150 |
| **合计** | 55,000 | — | 55,000 | **950** |

对比 FLAT 1:50（2%）：55,000 × 2% = 1,100。TIERED 节省 150 USD，体现"小仓位高杠杆、大仓位递减"的监管意图。

跨档加仓（从 45k → 55k）：增量 10k 全部落在 Tier 3（3%），增量保证金 = 300，非按平均率。

### C.4 与 `t_account_group_symbol_config` 优先级矩阵

账户组级品种配置 `t_account_group_symbol_config.max_leverage_override` 可对特定品种做账户组级封顶。优先级裁定：

```
生效杠杆 = min(
  LeverageResolver 八级解析结果,         // L0..L8
  t_account_group_symbol_config.max_leverage_override,  // 账户组×品种封顶（若存在）
  t_account_leverage_override.leverage   // 账户级覆盖（若存在）
)
生效 margin_rate = 对应生效杠杆的 margin_rate
```

- override 仅做 **cap**（封顶），不替换完整 LeverageBlock。
- 若 override 杠杆 < 解析结果，margin_rate 按 override 杠杆重算（更保守）。
- 审计：`resolved_from` 记录"OVERRIDE_BY_ACCOUNT_GROUP_SYMBOL"或"OVERRIDE_BY_ACCOUNT"。

### C.5 SPAN ETL 时序与失败保留

- SPAN 参数文件由清算所每日 T+1 提供，本域 ETL 在 06:00 UTC 拉取。
- ETL 失败：保留昨日 `span_margin_value`，告警；连续 3 日失败升级 P1。
- SPAN 计算结果写入 `t_margin_requirement_rule`，`calc_mode=SPAN`，`status=PENDING_REVIEW` → `PMS_RISK_OFFICER` 审批 → `ACTIVE`。
- 回滚：审批前可丢弃；审批后 `SUPERSEDED` 旧版本。

### C.6 增量失效（Bloom Filter + 版本戳）

旧稿 §10.3 失效粒度过大（监管变更失效全区账户）。v2.1 改进：

- 每个配置表含 `config_version` 版本戳。
- 缓存 key 含 `config_version`，版本变更自动 miss。
- Bloom Filter 记录"受影响 account_id 集合"，仅失效命中账户，避免全量失效。
- 配置广播经补充卷 §八灰度（SHADOW → CANARY → ACTIVE）。
