# 营销与推荐系统设计 — 终版 v2.0

> **版本**：v2.0 Final｜2026-07-22
> **定位**：经纪平台的**营销活动、客户分群推荐与 Desk 经营权威**——CRM 分群/归属、B2C 活动发奖（含交易竞赛）、个性化推荐、投诉工单。**不含**资讯 CMS 与 IM 广场（见《内容与社区系统设计》[15]），**不拥有**跟单执行与排行榜计算（见《跟单与排行系统设计》[13]），**不改**代理归因（见《代理系统设计》[07]）。
> **对标**：券商中台 CRM、零售 Growth/Loyalty、推荐投放（能力级，不照搬产品形态）。
> **运行时分层**：CRM Desk（分群/归属/任务/线索）· Segment & Tag（标签与人群圈选）· Recommend（召回/排序/频控）· Campaign Orchestrator（活动/资格/发奖/预算）· Ticket（工单）· Analytics（画像快照/看板）。触达走通知域；发奖走资金引擎契约。方向设计不锁定实现语言。
> **域地图**：见 [`[总览]经纪平台域地图.md`](../[总览]经纪平台域地图.md)。

## 一、系统定位与边界

### 1.1 系统定位

1. **CRM Desk**：分群、客户经理归属、任务跟进、生命周期阶段、经营标签、销售线索（存量客户跟进）
2. **Growth**：入金/交易赠金、优惠券、交易竞赛、邀请好友、Promo Code
3. **推荐**：基于分群/标签/行为的活动与内容卡片推荐意图（内容实体仍在内容域）
4. **画像**：客户标签体系与画像快照（衍生特征，非身份/资产权威）
5. **工单/投诉**：受理、路由、结案

以下概念**不属于**本系统：

| 概念 | 归属 |
|------|------|
| 资讯 CMS、IM 广场/群聊 | 内容与社区 [15] |
| 跟单执行、排行榜计算（含竞赛型榜单计分/强平/防刷） | 跟单与排行系统（社交交易域）[13] |
| 客户归因权威（`t_partner_attribution`）、IB 佣金计提 | 代理 [07] |
| 通知送达 | 通知 [03] |
| 余额直写、Order/Deal/Position | 资金 / 交易引擎 |
| KYC 审核结论、身份/证件权威 | 准入 / 用户 [04] |
| 账户资产、持仓、保证金明细 | 账户 [05] / 资金 [09] |

### 1.2 与代理的关系

| 事项 | 代理 [07] | 本域 [14] |
|------|----------|-----------|
| 客户归因（`t_partner_attribution` / `t_partner_attribution_current`） | **唯一权威** | 只读投影；若需改派，提交工单升级代理域审核 |
| IB 佣金 | **唯一计提** | 不参与 |
| B2C 赠金活动（含交易竞赛） | 不拥有 | **编排权威** |
| 内部客户经理 KPI | 不属于代理 | 本域内部激励（若做） |

禁止：改写 `t_partner_attribution`；Partner 代发 B2C Credit；本域自建 `referrer_id`/UTM 之类的归因字段（获客渠道判断直接读用户域 `t_user.register_channel`，值域 `WEB/IOS/ANDROID/API/ADMIN/OAUTH`；归因判定读代理域投影，不重复造字段）。

## 二、十二条不可破坏决策

1. **`entity_id` 贯穿**；禁止跨实体活动/分群串数据。
2. **不改交易真相**：禁止写 Order/Deal/Position/ledger。
3. **发奖只经资金契约**：`FundService.AdjustCredit` + `reward_id` 幂等键。
4. **不改代理归因**：`t_partner_attribution` 唯一真相，本域只读投影。
5. **营销触达遵守 opt-in**（通知域 `MARKETING` 类目）。
6. **推荐不替代内容权威**：推荐只产出「推哪篇/哪个活动/哪个 Provider」；正文在内容域。
7. **PII 最小必要**：本域不复制用户域的手机号/证件等身份字段，标签/画像/线索均以 `user_uid` 关联，不新建 PII 副本；Desk 导出审计。
8. **四眼/SoD**：大额发奖、归属批量转移。
9. **事件经 Outbox**。
10. **活动预算独立**：≠ Partner CPA 营销池。
11. **工单不改余额**：资金争议协同资金域冻结。
12. **不做**：IM、CMS、跟单执行与排行榜计算、第二套通知通道、自建归因字段。

## 三、跨域契约

### 3.1 契约引用声明

- **错误码**：遵循《平台跨域契约补充卷》§三，使用 `MKT_*` 前缀（完整列表见 §17）。
- **事件**：遵循补充卷 §四；本域产出/消费事件的登记状态见 §15。
- **角色**：客户经理属业务角色（补充卷 §5.3 `BIZ_SALES_REP`），运营管理引用补充卷 §5.2 `PMS_MARKETING_OPERATOR`（均已核实登记，非 `PMS_MARKETING_MANAGER`）。
- **ID**：遵循补充卷 §六，`campaign_id`/`reward_id`/`segment_id` 由本域生成。
- **容灾/容量/监控**：遵循补充卷 §十/§十一/§十二。

### 3.2 领域边界

| 系统 | 使用方式 |
|------|----------|
| 用户 / 准入 / 账户 | 只读身份、KYC 结论、账户归属、`register_channel`；经营标签/线索本域治理，不复制 PII |
| 资金 | 发奖（`AdjustCredit`）；消费 `deposit.completed.v1` 做资格判定 |
| 交易 | 只读 `deal.*`/`position.*` 做竞赛特征、推荐特征、画像快照原料 |
| 代理 | 只读 `t_partner_attribution` 归因投影 |
| 内容与社区 [15] | 推荐可引用 `content_id`；不写文章 |
| 跟单与排行系统（社交交易域）[13] | 交易竞赛场景下，本域编排「创建竞赛型榜单 + 报名 + 消费结算事件」；不拥有跟单订阅与排行计算，见 §8.3 |
| 通知 | 战役投递意图 |
| 权限 | RBAC/SoD，`PMS_MARKETING_OPERATOR` 审批活动/推荐位 |

### 3.3 发奖流（总览，全链路见 §9）

```text
Campaign → 资格校验 → reward_id 幂等 → FundService.AdjustCredit → campaign.reward.granted.v1
```

## 四、CRM 与客户画像

### 4.1 生命周期与基础模型

- Lifecycle：`REGISTERED / KYC_DONE / FUNDED / ACTIVE / DORMANT / CHURNED`（可配）。
- `Owner`（客户经理归属）、`Segment`（人群）、`Task`（跟进任务）、`Note`（备注）。
- 经营标签 ≠ 用户域安全标签；**禁止**用经营标签绕过交易封禁（交易权限的唯一开关是账户域 `trade_enabled` 等字段，本域标签只影响触达/推荐，不影响交易能力）。

### 4.2 标签体系（Tag System）

Segment 圈选与画像快照的基础设施；标签本身是本域治理的**衍生数据**，不是身份/资产权威（标签值可以来源于用户域/交易域的只读投影，但存储、生效期、置信度管理权威在本域）。

```text
t_crm_tag_definition：标签定义（编码、类目、计算方式、刷新周期）
  → 计算任务（实时事件驱动 / T+1 批处理 / 模型评分）
  → t_crm_customer_tag：客户标签宽表（带生效期、置信度、来源）
  → 供 Segment 规则引擎（§六）与推荐召回（§七）消费
```

- `tag_category`：`asset`（资产规模档位，只读投影自资金域快照）/ `behavior`（行为）/ `value`（CLV/贡献度）/ `risk`（风险等级，只读投影自准入域测评结论）/ `lifecycle`（生命周期）。
- `compute_type`：`1-实时`（事件驱动，如消费 `deposit.completed.v1` 后打「已入金」标签）/ `2-T+1` / `3-T+7` / `4-手动`（运营手动打标，需审计）。
- 标签**值本身**（如 `asset_scale_level`）可以引用其他域的原始数值，但**本域不重新存储该原始数值的权威版本**，只存计算后的档位/得分；如需追溯原值，回查对应域。

### 4.3 客户画像快照

`t_crm_customer_profile_snapshot`：按日快照的衍生指标集合（CLV 评分、流失概率、活跃度等级、生命周期阶段等），供 Desk 360 视图、推荐排序、活动资格引擎消费。

- **边界**：本表只存"计算结果"，不存资产明细/持仓/交易流水本身——那些字段永远只读来自账户域[05]/资金域[09]/交易域[10]的投影或实时查询，快照表里只落"经过打分的衍生指标"。
- 更新方式：T+1 批处理为主（`clv_score`/`churn_prob` 等模型评分成本较高），部分字段（`days_since_login`）可实时增量。

## 五、客户经理分配与销售线索

### 5.1 客户经理分配算法

```text
分配策略（可配）：
  - ROUND_ROBIN：轮询
  - LOAD_BALANCED：按当前负载（活跃客户数）
  - JURISDICTION：按辖区
  - LANGUAGE：按语言
  - SKILL：按客户等级（RETAIL/PRO/ECP）

分配时机：
  - 注册时：按策略分配
  - 转移：客户经理离职 → 其客户按策略重新分配
  - 手动：运营可手动指派
```

- 客户经理归属权威表是本域自己的 `t_crm_owner_assignment`（`user_uid` + `sales_rep_id` + 生效期），**不是**用户域字段——用户域/账户域全文核实均无 `sales_rep_id` 或客户经理归属列（账户域的 `owner_user_id` 是开户人/账户所有者，与「客户经理」是两个不同概念，不能混用）。
- 离职联动：[`[00]后台权限系统/权限系统设计-终版.md:1761`](../[00]后台权限系统/权限系统设计-终版.md) 显示 PMS 域自称产出 `pms.user.left.v1`，但《平台跨域契约补充卷》§四事件表目前**没有登记任何 `pms.*` 事件**，即该事件是「域内已声明、尚未进入平台级注册表」状态。本域按可用事件消费该事件做自动重新分配，**同时**在 Desk 提供人工/定时批处理兜底（按 `sales_rep_id` 关联的 PMS 账号状态定期扫描），避免事件迟迟未登记或延迟导致离职联动失效。

### 5.2 销售线索（存量客户跟进）

**边界拍板**：线索管理限定为「已注册客户」的存量转化/流失挽回跟进，**不做**未开户潜客的获客渠道追踪（那属于获客渠道归因，权威在代理域 `t_partner_attribution`，本域不重复建设，也不新增 `referrer_id`/UTM 类字段）。

- `lead_type`：`1-存量升级`（低活跃转高活跃）/ `2-流失挽回`（`DORMANT`/`CHURNED` 客户）。
- 分配、跟进节奏、转化标记复用 §5.1 的分配策略与 `t_crm_owner_assignment`。

## 六、Segment 规则引擎

```json
{
  "segment_id": "int64",
  "name": "高净值未入金",
  "rule_dsl": "tag:asset_scale_level >= 4 AND tag:lifecycle_stage = KYC_DONE AND tag:total_deposit_count = 0",
  "refresh_mode": "REALTIME|BATCH|HYBRID",
  "refresh_interval": "5m|1h|1d",
  "entity_id": "string"
}
```

- 规则表达式基于 §4.2 标签体系组合（`tag:xxx`），而不是直接拼裸 SQL 字段，好处是圈选逻辑与底层数据源解耦，标签计算口径变化不需要改所有 Segment。
- **实时**：事件驱动（`user.approved.v1`、`deposit.completed.v1`）触发标签重算后增量更新成员。
- **批处理**：日批全量重算（适合大 Segment）。
- **混合**：实时增量 + 日批全量校准。
- 成员变更审计：`t_crm_segment_member_change`。

## 七、推荐系统详设

### 7.1 场景与槻位

| 场景编码 | 位置 | 候选类型 |
|------|------|----------|
| `HOME_BANNER` | 首页 banner | 活动、内容、Provider |
| `DEPOSIT_PROMO` | 入金页 | 充值活动 |
| `SYMBOL_HOT` | 热门品种 | 品种卡片 |
| `PROVIDER_REC` | 推荐信号源 | 跟单 Provider 深链（数据来自 [13]） |
| `CONTENT_REC` | 推荐资讯 | 内容卡片 |

`t_recommend_scene` 把每个场景的召回/排序/过滤/缓存策略配置化，而不是硬编码：

```text
t_recommend_scene (
  scene_id, entity_id, scene_code UNIQUE, scene_position,
  recall_strategy_json COMMENT '多路召回配置: 活动/内容/Provider/冷启动 各路权重',
  rank_model COMMENT '规则打分 或 CTR模型版本号(P2)',
  filter_rules_json COMMENT '合规/风险过滤规则(如竞赛榜单不得推给非参赛账户)',
  top_k, cache_ttl, is_realtime, status, ...
)
```

### 7.2 召回→排序→重排

```text
召回（多路）：
  - 活动召回（符合资格的活动，含交易竞赛报名入口）
  - 内容召回（基于兴趣标签，§4.2）
  - Provider 召回（基于跟单偏好，只读 [13] 排行/业绩投影）
  - 冷启动召回（新用户默认）
→ 合并去重
→ 排序（规则/模型）：
  - 规则：优先级 + 权重
  - 模型：CTR 预估（P2）
→ 重排：
  - 频控（每用户每场景每日 ≤ N）
  - 去重（24h 内不重复曝光）
  - 多样性（同类型 ≤ 2）
  - A/B 分流（见 §7.5）
```

### 7.3 冷启动

- 新用户：默认场景（平台精选活动 + 热门品种 + 教育内容）。
- 探索/利用：ε-greedy（ε=0.1 探索新内容）。
- 兴趣标签：基于浏览/点击埋点构建（§4.2 标签体系的行为类标签来源）。

### 7.4 频控与去重

- `t_recommend_exposure`（`user_uid` × `scene_code` × day × `content_id`）。
- 频控：营销 PUSH 每日 ≤ 1，banner 每日 ≤ 5。
- 去重：24h 内同 `content_id` 不重复曝光。

### 7.5 A/B 实验

```text
t_ab_experiment (
  experiment_id, entity_id, name,
  hypothesis COMMENT '假设描述(如: 改版banner文案提升点击率)',
  variants_json COMMENT '分组配置: [{"group":"A","weight":50},{"group":"B","weight":50}]',
  primary_metric COMMENT '主指标: ctr/conversion_rate/deposit_amount',
  guardrail_metrics_json COMMENT '护栏指标(不能显著变差): 投诉率/退出率',
  significance_threshold DEFAULT 0.05,
  start_at, end_at, status /*1-DRAFT 2-RUNNING 3-ENDED*/,
  winner_variant NULL COMMENT '实验结束后人工/自动判定的获胜组',
  ...
)
```

- 埋点 `recommend.impression.v1` / `recommend.click.v1` 携带 `experiment_id` + `variant`。
- 效果归因：CTR、转化率、ROI，结束后按 `primary_metric` + `significance_threshold` 判定显著性。

## 八、Growth 活动

### 8.1 活动类型

| `campaign_type` | 说明 |
|-----------------|------|
| `DEPOSIT_BONUS` / `TRADE_REBATE` / `COUPON` | 赠金与返利 |
| `CONTEST` | 交易竞赛（大赛），本域编排；排行计算/强平/防刷在 [13]，见 §8.3 |
| `REFERRAL` / `PROMO_CODE` | 邀请与优惠码（≠ IB 归因） |

状态：`DRAFT → APPROVED → RUNNING ⇄ PAUSED → ENDED`。
预算：`t_campaign_budget` 与 Partner CPA 池分表。

### 8.2 资格规则引擎

```json
{
  "campaign_id": "int64",
  "campaign_type": "DEPOSIT_BONUS|TRADE_REBATE|COUPON|CONTEST|REFERRAL|PROMO_CODE",
  "eligibility_rule": {
    "kyc_level": ">= 2",
    "client_category": "IN [RETAIL, PROFESSIONAL]",
    "min_deposit": 1000,
    "min_trade_lots": 0,
    "symbol_scope": "ALL|LIST",
    "time_window": "2026-07-01 to 2026-08-01",
    "entity_id": "string"
  },
  "reward_rule": {
    "type": "BONUS|REBATE|COUPON|CASH|CONTEST_PRIZE",
    "amount": 100,
    "currency": "USD",
    "cap_per_user": 500,
    "cap_total": 100000,
    "rank_from": null,
    "rank_to": null
  }
}
```

各 `campaign_type` 详设：

| 类型 | 资格 | 发奖时机 | 上限 | 叠加 |
|------|------|----------|------|------|
| `DEPOSIT_BONUS` | 入金 ≥ 门槛 | 入金到账后 | 按比例封顶 | 不叠加同类型 |
| `TRADE_REBATE` | 交易手数达标 | 日/周结算 | 按手数封顶 | 可叠加 |
| `COUPON` | 领取 | 凭码使用 | 一人一码 | — |
| `CONTEST` | 报名（须持有大赛账号，见 §8.3） | 竞赛结算冻结后按名次区间 | 按 `rank_from`/`rank_to` 分层 | 与 `PERFORMANCE` 常规排行榜隔离 |
| `REFERRAL` | 邀请有效 | 被邀请入金 | 按邀请数封顶 | — |
| `PROMO_CODE` | 输入码 | 首次入金 | 一人一次 | — |

### 8.3 交易竞赛（CONTEST）跨域协同——与 [13] §8.4 严格对齐

**边界**：活动名称/赛制/时间/奖池/文案/大赛账号规格/名次奖励映射权威在本域；报名绑定、计分、防刷、到点强制平仓、结果冻结权威在 [`[13]跟单与排行系统/社交交易系统设计.md`](../[13]跟单与排行系统/社交交易系统设计.md) §8.4；名次→奖励映射判定与实际发奖在本域+资金域。

```text
本域创建活动（campaign_type=CONTEST，配置大赛账号组/起始资金规格/名次奖励映射 t_campaign_reward.rank_from~rank_to）
  → 调 [13] POST /api/v1/copy/leaderboards（board_type=CONTEST, campaign_id=本活动ID）创建榜单壳子 — [13] 状态机进入 ENROLLING
  → 本域完成参赛资格判定（KYC 等级/地域/是否需实名）
  → 本域编排账户域 [05] 开设大赛账号：调 POST /api/v1/trading-accounts/demo
      （请求体字段为 group_id(Long)+leverage，不是 group_code；本域配置竞赛账户组时按人类可读的 group_code 选定，
       调用前解析为对应 group_id 再传给账户域接口）
  → 本域调 [13] POST /api/v1/copy/leaderboards/{id}/enroll，把 {account_id, baseline_equity} 写入 [13] 的 t_copy_contest_enrollment
  → 报名截止，[13] 切换 RUNNING（滚动计分，仅供参赛者查看进度）
  → run_end_at 到达，[13] 切换 SETTLING：对全部参赛大赛账号强制平仓 → 防刷复核 → FINALIZED
  → 本域消费 [13] 事件 copy.leaderboard.contest_finalized.v1（含最终名次 + 盈亏列表，域内扩展事件，待补充卷登记）
  → 本域按 t_campaign_reward 的 rank_from/rank_to 区间匹配名次，计算奖励金额
  → FundService.AdjustCredit(reason_code=CONTEST_PRIZE, reward_id="{campaign_id}:{account_id}")
  → 发 campaign.reward.granted.v1
```

- `t_campaign_reward` 增加 `rank_from`/`rank_to`（如 `1~1`/`2~10`/`11~100` 对应不同奖金），仅 `CONTEST` 类型使用；其余类型该字段为 `NULL`。
- 大赛账号生命周期（保留/转普通模拟户/按 `demo_expire_days` 失效）交回账户域处理，本域不新增账户生命周期机制。
- 与常见误区「竞赛榜 = 跟单排行」的边界见 §18。

## 九、发奖全链路与预算熔断

### 9.1 AdjustCredit 全链路

```text
活动触发（入金/交易/领码/竞赛结算）
  → 资格校验（eligibility_rule，竞赛场景为消费 [13] contest_finalized 事件）
  → 幂等检查（reward_id = {campaign_id}:{user_uid 或 account_id}:{event_id}）
  → 预算扣减（原子 UPDATE t_campaign_budget，乐观锁）
  → 调资金域 FundService.AdjustCredit(reason_code=CAMPAIGN_BONUS|CONTEST_PRIZE|..., reward_id, amount, request_id)
  → 成功：发 campaign.reward.granted.v1 → 通知域
  → 失败：预算回滚 + 死信 + 告警
  → 日终对账：Σ reward vs Σ AdjustCredit vs ledger
```

- `AdjustCreditRequest.reason_code` 是字符串字段（非强制枚举），补充卷 proto 注释登记的示例值为 `CAMPAIGN_BONUS/CHARGEBACK_REVERSAL`；本域新增 `CONTEST_PRIZE` 作为域内扩展取值，语义为竞赛奖金。
- `reward_id` 幂等：资金域以此去重。
- 失败重试：N 次（指数退避）入死信，人工补发。
- 预算熔断：预算消耗 > 90% 告警，> 100% 自动拒绝新发奖（`MKT_LIMIT_BUDGET_EXCEEDED`）。

### 9.2 预算熔断与对账

- `t_campaign_budget`：`total_budget`、`used_budget`、`reserved_budget`、`status`。
- 熔断：`used + reserved > total × 0.95` → 告警；`> total` → 拒绝（`MKT_LIMIT_BUDGET_EXCEEDED`）。
- 日终对账：`Σ reward_amount` vs `Σ ledger(CAMPAIGN_BONUS/CONTEST_PRIZE)` vs `budget.used`，不一致告警。

## 十一、工单

```text
OPEN → IN_PROGRESS → PENDING_CUSTOMER → RESOLVED → CLOSED
```

| 优先级 | SLA | 类型 |
|--------|-----|------|
| P0 | 4h | 资金争议、发奖失败 |
| P1 | 24h | 投诉、活动咨询 |
| P2 | 72h | 一般咨询 |

- 路由：按类型 + 辖区 + 语言分配。
- 与资金争议协同：工单触发资金冻结（`can_withdraw=0`），结案后解冻。

## 十二、数据集成与特征工程

本域标签（§4.2）、画像快照（§4.3）、推荐召回特征（§七）均建立在**只读投影**之上，不产生对其他域权威数据的写权限：

```text
数据来源（只读消费）：
  交易域 deal.created.v1 / position.closed.v1  → 竞赛特征、行为标签原料
  资金域 deposit.completed.v1                   → 资格判定、"已入金"标签
  用户/准入域 user.approved.v1                  → 生命周期标签
  账户域 account.opened.v1                       → 生命周期标签
  跟单与排行系统 copy.leaderboard.contest_finalized.v1 → 竞赛发奖触发
  App 埋点（recommend.impression/click、页面浏览、搜索）→ 行为标签原料

数据分层（ODS → DWD → DWS → ADS）：
  ODS：事件原始落地（Kafka 消费落 ClickHouse/Hive，或 CDC 镜像只读表）
  DWD：清洗、标准化（按 entity_id/user_uid 对齐）
  DWS：按 user_uid 日聚合（登录/交易/入金等汇总指标）
  ADS：面向业务的应用层（t_crm_customer_tag、t_crm_customer_profile_snapshot、Segment 物化成员）
```

- **原则**：任何一层都不得反向修改交易域/资金域/用户域的权威表；本域故障不影响这些域的正常运行（松耦合）。
- 实时特征（Redis）与离线特征（批处理产出）双路，供 Segment 实时/批处理两种刷新模式（§六）使用。

## 十八、明确不做与常见误区

| 误区 | 正确做法 |
|------|----------|
| 「CRM 含群聊大厅」 | IM 在内容与社区 [15] |
| 「CRM = 代理」 | 代理是渠道分润，归因权威在 `t_partner_attribution` |
| 「推荐改文章」 | 正文在内容域 |
| 「竞赛榜 = 跟单排行」 | 竞赛（CONTEST）活动定义/名次奖励在本域，排行计分/强平/防刷权威在 [13]；跟单持续业绩排行（`PERFORMANCE`）完全不经过本域 |
| 「本域可以直接改派客户归因」 | 归因唯一真相在代理域 `t_partner_attribution`，改派须走工单升级代理域审核 |
| 「客户经理归属字段在用户表」 | 用户/账户域均无此字段，权威在本域 `t_crm_owner_assignment` |

**参考设计（`参考设计.md`）不采纳清单**（新增补充材料里可借鉴思路但不能照搬的部分，避免以后被误当作规范照抄）：

| 参考设计里的做法 | 为什么不采纳 |
|------|------|
| `crm_customer_base`/`crm_customer_account`/`crm_customer_risk_profile` 整表复制 | PII、账户资产、风险测评权威已在用户域[04]/账户域[05]/资金域[09]/准入域，本域重建等于重复权威，违反决策 2 |
| `mobile_hash`/`id_card_hash` 自建 PII 副本 | 违反决策 7「PII 最小必要」；本域一律以 `user_uid` 关联查询，不复制身份字段 |
| `referrer_id`/`register_source`/UTM 归因字段 | 与代理域 `t_partner_attribution` 归因权威冲突；获客渠道用 `t_user.register_channel` 已够用 |
| 基金/股票/债券产品推荐池 | 本平台是 CFD/FX 单品种体系，无多资产产品线，`SYMBOL_HOT` 场景已覆盖品种推荐 |
| "新客线索"（未开户潜客名片/手机号采集） | 会引入本域尚不具备权威的 PII 存储场景，本域线索功能限定为存量已注册客户跟进（§5.2） |
