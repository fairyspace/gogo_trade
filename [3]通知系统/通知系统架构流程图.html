📥 事件接入（Event Ingestion）
业务系统 交易/资金/风控/账户等通过 Outbox/CDC 或 Internal API 发布标准事件

通知引擎 执行：
  1. Schema 校验 + 来源白名单过滤
  2. 幂等校验：Redis GET idempotent:{entityId}:{key}
  3. 未命中 → Redis SET NX (PROCESSING, TTL=24h)
  4. 原始事件落库 → 按 priority 路由至对应 MQ Topic
📌 幂等双保险：Redis SET NX 为第一道防线，DB 唯一索引 (idempotency_key) 为最终兜底
📄 t_notify_event (CREATE) 🔑 Redis idempotent key (SET NX)
事件投递至 MQ Topic 📬
2
⚙️ 策略编排（Policy Orchestration）
通知引擎 消费 MQ 消息，依次执行六项检查：
1️⃣ 策略路由查询
查 t_notify_policy
确定通道计划 + 降级链 + 聚合规则
2️⃣ 用户偏好 + 同意
查 t_notify_preference & t_notify_consent
P0/P1 不可退订 → 强制发送
3️⃣ 抑制检查
查 t_notify_suppression
黑名单 / 投诉 / 法院禁联 → 抑制
4️⃣ 限频检查
Redis 滑动窗口
P0/P1 跳过限频，营销严格控制
5️⃣ 聚合检查
5分钟窗口内同类型 ≥ 5条 → 合并
P0/P1 始终逐条推送
6️⃣ 静默时段检查
用户时区 + quiet_hours
P0/P1 突破静默，营销必须遵守
📖 t_notify_policy (READ) 📖 t_notify_preference (READ) 📖 t_notify_consent (READ) 📖 t_notify_suppression (READ)
策略通过 → 模板渲染 🎨
3
🎨 模板渲染（Template Rendering）
通知引擎 执行：
  1. 查询活跃模板版本（template_code + locale → PUBLISHED 版本）
  2. 安全白名单引擎（Mustache/Handlebars）变量替换
  3. 生成各通道最终内容（Push/SMS/Email/Inbox/Voice）
  4. 计算 SHA256 内容哈希（合规防篡改）
  5. 渲染快照写入 rendered_snapshot_json（不可变）
  6. 消息主表入库
📌 模板版本化管理：每次变更需审批流 → DRAFT → PENDING_APPROVAL → APPROVED → PUBLISHED

📩 回执处理 & 梯次降级
通知引擎 监听回执（Webhook / 长轮询 / 客户端上报），处理三种结果：

记录 delivered_at

❌

可重试 → 指数退避 (1→2→4→8s)

⚡ 优先级 & SLA & 调度策略
优先级 典型场景 渠道策略 降级链 SLA MQ隔离
P0 紧急 强平/追保/异常登录 PUSH+INBOX并行 30s→SMS→5min→VOICE→10min→人工 INBOX <300ms VIP Topic
P1 重要 成交回报/资金到账 PUSH+INBOX并行 60s→SMS INBOX <1s Trade Topic
P2 普通 委托状态/行情预警 PUSH+INBOX 无降级 <10s Biz Topic
P3 低优 营销/投教/关怀 INBOX为主 无降级 分钟级 Offline Topic
📌 SLA 基准：Push 受 APNs/FCM 影响，峰值 5-10s。以 INBOX 落库时间作为系统可控 SLA 主要度量点
