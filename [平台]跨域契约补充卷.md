# 平台跨域契约补充卷（权威版）

> **版本**：v1.2
> **定位**：gogo-trade 全平台**跨域契约的单一权威源**。本文不拥有任何业务域的表结构或运行时权威，只定义**跨域共享的契约、注册表、命名规范、非功能基线**。各业务域文档凡涉及跨域引用，统一指向本文，禁止各自为政。
> **范围**：错误码注册表、事件注册表、角色注册表、ID 命名规范、entity_id 多实体策略、灰度发布规范、API 契约骨架、容灾 Runbook 模板、容量压测规范、监控指标清单。
> **原则**：契约先行、单一真相、版本化演进、不锁定实现语言。
> **适用**：[00]–[15] 全部 16 个业务域 + 总览域地图。

## 一、设计哲学与使用方式

### 1.1 为什么需要本卷

5 个探索子代理对 20 份设计稿的审阅发现：各域在**领域内**设计深度普遍较好，但在**跨域**层面存在三类系统性问题：

1. **命名漂移**：同一概念在不同域有不同命名（`COMPLIANCE_OFFICER` vs `COMPLIANCE_REVIEWER`；`user_uid` vs `trading_user_uid`）。
2. **契约缺失**：跨域调用只有文字描述，无 schema（`user.approved.v1` 无 payload；`realized_markup_revenue` 无生产方；`margin_call_buffer` 无定义）。
3. **非功能基线不一**：有的域有 SLO，有的没有；容灾/容量/灰度/监控各域各做一半。

本卷以**平台级单一真相源**的方式收敛上述问题。各域文档在修订时，凡涉及跨域内容，应**删除本地重复定义**并**引用本文相应章节**。

### 1.2 使用方式

| 场景 | 引用方式 |
|------|----------|
| 某域需要返回错误码 | 引用本文 §三 错误码注册表，使用 `DOMAIN_CODE` 前缀，禁止自创 |
| 某域需要发布/消费事件 | 引用本文 §四 事件注册表，使用 CloudEvents 1.0 envelope |
| 某域需要定义后台角色 | 引用本文 §五 角色注册表，使用统一 `role_code` |
| 某域需要定义实体 ID | 引用本文 §六 ID 命名规范，明确生成方与消费方 |
| 某域涉及多实体 | 引用本文 §七 entity_id 策略 |
| 某域上线灰度 | 引用本文 §八 灰度发布规范 |
| 某域定义对外 API | 引用本文 §九 API 契约骨架 |
| 某域定义容灾 | 引用本文 §十 容灾 Runbook 模板 |
| 某域做容量压测 | 引用本文 §十一 容量压测规范 |
| 某域定义监控 | 引用本文 §十二 监控指标清单 |

### 1.3 版本化与演进

- 本卷自身遵循**语义化版本**：MAJOR（破坏性契约变更）、MINOR（新增条目，向后兼容）、PATCH（修订说明）。
- 各域引用本卷时应记录所依赖的**最低版本号**，例如"依赖《平台跨域契约补充卷》≥ v1.2"。
- 注册表条目新增采用**追加**方式，已发布条目**不得改名或删除**，只能标记 `status=DEPRECATED` 并指向替代条目。

## 二、平台级术语与分层

### 2.1 L0–L7 分层（与总览域地图一致）

```text
L0 平台基座     权限 · 通知 · 多实体 · 审计
L1 客户与合规   用户 · 准入/KYC · 营销与推荐 · 内容与社区
L2 账户与资金   账户 · 资金
L3 市场与产品   品种 · 交易配置 · 行情
L4 交易执行核   交易/引擎 · 风控
L5 程序化与生态 量化 · 社交交易(跟单/排行榜)
L6 渠道         代理 IB/Affiliate
L7 只读经营层   结单/BI/监管报送
```

### 2.2 跨域交互三原则

1. **跨域只走事件或 RPC**：禁止跨域直连对方数据库表。
2. **一域一类权威数据**：每类真相只归一个域，其他域只读引用。
3. **方向稿不锁定实现语言**：本文所有 RPC/事件契约以 Protobuf/JSON Schema 表达，不绑定具体语言。

### 2.3 跨域调用矩阵（域 → 域）

下表为高频跨域调用，详细 schema 见后续章节。

| 调用方 | 被调方 | 方式 | 契约 |
|--------|--------|------|------|
| 准入 | 用户 | 事件 | `user.approved.v1`（§四） |
| 准入 | 账户 | RPC | `OpenAccount`（§九） |
| 准入 | 通知 | 事件 | `KYC_APPROVED/REJECTED`（§四） |

| 交易引擎 | 行情 | SHM/共享内存 | Core SymbolBook（§九） |

## 三、错误码注册表（平台级唯一权威）

### 3.1 错误码格式

```
<DOMAIN>_<CATEGORY>_<DETAIL>
```

- `DOMAIN`：域编号或缩写，全大写（如 `TRD`、`FUND`、`RISK`、`ACC`、`ONB`、`PMS`、`NTF`、`ROUTE`、`LEV`、`SESS`、`MD`、`SYM`、`IB`、`QNT`、`SOC`、`MKT`、`CMS`、`USER`、`PLATFORM`）。
- `CATEGORY`：类别，全大写（`VALIDATION` / `STATE` / `AUTH` / `LIMIT` / `CONFIG` / `EXTERNAL` / `CONFLICT` / `NOTFOUND` / `RATE` / `DEGRADED`）。
- `DETAIL`：具体细节，全大写下划线分隔。

### 3.2 标准响应体（所有 API 统一）

```json
{
  "code": "string",
  "message": "string",
  "retryable": false,
  "request_id": "uuid",
  "details": {}
}
```

- `code`：本注册表中的错误码。
- `retryable`：客户端是否可重试（幂等键不变）。
- `request_id`：服务端 trace id，贯穿日志/监控/审计。
- `details`：结构化补充（字段级错误等）。

HTTP 状态码映射：

| 类别 | HTTP |
|------|------|
| `VALIDATION` / `NOTFOUND` | 400 / 404 |
| `AUTH` | 401 / 403 |
| `CONFLICT` / `STATE` | 409 |
| `LIMIT` / `RATE` | 422 / 429 |
| `EXTERNAL` / `DEGRADED` | 502 / 503 |
| `CONFIG` | 500 |
| 成功 | 200 |

### 3.3 平台级错误码（前缀 `PLATFORM_`）

| code | 含义 | retryable | HTTP |

## 四、事件注册表（CloudEvents 1.0 权威）

### 4.1 事件信封（统一 envelope）

所有跨域事件**必须**采用 CloudEvents 1.0 规范：

```json
{
  "specversion": "1.0",
  "id": "uuid",
  "source": "/domain/<domain-code>",
  "type": "<domain>.<entity>.<verb>.v<major>",
  "time": "2026-07-21T02:03:04.567Z",
  "datacontenttype": "application/json",
  "subject": "<entity-id>",
  "data": { }
}
```

- `type` 命名：`<domain>.<entity>.<verb>.v<major>`，全小写，点分。`v<major>` 为破坏性版本号。
- `subject`：实体主键（如 `account_id`、`deal_id`）。
- `id`：事件唯一 id，消费方以此去重。
- 所有事件经 **Outbox + Kafka** 投递，禁止业务库直写后另起事务发事件。

### 4.2 事件投递语义

- **at-least-once**：消费方必须幂等。
- **有序**：同 `subject` 走同 Kafka partition（partition key = `subject`）。
- **重试**：消费失败进入重试 topic，N 次后入死信 topic `dlq.<domain>.<entity>`，告警 + 人工。
- **Schema Registry**：所有事件 schema 注册到 Confluent Schema Registry（或等价物），生产/消费双向校验。

| `config_version` | string | **配置域** | 交易/风控/行情 | 配置版本戳 |

### 6.2 关键裁定

1. **`user_uid` 由用户服务生成**：准入服务**不再**生成 `trading_user_uid`；账户域 `t_trading_account.owner_user_id` 引用 `user_uid`。
2. **注册即建用户**：注册流程同时建 `t_user(status=0)` 与 `t_onboarding_application`，二者通过 `user_uid` 关联；KYC 通过后发 `user.approved.v1`，账户域据此开户。
3. **`entity_id` 全表贯穿**：所有业务表必须含 `entity_id`，跨实体查询由 DB 行级安全或应用层强制隔离（品种/行情域需补 `entity_id` 维度，见 §七）。
4. **`magic_number` 全局唯一**：量化策略实例与社交跟单订阅各占一段命名空间，由生成方保证全局唯一，交易引擎据此归因。
5. **`client_order_id` 客户端生成**：交易引擎以 `(account_id, client_order_id)` 唯一索引去重。

### 6.3 ID 暴露规则

- 对外 API（App/Partner）只暴露 `user_uid` / `account_id` / `symbol_id` / `deal_id` 等业务 ID，**不**暴露内部自增主键 `id`（除审计需要）。
- 审计/对账场景可暴露 `id`，但需 PMS `PMS_AUDITOR` 权限。

## 七、entity_id 多实体策略（平台级唯一权威）

### 7.1 设计原则

1. **`entity_id` 是法律实体隔离的最高键**：所有客户数据、账户、订单、成交、资金、配置原则上按 `entity_id` 隔离。
2. **同一自然人可跨实体注册**：`t_user` 允许同邮箱/手机在不同 `entity_id` 下各注册一个账户，但 `user_uid` 不同。
3. **跨实体查询禁令**：业务库查询必须带 `entity_id` 过滤；DB 层通过行级安全（RLS）或应用层强制；审计/BI 走专用只读副本。
4. **全局共享数据**：品种主数据、行情、监管规则可全局共享，但品种可见性、账户组、营销活动按 `entity_id` 配置。

### 7.2 各域 entity_id 落地清单

| 域 | 是否含 entity_id | 缺口 |
|----|------------------|------|
| 用户 | 是 | — |
| 准入 | 是 | — |
| 权限 | 是（pms_*） | — |
| 通知 | 是 | — |
| 账户 | 是 | — |
| 资金 | 是 | — |
| 风控 | 是 | — |
| 交易 | 是 | — |
| 代理 | 是 | — |
| 营销 | 是 | — |
| 内容 | 是 | — |
| 社交 | 是 | — |
| 量化 | 是 | — |
| **品种** | **否** | **需补：品种可见性按 entity 配置（`t_symbol_visibility` 或 `t_account_group_symbol_config` 间接表达）** |
| **行情** | **否** | **需补：行情接入全局，但 Client Quote 的 markup_profile 按 entity/account_group 隔离（已通过路由网关覆盖）** |
| 交易配置 | 是（route_rule/leverage/session 按 entity） | — |

### 7.3 品种域 entity_id 方案（裁定）

品种主数据（`t_symbol`）保持**全局共享**（同一 FIGI/ISIN 全实体共用），通过 `t_account_group_symbol_config`（账户组级，含 `entity_id`）表达"哪些账户组可见哪些品种"。品种域文档需在通用约定中声明此假设，避免后期 Breaking Change。

### 7.4 行情域 entity_id 方案（裁定）

行情 Core Book 全局共享；Client Quote 的 `markup_profile` 通过路由网关 `t_route_rule` 按 `entity_id`/`account_group_id` 隔离。行情域文档需声明"行情接入全局，差异化在路由网关"。

## 八、灰度发布规范（平台级唯一权威）

### 8.1 灰度层级

| 层级 | 粒度 | 适用 |
|------|------|------|
| L1 配置灰度 | 配置版本按 entity/account_group 百分比 | 路由/杠杆/风控规则/品种规格 |
| L2 模板灰度 | 通知模板 `rollout_percentage` + `target_entity_ids` | 通知模板 |
| L3 代码灰度 | 服务实例按流量百分比 | 全部服务 |
| L4 功能开关 | feature flag 按 entity/user 百分比 | 新功能 |

### 8.2 配置灰度（最常用）

- 配置发布经 `t_config_version` 快照，新版本先标记 `status=SHADOW`（只读不生效）→ `CANARY`（指定 entity/account_group 生效）→ `ACTIVE`（全量）→ `SUPERSEDED`。
- 灰度期间同时跑**回放测试**：用历史请求对新旧版本求值，diff 告警。
- 回滚：将上一版本 `status` 改回 `ACTIVE`，新版本 `SUPERSEDED`，配置广播 < 3s 全网生效。

### 8.3 灰度发布 Runbook 模板

1. **预检**：新配置经静态校验 + 回放测试 diff < 阈值。
2. **审批**：四眼（`PMS_*_MANAGER` + `PMS_RISK_OFFICER`）。
3. **CANARY**：选 1–5% 流量，观察 30 min（监控 §十二 指标）。
4. **扩量**：10% → 50% → 100%，每档观察 30 min。
5. **回滚条件**：错误率 > 1%、P99 超 SLO 2 倍、对账不一致、业务告警。
6. **完成**：全量后归档旧版本，记录修订历史。

### 8.4 各域灰度要求

| 域 | 灰度对象 | 必须 |
|----|----------|------|
| 路由网关 | route_rule / markup_profile | 是 |
| 杠杆配置 | leverage_template / tier | 是 |
| 时间配置 | session_override | 是（PHASED 恢复） |
| 风控 | risk_rule / kill_switch | 是 |
| 品种 | symbol 规格变更 | 是（影响在途持仓） |
| 通知 | 模板 | 是（rollout_percentage） |
| 营销 | 活动 | 是（按 entity 灰度） |
| 内容 | 内容/审核规则 | 是 |
| 量化 | 策略版本 | 是（实例滚动） |
| 社交 | 排行榜算法 | 是（版本回滚） |

### 9.2 REST API 通用规范

- 路径：`/api/v<major>/<resource>`，版本化。
- 鉴权：Bearer Token（C 端）或 mTLS（服务间）。
- 幂等：写操作必须带 `Idempotency-Key` header。
- 分页：`?cursor=&limit=`，limit 上限 100。
- 时间：所有时间字段 RFC3339 UTC。
- 错误：统一 §3.2 响应体。
- Rate Limit：`X-RateLimit-Limit/Remaining/Reset` header。

## 十、容灾 Runbook 模板（平台级最低基线）

### 10.1 平台级容灾目标

| 维度 | 目标 |
|------|------|
| RPO（数据丢失） | 交易引擎 WAL = 0；资金 ledger = 0；其他域 < 1s |
| RTO（恢复时间） | 交易引擎 < 30s；资金 < 5min；其他域 < 15min |
| 多活 | 资金/交易核心同城双活 + 异地灾备；其他域同城主备 |
| 切换 | 自动健康检测 + 人工确认（Kill Switch 类需四眼） |

### 10.2 各域容灾要求

| 域 | RPO | RTO | 多活 | 关键 |
|----|-----|-----|------|------|
| 交易引擎 | 0（WAL） | 30s | 同城双活 | WAL 拍板，主故障只平不开 |
| 资金 | 0（ledger） | 5min | 同城双活 | Saga 中间态可补偿 |
| 风控 | < 1s | 30s | 同城双活 | Kill Switch 脑裂防护（共识广播） |
| 行情 | < 1s | 5s | Active-Standby | seq failover 防重复 |
| 路由网关 | < 1s | 30s | 同城双活 | 配置全网一致 < 3s |
| 杠杆/时间配置 | < 1s | 30s | 同城主备 | 缓存降级回源 MySQL |
| 账户 | < 1s | 5min | 同城主备 | 账户组配置缓存 |
| 用户/准入 | < 5s | 15min | 同城主备 | 注册/KYC 可降级 |
| 通知 | < 1s | 5min | 同城双活 | WORM 写入失败本地缓冲 |
| 品种 | < 5s | 15min | 同城主备 | 主数据缓存 |
| 代理 | < 1s | 15min | 同城主备 | Inbox 积压告警 |
| 量化 | < 5s | 15min | 同城主备 | Runtime 池耗尽降级 |
| 社交 | < 5s | 15min | 同城主备 | Copy Kill Switch |
| 营销 | < 5s | 15min | 同城主备 | 活动暂停降级 |
| 内容 | < 5s | 15min | 同城主备 | IM 只读降级 |
| 权限 | < 1s | 5min | 同城双活 | PDP 全集群不可用切跨机房 |

### 10.3 通用 Runbook 模板

1. **检测**：健康检查 + 业务指标（错误率/延迟/对账）超阈值。
2. **告警**：自动告警 + on-call 通知（通知域）。
3. **判定**：on-call 判定是否切换（自动切换需预设条件）。
4. **切换**：DNS/负载均衡切流；WAL/ledger 主备切换；Kill Switch 四眼确认。
5. **验证**：冒烟测试 + 对账。
6. **回切**：原主恢复后灰度回切。
7. **复盘**：72h 内 RCA + 修订 Runbook。

### 11.3 压测方法论

- **基准**：每域上线前必跑基线压测，记录 P50/P95/P99/P999 + 错误率。
- **容量拐点**：找到错误率 > 0.1% 或 P99 超 SLO 2 倍的拐点，留 50% 余量。
- **混沌**：注入下游超时、Redis 故障、网络分区、WAL 丢一条、LP 重复回报。
- **回归**：每次重大版本回归压测，对比基线。

## 十四、结语

本卷是 gogo-trade 全平台跨域协作的"宪法"。各域在追求自身深度时，凡涉及跨域，**一律以本卷为准**。本卷自身遵循语义化版本演进，注册表条目只追加不删改，确保契约稳定可依赖。

随着各域文档按本卷补齐引用，gogo-trade 将从"一组各自优秀的域设计稿"演进为"一套契约严密、边界清晰、非功能基线统一的世界顶级金融经纪平台设计体系"。
