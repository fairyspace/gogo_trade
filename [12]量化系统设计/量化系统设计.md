# 量化系统设计（权威版）

> **版本**：v1.0 Final  
> **定位**：零售 / 专业客户多资产 CFD·FX 经纪平台的**程序化交易入口权威** + **托管策略运行权威** + **券商内部量化分析权威**。  
> **运行时分层**：Trading Engine（成交权威）+ Quant API / Runtime Orchestrator + Strategy Runtime（沙箱）+ Backtest Worker + MySQL（业务态）+ Redis（限流/实例热态）+ ClickHouse（历史行情只读）+ Kafka Outbox（跨域事件）。**方向设计不锁定平台实现语言。**  
> **对标**：MetaQuotes / Spotware(cTrader) / Devexperts / Apex / DriveWealth / TradingView / TT / Broadridge / Talos / 恒生电子。  
> **跟单 / 排行榜权威**：已上收 [`[13]社交交易/社交交易系统设计.md`](../[13]社交交易/社交交易系统设计.md)；本域仅保留可选信号源适配与策略商城（代码侧），不再定义 Copy 执行真相。  
> **文档收敛**：早期调研稿《量化引擎设计》《量化引擎设计参考1》已移除；**以本文为唯一权威**。

## 一、系统定位与对标决策

### 1.1 先分清两件事（拍板）

市面上「量化」常被混为一谈。本平台**必须拆成两轨**，禁止混跑、混库、混权限：

```
┌─────────────────────────────────────────────────────────────────────┐
│  Track A — Client Quant（客户量化）                                  │
│  谁用：零售 / 专业客户 / 外部开发者                                   │
│  做什么：                                                            │
│    A1. 自有服务器用官方 SDK 调 OpenAPI 下单（对标 IB / Apex）         │
│    A2. 平台托管策略运行时（对标 MT5 EA / cTrader cBot）               │
│    A3. 回测 / 参数扫描（对标 Strategy Tester / LEAN）                │
│    A4. （后期）策略商城；若策略作信号源 → 输出给社交交易域跟单      │
│  数据权限：Client Quote + 自己账户的订单/持仓/成交                    │
│  下单路径：一律走引擎 SubmitOrder（与手工单同一 PreTrade）            │
├─────────────────────────────────────────────────────────────────────┤
│  Track B — Broker Quant（券商自用量化）                               │
│  谁用：风控 / 做市 / 对冲台 / 运营分析                                │
│  做什么：                                                            │
│    B1. 公司敞口聚合、对冲建议、报价/加点效果分析                      │
│    B2. 客户行为 / EA 异常检测（刷单、对倒、延迟套利嫌疑）              │
│    B3. 内部回测与研究（用 Core Book / House 数据，不对客）            │
│  数据权限：Core Book + House Exposure + 全站脱敏聚合                  │
│  下单路径：不对客账户下单；对冲指令走运营/对冲台专用通道               │
└─────────────────────────────────────────────────────────────────────┘
```

**本系统不是对冲基金**：不做自营 Alpha 全栈、不做客户资金池代客理财（PAMM 另立项）、不做美股 DTC / 期货 SPAN。

| # | 厂商 | 可借鉴 | 本设计采纳 / 取舍 |
|---|------|--------|------------------|

| 2 | **Spotware (cTrader)** | cBot、Automate REST/WS、Copy | 采纳 **API-first + 托管 bot**；Copy 权威在社交交易域 |
| 3 | **Devexperts** | 程序化入口与风控同路径 | **API/EA 订单与手工单同一 PreTrade** |
| 4 | **Apex Fintech** | API Key、权限、审计、幂等 | **`t_api_app` + HMAC + 限流 + 审计日志** |
| 5 | **DriveWealth** | 指令编排与执行分离；纸交易镜像 | 沙箱/Demo 账户走同一引擎语义；回测用影子引擎 |

| 阶段 | 交付 | 对标心智 |
|------|------|----------|

## 二、十六条不可破坏决策

1. **双轨隔离**：Client Quant 与 Broker Quant **分服务、分库权限、分网络策略**；客户沙箱永不可见 Core Book / House / 他户数据。  
2. **成交权威只在 Trading Engine**：量化域**禁止**直写 `t_order` / `t_deal` / `t_position` / `t_account_ledger`；只通过 `SubmitOrder` 等引擎 RPC。  
3. **同一 PreTrade**：本域 `source=API|EA` 与 `CLIENT` 走同一套风控/保证金/熔断；额外叠加 **API 限流 + 实例级限额**（不得替代账户级风控）。`SIGNAL` 由社交交易域提交，仍走同一引擎 PreTrade。  
4. **幂等强制**：量化下单必须带账户内唯一 `client_order_id` + 传输层 `request_id`；重复提交返回首次结果。  
5. **策略归因**：托管策略 / API 订单必须带 `magic_number`（实例级全局唯一）与 `source`；交易表需补齐该字段（见 §3.3）。跟单订单的 magic 由**社交交易域**分配。  
6. **回测 ≠ 实盘**：回测只读 ClickHouse 历史，在 **Backtest Shadow Matcher** 内撮合；**禁止**调用实盘引擎、禁止写实盘账本。  
7. **行情分层**：客户策略只订 Client Quote；估值/强平/B-Book 仍由引擎用 Core Book（策略不可覆盖）。  
8. **沙箱默认零信任**：无出站网络（除平台注入的行情/交易/时钟 API）、无宿主机文件系统、CPU/内存/回调超时硬限制；镜像白名单依赖。  
9. **代码与运行版本绑定**：实例绑定 `strategy_version`；改代码不影响已运行实例，需显式重启/滚动。  
10. **`entity_id` 贯穿**：API App、策略、实例、回测任务均带实体；禁止跨实体操作账户。  
11. **出金默认关闭**：API `perm_withdraw=0` 为默认；开启需独立审核与更严限流。  
12. **事件经契约**：策略启停、回测完成走 Kafka；可选向社交交易域发出信号适配事件；禁止「写库后尽力通知」。  
13. **Broker Quant 不对客下单**：内部研究/对冲建议产出信号给对冲台或运营工具；禁止静默改客户仓。  
14. **不拥有跟单执行与排行榜**：1→N 放大、订阅关系、排行榜权威在社交交易域；本域不得另立 Copy 服务真相。  
15. **合规披露**：回测/实盘业绩展示必须带风险声明；过去收益不保证未来（商城上架强制）。  
16. **不做的事**：自研 MQL5、完整 QuantConnect 云、客户侧 SPAN、未隔离的「服务器直接 exec 用户代码」、完整 IM/群聊。

## 三、边界与跨系统契约

### 3.1 领域边界

| 系统 | 权威数据 | 量化系统的使用方式 |
|------|---------|-------------------|
| 交易引擎 | Order/Deal/Position、SubmitOrder | **唯一写交易路径**；量化只做编排 |
| 行情 | Client Quote、历史 Tick/K 线 | 客户策略订 Client；回测读 CH；Broker Quant 可读 Core 投影 |
| 品种 / 时间 | digits、手数、`t_daily_session` | 回测与校验只读 |
| 账户 | 账户归属、账户组、EA 开关 | 绑定账户；校验 `user↔account` |
| 资金 | 出入金、ledger 结构 | API 出金极谨慎；默认关 |
| 风控 | PreTrade / Kill Switch | 引擎执行；量化叠加实例限额与 API 限流 |
| 用户 / 权限 | 用户、credential、RBAC | API Key 落 `t_user_credential`；运营审批走权限 SoD |
| 通知 | 模板送达 | 消费 `quant.*` / `strategy.*` 事件 |
| 代理 | IB 钱包 / 佣金 | 策略商城结算可复用钱包入账模式（非跟单权威） |
| 社交交易 | 订阅、Copy 放大、排行榜 | 本域策略实例可作**可选信号源**；不拥有跟单执行 |

### 3.2 量化 → 引擎下单契约

```protobuf
// 量化编排层填充后调用引擎 SubmitOrderRequest（字段对齐交易系统设计）
// 量化侧强制约定：
message QuantOrderContext {
  string source            = 1;  // API | EA | SIGNAL
  int64  magic_number      = 2;  // 策略实例归因；API 本地脚本可用 app 级 magic 或 0
  string strategy_id       = 3;  // 可选，写入 comment 或扩展字段
  string instance_id       = 4;  // 可选
  string client_order_id   = 5;  // 必填，账户内唯一
  string request_id        = 6;  // 必填
}
```

**生成规则（落地）**：

| 入口 | `source` | `magic_number` | `client_order_id` 建议 |
|------|----------|----------------|------------------------|
| 外部官方 SDK | `API` | `t_api_app.default_magic` 或 0 | SDK 生成 UUID |
| 托管策略实例 | `EA` | `t_strategy_instance.magic_number` | `{instance_id}:{seq}` |
| TradingView Webhook | `API` 或 `EA` | 绑定实例 magic | webhook delivery id |
| 跟单放大（非本域） | `SIGNAL` | 见《社交交易系统设计》 | 见社交交易域 |

### 3.3 交易域增量（必须合入交易权威版）

`t_order` / `t_order_history` / `t_deal` / `t_position` / `t_position_history` 增加：

```sql
`magic_number`    BIGINT UNSIGNED NOT NULL DEFAULT 0
    COMMENT '策略/API归因; 0=非策略; 与量化 t_strategy_instance.magic_number 对齐',
KEY `idx_account_magic` (`account_id`, `magic_number`)
```

引擎 `SubmitOrderRequest` 同步增加 `magic_number` 字段。无此字段则托管策略**无法审计归因**，量化 P1 不得上线；跟单归因字段由社交交易域同样依赖该列。

### 3.4 行情只读契约（回测）

| 数据 | 来源 | 用途 |
|------|------|------|
| Normalized Tick | ClickHouse `ticks` | Tick 模型回测（P2） |
| M1 K 线 | ClickHouse `klines_m1` | P1 默认回测 |
| 合约规格 | `t_symbol` | 手数/精度/盈亏 |
| 时段 | `t_daily_session` + 日历 | 可交易判断 |
| Swap / 佣金参数 | 品种 + 账户组 | 费用模拟 |
| Client markup | `t_markup_profile` | 可选：回测用客户可见点差 |

回测默认点差：**Core + 账户组成交点差模型**（可配置固定点差）；展示层 Client Quote 不得用于「更优成交」作弊。

### 3.5 跨域事件（Quant 生产）

| 事件 | 何时 | 消费者 |
|------|------|--------|
| `quant.api_app.created/revoked.v1` | App 生命周期 | 审计、通知 |
| `strategy.instance.started/stopped/error.v1` | 实例状态 | 通知、监控 |
| `backtest.completed/failed.v1` | 回测结束 | 通知、IDE |
| `quant.signal.candidate.v1`（可选） | 策略显式发信号意图 | **社交交易域**适配为 `signal.generated` / 拒绝 |
| `quant.anomaly.detected.v1` | Broker Quant | 风控运营 |

量化**只消费**交易域 `order.*` / `deal.*` / `position.*` 用于实例统计，不反向改成交。跟单扇出不在本域。

### 4.2 服务拆分（可部署单元）

| 服务 | 路径特征 | 职责 | 扩缩容特征 |
|------|---------|------|-----------|
| `quant-api` | 编排接入 | OpenAPI、App 管理、查询编排 | IO 密集，水平扩 |
| `quant-ws` | 连接态推送 | 行情/订单/持仓推送（可与行情 WS 共用网关主题） | 连接态 |
| `strategy-orchestrator` | 控制面 | 实例生命周期、调度、心跳、熔断停机 | 控制面 |
| `strategy-runtime` | 沙箱执行 | 跑用户 `on_tick/on_bar` | CPU/内存密集，按实例槽位扩 |
| `backtest-worker` | 批处理 | 回测/优化作业 | CPU 密集，队列消费 |
| `broker-quant` | 内网研究 | 内部分析作业与 API | 内网，只读为主 |
| `webhook-ingress` | 编排接入 | TV/外部信号入口 | 独立限流 |

> Copy / 排行榜进程归属社交交易域，不在本表。

**量化挂了不得影响**纯手工交易与引擎热路径（松耦合硬要求）。

### 4.3 与账户「EA 开关」协同

账户组 / 账户级应具备（账户域字段，量化只读）：

- `allow_ea`：是否允许 `source=EA`  
- `allow_api`：是否允许 `source=API`  
- 可选：`max_api_orders_per_sec`（与 App 限流取更严）

引擎 PreTrade 校验上述开关；量化编排层提前拒绝以改善 UX。

## 五、Track A — 客户量化详细设计

### 5.1 OpenAPI（P0 核心）

#### 5.1.1 能力清单

| 类别 | 接口（示意） | 权限位 |
|------|-------------|--------|
| 账户 | `GET /v1/accounts`, `GET /v1/accounts/{id}/state` | `perm_read_account` |
| 行情 | `GET /v1/quotes`, `GET /v1/klines`, WS `quote.subscribe` | `perm_read_market` |
| 交易 | `POST /v1/orders`, `DELETE /v1/orders/{id}`, `PATCH ...` | `perm_trade` |
| 持仓 | `GET /v1/positions`, `POST /v1/positions/{id}/close` | `perm_trade` / read |
| 历史 | `GET /v1/orders/history`, `GET /v1/deals` | `perm_read_account` |
| 策略（P1） | 实例启停、日志 | `perm_strategy` |

#### 5.1.2 认证与安全

1. `t_user_credential`（`cred_type=API_KEY`）存 Key 哈希；`t_api_app` 存权限/限流/绑定账户。  
2. 签名：推荐 **HMAC-SHA256**（timestamp + method + path + body）；时钟窗 ±30s。  
3. IP 白名单可选；生产 App 默认需审核（`status=审核中→启用`）。  
4. Secret 只展示一次；轮转发新凭证、旧凭证宽限期后作废。  
5. 所有写操作审计入 `t_api_call_log`（热计数 Redis，明细可冷存 CH）。

#### 5.1.3 限流（Redis）

```
api:rate:{app_id}:sec | min | order_sec
api:ws:connections:{app_id}
api:ws:subscriptions:{app_id}:{conn_id}
```

超限返回 HTTP 429 + `Retry-After`；**下单限流与 REST 总限流分离**（防行情轮询挤占下单）。

#### 5.1.4 Webhook 出站（可选 P0.5）

订单成交 / 强平等事件投递客户 URL：`t_api_webhook_delivery`，HMAC 签名，指数退避，至少一次投递。

### 5.2 官方 SDK（P0）

定位：客户**本地/自有 VPS** 运行，不经过平台沙箱。首期只交付**一种**脚本语言的官方 SDK（产品选型；实现阶段再定具体语言与包名，业界常见为 Python）。

```text
能力：
  - 同步/异步 REST
  - WebSocket 行情与执行回报
  - 自动签名、重试、幂等 client_order_id
  - 类型化 Order/Position/Deal
  - 纸交易与实盘同一 API，仅 base_url / app 环境不同
```

**API 心智（示意，语法随首期脚本语言而定）**：

```text
client = Client(api_key, api_secret, account_id)
client.on("quote", handler)          # 订阅 Client Quote
client.submit_order(...)             # 强制带 client_order_id
client.subscribe_quotes(["EURUSD"])
client.run_forever()
```

SDK **不是**策略框架；策略框架见 §5.3。

### 5.3 托管策略 Runtime（P1）

#### 5.3.1 策略框架

平台提供受限标准库 + 官方策略 SDK；回调语义固定为 `on_init` / `on_bar` / `on_tick` / `on_order`。下列为**语义示意**（语法随首期脚本语言而定）：

```text
class SmaCross(Strategy):
    on_init(ctx):  读取参数
    on_bar(ctx, bar):  仅允许调用 ctx.* ；禁止宿主机/公网能力
    on_tick(ctx, tick): ...
    on_order(ctx, event): ...
```

**注入 API（白名单）**：

| API | 说明 |
|-----|------|
| `ctx.buy/sell/close/...` | 下单，内部生成幂等 id + magic |
| `ctx.positions / ctx.orders` | 仅本账户 + 本 magic 视图（可配看全账户） |
| `ctx.quote / ctx.bars` | Client Quote / K 线窗口 |
| `ctx.account` | balance/equity/margin |
| `ctx.param / ctx.storage` | 参数与小 KV 持久化（限额） |
| `ctx.log` | 写入策略日志流 |
| `ctx.now` | 平台时钟（禁止信本地时间做关键逻辑） |

#### 5.3.2 隔离与资源

| 项 | P1 基线 | 说明 |
|----|---------|------|
| 隔离 | gVisor 或 Firecracker microVM | Docker 默认 runc **不够** |
| CPU | 默认 0.5 vCPU / 实例 | 可按套餐升 |
| 内存 | 默认 256MB | 超限杀实例 → ERROR |
| 回调超时 | `on_tick` 50ms / `on_bar` 500ms 可配 | 超时记错误，连续 N 次停机 |
| 网络 | 仅本地 IPC / RPC 到 sidecar | 无公网 |
| 依赖 | 平台 wheel 白名单 | 禁止任意 pip 公网装包（P2 可审核私有包） |

调度模型：

- Orchestrator 按 `symbol_key + timeframe` 将 bar/tick 事件推到实例 mailbox。  
- 单实例单线程事件循环（与 MT5 单图表 EA 心智一致），避免用户代码锁问题。  
- 节点心跳；失联 → 实例标记 ERROR 并告警（是否自动迁移可配，默认手动重启）。

#### 5.3.3 实例级风控（叠加层）

在账户风控之上：

- `max_positions` / `max_lots` / `max_daily_loss` / `max_total_loss`  
- 触发 → 停止开新仓并 `STOP` 实例（已有仓是否自动平由用户配置 `flatten_on_stop`）  
- Redis `quant:instance_risk:{id}` 热更新，编排层下单前检查

### 5.4 回测引擎（P1）

#### 5.4.1 原则

1. **同一套策略代码**：`Strategy` 在 Live / Backtest 切换 `Context` 实现。  
2. **影子撮合**：虚拟 balance/equity/margin、挂单、SL/TP、Swap、StopOut（参数取账户组快照）。  
3. **可复现**：固定 `data_version` + `code_hash` + `parameters` + `engine_version` → 同结果。  
4. **反过拟合提示**：报告展示样本内外可选（P2）；P1 至少展示交易次数与最大回撤。

#### 5.4.2 精度模型

| `tick_model` | P1 | 说明 |
|--------------|----|------|
| 3 Open only | 可选 | 最快，最不准 |
| 2 M1 OHLC 模拟 | **默认** | 用 OHLC 生成路径（如 O-H-L-C / 随机路径需声明） |
| 1 真实 Tick | P2 | 读 CH ticks，贵 |

#### 5.4.3 作业流

```
创建任务 → 安全扫描 → 估时排队 → Worker 拉 CH 数据
  → 事件循环（报价→挂单→SL/TP→策略回调→模拟成交）
  → 写 t_backtest_result / trades / equity 曲线对象存储
  → 事件 backtest.completed
```

队列：`backtest:queue`（ZSET）；VIP 可降 score。超时与并发：每用户并行回测上限（如 2）。

### 5.5 Web IDE（P1 最小）

- 浏览器编辑单个策略源文件 + 参数表单（JSON Schema）。  
- 保存 → 新 `t_strategy_version` → 静态扫描（禁 import 黑名单）。  
- 一键「回测」/「部署到 Demo 实例」。  
- **不做**首期完整 Jupyter/多文件 monorepo（P2）。

### 5.6 TradingView Webhook（P1）

- 策略类型 `WEBHOOK`：平台生成 `webhook_url` + token。  
- 验签/限流后映射为内部下单或注入运行中实例。  
- 仅操作绑定账户；失败可查 `t_api_webhook_delivery` 对偶的入站日志表（`t_webhook_ingress_log`）。

### 5.7 策略商城与信号源适配（P2，纲要）

| 能力 | 要点 |
|------|------|
| 策略商城上架 | 代码审计 + 风险声明 + 实盘展示实例（不可刷数） |
| 商城订阅（买策略/租用） | 月费/买断；结算可复用钱包入账模式 |
| 作为跟单信号源 | 经适配输出给**社交交易域**；订阅关系、放大、排行榜、跟单分成以《社交交易系统设计》为准 |
| 反向跟单 / 排行榜 | **不在本域实现** |

P2 前不要在本域建 Copy 表流程；原稿跟单表仅作社交交易域字段参考。

## 六、Track B — 券商自用量化

### 6.1 目标

让券商**自己**用量化手段运营平台，而不是替客户跑 Alpha：

| 域 | 作业示例 | 输出 |
|----|----------|------|
| House Risk | A/B Book 敞口、品种集中度、LP 延迟 | 看板 / 告警 / 对冲建议单 |
| Pricing | markup 敏感性、时段点差、滑点归因 | 调价建议（人工确认生效） |
| Client Behavior | EA 高频、对敲嫌疑、异常盈利聚类 | `quant.anomaly.detected` → 风控 |
| Ops Research | 规则变更前后对照实验 | 内部报告（不对客） |

### 6.2 架构要点

1. **内网部署**，SSO + RBAC（权限系统）；无客户 API Key。  
2. 数据：Kafka 交易/行情事件 + CH 历史 + 风控快照；**禁止**挂载客户策略代码目录。  
3. 作业调度：Airflow / 自研 cron + notebook / 作业制品库（内部）。  
4. 若产生对冲下单：走 **Dealer / Hedge Desk 工具** → 引擎 `source=DEALER` 或公司账户，不走客户 `EA` 通道。  
5. 与 Client Runtime **镜像仓库分离、凭证分离、VPC 分离**。

### 6.3 与风控域分工

| 事项 | 风控权威版 | Broker Quant |
|------|-----------|--------------|
| Kill Switch / PreTrade / StopOut | ✅ 执行权威 | 只分析与建议 |
| 公司限额配置 | ✅ | 可做「限额是否合理」研究 |
| 可疑行为标记 | 可落风控事件 | 模型产出嫌疑列表 |
| 客户策略代码 | ❌ | ❌（隐私）；只看订单行为特征 |

## 八、关键时序

### 8.1 API 下单（P0）

```
Client SDK → quant-api
  → 验签 / 限流 / 权限 / 账户绑定
  → 生成 request_id（若缺失）
  → SubmitOrder RPC(source=API, magic=app.default_magic, ...)
  → Engine PreTrade → 成交原子提交
  → 返回 Order 视图；WS/Webhook 异步推送
```

### 8.2 托管策略 Tick（P1）

```
Quote/Bar 事件 → Orchestrator 路由到实例 mailbox
  → Runtime 调 on_tick/on_bar
  → ctx.buy → Runtime Sidecar
       → 实例风控 → SubmitOrder(source=EA, magic=instance.magic)
  → 成交事件 → 更新实例统计 / 用户日志
```

### 8.3 回测（P1）

```
IDE/API 创建 backtest_task
  → Worker：CH 拉数 + Shadow Matcher + 同款 Strategy
  → 只写回测表 / 对象存储
  → 永不触碰实盘账户
```

## 九、安全、合规与运营

1. **密钥**：API Secret 加密存储；日志脱敏。  
2. **代码**：用户策略代码属用户资产；运营只读需审批；商城上架强制安全扫描。  
3. **滥用**：异常请求速率、多账户同 IP 刷单 → Broker Quant / 风控联动。  
4. **披露**：回测报告水印「模拟业绩」；Live 业绩防刷（最短展示天数）。  
5. **容量**：Runtime 槽位售卖/套餐（免费 Demo 限额 + 付费 Live 槽位）。  
6. **灾难**：Orchestrator 无状态；实例状态 DB+Redis；节点挂侧重拉，默认不自动重下历史意图单。

## 十三、明确不做与常见误区

| 误区 | 正确做法 |
|------|----------|
| 「先做策略商城引流」 | 先做稳定 API + 执行归因 |
| 「回测直接打实盘引擎」 | 影子撮合，保证可复现与安全 |
| 「客户策略读 Core Book 更公平」 | 客户只见 Client Quote；成交价规则在引擎 |
| 「仅靠解释器级沙箱就够安全」 | 生产要内核级隔离 |
| 「券商分析与客户 EA 共用集群」 | 双轨隔离 |
| 「量化服务写 MySQL 订单表」 | 只走引擎 RPC |
| 「一次支持 MQL/Pine/JS/多语言」 | 首期只一种 SDK/托管脚本语言，降低运行时爆炸 |
| 「方向稿先锁 C++/Go/Python」 | 方向稿只定职责与契约；实现语言另立项 |
| 「跟单/排行榜继续写在量化里」 | 权威在《社交交易系统设计》 |
