# 跟单与排行系统设计 — 终版 v2.0

> **版本**：v2.0 Final｜2026-07-22
> **定位**：零售 / 专业客户多资产 CFD·FX 经纪平台的**跟单（Copy）执行权威** + **信号源业绩与交易排行榜权威** + **跟单分成结算意图权威**。
> **对标**：Spotware cTrader Copy、MetaQuotes MQL5 Signals、eToro CopyTrader（能力级，不照搬产品形态）。
> **运行时分层**：Copy Orchestrator（订阅/放大/限速）+ Performance Projector（业绩/排行/竞赛计分）+ 既有 Trading Engine（成交权威）。方向设计不锁定实现语言。
> **域地图**：见 [`[总览]经纪平台域地图.md`](../[总览]经纪平台域地图.md)。
> **命名**：本文档目录为 `[13]跟单与排行系统`，全文以「**跟单与排行系统**」作为业务侧展示名；跨域契约、事件、错误码、角色码中使用的注册域代号仍是「**社交交易（Social Trading）**」（域地图、《平台跨域契约补充卷》、《量化系统设计》等 6 份以上文档均以此代号互相引用），二者指同一个域，本文档内不再重复区分。域内容 = 跟单（Copy）+ 排行榜（含竞赛型榜单）；完整 IM / 大厅广场见《内容与社区系统设计》[15]，营销活动/竞赛活动定义见《营销与推荐系统设计》[14]。

## 一、系统定位与边界

### 1.1 系统定位

本域负责：

1. **信号源（Provider）注册与可见性**（交易员账户 / 策略实例映射）
2. **订阅关系与跟单参数**（比例、固定手、净值比、最大滑点、品种过滤）
3. **信号生成与 1→N 放大下单**（经引擎 `SubmitOrder`，`source=SIGNAL`）
4. **业绩统计与交易排行榜**（可审计、带风险披露；含为营销域「交易竞赛」活动提供的竞赛型榜单计算能力）
5. **跟单分成 / 订阅费结算意图**（入账走资金域或 Partner 钱包模式契约，本域不算客户交易盈亏）
6. **可选：信号源主页轻量动态与评论**（非群聊）

以下概念**不属于**本系统：

| 概念 | 归属 |
|------|------|
| Order/Deal/Position 写权威、PreTrade/强平 | 交易 [10] / 风控 [11] |
| 完整群聊、大厅广场、私聊 | 内容与社区 [15] |
| IB/Affiliate 归因与返佣账本 | 代理体系 |
| OpenAPI / 托管策略运行时 / 回测引擎 | 量化系统 [12] |
| 出入金、Credit 发放规则 | 资金 [09] / 营销与推荐活动 [14] |
| **交易竞赛活动定义、报名资格、奖池、发奖规则、活动文案** | 营销与推荐 [14]（本域只提供排行榜计算引擎，见 §8.4） |
| 资讯头条 CMS | 内容与社区 [15] |

### 1.2 行业参考的正确使用方式

| # | 厂商 | 可借鉴 | 本设计采纳 / 取舍 |
|---|------|--------|------------------|
| 1 | **cTrader Copy** | 订阅、比例跟单、提供者业绩 | 采纳订阅+放大；不照搬其结算细节 |
| 2 | **MQL5 Signals** | 信号订阅、业绩展示、分成 | 采纳业绩披露与分成意图；不自研 MQL |
| 3 | **eToro** | Copy + 社交动态 | 采纳轻量动态；完整 IM 不进本域 |
| 4 | **MetaQuotes** | magic 归因 | 与交易域 `magic_number` 对齐（现状见 §3.2） |
| 5 | **Quant/API 平台** | 策略可作信号源 | 量化只生产可选信号，Copy 执行与排行在本域 |

### 1.3 产品定位（拍板）

- **核心**：跟单订阅 + 信号放大 + 排行榜（含竞赛型榜单计算） + 合规披露。
- **轻量社交**：信号源动态/评论可选。
- **不做**：聊天室、自动解析自然语言下单、未授权晒全账户持仓、定义/运营交易竞赛活动本身。

## 二、十五条不可破坏决策

1. **成交只经引擎**：本域禁止直写 `t_order` / `t_deal` / `t_position` / `t_account_ledger`。
2. **同一 PreTrade**：跟单订单与手工单同一风控/保证金/熔断；另叠加订阅级与账户级跟单限额。
3. **幂等强制**：每笔放大单带账户内唯一 `client_order_id` + `request_id`；建议 `{subscription_id}:{signal_id}`。
4. **强制归因**：`source=SIGNAL`，`magic_number` = 订阅级全局唯一 magic（现状为跨域协同推进中的依赖，见 §3.2）。
5. **限速放大**：1 信号 → N 订阅者必须经独立队列与账户节流，保护引擎。
6. **排行榜可复算**：业绩只基于已发生 Deal/权益曲线投影；禁止手填收益率；竞赛型榜单同样禁止手改名次。
7. **合规披露强制**：排行与信号源页必须带风险声明；过去收益不保证未来。
8. **`entity_id` 贯穿**：信号源、订阅、信号、分成、榜单均带实体；禁止跨实体跟单。
9. **反向跟单默认关**：开启需二次确认与额外风险提示。
10. **取消订阅不强制平仓**：默认保留已有仓；可选 `flatten_on_unsubscribe`。
11. **事件经 Outbox**：订阅变更、信号、放大结果、分成意图、榜单结算禁止「写库后尽力发 MQ」。
12. **IM 非本域**：群/大厅归内容与社区；本域最多深链落地页。
13. **晒单非自动**：持仓展示给第三方须用户授权；IM 侧卡片由内容与社区域 [15] 编排（事实来自本域事件）。
14. **不做的事**：凭聊天下单、把排行榜当营销刷数池、绕过引擎改仓。
15. **竞赛边界**：交易竞赛活动的定义、报名资格、奖池、发奖规则权威在营销域 [14]；本域只输出可复算的竞赛型榜单计算结果，不定义活动、不发奖（见 §8.4、§17）。

## 三、边界与跨系统契约

### 3.1 领域边界

| 系统 | 权威数据 | 本域使用方式 |
|------|---------|--------------|
| 交易引擎 | Order/Deal/Position、SubmitOrder | 唯一写交易路径 |
| 行情 | Client Quote | 展示；成交价规则仍在引擎（Core） |
| 账户 | 账户归属、`trade_enabled`（通用交易开关）、`allow_expert`（账户组级 EA 开关）/`expert_enabled`（账户级 EA 开关） | 绑定跟随账户；校验 user↔account。**账户域目前无独立 API 开关，API 来源与手工单共用 `trade_enabled`**（对齐 [`[12]量化系统设计/量化系统设计.md`](../[12]量化系统设计/量化系统设计.md) §3.3 已核实结论）；本域建议新增只读 `allow_copy`（账户或账户组级），供引擎 PreTrade 校验（P2 待账户域评估） |
| 资金 | 入账 RPC / Partner 出款 | 分成结算意图、竞赛奖金意图落地 |
| 量化 | 策略实例、可选 `quant.signal.candidate.v1` | 可作为 Provider 类型之一（STRATEGY_HOOK） |
| 代理 | 归因/佣金 | 弱耦合；跟单分成 ≠ IB 返佣 |
| 内容与社区 | IM、CMS | 深链与系统卡片；不拥有订阅 |
| 营销与推荐 | 活动、推荐、**交易竞赛活动定义与发奖** | 可深链本域 Provider；竞赛场景调用本域榜单计算能力（见 §8.4） |
| 通知 | 模板送达 | 订阅成功/放大失败/竞赛结算等事件 |
| 权限 | RBAC/SoD | 信号源上架审核、人工干预 |

### 3.2 下单契约

```text
Copy Orchestrator
  → 校验订阅有效、限额、品种过滤、账户 allow_copy（如已落地）
  → SubmitOrder(
       source=SIGNAL,
       magic_number=subscription.magic_number,
       client_order_id="{subscription_id}:{signal_id}",
       request_id=...,
       volume=sizing(provider_volume, mode, equity),
       max_deviation=max(subscription.max_deviation, symbol.default_slippage),
       ...
     )
```

| 入口 | `source` | `magic_number` | `client_order_id` 建议 |
|------|----------|----------------|------------------------|
| 跟单放大 | `SIGNAL` | `t_copy_subscription.magic_number` | `{subscription_id}:{signal_id}` |
| 信号源自成交（被跟） | `CLIENT`/`EA`/`API` | 其自身 magic 或 0 | 原路径不变 |

**跨域依赖现状（不写成既成事实）**：[`[10]交易系统/交易系统设计.md`](../[10]交易系统/交易系统设计.md) `t_order.source` 现状枚举为 `CLIENT/EA/API/SYSTEM/DEALER/ADMIN`，尚不含 `SIGNAL`；`magic_number` 字段目前在交易域五张权威表（`t_order`/`t_order_history`/`t_deal`/`t_position`/`t_position_history`）及 `SubmitOrderRequest` 中均不存在。该缺口已由《量化系统设计》§3.3 作为跨域强制依赖声明（与本域协同推进，量化侧的托管策略归因和本域的跟单归因共用同一批段位增量），本文档不重复代交易域改动其权威 DDL，仅声明：**`magic_number` 全局唯一分配规则中，跟单订单一段由本域分配**（对齐《平台跨域契约补充卷》§六）；`SIGNAL` 枚举值与 `magic_number` 字段的合入由本域与交易域、量化域三方协同推进，落地前本域文档中的 `SubmitOrder` 示例均为目标态。

`max_deviation` 字段名对齐 [`[10]交易系统/交易系统设计.md`](../[10]交易系统/交易系统设计.md) `t_order.max_deviation`（DDL 注释「最大不利滑点(point)」），本域历史版本曾用 `slippage_points` 命名，本版统一改为与交易域一致的 `max_deviation`。

| `sizing_mode` | 语义 |
|---------------|------|
| `FIXED_LOT` | 固定手数 |

## 五、状态机与信号源准入分级

### 5.1 信号源状态机与准入分级体系

```text
DRAFT → PENDING_REVIEW → LISTED ⇄ SUSPENDED → DELISTED
```

上架须：风险声明签署、最低交易历史/披露字段齐全、实体合规开关。

`LISTED` 内部再分四级，控制流量与信任等级：

| 等级 | 准入条件 | 权益 |
|------|----------|------|
| `LISTED_SHADOW`（试用） | KYC 通过 + 实盘 ≥ 30 天 | 可被小流量跟单（≤ 100 follower），不计入主排行 |
| `VERIFIED`（验证） | `LISTED_SHADOW` 满 60 天 + 最大回撤 < 20% + 夏普 > 1.0 | 计入主排行，无 follower 上限 |
| `PRO`（专业） | `VERIFIED` 满 180 天 + AUM > 100 万 + 合规审核 | 平台推荐位、分成上限提高 |
| `SUSPENDED` | 违规/回撤超限 | 暂停接受新跟单，现有 follower 可平仓 |

- 准入审核：`PMS_COMPLIANCE_REVIEWER`（对齐《平台跨域契约补充卷》§5.2 裁定的权威角色码）+ 风控复核。
- 试用期 `LISTED_SHADOW`：业绩展示但不进主排行，避免新号刷榜。
- 降级：连续 30 天回撤 > 30% 或违规 → 自动降级。

### 5.2 订阅状态机

```text
CREATED → ACTIVE ⇄ PAUSED → CANCELLED
```

`PAUSED`：不再放大新信号；已有仓按用户配置处理。

### 5.3 放大执行状态机

```text
QUEUED → SUBMITTED → ACCEPTED | REJECTED | SKIPPED
```

`SKIPPED`：过滤规则（品种/限额/账户只平）命中，不计失败风暴。

## 六、信号生成、映射与执行保护

### 6.1 信号生成模式

| 模式 | 说明 |
|------|------|
| `MIRROR_DEAL` | 消费 Provider 账户 `deal.created` 生成信号 |
| `STRATEGY_HOOK` | 量化策略实例经 `quant.signal.candidate.v1` 显式发信号（适配器） |
| `MANUAL_PROVIDER` | 提供者手动发信号（需审核开关，默认关） |

### 6.2 信号→订单映射矩阵

| Provider 动作 | Follower 行为 | 备注 |
|---------------|---------------|------|
| OPEN（开仓） | 按 sizing 开仓 | 经 PreTrade，保证金不足跳过 |
| CLOSE（平仓） | 按比例平仓 | 部分平仓按比例 |
| MODIFY_SLTP | 同步修改 SL/TP | Follower 可设 `max_deviation` |
| PARTIAL_CLOSE | 按比例部分平仓 | `volume × follower_ratio` |
| REVERSE（反手） | 平仓 + 反向开仓 | 两笔独立 |
| PENDING_ORDER | 可选跟随挂单 | 由订阅配置 |
| PENDING_CANCEL | 取消对应挂单 | |

- **MIRROR_DEAL**：消费 `deal.created.v1` 构造 signal，按上表映射。
- **STRATEGY_HOOK**：消费 `quant.signal.candidate.v1` 构造 signal，不经 deal 事件。
- 字段映射：`signal.action`、`signal.symbol_id`、`signal.volume`、`signal.sl`、`signal.tp`、`signal.source_deal_id`。

### 6.3 Sizing 模式公式与边界

| 模式 | 公式 | 边界 |
|------|------|------|
| `FIXED_LOT` | `follower_volume = config.lots` | 最小手数圆整 |
| `RATIO` | `follower_volume = provider_volume × config.ratio` | 除零保护（`provider_volume=0` 跳过） |
| `EQUITY_RATIO` | `follower_volume = follower_equity × config.pct / margin_per_lot` | 净值 < 0 跳过 |
| `MIRROR_BALANCE` | `follower_volume = provider_volume × (follower_balance / provider_balance)` | `provider_balance=0` 跳过 |

- 所有模式圆整到品种 `min_volume` 的整数倍，超 `max_volume` 截断。
- 品种合约差异：跨品种跟单按 notional 等价换算（P2）。

### 6.4 滑点保护状态机

```text
[信号到达] → 计算参考价（Provider 成交价）
  → 提交 SubmitOrder(max_deviation = max(config.slippage, symbol.default))
  → [WAITING_FILL]
  → 成交且 |fill_price - ref_price| <= max_deviation → [FILLED]
  → 引擎拒单（TRD_LIMIT_SLIPPAGE_EXCEEDED）→ [SKIPPED]（SOC_LIMIT_SLIPPAGE_EXCEEDED）
  → 超时未成交 → [SKIPPED]（SOC_LIMIT_SLIPPAGE_EXCEEDED）
```

- 默认滑点：可配，按品种 volatility 动态调整（P2）。
- 连续 N 次 `SKIPPED` → 订阅级熔断（暂停跟单 + 通知 follower）。

### 6.5 爆仓/保证金不足处理矩阵

| Follower 状态 | 处理 |
|---------------|------|
| 保证金充足 | 正常跟单 |
| `margin_call` 中 | 跳过新开仓信号（`SOC_LIMIT_FOLLOWER_THROTTLED`），允许平仓信号 |
| `stop_out` 触发 | 引擎自动平仓，跟单订阅保持，但跳过后续开仓 |
| 净值 < 0 | 订阅自动暂停，通知 follower |
| 连续 3 次 `SKIPPED` | 订阅熔断 1h，通知 |

## 七、Copy Orchestrator 扇出架构

```text
Provider Deal 事件
  → Kafka partition by provider_id
  → Copy Orchestrator Worker（按 provider 分区消费）
    ├── 加载该 Provider 的活跃订阅列表（Redis 缓存 + MySQL 回源）
    ├── 对每个 Follower：
    │   ├── 限速检查（账户级节流）
    │   ├── sizing 计算
    │   ├── 调 SubmitOrder(source=SIGNAL, magic_number=subscription.magic_number)
    │   └── 记录执行结果（FILLED/SKIPPED/REJECTED）
    └── 批量写 t_copy_execution_log
  → 发 copy.execution.skipped.v1（如有 SKIPPED）
```

- **并行度**：单 Worker 顺序处理单 Provider 的 follower（保序）；多 Provider 并行。
- **背压**：follower 队列积压 > 1000 → 丢弃非关键 follower（小仓位）+ 告警（需合规评审）。
- **万级 follower**：单 Provider 1 万 follower 时，按 follower 分片到多 Worker，保序靠 partition。
- **exactly-once**：`(subscription_id, source_deal_id)` 幂等键去重。

## 八、排行榜与竞赛型榜单

### 8.1 原则

1. 指标来自可复算投影：收益率、最大回撤、盈亏比、跟单人数、交易天数等。
2. 窗口：`7D/30D/90D/ALL` 可配；防刷：最小成交次数、最小交易天数。
3. 展示强制：风险声明、是否含模拟账户、是否跟单净值曲线。
4. **禁止**运营手工改名次；仅可下架违规 Provider（审计）。

### 8.2 投影流

```text
deal.*/equity 快照
  → Performance Worker 增量更新
  → t_copy_performance_snapshot
  → Leaderboard 物化视图 / Redis 排行 ZSET
```

### 8.3 多维度评分与防刷模型

**评分模型**（`board_type=PERFORMANCE` 适用）：

```text
score = w1 × return_rate + w2 × (1 / max_drawdown) + w3 × sharpe_ratio
      + w4 × calmar_ratio + w5 × log(aum) + w6 × trading_days
默认权重：w1=0.3, w2=0.2, w3=0.2, w4=0.1, w5=0.1, w6=0.1
```

- 窗口：7D / 30D / 90D / ALL，分别排名。
- AUM 门槛：AUM < 1 万的不进主榜（防小账户刷收益率）。

**防刷规则**（`PERFORMANCE` 与 `CONTEST` 两种榜单通用）：

| 规则 | 说明 |
|------|------|
| 最小成交次数 | 30D 内 < 10 笔不进榜 |
| 最小交易天数 | 30D 内 < 15 天不进榜 |
| 对倒检测 | 关联账户净敞口抵消 → 剔除 |
| 模拟账户隔离 | 依据 [`[05]账户体系/账户体系设计.md`](../[05]账户体系/账户体系设计.md) `t_account_group.environment=2` 判定为模拟账户，单独榜单，不与实盘混 |
| 自跟自检测 | Provider 与 Follower 同 `user_uid` → 剔除 |
| wash trade | 短时间高频对倒 → 标记 + 人工审核 |
| 跟单 AUM 加权 | AUM 高的 Provider 排名加权，反映真实影响力 |

**物化策略**：

- `t_copy_performance_snapshot`（按 provider × day 聚合）。
- Redis ZSET 实时排名，5min 刷新。
- 版本化：排行榜每次更新带 `version`，支持回滚。

### 8.4 竞赛型榜单（Contest Leaderboard）

**边界拍板**：交易竞赛（大赛）活动的名称、宣传文案、报名入口 UX、奖池设置、发奖规则、税务/合规文案，权威均在营销与推荐系统 [14]（[`[14]营销推荐系统/营销与推荐系统设计.md`](../[14]营销推荐系统/营销与推荐系统设计.md) 明确将「竞赛」列为营销域拥有的能力，并在其误区表中点名「竞赛榜 = 跟单排行」是错误认知）。**本域不定义竞赛活动本身**，只把已经具备的「排行榜计算引擎」扩展出一种新的 `board_type=CONTEST`，供营销域的竞赛活动场景复用，避免重复建设一套独立的计分/防刷/物化能力。

**「平时」与「大赛」两种榜单的本质区别**（`board_type` 两个取值对应两种完全不同的计分基础，不是同一份数据换个窗口）：

| 维度 | `PERFORMANCE`（平时排行，§8.1-8.3） | `CONTEST`（大赛排行，本节） |
|------|-----------------------------------|------------------------------|
| 交易载体 | 信号源自己日常使用的真实/模拟账户，历史滚动、无终点 | 每场大赛专属开设的**大赛账号**（见下），报名时清零起点，活动结束即定榜 |
| 计分窗口 | 滚动窗口（7D/30D/90D/ALL），持续更新，无「收官」动作 | 固定 `run_start_at ~ run_end_at`，到点**强制平仓**后一次性定榜，不再变化 |
| 计分方式 | 多维度评分模型（收益率+回撤+夏普+…，§8.3） | 默认按**大赛账号最终盈亏金额**排名（起点资金统一，见下），可选切换收益率 |
| 榜单归属 | 本域自主运营（信号源业绩） | 营销域发起活动，本域只提供计算引擎（§17 边界表） |

**大赛账号（Contest Account）**：大赛不复用参赛者现有的常规交易账户，而是要求参赛者持有一个**专属绑定本场大赛的账号**，理由：常规账户可能已有历史仓位/盈亏，直接拿"当前净值"做基准既不公平也难以审计；专属账号从开户瞬间起点唯一、干净。

- 开户方式：复用账户域既有开户能力（[`[05]账户体系/账户体系设计.md`](../[05]账户体系/账户体系设计.md) `POST /api/v1/trading-accounts/demo` 模拟户开户，或实盘户常规开户流程）。活动配置层用人类可读的 `group_code`（如 `CONTEST_STD_USD`）指定专属账户组，调用账户域开户接口前须解析为对应的 `account_group_id`（`int64`，账户域 `OpenAccountRequest`/demo 开户接口只接受 ID，不接受 `group_code`，与 [14]§8.3 一致的解析步骤）；起始资金统一取该账户组 `demo_initial_balance`（模拟赛）或活动规定的最低入金（实盘赛，入金本身仍走资金域正常出入金链路，本域不碰钱）。
- 账号与账户域的关系：大赛账号本质仍是账户域的一个合法 `t_trading_account`，只是其 `group_id` 指向的账户组打了竞赛标记；本域不新增账户实体，只在 `t_copy_contest_enrollment.account_id` 里记录这一层绑定关系。
- 赛后处置：账号是否保留、转为普通模拟户、还是按 `demo_expire_days` 自动失效，属账户域现有生命周期管理，本域不新增机制。

**能力边界**：

| 能力 | 归属 |
|------|------|
| 竞赛活动定义（名称、赛制、时间、奖池、文案、账户组/起始资金规格） | 营销域 [14] |
| 参赛资格判定（KYC 等级、地域、是否需实名） | 营销域 [14]（可读本域账户绑定关系） |
| 大赛账号开户（调账户域 API） | 营销域 [14] 编排，账户域 [05] 执行 |
| 报名绑定、计分、排名、防刷、到点强制平仓、结果冻结 | **本域**（§8.4 本节） |
| 名次→奖励映射、实际发奖 | 营销域 [14] 判定 + 资金域 [09] `AdjustCredit` 执行 |

**LeaderboardBoard 扩展字段**（见 §11 `t_copy_leaderboard` DDL）：`board_type`（`1-PERFORMANCE` 常规业绩榜 / `2-CONTEST` 竞赛型榜单）、`campaign_id`（仅 `CONTEST`，关联营销域活动 ID，弱引用不做外键）、`rank_metric`（`1-` 最终盈亏金额，默认 / `2-` 收益率，仅多起点资金规格场景使用）、报名与运行时间窗口字段。

**竞赛型榜单专属状态机**（`board_type=CONTEST` 才适用；`PERFORMANCE` 榜单沿用 §8.1-8.3 的滚动窗口逻辑，不经过此状态机）：

```text
ENROLLING → RUNNING → SETTLING（强制平仓收官） → FINALIZED → ARCHIVED
```

- `ENROLLING`：营销域已创建活动并调用本域接口开出榜单壳子，报名窗口开放；每个参赛者须先持有一个大赛账号，本域记录 `account_id + baseline_equity`（= 大赛账号开户时的初始资金，唯一且干净）到 `t_copy_contest_enrollment`。
- `RUNNING`：报名关闭，正式比赛期；实时按大赛账号当前净值 - `baseline_equity` 计算浮动盈亏，Redis ZSET 滚动刷新（仅供参赛者查看进度，非终榜）。
- `SETTLING`（**强制平仓收官**）：`run_end_at` 到达，Orchestrator 对每个参赛大赛账号下的**全部持仓**发起强平指令——复用交易引擎既有的系统强平通道（`SubmitOrder(source=SYSTEM, intent=CLOSE)`，与 [`[10]交易系统/交易系统设计.md`](../[10]交易系统/交易系统设计.md) Stop Out 系统单同源，本域不新造引擎能力），逐账号等待清仓 Deal 全部确认（消费 `position.closed.v1`）；清仓后以账户最终净值定榜，名次暂未冻结，进入防刷复核（对倒、自跟自检测、wash trade 人工审核）。
  - **强平失败处理**：STALE/无报价导致个别账户无法平仓 → 按退避重试；超过重试上限仍未清仓 → 该账户标记 `DISQUALIFIED` 并转人工处理（不得带着未平仓浮亏/浮盈强行定榜），不影响其余账户正常定榜。
- `FINALIZED`：名次冻结（基于强平后的最终盈亏 = `final_equity - baseline_equity`），产出 `copy.leaderboard.contest_finalized.v1`，营销域据此触发发奖意图。
- `ARCHIVED`：结果只读留存，供审计与合规查询；大赛账号生命周期交回账户域处理（见上）。

**计分方式**：默认 `rank_metric=1`（最终盈亏金额），因为同一场大赛的参赛账号起始资金统一，直接比较绝对盈亏即「谁赚的钱多」，语义清晰且与营销活动的直觉一致；若某场活动允许多档起始资金规格（如不同报名费对应不同起始资金），营销域可将 `rank_metric` 配置为收益率，两种口径均由本域计算，不需要额外系统。防刷规则复用 §8.3 表格（对倒检测、自跟自检测、模拟账户隔离、wash trade 审核）。

**跨域协作流程**：

```text
营销域创建竞赛活动（含账户组/起始资金规格）
  → 调本域 API 创建 CONTEST 榜单（携带 campaign_id）— ENROLLING
  → 营销域完成参赛资格判定 → 编排账户域为参赛者开设大赛账号
  → 营销域调本域接口，把 {account_id, baseline_equity} 写入 t_copy_contest_enrollment
  → 报名截止，本域切换 RUNNING，开始滚动计分（仅供展示，非终榜）
  → run_end_at 到达，本域切换 SETTLING：对全部参赛账号强制平仓 → 防刷复核 → FINALIZED
  → 本域发 copy.leaderboard.contest_finalized.v1（含最终名次与盈亏列表）
  → 营销域消费该事件，判定名次→奖励映射，生成发奖意图
  → 资金域执行 AdjustCredit（沿用 §9 分成结算模式：本域及营销域均只产出意图事件，实际入账权威在资金域）
```

## 九、分成与费用结算

### 9.1 费用类型

| 类型 | 说明 | 落地 |
|------|------|------|
| 订阅费 | 跟随者周期付费 | 资金扣款/站内结算意图 |
| 利润分成 | 高水位利润抽成 | 日/周结算意图事件 |
| 平台抽成 | 从上述费用切分 | 公司 journal（资金域） |

本域产出 `copy.fee.accrued.v1`；**实际扣款/入账**走资金引擎契约。可复用代理「钱包入账」模式，但账本主体不是 IB Program。

### 9.2 高水位结算算法

```text
高水位（High Water Mark）：
  performance_fee = fee_rate × max(0, current_profit - historical_max_profit)

  按结算周期（月/季）计算：
    若当前净值 > 历史最高净值：
      fee = fee_rate × (current_equity - hwm_equity)
      hwm_equity = current_equity
    否则：
      fee = 0（亏损期不收费，待回水）

  亏损回补：后续盈利需先补回亏损再计费
```

- 订阅费：固定，按周期收。
- 业绩分成：高水位算法，经 `copy.fee.accrued.v1` → 资金域 `AdjustCredit` 入账。
- 争议：Follower 可申诉，工单走营销域，退款经资金域。
- 幂等：`fee_id = {subscription_id}:{period}`。

## 十四、容灾与 Copy Kill Switch

- Copy Kill Switch：全域/Provider 级/Follower 级，`PMS_RISK_OFFICER`（对齐补充卷权威角色码）四眼触发。
- 触发后：暂停所有新跟单信号，允许平仓，通知所有 follower。
- 排行榜只读降级：Orchestrator 故障时排行榜（含竞赛型榜单）切只读快照。
- 积压降级：队列 > 10 万 → 丢弃非关键 follower（需合规评审）。

## 十五、合规披露

- 订阅页强制展示：CFD 风险、过往业绩不代表未来、计算方法说明。
- Provider 主页：业绩计算口径、AUM、最大回撤、夏普。

## 十七、明确不做与常见误区

| 误区 | 正确做法 |
|------|----------|
| 「跟单放在量化里一起做」 | 执行与排行在本域；量化只做可选信号源 |
| 「排行榜单独一个系统」 | 本域业绩投影 |
| 「群聊算社交交易」 | IM 在内容与社区 |
| 「聊天里说买入就下单」 | 禁止 |
| 「运营改排行名次冲活动」 | 禁止；活动发奖走营销与推荐 + 资金 |
| 「跟单绕过保证金」 | 同一 PreTrade |
| 「竞赛活动/发奖归本域」 | 竞赛活动定义、报名资格、奖池、发奖规则权威在营销域；本域只提供竞赛型榜单计算引擎（见 §8.4） |
| 「竞赛榜 = 跟单排行」 | 两者是不同的 `board_type`：`PERFORMANCE` 是信号源持续业绩榜，`CONTEST` 是营销活动驱动的时间窗口竞赛榜，底层复用同一计算引擎但业务归属不同 |
| 「大赛直接用参赛者现有账户」 | 大赛须用专属开设的大赛账号（起点资金统一、无历史仓位污染），见 §8.4 |
| 「大赛结束靠当前净值快照定榜」 | 必须先 `SETTLING` 强制平仓收官，用平仓后的最终净值定榜；不能用含未实现盈亏的净值定榜（浮盈浮亏可被瞬时报价波动或操纵影响） |
