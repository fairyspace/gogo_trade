# 内容与社区系统设计 — 终版 v2.0

> **版本**：v2.0 Final｜2026-07-22
> **定位**：经纪平台的**资讯/公告/投教内容权威** + **论坛发帖与话题广场权威** + **社群 IM（大厅/广场/群聊）权威**。
> **对标**：App 资讯流、个股论坛、投教中心、交易社群聊天（能力级，不照搬多资产证券社区产品形态）。
> **运行时分层**：Content CMS（资讯/公告/NLP处理）· Forum（论坛/话题/热度/Feed）· IM Service（会话/消息/审核）· Moderation（内容审核）；推送复用通知域。方向设计不锁定实现语言。
> **域地图**：见 [`[总览]经纪平台域地图.md`](../[总览]经纪平台域地图.md)。

---

## 一、系统定位与边界

### 1.1 系统定位

「社区」在本域特指**聊天与内容社区**（论坛发帖、话题广场、群聊大厅），不是 [13] 的跟单「社交交易（Social Trading）」。

1. **内容 CMS**：`NEWS`/`ANNOUNCEMENT`/`EDUCATION`/`LEGAL_NOTICE`/`EARNINGS` 等，版本化与审批发布，标的关联与情感分析
2. **论坛与话题广场**：个股/主题板块、发帖、多级评论、热度排行、Feed 信息流（新增，弥补旧稿只有 IM 没有异步内容社区的缺口）
3. **社群 IM**：大厅/频道/群/私聊、成员角色、禁言、举报审核、系统卡片
4. **社区身份**：公开人设（昵称/头像/简介）、关注关系、KOL/分析师认证治理
5. **外链 IM**（可选）：Telegram 等绑定与合规披露

### 1.2 不属于本系统

| 概念 | 归属 |
|------|------|
| 营销活动规则、发奖、CRM Desk | 营销与推荐 [14] |
| 「推荐哪个 content_id 放在运营位卡片」 | 营销与推荐 [14]（`CONTENT_REC` 场景只做卡片位推荐，不写正文；本域拥有正文与频道自身信息流排序，见 §八） |
| 跟单、排行榜、信号放大、Provider 主页轻量动态 | 社交交易 [13]（其 Provider 动态是"跟单信号源主页"专属，若需升级为群聊/@全体/已读/会话推送 → 迁入本域 IM，见 [13] §十） |
| 多通道 Push/短信送达 | 通知 [03] |
| 成交写权威、持仓真相 | 交易引擎 / 账户域 |
| 标的身份、合约规格 | 品种体系 [06]，本域只读引用 `symbol_id` |
| 行情报价权威 | 行情体系 [08]，本域标的情绪指数是衍生分析，不是报价 |

## 二、十二条不可破坏决策

1. **`entity_id` 贯穿**内容可见性与群；禁止跨实体拉群、跨实体发帖。
2. **内容版本化 + 审批**后方可 `PUBLISHED`。
3. **IM/论坛禁止自动下单**：聊天与帖子不得变为 `SubmitOrder`。
4. **晒单须授权**：交易只读脱敏 API + 用户授权 → `SHARE_CARD`；帖子/评论关联持仓同理。
5. **跟单仅深链**：跳转社交交易落地页；订阅/信号放大不在本域。
6. **诱导交易/喊单/内幕/操纵**：审核队列 + 关键词/举报 + 金融社区专项监控（见 §十二）。
7. **推送不拥有正文**：`content.published.v1` 触发通知；正文权威在本域。
8. **事件经 Outbox**。
9. **PII 最小必要**：社区身份是本域治理的公开人设数据，不是用户域身份 PII 的副本；消息/帖子可审计保留（按时限，见 §十八）。
10. **外链模式**：可不自研消息栈，绑定配置仍属本域。
11. **与营销/推荐解耦**：运营位卡片可引用 `content_id`；活动资格、推荐排序策略不在本域；本域自身的资讯/论坛信息流排序权威在本域（见 §八）。
12. **不做**：CRM、赠金引擎、跟单放大、排行榜计算、自建 PII 副本、虚拟积分/VIP 经济。

## 三、跨域契约

### 3.1 契约引用声明

- **错误码**：遵循《平台跨域契约补充卷》§三，使用 `CMS_*` 前缀（完整列表见 §19）。
- **事件**：遵循补充卷 §四；本域产出/消费事件的登记状态见 §16。
- **角色**：内容编辑/审核引用补充卷 §五 `PMS_CONTENT_EDITOR`、`PMS_CONTENT_REVIEWER`（二者互斥，均已核实登记）；执法调取审批引用 `PMS_COMPLIANCE_OFFICER`。
- **ID**：遵循补充卷 §六，`content_id`/`post_id`/`message_id`/`conv_id` 由本域生成。
- **容灾/容量/监控**：遵循补充卷 §十/§十一/§十二。

### 3.2 领域边界

| 系统 | 使用方式 |
|------|----------|
| 通知 [03] | 消费 `content.published.v1` 等做推送；本域不拥有送达通道 |
| 营销与推荐 [14] | 只读引用 `content_id` 做运营位卡片推荐（`CONTENT_REC` 场景）；本域不算推荐分，不参与活动资格判定 |
| 社交交易 [13] | IM 深链跳转 Provider 主页；可消费 `copy.execution.skipped.v1`/`deal.created.v1` 生成系统卡片；Provider 轻量动态与本域论坛互不合并 |

### 3.3 与 [14] 内容推荐的边界（显式声明）

[`[14]营销推荐系统/营销与推荐系统设计.md`](../[14]营销推荐系统/营销与推荐系统设计.md) 的 `CONTENT_REC` 场景**只做"推荐哪个 `content_id` 放在首页/App 内某个运营位卡片"**，不写文章正文，不拥有内容元数据权威。本域的"资讯频道信息流"（打开新闻 Tab 看到的列表）与"个股论坛帖子列表"是**频道自身的内容排序**，服务对象、触发路径、排序目标均与 [14] 的运营位卡片推荐不同，两者字段与表不重复（见 §八）。

## 四、内容 CMS

### 4.1 内容类型与发布流

| `content_type` | 用途 |
|-----------------|------|
| `NEWS` | 金融资讯头条 |
| `ANNOUNCEMENT` | 系统/产品公告 |
| `EDUCATION` | 投教 |
| `LEGAL_NOTICE` | 运营向提示（强制协议仍在准入域） |
| `EARNINGS` | 财报解读（标的关联，见 §4.3） |

发布流：`DRAFT → IN_REVIEW → PUBLISHED ⇄ UNPUBLISHED → SUPERSEDED`。

- **版本管理**：`t_content_version`，每次编辑生成新版本；已发布版本可回滚到上一版本（当前 `SUPERSEDED` → 目标版本 `PUBLISHED`）。
- **定时发布**：`publish_at` 字段，定时任务扫描发布。
- **多语言 fallback**：`locale` 优先匹配，缺失回退 `default_locale`。
- **辖区可见性**：`t_content_visibility`（`content_id` × `entity_id` × `jurisdiction`）矩阵控制。
- **附件安全**：图片/附件经病毒扫描 + XSS 消毒（富文本）+ CDN 分发。
- **SEO**：meta 字段（title/description/og）由编辑填写。

### 4.2 NLP 内容处理流水线

```text
原始内容输入（采集/编辑提交）
  → Step1 基础清洗：去 HTML 标签、去广告、编码统一、语言检测
  → Step2 NLP 理解：分词/词性标注、命名实体识别（公司名/人名/机构）、关键词提取、自动摘要
  → Step3 金融专项理解：
      标的关联 → 匹配 [06] symbol_id/symbol_code（本域不新增 market/symbol_type 分类）
      事件分类 → EARNINGS/MERGER/DIVIDEND/POLICY/MACRO
      情感分析 → 正面/负面/中性 + 情感强度
      影响评估 → 短期/长期
  → Step4 内容标签化：行业标签、主题标签、重要程度(1-5)、时效性标记
  → Step5 合规审核：违禁词过滤、虚假信息识别、内幕信息检测（见 §十二）、投资建议合规标注
  → 写入存储 + 建立索引 + 触发 content.published.v1
```

- NLP 处理结果落入 `t_content_article` 的 `keywords`/`entities`/`sentiment`/`event_type` 等字段（见 §14 DDL）。
- 标的关联落入独立关联表 `t_content_symbol_relation`（一篇内容可关联多个标的，含相关度与主关联标记）。

### 4.3 财报解读（`EARNINGS` 子类型）

精简版结构化解读，服务于本平台支持的股票类 CFD 标的（`symbol_id` 关联 [06]，不做多市场/多品种分类）：

```text
财报采集（人工录入或对接数据源）
  → 结构化字段落 t_content_earnings_detail（营收/净利润/EPS/ROE 等核心指标 + 同比）
  → AI 生成摘要/亮点/风险点/评级（beat/meet/miss）
  → 主表 t_content_article（content_type=EARNINGS）关联发布，走 §4.1 标准发布流
```

## 五、社区身份与关注关系

### 5.1 社区身份档案（非 PII）

`t_content_user_profile`：本域治理的**公开社区人设**，与用户域身份数据是两个概念——[`[04]用户体系/用户体系设计.md`](../[04]用户体系/用户体系设计.md) 的 `t_user_oauth.nickname` 仅是"第三方昵称(首次授权拉取)"，不是可自定义的社区公开昵称；本表不复制手机号/证件等身份字段，仅以 `user_uid` 关联。

- 字段：昵称、头像、简介、社区等级（按活跃度/发帖质量累计）、关注数/粉丝数、发帖数、获赞数。
- **明确不采纳**：参考设计里的金币余额、VIP 到期时间等虚拟积分/会员经济字段——本平台当前无此会员体系，不由本域定义，避免虚构不存在的经济系统。

### 5.2 KOL / 分析师认证治理

- `verify_type`：`ANALYST`/`FUND_MANAGER`/`EXPERT`/`INSTITUTION`。
- 认证流程：用户提交执业资格材料 → `PMS_CONTENT_REVIEWER` 人工审核 → 通过后打 `verify_badge` 展示标签。
- **明确边界**：认证是本域对"社区展示身份"的人工治理标识，**不对接**外部监管机构执业资格数据库做实时核验，材料真实性由审核流程与用户承诺函兜底。
- 认证历史与历史发帖表现公开展示，防止虚假收益晒单（晒单本身仍需 §十经授权脱敏）。

### 5.3 用户关注关系

`t_content_user_follow`：社区社交图谱（关注/粉丝），用于 §八 Feed 个性化召回。

**明确边界**：与 [13] 跟单域的"关注/订阅 Provider"是两个不同的关注关系——本域关注是"关注某用户的论坛发帖/动态"，[13] 的订阅是"跟单执行意图"，两表不合并、不互相同步。

## 六、论坛与话题广场（新增）

### 6.1 定位

当前旧稿"社群 IM"章节只有会话制聊天，缺失"发帖 + 评论 + 板块 + 话题"这种异步信息流内容社区能力，本次补齐这一真实缺口。

- **板块**：个股板块（关联 `symbol_id`）、主题板块、综合板块。
- **帖子**：图文/长文，可关联标的、可标注看多看空方向（用户主观观点，非投资建议）。
- **评论**：多级（最多 3 级），支持 @回复。
- **话题**：`#话题#` 聚合，官方话题（运营创建）与用户话题。

### 6.2 与 [13] Provider 轻量动态的边界

[`[13]跟单与排行系统/社交交易系统设计.md`](../[13]跟单与排行系统/社交交易系统设计.md) §十已声明 `t_copy_provider_post`/`t_copy_provider_comment` 是"信号源主页轻量动态"，专属于该 Provider 的业绩/观点展示页，**不是**通用论坛；本域的板块/帖子/评论是面向全体用户的公开论坛，两者数据不互通、不合并。若产品侧后续需要把 Provider 动态升级为可群聊/@全体/会话推送的能力，按 [13] 既有声明迁移到本域 IM，而不是并入本域论坛。

### 6.3 热度与发布流

- 发布流：`审核前置校验（§十一）→ 发布（PUBLISHED）→ 置顶/加精（运营操作）→ 用户可删除（软删，审计保留见 §十八）`。
- 热度分算法见 §八（帖子与新闻共用同一套热度模型，权重系数按内容类型区分）。

## 七、社群 IM（大厅 / 广场）

### 7.1 能力

- 广场/大厅频道、主题群、私聊
- 角色：`OWNER`/`ADMIN`/`MEMBER`/`GUEST`（私聊）
- 禁言、踢人、举报、审核台
- 系统消息与晒单卡片

### 7.2 模型纲要

```text
ImConversation（DIRECT / GROUP / PLAZA_CHANNEL）
ImMember
ImMessage（text/image/system/share_card）
ImModerationCase → 统一并入 §十一 t_content_moderation_case（content_type=IM_MESSAGE）
ImExternalBinding
ImShareAuthorization
```

### 7.3 连接层架构

```text
客户端 ──WSS──► [IM Gateway 集群]
                │ 鉴权：Bearer Token
                │ 心跳：30s ping/pong
                │ 连接路由：按 user_uid → node_id（Redis 注册中心）
                ▼
            [消息路由 Worker]
                │ 按 conversation_id partition 消费
                │ 写消息存储 + 推送到成员连接节点
                ▼
            [Gateway 节点] ──推送──► 在线成员
                │ 离线成员：写离线消息表 + 通知域推送
                ▼
            [消息存储]（分片存储 + 热缓存）
```

- **连接数**：单节点 5 万，集群 50 万。
- **重连**：客户端带 `last_msg_id` 重连，Gateway 补发未读。
- **消息 ID 游标**：每会话单调递增 `msg_seq`，客户端按 seq 去重 + 排序。

### 7.4 消息投递语义

| 语义 | 实现 |
|------|------|
| at-least-once | 客户端 ack 后服务端才标记送达；未 ack 重发 |
| 有序 | 同会话按 `msg_seq` 单调，按 `conversation_id` partition |
| 幂等发送 | 客户端 `client_msg_id`，服务端去重 |
| 编辑 | 消息可编辑（2min 内），标记 `edited_at`，保留原内容审计 |
| 撤回 | 消息可撤回（2min 内），标记 `recalled_at`，客户端移除展示 |

### 7.5 成员角色与权限

| 角色 | 权限 |
|------|------|
| OWNER | 全部，含转让、解散 |
| ADMIN | 踢人、禁言、设管理员、发系统消息 |
| MEMBER | 发消息、@、举报 |
| GUEST（私聊） | 发消息 |

- 入群审批：`join_mode`（`OPEN`/`INVITE`/`APPROVAL`）。
- 邀请链接：`invite_link`（含过期时间 + 使用次数）。
- 全员禁言：ADMIN 可开启，仅 ADMIN 可发言。

## 八、内容 Feed 个性化（新增）

### 8.1 与 [14] 运营位推荐的边界（重申）

本章是**内容频道自身的信息流排序**（资讯 Tab 列表、个股论坛帖子列表），与 [14] 的运营位卡片推荐（`CONTENT_REC` 场景，只读引用 `content_id`）完全独立：本域拥有正文与互动数据权威，排序方法可以类似（多路召回+排序），但表与曝光日志不共用，触发路径也不同（本域是用户主动进入频道浏览，[14] 是运营位卡片曝光）。

### 8.2 热度分算法

```text
热度分 = (互动分 × 来源权重因子) / (发布时间距今小时数 + 2)^衰减系数 × 内容质量系数

互动分权重：view=0.1, like=3, comment=5, share=8, collect=4, follow_from=10, report=-10
来源权重：官方媒体=1.5，认证分析师/KOL=1.3，普通用户=1.0，历史违规用户=0.5
衰减系数：新闻类=1.8（快速衰减），财报类=1.2（慢速衰减），论坛帖子=1.5（中速衰减）
内容质量系数：精华帖=1.5，优质标签=1.2，违规历史=0.3
```

### 8.3 Feed 召回与排序

```text
多路并行召回：
  - 关注内容召回（关注用户的帖子/资讯，§5.3 关注关系）
  - 持仓/自选标的相关召回（读账户域持仓 + 标的关联表）
  - 行为兴趣召回（浏览/点击历史相似内容）
  - 热门内容兜底召回
  - 突发重要新闻插入（`is_breaking=1` 实时插入，不受召回排序延迟）
→ 合并去重
→ 排序（规则打分，模型排序为 P2）
→ 策略过滤：频控（§8.4）、去重、多样性、合规过滤（下架/审核中内容不进入 Feed）
→ 输出 Feed（分页加载）
```

### 8.4 频控与去重

`t_content_feed_exposure`（`user_uid` × `content_id` × 曝光时间）：24h 内同内容不重复曝光；此表服务本域频道自身 Feed，与 [14] 的 `t_recommend_exposure`（运营位卡片曝光）是两张独立表，不合并。

## 九、标的情绪指数（新增）

`t_content_sentiment_daily`：按 `symbol_id` + 日期统计的综合情绪指数，**明确是本域治理的衍生分析数据**，不是行情权威（行情报价权威在 [08]），不进入 quote-engine 热路径。

```text
新闻情绪：当日相关新闻数、正/负/中性新闻数、新闻情感得分(-1~1)
社区情绪：论坛帖子数、看多/看空帖子数、多空比、社区情感得分
综合情绪：综合情绪得分、情绪等级(1-5)、关注热度得分、热度排名
```

- 计算频率：T+1 批处理为主，突发事件可触发当日重算。

## 十、系统卡片与晒单授权

- 消费 [13] `copy.execution.skipped.v1`、交易域 `deal.created.v1`（经授权）→ 生成系统卡片，推送到指定频道/私聊。
- 卡片 schema：`{type, provider_id/deal_id, summary, timestamp}`。
- **禁止**卡片含下单指令；跟单仅深链 [13]。

```sql
CREATE TABLE t_content_share_authorization (
  id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_uid VARCHAR(64) NOT NULL COMMENT '发起晒单授权的用户UID',

  expires_at BIGINT UNSIGNED NOT NULL COMMENT '默认 7 天',
```

- 授权生命周期：授权 → 展示 → 过期/撤销。
- 脱敏：金额可配精度、账户 ID 哈希。
- 交易只读 API：`GET /api/v1/internal/share/{auth_id}/deal`（mTLS），返回脱敏字段。
- 论坛发帖关联晒单同样经此授权链路，不单独建流程。

## 十一、内容审核与反垃圾

### 11.1 审核流程

```text
用户发消息/发帖/评论/提交内容
  → 机审（关键词 + NLP 模型 + 图片识别）
    ├── 命中 → 标记 pending_review → 人审队列
    │   ├── 人审通过 → 发布
    │   └── 人审拒绝 → 删除 + 通知作者
    └── 未命中 → 发布
  → 举报触发 → 入人审队列
```

统一审核记录表 `t_content_moderation_case`（覆盖 `content_type ∈ {ARTICLE, POST, COMMENT, IM_MESSAGE}`），替代旧稿分散的 `ImModerationCase`，避免多张相似审核表分裂治理口径。

### 11.2 诱导交易判定规则

| 规则 | 处置 |
|------|------|
| 含"稳赚""包赔"等绝对化用语 | 拦截 |
| 含具体标的 + 买卖方向 + 价格 | 标记 + 人审 |
| 未授权投资建议（无牌照） | 拦截 + 通知合规 |
| 喊单刷屏（同用户高频同质内容） | 自动禁言 1h |

### 11.3 处罚梯度

| 次数 | 处罚 |
|------|------|
| 1 | 警告 + 删除 |
| 2 | 禁言 1h |
| 3 | 禁言 24h |
| 4 | 禁言 7 天 |
| 5 | 封号（转用户域冻结） |

### 11.4 反垃圾

| 策略 | 说明 |
|------|------|
| 速率限制 | 新用户 1min 内 ≤ 5 条；老用户 ≤ 30 条/min |
| 相似度检测 | 5min 内同板块/群相似内容 > 80% → 折叠 + 告警 |
| 链接/二维码过滤 | 新用户禁发链接；链接经安全扫描 |
| 举报聚合 | 同用户被举报 ≥ 3 次 → 自动禁言待人审 |
| 蜜罐 | 蜜罐频道/板块诱捕机器人 → 自动封 |

## 十二、金融社区合规增强

在 §十一 通用审核之上，金融社区场景的专项监控：

| 规则 | 说明 |
|------|------|
| 内幕信息监控 | 关键词监控财报/并购提前泄露；发帖前后异常交易行为关联分析（只读交易域时间线，不做交易行为权威判定，仅作合规线索） |
| 市场操纵防控 | 拉群炒股（同群多账号集中喊多/喊空同一标的）监控；社区情绪与异常行情联动监控（消费 §九情绪指数 + 行情异常告警） |
| 投资建议合规 | 非持牌用户不得发布明确买卖建议；发布投资相关内容需带"不构成投资建议"标注；AI 检测违规建议自动拦截 |
| KOL 言论加权审核 | 认证 KOL 的内容默认进入更严格的人审复核（避免"权威账号"造成更大误导） |

- 触发上述规则的内容进入 `t_content_moderation_case`，`trigger_type=MACHINE`，风险类型标注 `INSIDER_INFO`/`MARKET_MANIPULATION`。
- 审计保留时限与 §十八 一致（**7 年**），不采用参考设计里"6 个月"的更短留存（本域取更严格标准）。

| 审核队列 SLA | < 4h |
| 单频道/板块上限 | 10 万成员 |
| 全站并发连接 | 50 万 |
| 消息 TPS | 1 万 |

| 场景 | 降级 |
|------|------|
| IM Gateway 故障 | 切备集群，客户端重连 |

| Feed 服务故障 | 返回默认热门列表 |

## 十八、审计保留与执法协同

- 消息/帖子/评论审计保留：**7 年**（合规），按月分区 + 冷归档。
- 执法调取：经 `PMS_COMPLIANCE_OFFICER` 审批 + 法律流程，导出指定会话/帖子。
- 删除保留：用户删除仅前端隐藏，后端保留审计 + 标记 `deleted_at`。
- §十二 金融社区专项监控命中的审核记录同等适用 7 年留存，不因内容类型缩短时限。

## 二十、明确不做与常见误区

| 误区 | 正确做法 |
|------|----------|
| 「IM/论坛算社交交易/排行社区」 | 本域；跟单排行在 [13] |
| 「新闻模块附带 CRM」 | CRM 在营销与推荐 [14] |
| 「群里/帖子喊单自动跟」 | 禁止 |
| 「通知系统存文章正文」 | 正文权威在本域 |
| 「本域信息流排序 = [14] 运营位推荐」 | 两者服务对象不同，见 §3.3，字段与曝光日志不共用 |
| 「标的关联自建市场/品种分类」 | 一律引用 [06] `symbol_id`，不新增 market/symbol_type |
| 「社区昵称直接用用户域 `t_user_oauth.nickname`」 | 本域自建公开人设档案 `t_content_user_profile`，不复用第三方昵称 |

**参考设计（`头条社区参考.md`）不采纳清单**：

| 参考设计里的做法 | 为什么不采纳 |
|------|------|
| 多市场（SH/SZ/HK/US）+ 多品种类型（stock/fund/futures）分类 | 本平台标的模型统一在 [06]，用 `symbol_id` 已覆盖 FX/贵金属/加密/股票 CFD 等，不需要重建市场分类 |
| 金币余额、VIP 到期等虚拟积分/会员经济 | 本平台当前无此会员体系，非本域权威可自行定义 |
| 自选品种/watchlist | 平台尚无归属权威表（[08] `t_price_alert` 语义不同），本域不新建以免制造新的归属歧义 |
| 红包消息、投顾专属频道类型 | 超出当前"聊天与内容社区"定位，未有业务需求验证 |
| IM 专用存储技术选型（特定 MQ/自研协议） | 遵循方向设计不锁定实现语言的既有原则 |
| `mobile_hash`/`id_card_hash` 类自建 PII 副本思路 | 本域社区身份不复制用户域身份字段，一律以 `user_uid` 关联 |
