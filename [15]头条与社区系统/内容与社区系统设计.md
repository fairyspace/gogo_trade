# 内容与社区系统设计（权威版）

> **版本**：v1.0 Final  
> **定位**：经纪平台的**资讯/公告/投教内容权威** + **社群 IM（大厅/广场/群聊）权威**。  
> **对标**：App 资讯流、投教中心、交易社群聊天（能力级）。  
> **运行时分层**：Content CMS · IM Service（会话/消息/审核）；推送复用通知域。**方向设计不锁定实现语言。**  
> **域地图**：见 [`[总览]经纪平台域地图.md`](../[总览]经纪平台域地图.md)。

---

## 〇、为何独立成域

| 本域 | 不放这里 |
|------|----------|
| 新闻头条、公告、投教 CMS | CRM 任务 / 赠金活动规则（营销与推荐） |
| 群聊大厅、广场频道、私聊 | 跟单订阅 / 交易排行榜（社交交易） |
| 消息审核、晒单授权卡片 | 通知通道实现（通知域只送达） |

「社区」在本域指 **聊天与内容社区**，不是跟单「社交交易」。

---

## 一、系统定位

1. **内容 CMS**：`NEWS` / `ANNOUNCEMENT` / `EDUCATION` 等，版本化与审批发布  
2. **社群 IM**：大厅/频道/群/私聊、成员角色、禁言、举报审核、系统卡片  
3. **外链 IM**（可选）：Telegram 等绑定与合规披露  

**不属于本系统**：

| 概念 | 归属 |
|------|------|
| 营销活动规则、发奖、CRM Desk | 营销与推荐 [14] |
| 跟单、排行榜、信号放大 | 社交交易 [13] |
| 推荐「推哪篇文章」的策略 | 营销与推荐（可引用本域 content_id） |
| 多通道 Push/短信送达 | 通知 [3] |
| 成交写权威 | 交易引擎 |

## 二、十二条不可破坏决策

1. **`entity_id` 贯穿**内容可见性与群；禁止跨实体拉群。  
2. **内容版本化 + 审批**后方可 `PUBLISHED`。  
3. **IM 禁止自动下单**：聊天不得变为 `SubmitOrder`。  
4. **晒单须授权**：交易只读脱敏 API + 用户授权 → `SHARE_CARD`。  
5. **跟单仅深链**：跳转社交交易落地页；订阅不在本域。  
6. **诱导交易/喊单**：审核队列 + 关键词/举报。  
7. **推送不拥有正文**：`content.published` 触发通知；正文权威在本域。  
8. **事件经 Outbox**。  
9. **PII/留痕**：消息可审计保留（按时限）。  
10. **外链模式**：可不自研消息栈，绑定配置仍属本域。  
11. **与营销解耦**：活动文案可链到文章，活动资格不在本域。  
12. **不做**：CRM、赠金引擎、跟单放大、排行榜。

## 八、跨域契约对齐与扩写（v1.0 补充）

> 本节为 v1.0 对齐《平台跨域契约补充卷》v1.x 并扩写至生产级。旧稿 §一–§七 保留，本节补齐缺失章节。

### 8.1 跨域契约引用声明

- **错误码**：遵循补充卷 §三，使用 `CMS_*` 前缀。
- **事件**：遵循补充卷 §四。本域产出 `content.published.v1`、`content.unpublished.v1`、`im.message.sent.v1`、`im.message.flagged.v1`。
- **角色**：内容编辑/审核引用补充卷 §五 `PMS_CONTENT_EDITOR`、`PMS_CONTENT_REVIEWER`。
- **ID**：遵循补充卷 §六，`content_id`/`message_id` 由本域生成。
- **API**：见 §8.10。
- **容灾/容量/监控**：遵循补充卷 §十/§十一/§十二。

### 8.2 CMS 版本管理与发布（补缺）

- **版本**：`t_content_version`，每次编辑生成新版本，`status=DRAFT/IN_REVIEW/PUBLISHED/UNPUBLISHED/SUPERSEDED`。
- **定时发布**：`publish_at` 字段，定时任务扫描发布。
- **回滚**：已发布版本可回滚到上一版本（`SUPERSEDED` 当前 → `PUBLISHED` 上一）。
- **多语言 fallback**：`locale` 优先匹配，缺失回退到默认语言（`default_locale`）。
- **辖区可见性**：`t_content_visibility`（content_id × entity_id × jurisdiction），矩阵控制。
- **附件安全**：图片/附件经病毒扫描（ClamAV）+ XSS 消毒（富文本）+ CDN 分发。
- **SEO**：meta 字段（title/description/og）由编辑填写。

### 8.3 IM 连接层架构（补缺）

```text
客户端 ──WSS──► [IM Gateway 集群]
                │ 鉴权：Bearer Token
                │ 心跳：30s ping/pong
                │ 连接路由：按 user_uid → node_id（Redis 注册中心）
                ▼
            [消息路由 Worker]
                │ 按 conversation_id partition 消费
                │ 写消息存储 + 推送到成员连接节点
                ▼
            [Gateway 节点] ──推送──► 在线成员
                │ 离线成员：写离线消息表 + 通知域推送
                ▼
            [消息存储]（MySQL 分片 + Redis 热缓存）
```

- **连接数**：单节点 5 万，集群 50 万。
- **重连**：客户端带 `last_msg_id` 重连，Gateway 补发未读。
- **消息 ID 游标**：每会话单调递增 `msg_seq`，客户端按 seq 去重 + 排序。

### 8.4 消息投递语义（补缺）

| 语义 | 实现 |
|------|------|
| at-least-once | 客户端 ack 后服务端才标记送达；未 ack 重发 |
| 有序 | 同会话按 `msg_seq` 单调，Kafka partition by conversation_id |
| 幂等发送 | 客户端 `client_msg_id`，服务端去重 |
| 编辑 | 消息可编辑（2min 内），标记 `edited_at`，保留原内容审计 |
| 撤回 | 消息可撤回（2min 内），标记 `recalled_at`，客户端移除展示 |

### 8.5 成员角色与权限（补缺）

| 角色 | 权限 |
|------|------|
| OWNER | 全部，含转让、解散 |
| ADMIN | 踢人、禁言、设管理员、发系统消息 |
| MEMBER | 发消息、@、举报 |
| GUEST（私聊） | 发消息 |

- 入群审批：`join_mode`（OPEN/INVITE/APPROVAL）。
- 邀请链接：`invite_link`（含过期时间 + 使用次数）。
- 全员禁言：ADMIN 可开启，仅 ADMIN 可发言。

### 8.6 系统卡片与 [13] 事件消费（补缺）

- 消费 `copy.execution.skipped.v1`、`deal.created.v1`（授权晒单）→ 生成系统卡片。
- 卡片 schema：`{type, provider_id, deal_summary, timestamp}`。
- 卡片由 IM Gateway 推送到指定频道/私聊。
- 禁止卡片含下单指令；跟单仅深链 [13]。

### 8.7 晒单授权与脱敏（补缺）

```sql
CREATE TABLE t_im_share_authorization (
  id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_uid VARCHAR(64) NOT NULL,
  deal_id BIGINT UNSIGNED NOT NULL,
  authorized_at BIGINT UNSIGNED NOT NULL,
  expires_at BIGINT UNSIGNED NOT NULL COMMENT '默认 7 天',
  scope VARCHAR(32) COMMENT 'POSITION/PNL/DEAL_DETAIL',
  revoked_at BIGINT UNSIGNED NULL,
  INDEX idx_user (user_uid),
  INDEX idx_deal (deal_id)
) COMMENT='晒单授权';
```

- 授权生命周期：授权 → 展示 → 过期/撤销。
- 脱敏：金额可配精度（如隐藏小数位）、账户 ID 哈希。
- 交易只读 API：`GET /api/v1/internal/share/{auth_id}/deal`（mTLS），返回脱敏字段。

### 8.8 内容审核与反垃圾（补缺）

#### 内容审核流程

```
用户发消息/内容
  → 机审（关键词 + NLP 模型 + 图片识别）
    ├── 命中 → 标记 pending_review → 人审队列
    │   ├── 人审通过 → 发布
    │   └── 人审拒绝 → 删除 + 通知作者
    └── 未命中 → 发布
  → 举报触发 → 入人审队列
```

#### 诱导交易判定规则

| 规则 | 处置 |
|------|------|
| 含"稳赚""包赔"等绝对化用语 | 拦截 |
| 含具体品种 + 买卖方向 + 价格 | 标记 + 人审 |
| 未授权投资建议（无牌照） | 拦截 + 通知合规 |
| 喊单刷屏（同用户高频同质内容） | 自动禁言 1h |

#### 处罚梯度

| 次数 | 处罚 |
|------|------|
| 1 | 警告 + 删除 |
| 2 | 禁言 1h |
| 3 | 禁言 24h |
| 4 | 禁言 7 天 |
| 5 | 封号（转用户域冻结） |

#### 反垃圾

| 策略 | 说明 |
|------|------|
| 速率限制 | 新用户 1min 内 ≤ 5 条；老用户 ≤ 30 条/min |
| 相似度检测 | 5min 内同群相似内容 > 80% → 折叠 + 告警 |
| 链接/二维码过滤 | 新用户禁发链接；链接经安全扫描 |
| 举报聚合 | 同用户被举报 ≥ 3 次 → 自动禁言待人审 |
| 蜜罐 | 蜜罐频道诱捕机器人 → 自动封 |

### 8.9 外链 IM 绑定安全（补缺）

- Telegram 绑定：OAuth 授权 + bot token 关联。
- 消息同步边界：仅同步系统公告单向推送，不同步用户对话。
- 解绑：用户可随时解绑，撤销 bot token。

### 8.11 SLO、容量、容灾（补缺）

| 指标 | SLO |
|------|-----|
| 消息投递 P99 | < 100ms |
| CMS 发布延迟 | < 5s |
| 审核队列 SLA | < 4h |
| 单频道上限 | 10 万成员 |
| 全站并发连接 | 50 万 |
| 消息 TPS | 1 万 |

| 场景 | 降级 |
|------|------|
| IM Gateway 故障 | 切备集群，客户端重连 |
| 消息存储故障 | 写本地缓冲 + 告警，恢复后补 |
| 审核积压 | 自动处置规则收紧（提高拦截阈值） |
| 全站只读降级 | 禁发消息，仅展示 |

### 8.12 审计保留与执法协同（补缺）

- 消息审计保留：7 年（合规），按月分区 + 冷归档。
- 执法调取：经 `PMS_COMPLIANCE_OFFICER` 审批 + 法律流程，导出指定会话。
- 删除保留：用户删除消息仅前端隐藏，后端保留审计 + 标记 `deleted_at`。
