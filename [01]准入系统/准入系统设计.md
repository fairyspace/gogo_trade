# 准入系统设计（权威版）v1.0

> **版本**：v1.0 Final（重构稿，对齐《平台跨域契约补充卷》v1.x）
> **定位**：经纪平台的**客户准入与 KYC/AML 编排权威**。负责开户申请、身份核验（IDV）、反洗钱筛查（AML）、协议签署、审核路由、开户触发。
> **跨域契约**：遵循《平台跨域契约补充卷》v1.x——错误码（§三 `ONB_*`）、事件（§四 `user.approved.v1` 等）、角色（§五 `PMS_COMPLIANCE_REVIEWER`）、ID（§六 `user_uid` 由用户服务生成）、API（§九）、容灾/容量/监控（§十/§十一/§十二）。
> **对标**：Sumsub / Onfido / Jumio（IDV）、ComplyAdvantage / World-Check（AML）、Broadridge（KYC 编排）、IB / DriveWealth（适当性）。
> **不做**：用户主体存储（用户域）、账户开户执行（账户域）、通知送达（通知域）、交易（交易域）、资金（资金域）。

## 一、系统定位与边界

### 1.1 系统定位

Onboarding Service 是经纪平台的：

1. **开户申请权威**（`t_onboarding_application`）
2. **KYC/IDV 编排方**（调用第三方 Sumsub/Onfido/Jumio）
3. **AML 筛查编排方**（调用 ComplyAdvantage/World-Check）
4. **协议签署权威**（`t_onboarding_agreement`）
5. **审核路由权威**（规则引擎 + 人工审核任务）
6. **开户触发方**（KYC 通过后调账户域 `OpenAccount`，发 `user.approved.v1`）

### 1.2 边界表

| 模块 | 准入负责 | 准入不负责 |
|------|----------|------------|
| 用户体系 | 申请关联 `user_uid` | 用户主体存储、凭证、登录 |
| 账户体系 | KYC 通过后调 `OpenAccount` | 账户主数据、组、杠杆 |
| 通知系统 | 发标准事件触发通知 | 模板、通道、送达 |
| 权限系统 | `assign_to_role` 引用 `PMS_COMPLIANCE_REVIEWER` | RBAC/ABAC 决策 |
| 资金系统 | 无直接交互 | 出入金、ledger |
| 代理系统 | `referral_code` 字段透传 | 归因、佣金 |
| 营销系统 | KYC 状态供 CRM Lifecycle 同步 | 活动发奖 |

## 二、整体架构

```text
客户端
  │
  ▼
[注册] ──建 t_user(status=0) + t_onboarding_application──► 用户域 + 准入域
  │
  ▼
[信息完善] ──匹配 KYC 规则（国家/收入 → KYC 等级/材料）
  │
  ▼
[协议签署] ──t_onboarding_agreement（版本化）
  │
  ▼
[文件上传] ──t_onboarding_kyc_document（证件照片）
  │
  ▼
[IDV + AML] ──异步调用第三方
  │   ├── IDV: Sumsub/Onfido/Jumio（OCR + 活体）
  │   └── AML: ComplyAdvantage/World-Check（制裁/PEP/负面媒体）
  │
  ▼
[规则引擎] ──t_onboarding_review_rule（优先级短路）
  │   ├── P=10 制裁命中 → 自动拒绝
  │   ├── P=20 IDV 失败 → 补材料
  │   ├── P=100 自动通过条件满足 → 自动通过
  │   └── P=999 兜底 → 转人工
  │
  ▼
[人工审核] ──t_onboarding_review_task（SLA + 升级）
  │
  ▼
[最终检查 + 开户] ──调账户域 OpenAccount（幂等 + 乐观锁）
  │
  ▼
[发事件] ──user.approved.v1 / user.rejected.v1 ──► 通知/营销/代理
```

## 三、用户创建时机（关键裁定）

旧稿与用户域在用户创建时机上不一致。v1.0 裁定：

- **注册即建 `t_user(status=0)` + `t_onboarding_application`**，二者通过 `user_uid` 关联。
- `user_uid` 由**用户服务生成**（补充卷 §六），准入服务**不**生成 `trading_user_uid`。
- KYC 通过后，准入发 `user.approved.v1`，账户域消费此事件调 `OpenAccount`，`owner_user_id` 引用 `user_uid`。
- OAuth 注册：用户域建 `t_user(status=0)`，准入域同步建 application，引导补全。

时序图：

```
客户端 ──注册──► 用户域(建 t_user, 返 user_uid)
       ──申请──► 准入域(建 t_onboarding_application, 关联 user_uid)
       ──完善+KYC──► 准入域(规则匹配/IDV/AML/审核)
       ◄──结果── 准入域(发 user.approved.v1)
通知域 ◄──事件── 准入域(KYC_APPROVED/REJECTED 模板)
账户域 ◄──事件── 准入域(消费 user.approved.v1, 调 OpenAccount)
营销域 ◄──事件── 准入域(Lifecycle 同步)
代理域 ◄──事件── 准入域(归因 introducing_broker_id)
```

## 四、三维状态机

### 4.1 `t_onboarding_application.status`（准入状态）

| 值 | 含义 |
|----|------|
| 0 | DRAFT（信息完善中） |
| 1 | PENDING_KYC（待 IDV/AML） |
| 2 | PENDING_REVIEW（待人工审核） |
| 3 | REJECTED（拒绝） |
| 4 | APPROVED（已通过，待开户） |
| 5 | ACCOUNT_OPENED（已开户完成） |

### 4.2 `t_onboarding_kyc.status`

| 值 | 含义 |
|----|------|
| 0 | NOT_STARTED |
| 1 | IN_PROGRESS |
| 2 | PASSED |
| 3 | FAILED |
| 4 | MANUAL_REVIEW |

### 4.3 组合状态转换

```
(DRAFT, NOT_STARTED)
  → (PENDING_KYC, IN_PROGRESS)  [提交文件]
  → (PENDING_KYC, PASSED/FAILED/MANUAL_REVIEW)  [第三方返回]
  → (PENDING_REVIEW, MANUAL_REVIEW)  [规则转人工]
  → (APPROVED, PASSED)  [人工通过]
  → (REJECTED, FAILED)  [拒绝]
  → (ACCOUNT_OPENED, PASSED)  [账户域 OpenAccount 成功回调]
```

## 六、核心表（保留旧稿 DDL，补 1 张）

旧稿 8 张表 DDL 保留（`t_onboarding_application`、`t_onboarding_kyc`、`t_onboarding_kyc_document`、`t_onboarding_agreement_template`、`t_onboarding_agreement`、`t_onboarding_kyc_rule`、`t_onboarding_review_rule`、`t_onboarding_config_change_log`），字段修订：

| 表 | 修订 |
|----|------|
| `t_onboarding_application` | `status` 注释补齐状态 4 定义；`user_uid` 字段引用用户服务生成；`trading_user_uid` 字段废弃，改引用 `user_uid` |
| `t_onboarding_review_rule` | `assign_to_role` 改为 `PMS_COMPLIANCE_REVIEWER`（补充卷 §五） |

### 6.1 新增 `t_onboarding_review_task`（人工审核任务表）

```sql
CREATE TABLE t_onboarding_review_task (
  id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  application_id BIGINT UNSIGNED NOT NULL,
  reviewer_id BIGINT UNSIGNED NULL COMMENT '审核员 pms_user_id',
  assigned_at BIGINT UNSIGNED NOT NULL,
  sla_due_at BIGINT UNSIGNED NOT NULL COMMENT 'SLA 到期时间',
  status TINYINT UNSIGNED DEFAULT 0 COMMENT '0-待领取 1-进行中 2-已决策 3-已升级 4-已超时',
  decision TINYINT UNSIGNED NULL COMMENT '1-通过 2-拒绝 3-补材料',
  decision_reason VARCHAR(512) NULL,
  escalated_from_id BIGINT UNSIGNED NULL COMMENT '升级来源任务',
  created_at BIGINT UNSIGNED NOT NULL,
  updated_at BIGINT UNSIGNED NOT NULL,
  INDEX idx_app (application_id),
  INDEX idx_status_sla (status, sla_due_at)
) COMMENT='人工审核任务（含 SLA 与升级）';
```

- SLA：默认 24h，超时自动升级 P1（`PMS_COMPLIANCE_OFFICER`）。
- 升级链：审核员 → 合规官 → 合规总监，每级 SLA 递减。

## 七、API 契约（遵循补充卷 §九）

### 7.1 客户端 API（REST）

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/onboarding/applications` | POST | 创建申请（关联 user_uid） |
| `/api/v1/onboarding/applications/{id}` | GET | 查询申请状态 |
| `/api/v1/onboarding/applications/{id}/profile` | PUT | 完善信息 |
| `/api/v1/onboarding/applications/{id}/documents` | POST | 上传证件 |
| `/api/v1/onboarding/applications/{id}/agreements` | GET/POST | 查询/签署协议 |
| `/api/v1/onboarding/applications/{id}/kyc/submit` | POST | 提交 KYC（触发 IDV/AML） |

### 7.2 审核员 API（REST，需 `PMS_COMPLIANCE_REVIEWER`）

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/admin/review/tasks` | GET | 领取任务列表 |
| `/api/v1/admin/review/tasks/{id}/claim` | POST | 领取任务 |
| `/api/v1/admin/review/tasks/{id}/decide` | POST | 决策（通过/拒绝/补材料） |

### 7.3 内部 RPC

- 消费 `user.registered.v1`（用户域）→ 建申请
- 产出 `user.approved.v1` / `user.rejected.v1`（账户/营销/代理消费）
- 调账户域 `AccountService.OpenAccount`（开户）

## 八、AML 降级策略

| 场景 | 降级 |
|------|------|
| AML 主供应商超时 | 切备供应商（ComplyAdvantage ↔ World-Check） |
| 主备均不可用 | 转人工审核（`ONB_EXTERNAL_AML_PROVIDER_DOWN`），SLA 48h |
| IDV 超时 | 转人工核对证件 |
| 证件 OCR 全失败 | 要求重新上传（`ONB_VALIDATION_KYC_DOCS_REQUIRED`） |
| 制裁命中（P=10） | 自动拒绝（`ONB_STATE_AML_SANCTION_HIT`），不可人工覆盖 |
| PEP 命中 | 转人工 + 合规官终审 |

## 九、KYB 企业客户流程（`user_type=2`）

旧稿仅字段存在。v1.1 补企业 KYB：

- 企业主体：营业执照、注册地、受益所有人（UBO，> 25% 持股穿透）、授权代表。
- KYB 文档：注册证书、章程、董事名册、UBO 证件。
- 第三方：Kyckr / Dun & Bradstreet 企业核验。
- 审核路由：企业申请一律转人工 + 合规官终审。
- 多操作员：企业账户可挂多个授权操作员，各自 KYC。

| 指标 | SLO |

| KYC 自动通过率 | > 70% |

## 十一、容灾

| 场景 | 降级 |
|------|------|
| 第三方 IDV/AML 不可用 | 转人工，SLA 放宽 |
| 准入服务故障 | 申请暂存，恢复后继续 |
| 账户域 OpenAccount 不可用 | 事件积压 Outbox，恢复后补发 |

## 十四、结语

准入系统是客户进入经纪平台的第一道闸门。v1.0 以"注册即建用户 + 申请关联"统一了与用户域的创建时机，以三维状态机消除旧稿状态枚举矛盾，以事件契约替代直连通道，以人工审核任务表支撑 SLA，使准入域具备持牌运营的可落地水准。
