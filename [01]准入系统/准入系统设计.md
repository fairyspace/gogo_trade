### Onboarding Service（8张表）


####  t_onboarding_agreement_template

```sql
-- ----------------------------------------------------------------
-- 协议模板定义
--
-- 管理券商需要客户签署的所有法律文件
-- 券商管理员在后台维护协议内容和版本
--
-- 业务流程:
--   1. 管理员上传新版协议(如风险揭示书v3.0)
--   2. 设置生效时间
--   3. 新开户用户会看到最新版本并签署
--   4. 已有用户如果force_resign=1, 下次登录时要求重新签署
--
-- 示例数据:
--   CLIENT_AGREEMENT    客户协议          v2.1  必签  无需重签
--   RISK_DISCLOSURE     风险揭示书        v3.0  必签  需要重签
--   PRIVACY_POLICY      隐私政策          v1.5  必签  无需重签
--   LEVERAGE_WARNING    杠杆风险警告      v1.0  必签  无需重签
--   CRYPTO_RISK         加密货币风险补充   v1.0  可选  无需重签
-- ----------------------------------------------------------------
CREATE TABLE `t_onboarding_agreement_template` (
    `id`                    BIGINT UNSIGNED     NOT NULL AUTO_INCREMENT COMMENT '自增主键',
    `entity_id`             BIGINT UNSIGNED     NOT NULL COMMENT '法律实体ID(对应牌照主体,如AU牌照=1001/UK牌照=1002,单一牌照时为固定值)',

    `agreement_code`        VARCHAR(32)         NOT NULL
        COMMENT '协议代码: CLIENT_AGREEMENT/RISK_DISCLOSURE/PRIVACY_POLICY/LEVERAGE_WARNING/CRYPTO_RISK',
    `agreement_name`        VARCHAR(128)        NOT NULL
        COMMENT '协议名称,面向用户展示,如"客户服务协议"',
    `current_version`       VARCHAR(16)         NOT NULL
        COMMENT '当前生效版本号,如v3.0',

    -- ==================== 内容 ====================
    `content_url`           VARCHAR(512)        DEFAULT NULL
        COMMENT '协议内容文件地址(OSS/S3上的PDF或HTML)',
    `content_text`          TEXT                DEFAULT NULL
        COMMENT '协议纯文本内容(可选,用于App内直接展示,不走文件)',
    `content_locale`        VARCHAR(8)          NOT NULL DEFAULT 'en'
        COMMENT '内容语言,ISO 639-1,如en/zh/ar',

    -- ==================== 规则 ====================
    `is_required`           TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '开户时是否必须签署: 0-可选 1-必签(不签不能开户)',
    `force_resign`          TINYINT UNSIGNED    NOT NULL DEFAULT 0
        COMMENT '版本更新后是否要求已有用户重新签署: 0-不要求 1-下次登录时强制弹出',

    -- ==================== 适用范围 ====================
    `applicable_countries`  VARCHAR(512)        DEFAULT NULL
        COMMENT '适用国家/地区,逗号分隔ISO代码,如AU,NZ,NULL=全部',
    `applicable_environment` TINYINT UNSIGNED   DEFAULT NULL
        COMMENT '适用环境: NULL=全部 1-仅真实账户 2-仅模拟账户',

    -- ==================== 排序与状态 ====================
    `sort_order`            INT                 NOT NULL DEFAULT 0
        COMMENT '展示排序',
    `status`                TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '状态: 0-停用 1-启用',
    `effective_at`          DATETIME            NOT NULL DEFAULT CURRENT_TIMESTAMP
        COMMENT '当前版本生效时间',

    `created_at`            DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    `updated_at`            DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_entity_code_locale` (`entity_id`, `agreement_code`, `content_locale`),
    KEY `idx_entity_status` (`entity_id`, `status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='协议模板定义(按法律实体配置)';
```

####  t_onboarding_kyc_rule


```sql
-- ----------------------------------------------------------------
-- KYC准入规则
--
-- 定义不同条件下的KYC等级要求和所需材料
-- 券商合规团队在后台维护，规则变更有审计记录
--
-- 匹配逻辑（按优先级从高到低）:
--   1. 精确匹配: 国家 + 用户类型 + 收入范围 全部命中
--   2. 部分匹配: 国家 + 用户类型 命中
--   3. 兜底规则: match_country = '*' 的默认规则
--
-- 示例数据:
--   规则1: US + 个人 + 任意收入 → L3 → 需要护照+地址证明+银行流水+自拍
--   规则2: AU + 个人 + 年收>50万 → L2 → 需要护照+地址证明
--   规则3: AU + 个人 + 年收<50万 → L1 → 需要身份证正反面
--   规则4: * + 企业 + 任意 → L3 → 需要营业执照+法人证件+公司章程
--   规则5: * + 个人 + 任意（兜底）→ L1 → 需要身份证正反面
-- ----------------------------------------------------------------
CREATE TABLE `t_onboarding_kyc_rule` (
    `id`                    BIGINT UNSIGNED     NOT NULL AUTO_INCREMENT COMMENT '自增主键',
    `entity_id`             BIGINT UNSIGNED     NOT NULL COMMENT '法律实体ID(不同牌照可有不同KYC规则)',
    `rule_code`             VARCHAR(64)         NOT NULL COMMENT '规则编码,唯一标识,如US_INDIVIDUAL_DEFAULT',

    -- ==================== 匹配条件 ====================
    `match_country`         VARCHAR(512)        NOT NULL DEFAULT '*'
        COMMENT '匹配国家,逗号分隔ISO代码,*=全部. 如: US,CA 或 AU 或 *',
    `match_user_type`       TINYINT UNSIGNED    DEFAULT NULL
        COMMENT '匹配用户类型: NULL=全部 1-个人 2-企业',
    `match_nationality`     VARCHAR(512)        DEFAULT NULL
        COMMENT '匹配国籍(有些场景国籍比居住国更重要),逗号分隔,NULL=不限',
    `match_income_range_min` TINYINT UNSIGNED   DEFAULT NULL
        COMMENT '匹配最低收入范围(>=),对应application表的annual_income_range,NULL=不限',
    `match_income_range_max` TINYINT UNSIGNED   DEFAULT NULL
        COMMENT '匹配最高收入范围(<=),NULL=不限',

    -- ==================== 输出: KYC要求 ====================
    `required_kyc_level`    TINYINT UNSIGNED    NOT NULL
        COMMENT '该条件下要求的最低KYC等级: 1-L1 2-L2 3-L3',
    `required_doc_types`    VARCHAR(512)        NOT NULL
        COMMENT '【全部必须】需要上传的文件类型,逗号分隔. 如: ID_FRONT,ID_BACK,SELFIE 表示每种都必须上传',
    `required_id_types`     VARCHAR(128)        DEFAULT NULL
        COMMENT '【多选一】接受的证件类型,逗号分隔. 如: 1,2,3 表示身份证/护照/驾照任选其一提交. NULL=接受全部类型',
    `required_agreements`   VARCHAR(512)        DEFAULT NULL
        COMMENT '额外需要签署的协议代码,逗号分隔. 如: LE
        VERAGE_WARNING,CRYPTO_RISK. NULL=仅基础必签协议',


    -- ==================== 优先级与状态 ====================
    `priority`              INT                 NOT NULL DEFAULT 100
        COMMENT '优先级,数字越小优先级越高. 多条规则匹配时取优先级最高的',
    `status`                TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '状态: 0-停用 1-启用',
    `effective_from`        DATETIME            DEFAULT NULL
        COMMENT '规则生效时间,NULL=立即生效',
    `effective_to`          DATETIME            DEFAULT NULL
        COMMENT '规则失效时间,NULL=永不过期',

    -- ==================== 审计 ====================
    `created_by`            VARCHAR(64)         DEFAULT NULL COMMENT '创建人(合规管理员)',
    `updated_by`            VARCHAR(64)         DEFAULT NULL COMMENT '最后修改人',
    `remark`                VARCHAR(512)        DEFAULT NULL COMMENT '规则说明/合规依据',

    `created_at`            DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    `updated_at`            DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_entity_rule_code` (`entity_id`, `rule_code`),
    KEY `idx_entity_status_priority` (`entity_id`, `status`, `priority`),
    KEY `idx_effective` (`effective_from`, `effective_to`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
COMMENT='KYC准入规则(决定不同用户需要什么KYC等级和材料)';
```

#### t_onboarding_review_rule

  **解决什么问题**：KYC材料提交后，系统怎么决定自动通过还是转人工？什么条件触发升级审核？

```sql
-- ----------------------------------------------------------------
-- 审核路由规则
--
-- 定义KYC审核的路由决策: 自动通过 / 自动拒绝 / 转人工
-- 规则按优先级从高到低执行,命中即停止(短路逻辑)
--
-- 执行流程:
--   1. KYC材料提交后,系统依次检查所有启用的规则
--   2. 如果命中某条规则,执行该规则指定的动作(通过/拒绝/转人工)
--   3. 如果所有规则都未命中,执行 fallback_action(默认动作)
--
-- 示例数据:
--   P=10  AML_SANCTION_HIT     aml_hit_type=SANCTION         → 自动拒绝  "制裁名单命中直接拒绝"
--   P=20  AML_PEP_HIT          aml_hit_type=PEP              → 转人工    "PEP需要合规主管审核"
--   P=30  HIGH_RISK_SCORE      aml_risk_score>=80            → 转人工    "高风险评分"
--   P=40  HIGH_RISK_COUNTRY    country IN (IR,KP,SY,CU)      → 自动拒绝  "OFAC制裁国家"
--   P=50  MEDIUM_RISK_COUNTRY  country IN (MM,VE,LB)         → 转人工    "中等风险国家需人工审核"
--   P=60  DOC_OCR_MISMATCH     ocr_match_score<0.7           → 转人工    "证件OCR匹配度低"
--   P=70  DUPLICATE_ID         id_hash已存在其他application   → 转人工    "证件号重复"
--   P=80  LARGE_DEPOSIT_INTENT expected_deposit>100000       → 转人工    "大额入金意向"
--   P=999 DEFAULT_PASS         (兜底)                        → 自动通过  "正常用户自动通过"
-- ----------------------------------------------------------------
CREATE TABLE `t_onboarding_review_rule` (
    `id`                    BIGINT UNSIGNED     NOT NULL AUTO_INCREMENT COMMENT '自增主键',
    `entity_id`             BIGINT UNSIGNED     NOT NULL COMMENT '法律实体ID(不同牌照可有不同审核规则)',
    `rule_code`             VARCHAR(64)         NOT NULL COMMENT '规则编码,如AML_SANCTION_HIT',
    `rule_name`             VARCHAR(128)        NOT NULL COMMENT '规则名称,如"制裁名单命中"',

    -- ==================== 匹配条件 ====================
    -- 使用JSON存储灵活的条件表达式,应用层解析
    -- 这样不需要为每种条件加字段,规则引擎统一处理
    `condition_type`        VARCHAR(32)         NOT NULL
        COMMENT '条件类型,辅助程序快速分类:
                 AML_HIT       - AML命中类型匹配
                 AML_SCORE     - AML分数阈值
                 COUNTRY       - 国家黑白名单
                 DOC_QUALITY   - 文件质量检查
                 DUPLICATE     - 重复检测
                 AMOUNT        - 金额阈值
                 COMPOSITE     - 组合条件
                 DEFAULT       - 兜底规则(始终匹配)',

    `condition_expression`  JSON                NOT NULL
        COMMENT '条件表达式(JSON).
                 示例1: {"field":"aml_hit_type","operator":"IN","value":["SANCTION"]}
                 示例2: {"field":"aml_risk_score","operator":">=","value":80}
                 示例3: {"field":"country_of_residence","operator":"IN","value":["IR","KP","SY"]}
                 示例4: {"field":"ocr_match_score","operator":"<","value":0.7}
                 示例5: {"operator":"AND","conditions":[
                           {"field":"aml_risk_score","operator":">=","value":60},
                           {"field":"annual_income_range","operator":"<=","value":2}
                         ]}
                 示例6(兜底): {"operator":"ALWAYS_TRUE"}',

    -- ==================== 输出: 动作 ====================
    `action`                TINYINT UNSIGNED    NOT NULL
        COMMENT '命中后的动作: 1-自动通过 2-自动拒绝 3-转人工审核',
    `assign_to_role`        VARCHAR(32)         DEFAULT NULL
        COMMENT '转人工时分配给哪个角色: COMPLIANCE_OFFICER/SENIOR_COMPLIANCE/MANAGER. 仅action=3时有效', --指向后台管理admin.role_code
    `auto_reject_reason`    VARCHAR(256)        DEFAULT NULL
        COMMENT '自动拒绝时的标准拒绝原因文案. 仅action=2时有效',
    `sla_hours`             INT UNSIGNED        DEFAULT NULL
        COMMENT '转人工后的SLA时限(小时),超时升级. 仅action=3时有效',

    -- ==================== 优先级与控制 ====================
    `priority`              INT                 NOT NULL DEFAULT 100
        COMMENT '执行优先级,数字越小越先执行. 命中即停止(短路)',
    `status`                TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '状态: 0-停用 1-启用',
    `effective_from`        DATETIME            DEFAULT NULL COMMENT '规则生效时间',
    `effective_to`          DATETIME            DEFAULT NULL COMMENT '规则失效时间',

    -- ==================== 审计 ====================
    `created_by`            VARCHAR(64)         DEFAULT NULL COMMENT '创建人',
    `updated_by`            VARCHAR(64)         DEFAULT NULL COMMENT '最后修改人',
    `remark`                VARCHAR(512)        DEFAULT NULL
        COMMENT '规则说明/合规依据,如"根据FATF建议第19条"',

    `created_at`            DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    `updated_at`            DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_entity_rule_code` (`entity_id`, `rule_code`),
    KEY `idx_entity_status_priority` (`entity_id`, `status`, `priority`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
COMMENT='审核路由规则(决定KYC走自动通过还是转人工)';
```


审核分配的逻辑在应用层完成：

1. 规则命中 → `action=3(转人工)`，`assign_to_role='COMPLIANCE_OFFICER'`
2. 应用层调用 Admin 服务，查询角色为 `COMPLIANCE_OFFICER` 的在线人员
3. 按负载均衡/地区/语言分配具体审核员
4. 写入 `t_onboarding_kyc.reviewer_id`

### 开户流程完整描述（修订版）


#### 1. 用户访问与初步注册 (Initial Registration)

**用户行为：** 用户访问券商官网或App，点击"注册"按钮。填写基础信息（邮箱或手机号 + 密码），系统发送验证码进行验证。

**系统动作：** 验证成功后，系统为该用户创建一个初级账户，但此时不开通任何交易功能。

**涉及数据库表：**
- **`t_onboarding_application`：** 创建一条开户申请记录，初始状态 `status = 0`（填写中），`account_status = 0`（未开户）。

---

#### 2. 信息完善与KYC等级判定 (Information Completion & KYC Level Determination)

**用户行为：** 注册成功后，系统引导用户进入"个人信息完善"页面，填写详细资料：
- 姓名（Legal First Name / Last Name）
- 国家/地区（Country of Residence）
- 出生日期（Birth Date）
- 地址（Address Line / City / State / Postal Code）
- 证件号码（ID Number）
- 职业（Employment Status）
- 年收入范围（Annual Income Range）
- 净资产范围（Net Worth Range）
- 投资经验年数（Trading Exp Years）
- 风险承受能力（Risk Tolerance）
- 税务居住国（Tax Residency Country，CRS/FATCA合规）
- 税务识别号（Tax ID / TIN，加密存储）
- 是否政治公众人物（PEP Self-Declaration，FATF合规）
- 雇主名称 / 行业（Employer，内幕交易筛查）
- 交易知识水平（Trading Knowledge，MiFID II适当性评估）
- 预计初始入金范围（Expected Deposit，可疑交易监控基准）

**系统动作：** 系统根据用户填写的信息（国家/地区、用户类型、收入等），查询 `t_onboarding_kyc_rule` 中预设的规则进行匹配。按照规则优先级（`priority`）查找最匹配的规则（精确匹配 > 部分匹配 > 兜底规则），确定：
- 该用户需要完成的**最低KYC等级**（`required_kyc_level`）
- 需要上传的**文件类型**（`required_doc_types`）
- 需要额外签署的**协议列表**（`required_agreements`）

**涉及数据库表：**
- **`t_onboarding_application`：** 更新该条记录，填入用户提交的详细个人信息。
- **`t_onboarding_kyc_rule`：** 用于匹配用户信息，确定KYC要求。

---
#### 3. 协议签署 (Agreement Signing)

**用户行为：** 在KYC文件上传完成后（或与KYC并行进行），系统引导用户进入协议签署页面。用户需要阅读并同意一系列法律文件。

**系统动作：**
1. 根据 `t_onboarding_kyc_rule` 的匹配结果和用户的国家/地区，从 `t_onboarding_agreement_template` 表中查询出当前需要签署的协议列表（包括基础必签协议和规则要求的额外协议）
2. 展示协议的最新有效版本（`current_version`）内容给用户阅读
3. 用户勾选"我已阅读并同意"并点击"签署"后，系统遍历需要签署的协议列表，在 `t_onboarding_agreement` 表中为每一份协议创建一条记录

**涉及数据库表：**
- **`t_onboarding_agreement_template`：** 查询需要签署的协议模板及其版本。
- **`t_onboarding_agreement`：** 创建签署记录，记录 `application_id`、`agreement_code`、`agreement_version`、`is_accepted = 1`、`accepted_at`。


---

#### 4. AML与IDV检查 (AML & IDV Checks)

**系统动作：** KYC材料提交完成后，系统调用第三方服务进行检查：

**身份验证（IDV）：**
- 调用第三方服务（如Sumsub、Onfido、Jumio）
- 发送：证件照片 + 自拍照 + 用户填写的姓名/生日
- 返回：OCR识别结果、信息匹配度（`idv_match_score`）、活体检测结果（`idv_liveness_result`）
- 验证证件的真实性、有效性，OCR提取信息与用户填写信息比对

**反洗钱筛查（AML）：**
- 调用第三方服务（如ComplyAdvantage、World-Check）
- 发送：姓名 + 国籍 + 出生日期
- 返回：风险评分（`aml_risk_score`）、命中类型（`aml_hit_type`：CLEAN/PEP/SANCTION/ADVERSE_MEDIA）
- 将用户信息与制裁名单、PEP名单等进行比对

| 数据源 | 内容 | 数据量 |
|--------|------|--------|
| UN安理会制裁名单 | 恐怖主义/武器扩散相关人员 | ~1,000+ |
| OFAC SDN名单（美国） | 美国制裁的个人和实体 | ~12,000+ |
| EU制裁名单 | 欧盟制裁 | ~2,000+ |
| FATF高风险国家 | 高风险和不合作地区 | 持续更新 |
| PEP数据库 | 全球政治公众人物及其亲属 | ~300万+ |
| 负面新闻（Adverse Media） | 涉及洗钱/欺诈/犯罪的新闻报道 | 海量 |

系统将第三方返回的详细结果（包括原始JSON `aml_raw_response`）写入 `t_onboarding_kyc` 表。

**涉及数据库表：**
- **`t_onboarding_kyc`：** 更新AML/IDV相关字段。

---

#### 5. KYC文件上传 (KYC Document Upload)

**用户行为：** 系统根据上一步的规则判定结果，引导用户进入KYC认证页面，要求上传指定类型的文件。例如规则要求L2级别认证和 `ID_FRONT, ID_BACK, PROOF_OF_ADDRESS`，则用户需按提示逐一上传。

**系统动作：**
1. 接收文件，进行初步的格式和大小校验
2. 将文件存储到安全的云存储（S3/OSS）
3. 计算文件哈希值用于防篡改
4. 在 `t_onboarding_kyc` 表中创建一条KYC申请记录，记录申请级别（`kyc_level`）、证件类型（`id_type`）、证件号（加密/哈希存储）、签发国（`id_issuing_country`）等，状态设为 `status = 0`（待审核）
5. 在 `t_onboarding_kyc_document` 表中为每个上传文件创建记录，关联 `kyc_id` 和 `application_id`，记录文件类型（`doc_type`）、存储路径（`file_path`）、哈希值（`file_hash`），审核状态设为 `review_status = 0`（待审）

**涉及数据库表：**
- **`t_onboarding_kyc`：** 创建KYC主记录。
- **`t_onboarding_kyc_document`：** 创建KYC文件明细记录。

---

#### 6. 审核路由规则引擎决策 (Review Rule Engine Decision)

**系统动作：** KYC材料（包括AML/IDV结果）和协议签署完成后，系统更新 `t_onboarding_application.status = 1`（待KYC审核），然后调用审核路由规则引擎进行决策。

1. 收集本次申请的所有相关信息（来自 `application`、`kyc`、`kyc_document`、`agreement` 等表，特别是 `aml_hit_type`、`aml_risk_score`、`idv_match_score` 等）
2. 查询 `t_onboarding_review_rule` 表，获取所有已启用（`status = 1`）的规则，按优先级（`priority`）从小到大排序
3. 依次将申请信息与每条规则的 `condition_expression` 进行匹配（**短路逻辑，命中即停止**）

**匹配结果处理：**

- **`action = 1`（自动通过）：** 更新 `t_onboarding_kyc.status = 1`（自动通过），流程进入**步骤8**
- **`action = 2`（自动拒绝）：** 更新 `t_onboarding_kyc.status = 3`（被拒），记录 `reject_reason`（`auto_reject_reason`）。同时更新 `t_onboarding_application.status = 3`（KYC被拒）。流程结束，通知用户
- **`action = 3`（转人工审核）：** 保持 `t_onboarding_kyc.status = 0`（待审核），根据 `assign_to_role` 将申请推入对应角色的审核队列。流程进入**步骤7**

**涉及数据库表：**
- **`t_onboarding_review_rule`：** 规则匹配。
- **`t_onboarding_kyc`：** 更新审核状态。
- **`t_onboarding_application`：** 被拒时同步更新主表状态。

---

#### 7. 后台人工审核 (Manual Review)

> 仅当步骤6中规则命中 `action = 3`（转人工）时触发

**审核员行为：**
1. 申请被推送到审核员的待办任务列表
2. 审核员查看系统展示的完整信息：用户填写的资料、上传的文件、AML/IDV结果、触发的规则
3. 审核员做出判断并操作

**系统动作：**
- **审核通过：** 更新 `t_onboarding_kyc.status = 2`（人工通过），记录 `reviewer_id`、`reviewed_at`、`approved_at`。流程进入**步骤8**
- **审核拒绝：** 更新 `t_onboarding_kyc.status = 3`（被拒），记录 `reviewer_id`、`reviewed_at`、`reject_reason`。同时更新 `t_onboarding_application.status = 3`（KYC被拒）。通知用户
- **要求补充材料：** 更新 `t_onboarding_kyc.status = 4`（需补充材料），通知用户补充后重新提交，回到**步骤3或步骤6**

审核员也可能更新 `t_onboarding_kyc_document` 中具体文件的 `review_status`（合格/不合格）。

**涉及数据库表：**
- **`t_onboarding_kyc`：** 更新 `status`、`reviewer_id`、`reviewed_at`、`reject_reason`。
- **`t_onboarding_kyc_document`：** 可能更新 `review_status`。
- **`t_onboarding_application`：** 拒绝时同步更新。

---

#### 8. 开户条件最终检查 (Final Eligibility Check)

> KYC通过后（自动通过或人工通过），系统执行最终的开户条件检查

**系统动作：** 检查以下所有条件是否全部满足：

| 检查项 | 数据来源 | 通过条件 |
|--------|----------|----------|
| KYC审核状态 | `t_onboarding_kyc.status` | `IN (1, 2)`（自动通过或人工通过） |
| 必要文件验证 | `t_onboarding_kyc_document.review_status` | 所有必要文件 `= 1`（合格） |
| 必签协议签署 | `t_onboarding_agreement.is_accepted` | 所有必签协议 `= 1`（已签署） |

- **全部通过：** 更新 `t_onboarding_application.status = 2`（KYC已通过），流程进入**步骤9**
- **存在未满足项：** 通知用户补充缺失项，回到对应步骤

---

#### 9. 调用交易引擎开户 (Account Creation)

> 这是从"准入通过"到"真正拥有交易账户"的关键步骤

##### Step 9.1：幂等性前置检查

```
IF status = 2 AND account_status = 0:
    → 允许发起开户，进入 Step 9.2
IF account_status = 1:
    → 开户中，查询上次结果（防重复提交）
IF account_status = 2:
    → 已完成，直接返回成功
```

##### Step 9.2：标记开户中

```sql
UPDATE t_onboarding_application
SET account_status = 1,       -- 开户中
    updated_at = NOW()
WHERE id = ? AND account_status = 0;   -- 乐观锁，防并发
```

如果 `affected_rows = 0`，说明并发冲突，退出。

##### Step 9.3：调用交易引擎API

```
POST /api/trading/create-user
Body: {
    application_no: "APP-2024-xxxxx",
    entity_id: 1001,
    email: "user@example.com",
    legal_name: "Zhang San",
    ...
}
```

##### Step 9.4：根据返回结果更新状态

**调用成功：**
```sql
-- 一个事务内同时更新 trading_user_uid 和状态
UPDATE t_onboarding_application
SET account_status = 2,                          -- 开户成功
    trading_user_uid = 'TU-20240115-ABCDEF',     -- 回填交易引擎返回的UID
    account_created_at = NOW(),
    status = 4                                   -- 已开户完成（可选，见下方说明）
WHERE id = ?;
```

> **关键原则：必须在拿到 `trading_user_uid` 之后，在同一个事务内更新 `account_status` 和 `trading_user_uid`。绝不能先改状态再调接口，也不能分两次事务更新。**

**调用失败或超时：**
```sql
UPDATE t_onboarding_application
SET account_status = 3,                          -- 开户失败
    account_error_msg = 'Connection timeout...',  -- 记录错误信息
    account_retry_count = account_retry_count + 1
WHERE id = ?;
```

系统触发告警，定时任务后续扫描 `account_status = 3` 的记录进行重试（最大重试3次，超过后转人工介入）。

##### 状态流转总结

```
准入阶段 (status):
    0(填写中) → 1(待KYC审核) → 2(KYC已通过) 
                              → 3(KYC被拒)

开户阶段 (account_status):
    仅 status=2 时才能进入
    0(未开户) → 1(开户中) → 2(开户成功，同时写入trading_user_uid)
                          → 3(开户失败，写入error_msg，等待重试)
                               → 1(重试开户中)

最终完成判定:
    status = 2 AND account_status = 2 AND trading_user_uid IS NOT NULL
```

**涉及数据库表：**
- **`t_onboarding_application`：** 更新 `account_status`、`trading_user_uid`、`account_created_at`、`account_retry_count`、`account_error_msg`。

---

### 10. 开户完成通知 (Completion Notification)

**系统动作：** 开户成功后（`account_status = 2`），系统向用户发送开户成功通知（邮件/短信/App推送），告知用户可以开始使用交易功能。同时可以触发下游事件：
- 通知CRM系统记录新客户
- 触发欢迎奖励/新手引导流程
- 同步用户信息到其他业务系统
