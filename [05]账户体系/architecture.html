📐 三大核心设计原则
Design Principles — "Why we do it this way."
🎯
原则 1: 账户组 = 参数中枢(MT5 Group 思路)
一个账户组 = 监管实体 + 保证金模型 + 杠杆策略 + 风控阈值 + 可交易品种 + 手续费规则的完整组合。开户即归组继承,99% 参数无需在账户级重复维护。 这种"参数模板化"是 MT5 / cTrader 通行做法,既支持多业务线快速上线,又便于合规审计(改组=改一类账户)。
🔗
原则 2: 资金变动唯一入口 — t_account_ledger
所有余额变动仅经 t_account_ledger(资金系统已定义):入金、出金、内部转账、平仓盈亏、手续费、隔夜利息、不活跃费、调账、分红、信用额发放/回收。
交易账户表的 balance / equity / margin 字段仅是撮合引擎定期同步的快照,不是权威值。
这意味着:账户体系不直接 UPDATE 资金字段,只写 ledger。
🛡️
原则 3: 身份-账户分离 + MANAGER 资金隔离
t_user 与 t_trading_account 通过 t_user_account_relation 多对多连接,角色 OWNER / MANAGER / VIEWER。
MANAGER 永远不能出金(can_withdraw=0 硬约束),即资管经理/跟单管理者/家人代操——只能交易,不能划走客户资金。这是客户资金隔离的硬底线。
💡 附加原则:④ 多实体隔离(entity_id 贯穿) ⑤ 关键属性变更全部走 audit_log ⑥ 异步通知用 MQ + 幂等 + 乱序防护 ⑦ Demo 与 Live 走相同代码路径,仅 environment 字段区分

🔢 实际杠杆计算公式
# 实际杠杆 = MIN(账户, 组, 品种, 动态模板)

final_leverage = MIN(
    account.leverage,                  # 账户选定杠杆
    group.max_leverage,                # 组允许最大
    client_category_max,               # ESMA 零售30:1, 专业500:1, ECP不限
    COALESCE(symbol_cfg.max_leverage_override, 99999),  # 品种覆盖
    COALESCE(tier_leverage, 99999)     # 动态杠杆模板(若启用)
)

# 保证金计算(按 margin_calc_model 分支)
if margin_calc_model == 1:  # 杠杆名义额(默认)
    margin = notional / final_leverage
elif margin_calc_model == 2:  # Reg-T(美股)
    margin = regt_calc(price, volume)
elif margin_calc_model == 3:  # 组合保证金
    margin = portfolio_margin(positions, prices)
elif margin_calc_model == 4:  # SPAN(期货)
    margin = span_calc(scenario_array)

🔄 资金快照与 ledger 一致性(账户表 vs ledger)
# t_trading_account.balance 是 SNAPSHOT,不是权威值

权威值推导(实时精确):
  balance = SELECT balance_after FROM t_account_ledger
            WHERE account_id = ? ORDER BY id DESC LIMIT 1

快照同步(撮合引擎定时任务):
  every 1s:
    for each 活跃账户:
      actual_balance = compute from t_account_ledger
      if actual_balance != account.balance:
        UPDATE t_trading_account SET balance = actual_balance, synced_at = NOW()

⭕ 账户表 balance / equity / margin_used / margin_free / floating_pnl / frozen_balance
   全部为快照,撮合引擎负责定期同步,本体系不维护

🔒 数据一致性策略
⭕ 强一致(本地事务):
  · 开户: 同时写 t_trading_account + t_user_account_relation + t_audit_log
            → 单库事务, 全部成功或全部回滚
  · 授权: 写 relation + 自动 check 至少保留一种凭证 + audit_log
            → 单库事务
  · 状态变更: 改 status + audit_log
            → 单库事务

⭕ 最终一致(异步 MQ):
  · 下游通知(撮合/风控/通知): 发 MQ, 失败重试 3 次, 入死信
  · 账户激活: FirstDepositEvent 消费端幂等激活
  · 缓存失效: 改组后推 MQ + 主动 DEL Redis

⭕ 弱一致(快照):
  · 账户资金字段(balance/equity): 撮合引擎定期同步(1s), 非实时
  · 资管汇总字段(AUM/investor_count): 资管结算子定期写(1min)

FirstDepositEvent 丢失 账户长期 status=0, 用户无法交易 MQ 重试 3 次 + 死信;定时对账任务扫描 status=0 且 ledger 有入金记录的账户补激活
账户参数变更事件丢失 撮合引擎用旧 status / leverage MQ 重试 + 兜底定时任务每 5 分钟对账

Redis 缓存击穿 DB 被打爆 缓存预热 + 单飞(SingleFlight)防穿透 + 永不过期+异步刷新
用户密码暴力破解 账户被盗 fail_count 5 次锁定 30min;2FA 必须;改密刷 audit
MANAGER 试图出金 资金风险 can_withdraw=0 硬约束 + 资金系统二次校验 MANAGER 关系
不活跃费重复扣 用户投诉 分布式锁 + Redis SET per_day_charged + 幂等 ledger 流水号
