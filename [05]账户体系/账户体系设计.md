# 账户体系设计稿（优化版）

> 对标全球主流交易/清算/经纪科技厂商进行优化。优化目标：**在不破坏现有系统（用户体系 / 资金系统 / 撮合引擎）约束的前提下，补齐保证金模型、动态杠杆、资管(MAM/PAMM)、多实体隔离、休眠与不活跃管理、客户监管分类等行业标准能力**。

---

## 一、设计理念与全球对标

### 1.1 对标厂商与采纳决策

| # | 厂商 | 其核心能力 | 本设计采纳 / 取舍 |
|---|------|-----------|------------------|
| 1 | **MetaQuotes (MT4/MT5)** | Group 承载全套交易参数；Account/Login 主体；动态杠杆(Dynamic Leverage)；MAM/PAMM；Netting/Hedging | **主架构基线**：账户组=一套完整参数；账户=Login 主体；采纳动态杠杆模板与 MAM/PAMM |
| 2 | **Spotware (cTrader)** | cTID 身份与交易账户分离；一身份多账户；Netting 优先；Asset 多币种 | 采纳"身份-账户分离"（`t_user_account_relation`）；**多币种 Asset 模型不采纳**（见 1.3 决策） |
| 3 | **Devexperts (DXtrade)** | 多资产、组合保证金(Portfolio Margin)、风险维度保证金 | 抽象 `margin_calc_model`，预留组合保证金枚举 |
| 4 | **Apex Fintech** | 美股清算托管；账户类型体系(Cash/Margin/IRA/Joint/Trust/Corp)；Reg-T 保证金；受益所有人；不活跃/休眠 | 采纳账户类型体系 `account_type` 与不活跃/休眠管理；Reg-T 纳入 `margin_calc_model` |
| 5 | **DriveWealth** | API-First；Money Manager/RIA 程序化资管；碎股 | 资管以 `t_money_manager` 承载；碎股在 CFD 场景由 `lot_step` 覆盖，不单列 |
| 6 | **TradingView** | 多券商接入、统一前端、账户别名展示 | 采纳账户别名(`display_name`)与可见性/排序展示位 |
| 7 | **Trading Technologies (TT)** | 期货 FCM；账户级盘前风控(持仓/敞口/信用/日内亏损)、自成交防护 | 账户级风控阈值下沉到账户组 + 账户级覆盖；SPAN 纳入 `margin_calc_model` |
| 8 | **Broadridge** | 多实体账簿(Books & Records)；主-子账户层级；账户编号规范 | 账户主体落 `entity_id` 实现分实体账簿；MAM 以主-子结构承载 |
| 9 | **Talos** | 机构加密；子账户、多场所、Prime 经纪 | 子账户复用 MAM 主-子结构；多场所属撮合/路由网关范畴，不在账户体系 |
| 10 | **恒生电子 (Hundsun)** | 资金账户/证券账户分离；三方存管；客户资金隔离 | 采纳"客户资金隔离/托管实体"标识；不引入证券账户拆分（CFD 单账户即资金+持仓主体） |

### 1.2 核心设计原则

1. **账户组即参数中枢（MT5 Group 思路）**：一个账户组 = 监管实体 + 保证金模型 + 杠杆策略 + 风控阈值 + 可交易品种 + 手续费规则的完整组合，开户即归组继承。
2. **身份与账户解耦（cTrader cTID 思路）**：用户(`t_user`) 与交易账户(`t_trading_account`) 多对多，经 `t_user_account_relation` 承载 OWNER/MANAGER/VIEWER 多角色，支撑资管代客与观摩。
3. **多监管实体隔离（Broadridge / ESMA-FCA-ASIC）**：账户组与账户均落 `entity_id`，开户时按用户实体过滤可选组，账簿天然分实体，满足分牌照监管与客户资金隔离。
4. **资金变动唯一入口不变**：所有余额变动仍只经 `t_account_ledger`（资金系统已定义），账户表资金字段为引擎同步快照，**本次优化不改变此契约**。
5. **保证金模型可插拔**：以 `margin_calc_model` 抽象杠杆名义额 / Reg-T / 组合保证金 / SPAN，当前 CFD 场景默认杠杆名义额，预留多资产演进位。
6. **合规留痕完整**：关键属性变更、生命周期事件全部进审计日志，满足 FCA/ASIC/CySEC 追溯要求。

### 1.3 关键设计决策（显式声明取舍）

- **【决策一】单账户单币种 ✅，不引入 cTrader 式多币种 Asset 钱包。**
  理由：现有 `t_account_ledger`、资金系统(入金/出金/内部转账/原路退回)、撮合引擎内存均以"账户单一结算币种"为前提。引入多币种钱包将级联重构资金与撮合两大子系统，收益（少量跨币种便利）远小于成本。跨币种需求由"同用户多账户 + 内部转账换汇"满足（资金系统已支持）。
- **【决策二】MAM/PAMM 仅承载"关系与配置"，盈利分配/业绩费计提/高水位线计算交由独立"资管结算"子系统。**
  理由：账户体系应保持轻量稳定；分配算法与业绩费计提是高频计算与复杂结算逻辑，属独立域。
- **【决策三】账户 owner 冗余落库（`owner_user_id`），但 `t_user_account_relation` 仍为多角色权威来源。**
  理由：99% 查询需要"账户归属人"，冗余 owner 避免每次回查关系表；多人多角色场景仍以关系表为准。

## 四、核心表 SQL

### 表1: 账户组 `t_account_group`（增强）

> 优化点：新增 `entity_id`(多实体)、`account_type`(账户类型体系)、`margin_calc_model`(保证金模型)、不活跃/休眠管理字段；修复原稿 `leverage_template_id` 后缺失逗号的语法问题。

```sql
-- ----------------------------------------------------------------
-- 账户组定义
--
-- 设计依据:
--   MT5:        Group 包含 Leverage/Margin/Commission/Symbols 全套参数
--   Apex:       账户类型体系 (Cash/Margin/Corp/Joint/Trust)
--   Devexperts: 保证金模型抽象 (杠杆名义额/组合保证金)
--   Broadridge: 多实体账簿, 账户组绑定监管实体
--
-- 命名规范: {环境}_{类型}_{币种}, 如 LIVE_STD_USD / LIVE_ECN_USD / DEMO_STD_USD
-- ----------------------------------------------------------------
CREATE TABLE `t_account_group` (
    `id`                        BIGINT UNSIGNED     NOT NULL AUTO_INCREMENT COMMENT '自增主键',
    `group_code`                VARCHAR(32)         NOT NULL COMMENT '组代码,全局唯一,如LIVE_STD_USD',
    `group_name`                VARCHAR(128)        NOT NULL COMMENT '组名称,如Standard USD',
    `description`               VARCHAR(512)        DEFAULT NULL COMMENT '组描述',

    -- ==================== 分类与监管 ====================
    `entity_id`                 BIGINT UNSIGNED     NOT NULL DEFAULT 1
        COMMENT '[新增]所属监管实体ID,与用户体系t_user.entity_id对齐;开户时按用户实体过滤可选组,实现牌照隔离与客户资金隔离',
    `regulatory_entity`         VARCHAR(32)         DEFAULT NULL
        COMMENT '监管实体标签(冗余,人类可读): ASIC/CYSEC/FCA/SVG/VFSC/NONE, NULL=未指定',
    `environment`               TINYINT UNSIGNED    NOT NULL
        COMMENT '环境: 1-真实(Live) 2-模拟(Demo)',
    `account_type`              TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '[增强]账户类型(对标Apex): 1-个人保证金 2-个人现金 3-企业/机构 4-联名 5-信托;CFD场景以1/2为主',
    `margin_calc_model`         TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '[新增]保证金计算模型(对标Devexperts/Apex/TT): 1-杠杆名义额(Forex/CFD,MT5默认) 2-RegT法定保证金(美股) 3-组合保证金(PortfolioMargin) 4-SPAN期货保证金',
    `currency`                  VARCHAR(8)          NOT NULL DEFAULT 'USD'
        COMMENT '结算币种,ISO4217,如USD/EUR/BTC;本体系单账户单币种(见设计决策一)',

    -- ==================== 杠杆 ====================
    `default_leverage`          INT UNSIGNED        NOT NULL DEFAULT 100
        COMMENT '默认杠杆,如100表示1:100',
    `max_leverage`              INT UNSIGNED        NOT NULL DEFAULT 500
        COMMENT '该组允许的最大杠杆',
    `leverage_options`          VARCHAR(256)        DEFAULT NULL
        COMMENT '可选杠杆值,逗号分隔,如: 1,10,50,100,200,500',
    `leverage_template_id`      BIGINT UNSIGNED     DEFAULT NULL
        COMMENT '[修复]动态杠杆模板ID,关联t_leverage_template;非空=按敞口/净值分级浮动杠杆(MT5 Dynamic Leverage),NULL=用default_leverage固定杠杆',

    -- ==================== 风控参数 ====================
    `margin_call_level`         DECIMAL(6,2)        NOT NULL DEFAULT 100.00
        COMMENT '追保预警比例(%),保证金水平(净值/已用保证金)低于此值触发预警',
    `stop_out_level`            DECIMAL(6,2)        NOT NULL DEFAULT 50.00
        COMMENT '强平比例(%),保证金水平低于此值系统自动平仓',
    `stop_out_mode`             TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '强平模式: 1-亏损最大优先 2-保证金占用最大优先 3-Smart部分平仓(与《风控系统设计》对齐)',
    `max_positions`             INT UNSIGNED        NOT NULL DEFAULT 200
        COMMENT '最大同时持仓订单数(已成交的活跃持仓)',
    `max_pending_orders`        INT UNSIGNED        NOT NULL DEFAULT 200
        COMMENT '最大挂单数(未触发的限价单/止损单)',
    `max_lots`                  DECIMAL(12,2)       DEFAULT NULL
        COMMENT '最大总持仓手数,NULL=不限制',
    `daily_loss_limit`          DECIMAL(18,2)       DEFAULT NULL
        COMMENT '[新增]组级单日最大亏损(结算币种),对标TT账户级风控,触发禁开新仓;NULL=不限,账户级可覆盖',
    `negative_balance_protection` TINYINT UNSIGNED  NOT NULL DEFAULT 1
        COMMENT '负余额保护: 0-关闭(客户可能欠券商钱) 1-开启(余额不会低于零,ESMA零售客户强制开启)',

    -- ==================== 交易能力 ====================
    `allow_expert`              TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '是否允许EA/自动交易: 0-禁止 1-允许',
    `allow_hedging`             TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '持仓记账模式(对标MT5 Netting/Hedging): 0-净额(Netting,同品种合并) 1-对冲(Hedging,多空并存)',

    -- ==================== 入金与不活跃管理 ====================
    `min_deposit`               DECIMAL(18,2)       NOT NULL DEFAULT 0
        COMMENT '最低入金要求(结算币种)',
    `inactivity_days`           INT UNSIGNED        DEFAULT NULL
        COMMENT '[新增]连续无交易达此天数视为不活跃,触发不活跃费(对标Apex/经纪商通行做法),NULL=不收取',
    `inactivity_fee`            DECIMAL(18,2)       DEFAULT NULL
        COMMENT '[新增]不活跃费(每月,结算币种),经t_account_ledger以FEE类型扣收',
    `dormant_days`              INT UNSIGNED        DEFAULT NULL
        COMMENT '[新增]连续无活动达此天数自动转休眠(dormant),NULL=不自动休眠',

    -- ==================== 模拟账户专属 ====================
    `demo_initial_balance`      DECIMAL(18,2)       DEFAULT NULL
        COMMENT '模拟账户初始虚拟资金,仅Demo环境有效',
    `demo_expire_days`          INT UNSIGNED        DEFAULT NULL
        COMMENT '模拟账户有效天数,NULL=永不过期,仅Demo环境有效',

    -- ==================== 开户展示 ====================
    `is_default`                TINYINT UNSIGNED    NOT NULL DEFAULT 0
        COMMENT '是否默认开户组,开户页面未选择时使用,同一(entity_id,environment)下最多一个',
    `is_visible`                TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '是否在开户页面展示: 0-隐藏(仅管理员可分配) 1-展示(用户可自选)',

    -- ==================== 状态 ====================
    `status`                    TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '组状态: 0-停用(不允许新开户) 1-启用',
    `sort_order`                INT                 NOT NULL DEFAULT 0
        COMMENT '排序权重,用于开户页面展示顺序',

    `created_at`                DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    `updated_at`                DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_group_code` (`group_code`),
    KEY `idx_entity_env` (`entity_id`, `environment`, `status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='账户组定义';
```

### 表2: 账户组手续费规则 `t_account_group_commission`（保持）

> 手续费规则单独建表，因为一个账户组对不同品种类别可有不同手续费结构。MT5 同样把 Commission 作为 Group 的子配置。

```sql
-- ----------------------------------------------------------------
-- 账户组手续费规则
--
-- 设计依据:
--   MT5: 每个Group可按Symbol Group配置不同Commission
--   IB:  按Asset Class有不同佣金阶梯
--
-- 计算示例:
--   FOREX, charge_type=1, charge_value=7.00 → 交易1手EURUSD佣金7.00 USD
--   STOCK, charge_type=3, charge_value=0.001 → 交易10000 USD的AAPL佣金10.00 USD
-- ----------------------------------------------------------------
CREATE TABLE `t_account_group_commission` (
    `id`                        BIGINT UNSIGNED     NOT NULL AUTO_INCREMENT COMMENT '自增主键',
    `group_id`                  BIGINT UNSIGNED     NOT NULL COMMENT '关联账户组ID',

    `symbol_category`           VARCHAR(32)         NOT NULL
        COMMENT '适用品种类别: ALL/FOREX/METALS/ENERGIES/INDICES/STOCKS/CRYPTO',
    `symbol_pattern`            VARCHAR(64)         DEFAULT NULL
        COMMENT '品种匹配模式(可选,更精确): 如EURUSD,GBPUSD或留空表示该类别全部',

    `charge_type`               TINYINT UNSIGNED    NOT NULL
        COMMENT '收费类型: 1-每手固定金额 2-每笔固定金额 3-按成交金额比例 4-按成交点数',
    `charge_value`              DECIMAL(18,6)       NOT NULL
        COMMENT '收费值: 类型1为每手金额, 类型3为费率(0.001=千分之一)',
    `charge_currency`           VARCHAR(8)          NOT NULL DEFAULT 'USD'
        COMMENT '手续费币种(固定金额时有效)',
    `charge_moment`             TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '收取时机: 1-开仓时全额 2-开平各半 3-平仓时全额',

    `min_charge`                DECIMAL(18,6)       DEFAULT NULL COMMENT '单笔最低手续费,NULL=无下限',
    `max_charge`                DECIMAL(18,6)       DEFAULT NULL COMMENT '单笔最高手续费,NULL=无上限',

    `created_at`                DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    `updated_at`                DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_group_category` (`group_id`, `symbol_category`, `symbol_pattern`),
    KEY `idx_group_id` (`group_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='账户组手续费规则';
```

### 表3: 账户组品种配置 `t_account_group_symbol_config`（保持）

> 控制每个账户组可交易哪些品种，以及品种级参数覆盖。对应 MT5 中 Group → Securities 配置。

```sql
-- ----------------------------------------------------------------
-- 账户组品种配置
--
-- 设计依据: MT5 每个Group的Securities设置,控制可交易品种及品种级杠杆/手数限制
--
-- 使用方式:
--   无配置记录 → 该组不能交易此类品种
--   有记录且enabled=1 → 可交易,使用此处参数覆盖账户组默认值
-- ----------------------------------------------------------------
CREATE TABLE `t_account_group_symbol_config` (
    `id`                        BIGINT UNSIGNED     NOT NULL AUTO_INCREMENT COMMENT '自增主键',
    `group_id`                  BIGINT UNSIGNED     NOT NULL COMMENT '关联账户组ID',

    `symbol_category`           VARCHAR(32)         NOT NULL
        COMMENT '品种类别: FOREX/METALS/ENERGIES/INDICES/STOCKS/CRYPTO',
    `symbol_pattern`            VARCHAR(64)         DEFAULT NULL
        COMMENT '具体品种(可选): 如BTCUSD, 留空表示该类别全部',

    `enabled`                   TINYINT UNSIGNED    NOT NULL DEFAULT 1
        COMMENT '是否允许交易此品种: 0-禁止 1-允许',

    -- ==================== 品种级参数覆盖(NULL=用账户组默认) ====================
    `max_leverage_override`     INT UNSIGNED        DEFAULT NULL
        COMMENT '该品种最大杠杆覆盖,如加密货币限制1:5',
    `min_lot`                   DECIMAL(12,4)       DEFAULT NULL COMMENT '最小交易手数,如0.01',
    `max_lot`                   DECIMAL(12,4)       DEFAULT NULL COMMENT '单笔最大交易手数,如100',
    `lot_step`                  DECIMAL(12,4)       DEFAULT NULL COMMENT '手数步进,如0.01',

    `trade_mode`                TINYINT UNSIGNED    DEFAULT NULL
        COMMENT '交易模式覆盖: 1-完整交易 2-仅平仓 3-禁止交易',

    `created_at`                DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    `updated_at`                DATETIME(3)         NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_group_symbol` (`group_id`, `symbol_category`, `symbol_pattern`),
    KEY `idx_group_id` (`group_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='账户组品种配置';
```

### 表4: 动态杠杆模板头 `t_leverage_template`（新增）

> 原稿 `t_account_group.leverage_template_id` 为悬空引用，此处补齐。对标 MT5 Dynamic Leverage、cTrader 与主流外汇商"敞口越大杠杆越低"的风险控制。

```sql
-- ----------------------------------------------------------------

## 七、关键链路

### 7.1 下单参数查询链路（增强：保证金模型 + 动态杠杆）

```
用户下单 → 确定交易账户
  → 查 t_trading_account: status/trade_enabled/leverage/client_category/group_id/currency
  → 查 t_account_group: margin_calc_model/margin_call_level/stop_out_level/allow_hedging/leverage_template_id
  → 查 t_account_group_symbol_config: 该品种是否allowed, 手数限制, max_leverage_override
  → 若 leverage_template_id 非空(动态杠杆):
        计算当前基准(净值或名义敞口) → 查 t_leverage_tier 落入区间 → 得 tier_leverage
  → 实际杠杆 = MIN(账户leverage, 账户组max_leverage, 品种max_leverage_override, tier_leverage)
  → 按 margin_calc_model 计算所需保证金:
        模型1(杠杆名义额): margin = 名义额 / 实际杠杆
        模型2/3/4: 预留(RegT/组合保证金/SPAN)
  → 查 t_account_group_commission 计算佣金
  → 校验: 持仓数<max_positions, 手数∈[min_lot,max_lot], 可用保证金充足, 客户分类杠杆合规
  → 通过 → 提交撮合引擎
```

### 7.2 开户链路（多实体隔离）

```
用户申请开户(已通过Onboarding/KYC)
  → 取 t_user.entity_id
  → 查 t_account_group: WHERE entity_id=用户实体 AND is_visible=1 AND status=1 (按实体过滤,牌照隔离)
  → 用户选择组(或用is_default组)
  → 校验: 客户分类(client_category)是否允许该组杠杆(ESMA零售限杠杆)
  → 创建 t_trading_account: 继承组的entity_id/currency/leverage, 生成account_no, owner_user_id=申请人
  → 创建 t_user_account_relation: relation_type=OWNER, can_trade=1, can_withdraw=1
  → 写 t_trading_account_audit_log: action=STATUS_CHANGED(创建)
  → status=0(待激活), 首次入金后转1(活跃)
```

### 7.3 资管(MAM/PAMM)链路

```
1. 基金经理开操盘主账户 → 创建 t_money_manager(master_account_id, 收费配置)
2. 投资人加入: 创建 t_mam_allocation(manager_id, investor_account_id, allocation_method)
                + t_user_account_relation(基金经理对investor账户: MANAGER, can_trade=1, can_withdraw=0)
3. 经理在master下单成交 → 分配引擎按allocation_method分摊到各investor账户(独立子系统)
4. 结算周期到 → 资管结算子系统按high_water_mark计提业绩费/管理费, 经t_account_ledger以FEE扣收
   (账户体系仅提供关系与配置, 不做份额与业绩费计算)
```

### 7.4 不活跃与休眠处理（定时任务）

```
每日定时任务:
  → 扫描 t_trading_account WHERE status=1
  → 若 NOW()-last_activity_at ≥ 组.inactivity_days: 经ledger扣不活跃费(FEE)
  → 若 NOW()-last_activity_at ≥ 组.dormant_days: status→5(休眠), dormant_at=NOW(),
        写审计 action=DORMANT
  → 休眠账户再次登录/入金: status→1, dormant_at=NULL, 写审计 action=REACTIVATED
```

## 八、与其他系统的边界交互

| 系统 | 交互方式 |
|------|----------|
| **用户体系** | 账户 `owner_user_id`/`entity_id` 源自 `t_user`；`client_category` 由 `t_user_profile_snapshot.is_pro_client` 映射；开户按 `entity_id` 过滤可选组。 |
| **资金系统** | 余额变动经 `t_account_ledger` 唯一入口；`frozen_balance` 快照对应 `t_account_fund_freeze` 明细；不活跃费以 `FEE` 类型入账。 |
| **撮合引擎** | 账户资金字段(balance/equity/margin)为引擎内存的定期同步快照；下单按 §7.1 链路取参；动态杠杆与保证金模型在引擎侧实时计算。 |
| **风控系统** | 账户/组级 `daily_loss_limit`、`stop_out_level` 等硬阈值同步风控引擎；`blocked_reason` 由风控/合规回写。 |
| **资管结算子系统** | 读取 `t_money_manager`/`t_mam_allocation` 的关系与收费配置，负责份额/业绩费/高水位线计算并经 ledger 计提，账户体系不参与计算。 |
| **代理(IB)系统** | 账户归属经 `owner_user_id` → `t_user.introducing_broker_id` 溯源分润，账户体系仅提供归属。 |
