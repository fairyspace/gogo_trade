# 行情系统设计 — 终版 v2.0

> **版本**：v2.0 Final（与品种体系 v2.0 / 路由网关 v2.x / 时间配置 v2 对齐）
> **范围**：FX / 贵金属 / 能源 / 股指 / CFD / 加密 / 美股等可报价品种的实时与历史行情
> **定位**：行情系统是**报价处理与分发权威源**——产出 Core SymbolBook（引擎用）与 Client Quote（客户可见），不拥有品种规格、路由规则、交易时段的配置权威。
> **运行时分层**：热路径（ingest / quote-engine）+ WS Gateway + Redis 热态 + ClickHouse 历史 + Kafka 异步扇出；内部序列化可采用 FlatBuffers 等零拷贝格式；配置权威在 MySQL。**方向设计不锁定实现语言。**

---

## 一、设计哲学与全球对标

### 1.1 六条设计原则

1. **单一处理链**——映射 / 池聚合 / 品种过滤只在 quote-engine 做一次，禁止 ingest、路由文档、引擎各做一遍。
2. **两层报价**——Core Book（无客户 markup）供撮合/估值/强平；Client Quote（Core + `t_markup_profile`）供图表与下单展示。
3. **热路径不碰库**——配置启动全量入内存，变更经 `t_config_version` 原子切换；Tick 落库 / Redis / WS 全部异步。
4. **可复算**——历史只存 Normalized Tick（客户加点前）；客户可见价可用 profile 复算，不落 N×品种副本。
5. **源可降级**——池成员 STALE / 偏离过滤 / `min_active_sources` 降级；断流标 STALE，不静默用过期价成交。
6. **边界不越界**——时段权威在时间配置；规格在品种；商业加点配置在路由；行情只执行与分发。

### 1.2 全球对标与采纳决策

| # | 厂商 | 采纳 |
|---|------|------|
| 1 | **MetaQuotes** | Point/Tick；Tick flags；Group/Profile 分层加点思想 → Core vs Client |
| 2 | **Spotware (cTrader)** | 多资产归一化；订阅主题；深度增量 |
| 3 | **Devexperts** | 多场所符号；Book 快照 + 增量恢复 |
| 4 | **Talos / TT** | 多 LP 映射 + 聚合；源健康度与 failover |
| 5 | **TradingView** | REST 拉历史 K 线 + WS 推增量；`t/o/h/l/c/v` 契约 |
| 6 | **DriveWealth / Apex** | 交易状态驱动行情状态（OPEN / HALT / CLOSE_ONLY） |
| 7 | **Broadridge / 恒生** | 公司行动归品种；行情只消费调整后价格序列，不内嵌除权引擎 |

### 1.3 八个定稿决策

| # | 决策 | 定稿 |
|---|------|------|
| 1 | 价格内部表示 | 热路径 `int64` 定点：`scaled = decimal × 10^digits`；API/Redis 对外十进制字符串 |
| 2 | 落库 Tick | Normalized Tick = 映射 + 池聚合 + 品种过滤后、**客户加点前**；含 `seq` / `gateway_id` / `pool_id` |
| 3 | 价格预警 | `t_price_alert` 由 Alert Worker 消费 Kafka/RingBuffer 副本，**不进** quote-engine 热循环 |
| 4 | CH 分区 | `PARTITION BY toYYYYMMDD(event_date)`（或周）；`ORDER BY (symbol_id, tick_time, seq)` |
| 5 | K 线 | 内存维护未收盘 M1；收盘写 `klines_m1`；M5+ 异步升采样；Redis 仅最近 N 根 |
| 6 | 深度 | MVP：BBO 必做；L2 快照/增量可选（`market_depth>0` 且源有深度） |
| 7 | 序列与恢复 | 每 `symbol_id` 单调 `seq`；WS：snapshot + gap-fill；过期标 STALE |
| 8 | SLO | Tick→Book P99 &lt; 1ms；Book→Redis/WS P99 &lt; 50ms（同机房）；CH 落库允许秒级，不阻塞交易 |

## 二、模块边界与依赖契约

### 2.1 行情系统负责 / 不负责

| 职责 | 行情 | 归属 |
|------|:----:|------|
| LP/交易所协议接入与归一化 | ✅ | 本模块 `quote-ingest` |
| 第一层 mapping 加点 / 精度缩放 | ✅ | 执行路由表配置 |
| 流动性池聚合执行 | ✅ | 配置权威在路由 `t_liquidity_pool*` |
| 品种过滤 / 跳空 / 时段点差 schedule | ✅ | 品种字段 + 本模块 schedule 表 |
| Core SymbolBook | ✅ | 本模块 |
| 异步 Redis / ClickHouse / K 线 | ✅ | 本模块 |
| Client Quote 编排与 WS 推送 | ✅ | 本模块 + WS Gateway |
| 报价质量监控 | ✅ | 本模块 |
| `t_markup_profile` 配置权威 | ❌ | 路由网关 v2 |
| 交易时段 / DST | ❌ | 时间配置 v2（`t_daily_session`） |
| 品种规格 digits/tick_size/gap | ❌ | 品种体系 v2（行情只读） |
| 成交 / 挂单触发 / 强平 | ❌ | 交易引擎（消费 Core Book） |
| 公司行动除权 | ❌ | 品种体系；行情不重算历史除权 |

### 2.2 只读依赖字段清单

**`t_symbol`（品种）**

| 字段 | 用途 |
|------|------|

| `spread_mode`, `spread_fixed`, `spread_limit_min/max`, `spread_balance` | 点差约束与平滑 |
| `filter_bad_ticks`, `allow_negative_price` | 坏 Tick / 负价 |
| `gap_mode`, `gap_threshold_l1/l2/l3` | 跳空 |
| `market_depth`, `chart_mode` | 深度档数、K 线用 Bid 或 Last |
| `primary_exchange`, `primary_timezone`, `dst_profile` | 关联时段解析（不存时段本身） |

**`t_gateway` + `t_gateway_symbol_mapping`（路由）**

| 字段 | 用途 |
|------|------|
| `gateway_type=QUOTE`, `protocol`, `heartbeat_interval_ms`/`request_timeout_ms` | 接入 |
| `remote_symbol`, `price_digits_remote`, `price_scale_factor`, `volume_scale_factor` | 归一化 |
| `bid_markup_points`, `ask_markup_points` | **第一层**加点 |
| `max_spread_points`, `max_deviation_points`, `quote_depth` | 源侧过滤 |
| `use_for_quote` | 是否参与报价 |

**`t_liquidity_pool` + `t_liquidity_pool_member`（路由）**

| 字段 | 用途 |
|------|------|
| `aggregation_mode`, `stale_quote_threshold_ms`, `min_active_sources`, `price_deviation_threshold_bps` | 聚合与健康度 |
| `priority`, `weight`, `status` | 成员选择 |

**`t_markup_profile`（路由，仅 Client 路径）**

| 字段 | 用途 |
|------|------|
| `default_bid_markup_points`/`default_ask_markup_points`, `symbol_overrides`, `category_overrides`, `session_overrides` | 第二层客户加点 |

**时间配置**：`t_daily_session`（及 `t_session_override`）→ 写入 `quote:status`。

### 2.3 点差两层模型（与路由对齐）

```
上游原始报价
  → mapping: 缩放 + 第一层 bid/ask_markup_points
  → (可选) pool 聚合
  → spread_schedule: 时段附加 / override（流动性侧）
  → symbol: spread_limit_min/max、坏 Tick、跳空
  → Core SymbolBook                    ← 引擎/估值/强平
  → + markup_profile（客户侧）          ← Client Quote → WS
```

- **`t_symbol_spread_schedule`**：对 Core Book 生效（全站流动性侧点差）。
- **`t_markup_profile.session_overrides`**：仅 Client Quote（VIP/标准组等商业差异）。
- A-Book 执行吃上游成交价，不吃 Client 展示价；B-Book 成交参考 **Core Book**（或规则另行规定的内部价），展示仍可用 Client Quote。

## 三、整体运行时架构

### 3.1 数据流

```
  上游 LP / 交易所
       │
       ▼
  quote-ingest (可多实例, 按 gateway 拆分)
       │  FIX / WS / REST → FlatBuffers NormalizedRaw
       ▼
  quote-engine (Active-Standby)
       │  映射 → 池聚合 → 过滤 → Core Book
       │
       ├──► SHM SPSC ──────────────► Trading Engine (挂单/SLTP/浮盈)
       │
       └──► Async Fanout (不阻塞 Book)
              ├── Redis (Core 热态)
              ├── Kafka topic: md.tick.normalized
              ├── ClickHouse Writer (批量)
              ├── Kline Aggregator
              ├── Client Quote Service → WS Gateway
              └── Alert Worker (价格预警)
```

### 3.2 进程职责

| 进程 | 路径特征 | 职责 |
|------|---------|------|
| `quote-ingest` | 热路径接入 | 协议适配、心跳、原始接收时间戳、写入引擎入口队列 |
| `quote-engine` | 热路径权威 | 归一化管线、Core Book、seq 分配、SHM 输出、扇出队列 |
| `md-fanout` | 异步扇出 | 批量写 Redis / Kafka / CH |
| `kline-aggregator` | 近热路径 | 未收盘 M1 内存；收盘落库；异步升 M5+ |
| `client-quote` + `ws-gateway` | 连接态推送 | 按连接绑定的 markup_profile 加点推送 |
| `alert-worker` | 异步消费 | 消费 `md.tick.normalized`，匹配 `t_price_alert` |

### 3.3 配置热更新

1. 启动：从 MySQL 加载品种报价参数、mapping、pool、spread_schedule、markup_profile 摘要 → 内存。
2. 变更：管理端写库 → bump `t_config_version` → 引擎拉全量或增量 → **双缓冲原子切换**。
3. 实时路径：**零** MySQL / 零远程配置中心同步调用。

### 3.4 HA、重建与 seq 分配防重复

- quote-engine：**Active-Standby**，切换目标 &lt; 5s。
- **脑裂防护**：Standby 接管前须确认 Active 心跳超时（3 次心跳 = 15s）。
- **Standby 接管时 seq 分配流程**：

```
1. 从 Redis quote:seq:{symbol_id} 读取最后 seq
2. 从 Kafka md.tick.normalized 读取该 symbol 最大 seq
3. 取 max(redis_seq, kafka_seq) + 1 作为新起点
4. 防重复：下游消费者按 (symbol_id, seq) 去重
```

- **Book 重建**：Redis Core 快照 + 追 Kafka `md.tick.normalized` 至最新 seq。
- ingest 无状态可水平扩；同一 `gateway_id` 同一时刻仅一个 Active 写，避免双写。
- 时钟：机房 NTP/PTP；记录 `venue_time` / `recv_time` / `book_time` 三段，用于源偏移监控（时钟漂移处置见十、10.5）。

## 四、协议接入与连接状态管理

上游/下游连接都有明确的会话生命周期、断线恢复与流控策略，属于 `quote-ingest` 与 `ws-gateway` 的正式设计内容，而非事后补丁。

### 4.1 FIX Session 状态机（上游接入）

```
DISCONNECTED
  → LOGON_SENT
  → LOGON_RECEIVED（序列号协商）
  → ESTABLISHED（Heartbeat 30s）
  → LOGOUT_SENT / LOGOUT_RECEIVED
  → DISCONNECTED
```

- **序列号恢复**：启动时从持久化存储加载上次 `incoming_seq`/`outgoing_seq`；不重置。
- **Resend Request**：检测 gap 时发 ResendRequest(begin, end)，对端重发缺失消息。
- **Sequence Reset**：对端主动发 Reset 跳号。
- **Heartbeat 超时**：&gt; 60s 无 TestRequest 响应 → 断连 + 指数退避重连（1s/2s/4s/8s/16s/30s 上限）。
- **Gap Fill**：重发期间非业务消息用 GapFill 跳过。

### 4.2 WebSocket / REST 接入（上游）

- WS：鉴权 + 心跳 15s；断连重连状态机见 4.3。
- REST：轮询兜底，限流规则见八、8.4。

### 4.3 WS 断线重连完整状态机（面向下游客户端）

```
[DISCONNECTED]
  → connect(ws_url, token)
  → [CONNECTING]
  → auth(token) → [AUTHENTICATING]
  → subscribe(channels) → [SUBSCRIBING]
  → server snapshot(seq) → [SYNCING]
  → gap-fill(本地 seq → server seq)
  → [LIVE]
  → stale 检测（> 5s 无 tick）→ [STALE]
  → 断连 → 指数退避 → [CONNECTING]
```

- **resume**：客户端带 `last_seq` 重连，服务端从 `last_seq+1` 补发（最多 1000 条，超限发 snapshot）。
- **snapshot**：服务端推当前全量 `seq` + `bid`/`ask`，客户端覆盖本地。
- **STALE 降级**：客户端展示「行情延迟」提示，不阻止平仓操作。

### 4.4 回压与流控

| 层 | 策略 |
|----|------|
| ingest 队列 | 水位 &gt; 80% 丢低优先级（如深度增量），保 tick；&gt; 95% 全丢并告警 |
| quote-engine 内部 | SPSC 无锁队列，背压反压 ingest |
| fanout Kafka | 生产者缓冲满 → 阻塞 ingest（反压） |
| WS 每连接 | pending 消息 &gt; 1000 → 踢出慢消费者（关闭连接，客户端重连） |
| Redis 写入 | 异步批量，背压不阻塞热路径 |

## 五、归一化 Tick / Book 模型

### 5.1 内部 NormalizedTick（FlatBuffers / 内存）

| 字段 | 类型 | 说明 |
|------|------|------|
| `symbol_id` | uint64 | 品种 ID |
| `seq` | uint64 | 每品种单调递增，引擎分配 |
| `venue_time` | int64 ms | 上游时间（无则 = recv_time） |
| `recv_time` | int64 ms | ingest 收到 |
| `book_time` | int64 ms | 写入 Core Book 完成 |
| `bid`, `ask` | int64 | 定点价；外汇必填 |
| `last` | int64 | 可 0=无；股票/加密常有 |
| `bid_size`, `ask_size`, `last_size` | int64 | 定点手数/数量 |
| `flags` | uint32 | bit0=bid bit1=ask bit2=last bit3=size |
| `gap_level` | int8 | 0 正常 / 1/2/3 |
| `stale` | bool | 超过有效期或源不足 |
| `gateway_id` | uint64 | 主导源（池聚合后可为胜出源） |
| `pool_id` | uint64 | 0=未走池 |

**不存**：客户 markup、day OHLC、累计 day_volume（摘要在 Redis `quote:daily`）。**不含** `entity_id`：本模型是全局归一化口径，客户/实体维度差异化只发生在 Client Quote 推送层，见十三、13.5。

### 5.2 行业映射

| 来源 | 映射 |
|------|------|
| MT5 Tick | bid/ask/last/volume → 本模型；flags 对齐 |
| FIX MDEntryType BID/OFFER/TRADE | 更新 bid/ask/last + flags |
| 交易所逐笔成交 | last + last_size；BBO 另通道或同包 |
| Binance bookTicker | bid/ask + sizes；aggTrade → last |

### 5.3 Core SymbolBook（每品种一份）

```
SymbolBook {
  symbol_id, digits, point_size
  bid, ask, last, sizes
  seq, venue_time, book_time, stale
  gap_level, gateway_id, pool_id
  day_open/high/low/close, day_volume, day_tick_count  // 交易日摘要（内存）
  trading_status  // 来自时间配置 runtime
}
```

## 六、处理管线规格

### 6.1 伪代码（quote-engine 热路径）

```
onRawQuote(raw):
  mapping = maps[raw.gateway_id][raw.remote_symbol]
  if mapping == null or !mapping.use_for_quote: drop

  q = scale(raw, mapping)                    // price_scale_factor, digits
  q.bid -= mapping.bid_markup_points * point // 第一层（符号约定与路由文档一致）
  q.ask += mapping.ask_markup_points * point

  if mapping.max_spread_points and spread(q) > max: drop
  if badDeviation(q, book, mapping.max_deviation_points): drop_or_alert

  if symbol uses pool P:
    updateSourceState(P, gateway, q)
    q = aggregate(P)                         // BEST_PRICE / VWAP / DEPTH_MERGE
    if activeSources(P) < min_active_sources: q.stale = true

  if age(q) > max_quote_age_ms: q.stale = true

  applySpreadSchedule(symbol, now, q)        // extra / override → Core
  applySymbolSpreadMode(symbol, q)           // fixed / balance
  clampSpreadLimits(symbol, q)               // min/max；无法满足则 drop 或 widen 策略见下

  if !symbol.allow_negative and q.bid < 0: drop
  if filter_bad_ticks and jumpPoints(q, book) > threshold: drop; bad_tick++

  gap = classifyGap(symbol, q, book)
  if gap_mode==IGNORE: pass
  if gap_mode==APPLY: q.gap_level = gap; pass
  if gap_mode==ADJUST: adjustTowardPrev(q, book, gap); q.gap_level = gap

  seq = ++book.seq
  commit Core SymbolBook
  shm.publish(book)                          // 引擎
  async.fanout(NormalizedTick)               // Redis/Kafka/CH/Kline/Client/Alert
```

**点差无法满足 `spread_limit_min` 时（定稿）**：向两侧对称拓宽至 min（不做交易的静默），并计入 quality 指标；超过 `spread_limit_max`：**丢弃**该 Tick 并告警（防止极端宽点差进 Book）。

### 6.2 池聚合（执行路由配置）

| `aggregation_mode` | 行为 |
|--------------------|------|
| `BEST_PRICE` | bid=max(有效源 bid)，ask=min(有效源 ask)；交叉则按优先级/中位数修正或丢弃劣源 |
| `VWAP` | 按 size 加权（无 size 则等权） |
| `DEPTH_MERGE` | 合并多源深度再取 BBO / 输出 L2 |

有效源：`now - source.recv_time <= stale_quote_threshold_ms`，且未超 `price_deviation_threshold_bps`。

### 6.3 Client Quote（推送路径，非引擎）

```
clientBid = core.bid + f(profile.bid_markup...)   // 解析顺序: symbol_overrides > category > default
clientAsk = core.ask + f(profile.ask_markup...)
+ session_overrides(OVERNIGHT/WEEKEND/...)
```

- Redis **不**为每个 profile 存一份报价。
- WS 连接建立时解析账户组 → `route_rule` → `markup_profile_id`，推送时计算。
- 历史图表默认用 Core 价序列（与引擎一致）；若产品要求「客户所见即所得」回放，用同一 profile 对 Tick 复算，仍不落库。

## 七、存储设计

### 7.1 分层

| 介质 | 存什么 |
|------|--------|
| MySQL | spread_schedule、price_alert、QUOTE 系统配置 |
| Redis | Core 实时报价、当日摘要、深度、状态、K 线热缓存、seq |
| ClickHouse | Normalized Tick、K 线、质量小时表 |
| Kafka | `md.tick.normalized` 供扇出/告警/重建 |

### 7.2 MySQL — `t_symbol_spread_schedule`

```sql
-- 流动性侧时段点差（作用于 Core Book）
-- 最终 Core 点差 ≈ 聚合后点差 + schedule.extra 或 schedule.override
-- 再受 t_symbol.spread_limit_min/max 约束
-- 客户侧时段加点用 t_markup_profile.session_overrides，不进本表
-- 本表不含 entity_id：属 Core 侧全局配置，客户/实体差异化在路由网关 markup_profile（见十三、13.5）
CREATE TABLE `t_symbol_spread_schedule` (
    `id`                        BIGINT UNSIGNED     NOT NULL AUTO_INCREMENT COMMENT '自增主键',
    `symbol_id`                 BIGINT UNSIGNED     NOT NULL COMMENT '关联品种ID',

    `schedule_name`             VARCHAR(64)         NOT NULL
        COMMENT '时段名称: Asian Session, Sunday Open, Low Liquidity',
    `day_of_week`               TINYINT UNSIGNED    DEFAULT NULL
        COMMENT '星期几: 1~7, NULL=每天都生效',
    `time_from`                 TIME                NOT NULL COMMENT '起始时间(与服务器/交易日日历约定一致, 建议UTC)',
    `time_to`                   TIME                NOT NULL COMMENT '结束时间',

    `extra_spread_points`       INT                 NOT NULL DEFAULT 0
        COMMENT '额外点差(point): 在当前点差上追加, 可为负',
    `spread_override`           INT                 DEFAULT NULL
        COMMENT '非NULL时直接作为目标点差(point), 忽略基础+extra',
    `priority`                  INT                 NOT NULL DEFAULT 0
        COMMENT '优先级: 同一symbol多条规则时段重叠时, 取值大的生效, 避免冲突歧义(原表缺失, 本次补充)',

    `status`                    TINYINT UNSIGNED    NOT NULL DEFAULT 1 COMMENT '0-停用 1-启用',
```

## 九、下游消费与联动流程

### 9.1 停牌 / HALT 联动

```
时间配置 runtime_state=HALT 或 品种 status=SUSPENDED
  → quote-engine 写 quote:status=HALT
  → 发 md.control（status=HALT）
  → 撮合引擎消费 → 拒开仓（TRD_VALIDATION_SYMBOL_NOT_TRADABLE）
  → WS 客户端消费 → 展示"停牌"
  → 恢复时 quote:status=LIVE → md.control（status=LIVE）
```

- 与品种域 `symbol.status_changed.v1` 联动：品种 SUSPENDED → 行情 HALT。
- 与时间域 `t_session_override` 联动：时段 HALT → 行情 HALT。

### 9.2 下游消费契约

| 消费方 | 方式 | 契约 |
|--------|------|------|
| 撮合引擎 | SHM SPSC | Core Book，seq 单调，禁止 Client Quote 做强平 |
| 风控 | SHM 或 Kafka `md.tick.normalized` | 用于 margin_level 计算；STALE 触发强平暂停 |
| 量化 | Kafka `md.tick.normalized` | 回测读 ClickHouse；实盘订阅 |
| 跟单 | Kafka `md.tick.normalized` | 信号源展示用 Client Quote |
| WS 客户端 | WS | Client Quote + K 线 + 深度 |
| 估值/资金 | SHM Core Book | 保证金计算用 Core 价 |

延迟 SLA：Tick → Book P99 &lt; 1ms；Book → Kafka P99 &lt; 5ms；Book → WS P99 &lt; 50ms。

### 9.3 公司行动价格切换

- 消费品种域 `symbol.corporate_action_executed.v1`。
- 除权日 engine 原子切换 book：旧 book 标记 `superseded`，新 book 生效。
- 发 `md.control`（action=CORP_ACTION，new_seq_base）。
- 历史 K 线复权：行情域消费事件后重算 ClickHouse 历史 K 线（前复权/后复权模式可配）。

## 十、可观测性、容灾与运维

### 10.1 延迟三段

| 指标 | 定义 | 目标 |
|------|------|------|
| `md_latency_recv_to_book_ms` | recv_time → book_time | P99 &lt; 1ms |
| `md_latency_book_to_ws_ms` | book_time → WS send | P99 &lt; 50ms（同机房） |
| `md_source_skew_ms` | \|venue_time - recv_time\| | 监控告警，按源定阈值 |

### 10.2 质量指标

- 每源/每品种：tick_count、bad_tick_count、gap_count、stale_ratio、source_switch_count
- 池：active_sources、聚合交叉次数
- CH 写入延迟与堆积（fanout 队列深度）

### 10.3 验收清单

```
□ 1. 单源品种：映射缩放 + 第一层 markup 后 Core 价与手工计算一致
□ 2. 多源池 BEST_PRICE：人为制造一侧 STALE，BBO 切换到另一源且 seq 连续
□ 3. spread_schedule 生效于 Core；markup_profile 仅改变 WS Client 价，引擎 SHM 不变
□ 4. filter_bad_ticks / gap L1~L3 行为符合 t_symbol 配置
□ 5. 超过 max_quote_age_ms 后 Book.stale=true，引擎拒绝用过期价开仓（联调）
□ 6. Redis 仅有 quote:tick Core；两账户不同 profile 同品种 WS 价差 = markup 差
□ 7. M1 收盘写入 CH；杀进程恢复后 Redis 快照 + Kafka 追平 seq
□ 8. CH 分区数为日期级，无按品种分区；90 天 TTL 策略可执行
□ 9. 价格预警：引擎 CPU 剖析无 alert 表扫描；worker 可触发通知
□ 10. 与品种 v2 / 路由 v2 / 时间配置 v2 联通：无 t_symbol_session 依赖
□ 11. SLO 压测：目标品种数下 Tick→Book P99 < 1ms，Book→WS P99 < 50ms
```

### 10.4 运维 Runbook

| 告警 | 处置 |
|------|------|
| `stale_ratio > 10%` | 检查上游 LP 连接，自动切备用源 |
| `source_switch_total 突增` | 排查 LP 不稳定，联系供应商 |
| `tick_to_book_ms P99 > 1ms` | 检查 quote-engine CPU/内存，扩容 |
| `ws_pending_backlog > 1000` | 踢慢消费者，检查客户端 |
| `seq_gap_detected` | 触发 ResendRequest，若持续切主源 |
| `ingest_queue_watermark > 80%` | 扩容 ingest worker，检查上游暴量 |

### 10.5 时钟漂移处理

- NTP 同步：所有节点 chrony 强制 &lt; 10ms 漂移。
- 失步检测：`venue_time` 与本地 `recv_time` 偏差 &gt; 100ms → 告警，标记 tick `time_quality=LOW`。
- 严重失步（&gt; 1s）：暂停该源，切备用。

| **合计** | ~500 | 峰值 50k | 50万 | ~1.3GB |

### 12.2 建议落地顺序

1. **P0**：单源 ingest → 映射/过滤 → Core Book → SHM + Redis + WS（无 profile 加点）
2. **P0**：ClickHouse ticks + M1 + 质量日志
3. **P1**：流动性池聚合 + STALE
4. **P1**：Client markup 推送 + spread
5. **P2**：L2 深度、Alert Worker、高周期升采样与补偿任务
6. **P2**：多机房源就近接入（`t_gateway.data_center`）

## 十三、跨域契约对齐

本域遵循《跨域契约补充卷》的平台级约定，具体对齐如下：

### 13.1 错误码

遵循补充卷 §三命名规则 `<DOMAIN>_<CATEGORY>_<DETAIL>`，使用 `MD_*` 前缀（见八、8.4）；新增错误码需按补充卷 MINOR 版本追加登记，不在域内文档自行定义未登记码。

### 13.2 事件与 Kafka Topic

`md.tick.normalized`、`md.control` 为本域内部事件与 Kafka topic；若下游其它域需要将其作为跨域权威契约直接订阅，应按补充卷 §四 CloudEvents 命名规则补充登记为 `md.tick.normalized.v1` / `md.control.v1`。本域同时作为消费方，接收品种域 `symbol.status_changed.v1`、`symbol.corporate_action_executed.v1`、`symbol.spec_changed.v1`。

### 13.3 角色与权限

停牌等运维操作引用补充卷 §五 `PMS_SYMBOL_MANAGER`（品种域角色）；行情域自身不新增角色，避免与品种域同义角色冲突。

### 13.4 ID 与序列号

`symbol_id` 遵循补充卷 §六由品种服务生成；`seq` 为行情域内分配（每 `symbol_id` 单调递增），failover 防重复流程见三、3.4。

### 13.5 entity_id 裁定

遵循补充卷 §7.4：**行情 Core Book 全局共享，不含 entity_id**；Client Quote 的 `markup_profile` 差异化通过路由网关 `t_route_rule` 按 `entity_id`/`account_group_id` 隔离。本域内唯一贯穿 `entity_id` 的表是用户业务表 `t_price_alert`（见七、7.3）。

### 13.6 容灾 / 容量 / 监控基线

- 容灾：RPO &lt; 1s，RTO &lt; 5s，Active-Standby（对齐补充卷 §十行情行，比平台通用基线更紧）。
- 容量假设：500 品种、峰值 50k tick/s、10 万 WS 连接（对齐补充卷 §十一）。
- 监控：平台通用指标（`requests_total`、`request_duration_seconds` 等，补充卷 §十二）之外，行情域增补 `stale_ratio`、`source_switch_total`、`seq_gap_detected` 等域内指标（见十、10.2/10.4）。

## 结语

本设计把行情收敛为一条可落地的处理链：**接入归一化 → 池聚合 → 品种过滤 → Core Book → 异步扇出**，并用 **Core / Client 两层报价**对齐 MetaQuotes / cTrader 的商业加点与机构侧引擎计价分离。存储上采用 MySQL 配置 / Redis 热态 / ClickHouse 历史三层；协议接入层明确 FIX Session、WS 重连、回压流控的完整状态机；对外通过下游消费契约、停牌联动、公司行动价格切换与撮合、风控、量化、跟单等域对齐。全篇字段级对齐已定稿的品种 / 路由 / 时间配置模块，实现语言与部署拓扑另立项，不写入本方向稿。
