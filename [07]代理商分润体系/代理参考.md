### 1.1 业务全景与角色定义

```
金融代理商生态角色

券商/平台 (甲方)
    │
    ├── 一级代理商 (IB/渠道商)
    │       ├── 二级代理商 (子代理)
    │       │       └── 三级代理商 (个人推广员)
    │       │               └── 终端用户 (被推荐客户)
    │       └── 直客 (代理商直接开发客户)
    │
    ├── 机构代理 (基金销售/保险代理)
    │       └── 终端用户
    │
    └── 个人KOL代理
            └── 终端用户

角色说明：
┌────────────────┬────────────────────────────────────────────┐
│  角色           │  说明                                       │
├────────────────┼────────────────────────────────────────────┤
│ 平台方          │ 券商/基金公司，制定分润规则，最终结算方      │
│ 一级代理商      │ 与平台签约，有独立代理协议，佣金最高         │
│ 二级代理商      │ 由一级发展，分润从一级佣金中分出            │
│ 三级代理商      │ 最末级代理，通常为个人推广                   │
│ 终端客户        │ 实际投资用户，产生交易行为                   │
│ 结算主体        │ 代理商关联的法人/个人(用于税务结算)          │
└────────────────┴────────────────────────────────────────────┘
```

### 2.2 分润计算核心流程

```
分润计算完整流程

交易系统产生成交
        │
        ▼
┌───────────────────────────────────────────────────────────┐
│  Step1: 事件消费 & 幂等校验                                 │
│  ├── 消费 Kafka trade.done 事件                            │
│  ├── 幂等Key: trade_id + rule_version → Redis SETNX       │
│  └── 重复事件直接丢弃                                       │
└────────────────────────┬──────────────────────────────────┘
                         ▼
┌───────────────────────────────────────────────────────────┐
│  Step2: 客户→代理归因查询                                   │
│  ├── customer_id → 查 agent_customer_relation             │
│  ├── 获取归因代理商ID + 归因时间 + 归因类型                 │
│  └── 若无归因 → 平台自营，不计算代理分润                    │
└────────────────────────┬──────────────────────────────────┘
                         ▼
┌───────────────────────────────────────────────────────────┐
│  Step3: 分润规则匹配                                        │
│  ├── 根据 [代理商等级 + 产品类型 + 计费基础] 匹配规则       │
│  ├── 获取生效规则版本 (按时间版本)                          │
│  └── 规则优先级: 个性化规则 > 等级规则 > 默认规则           │
└────────────────────────┬──────────────────────────────────┘
                         ▼
┌───────────────────────────────────────────────────────────┐
│  Step4: 多级分润计算                                        │
│                                                            │
│  平台佣金收入 = 成交金额 × 基础佣金率                        │
│                                                            │
│  一级代理分润 = 平台佣金 × 一级分成比例                      │
│  二级代理分润 = 一级代理分润 × 二级分成比例 (从一级中扣)     │
│  三级代理分润 = 二级代理分润 × 三级分成比例 (从二级中扣)     │
│                                                            │
│  举例：                                                     │
│  成交金额 100万 × 基础佣金率 0.03% = 300元 平台佣金         │
│  300元 × 60%(一级比例)  = 180元 → 一级代理                 │
│  180元 × 30%(二级比例)  =  54元 → 二级代理(从180中扣)      │
│   54元 × 20%(三级比例)  =  10.8元 → 三级代理               │
└────────────────────────┬──────────────────────────────────┘
                         ▼
┌───────────────────────────────────────────────────────────┐
│  Step5: 阶梯/特殊规则处理                                   │
│  ├── 月累计交易量阶梯加成                                    │
│  ├── 新客奖励 (首月额外奖励)                                 │
│  ├── 活动期间临时加成                                        │
│  └── 封顶规则检查                                           │
└────────────────────────┬──────────────────────────────────┘
                         ▼
┌───────────────────────────────────────────────────────────┐
│  Step6: 写入分润明细 & 更新虚拟账户                          │
│  ├── 写 commission_detail (分润明细)                       │
│  ├── 更新 agent_account (虚拟账户余额)                      │
│  ├── 写 account_flow (资金流水)                            │
│  └── 发送分润通知消息                                       │
└────────────────────────────────────────────────────────────┘
```

## 三、完整表结构设计

### 3.1 代理商基础域

```sql
-- ① 代理商主表
CREATE TABLE `agent_info` (
  `agent_id`          BIGINT       NOT NULL AUTO_INCREMENT COMMENT '代理商ID',
  `agent_code`        VARCHAR(32)  NOT NULL COMMENT '代理商编号(业务主键)',
  `agent_name`        VARCHAR(128) NOT NULL COMMENT '代理商名称',
  `agent_type`        TINYINT      COMMENT '类型 1-企业 2-个人 3-KOL 4-机构',
  `agent_level`       TINYINT      COMMENT '代理层级 1-一级 2-二级 3-三级',
  `parent_agent_id`   BIGINT       DEFAULT 0 COMMENT '上级代理商ID(0表示顶级)',
  `root_agent_id`     BIGINT       DEFAULT 0 COMMENT '顶级代理商ID',
  `agent_path`        VARCHAR(512) COMMENT '代理层级路径 /1/2/3/ (便于树查询)',
  -- 企业信息
  `company_name`      VARCHAR(256) COMMENT '公司名称',
  `business_license`  VARCHAR(64)  COMMENT '营业执照号',
  `tax_no`            VARCHAR(64)  COMMENT '税务登记号',
  `legal_person`      VARCHAR(64)  COMMENT '法定代表人',
  -- 个人信息
  `contact_name`      VARCHAR(64)  COMMENT '联系人姓名',
  `contact_mobile`    VARCHAR(11)  COMMENT '联系手机(加密)',
  `contact_email`     VARCHAR(128) COMMENT '联系邮箱',
  `id_card_no`        VARCHAR(64)  COMMENT '身份证号Hash',
  -- 银行账户(结算用)
  `bank_name`         VARCHAR(64)  COMMENT '开户银行',
  `bank_account`      VARCHAR(64)  COMMENT '银行账号(加密)',
  `bank_account_name` VARCHAR(64)  COMMENT '开户名',
  `bank_branch`       VARCHAR(128) COMMENT '开户支行',
  -- 合作信息
  `contract_no`       VARCHAR(64)  COMMENT '合同编号',
  `contract_start`    DATE         COMMENT '合同开始日期',
  `contract_end`      DATE         COMMENT '合同结束日期',
  `commission_level`  VARCHAR(32)  COMMENT '佣金等级代码(关联规则)',
  `region_code`       VARCHAR(32)  COMMENT '负责区域编码',
  `channel_code`      VARCHAR(64)  COMMENT '渠道标识码(唯一)',
  -- 资质审核
  `kyc_status`        TINYINT      DEFAULT 0 COMMENT 'KYC状态 0-未审 1-通过 2-拒绝',
  `kyc_time`          DATETIME     COMMENT 'KYC审核时间',
  `kyc_operator`      VARCHAR(64)  COMMENT '审核人',
  `kyc_remark`        VARCHAR(512) COMMENT '审核备注',
  -- 状态
  `status`            TINYINT      DEFAULT 0 COMMENT '0-待审 1-正常 2-暂停 3-终止',
  `activate_time`     DATETIME     COMMENT '激活时间',
  `suspend_time`      DATETIME     COMMENT '暂停时间',
  `suspend_reason`    VARCHAR(256) COMMENT '暂停原因',
  -- 统计(冗余)
  `total_customer`    INT          DEFAULT 0 COMMENT '累计客户数',
  `active_customer`   INT          DEFAULT 0 COMMENT '活跃客户数',
  `total_commission`  DECIMAL(20,4) DEFAULT 0 COMMENT '累计分润金额',
  `create_time`       DATETIME     DEFAULT CURRENT_TIMESTAMP,
  `update_time`       DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`agent_id`),
  UNIQUE KEY `uk_agent_code` (`agent_code`),
  UNIQUE KEY `uk_channel_code` (`channel_code`),
  KEY `idx_parent_agent` (`parent_agent_id`),
  KEY `idx_root_agent` (`root_agent_id`),
  KEY `idx_agent_path` (`agent_path`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB COMMENT='代理商主表';


-- ② 代理商层级关系表 (闭包表，支持任意层级查询)
CREATE TABLE `agent_hierarchy_closure` (
  `id`              BIGINT   NOT NULL AUTO_INCREMENT,
  `ancestor_id`     BIGINT   NOT NULL COMMENT '祖先代理商ID',
  `descendant_id`   BIGINT   NOT NULL COMMENT '后代代理商ID',

  UNIQUE KEY `uk_ancestor_descendant` (`ancestor_id`, `descendant_id`),
  KEY `idx_descendant_id` (`descendant_id`)

-- ③ 代理商资质文件表

  `doc_name`        VARCHAR(128) COMMENT '文件名称',
  `file_url`        VARCHAR(512) COMMENT '文件URL(OSS)',
  `expire_date`     DATE         COMMENT '有效期',
  `is_expired`      TINYINT      DEFAULT 0 COMMENT '是否过期',
  `verify_status`   TINYINT      DEFAULT 0 COMMENT '0-待审 1-通过 2-拒绝',
  `verify_operator` VARCHAR(64),
  `verify_time`     DATETIME,
  `create_time`     DATETIME     DEFAULT CURRENT_TIMESTAMP,

  KEY `idx_agent_id` (`agent_id`),
  KEY `idx_expire_date` (`expire_date`)

-- ④ 代理商操作日志表
CREATE TABLE `agent_operation_log` (
  `id`              BIGINT       NOT NULL AUTO_INCREMENT,
  `agent_id`        BIGINT       NOT NULL,
  `op_type`         VARCHAR(64)  COMMENT '操作类型 create/approve/suspend/resume/terminate',
  `op_content`      TEXT         COMMENT '操作内容(JSON记录变更前后)',
  `operator`        VARCHAR(64)  COMMENT '操作人',
  `operator_role`   VARCHAR(32)  COMMENT '操作人角色',
  `ip_address`      VARCHAR(64)  COMMENT 'IP地址',
  `remark`          VARCHAR(512) COMMENT '备注',
  `create_time`     DATETIME     DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_agent_id` (`agent_id`),
  KEY `idx_create_time` (`create_time`)
```

### 3.2 渠道归因域

```sql
-- ⑤ 推广渠道表
CREATE TABLE `agent_channel` (
  `channel_id`      BIGINT       NOT NULL AUTO_INCREMENT,
  `agent_id`        BIGINT       NOT NULL COMMENT '代理商ID',
  `channel_code`    VARCHAR(64)  NOT NULL COMMENT '渠道码(URL参数)',
  `channel_name`    VARCHAR(128) COMMENT '渠道名称',
  `channel_type`    VARCHAR(32)  COMMENT '渠道类型 qrcode/link/banner/sms/offline',
  `promo_url`       VARCHAR(1024) COMMENT '推广URL',
  `qr_code_url`     VARCHAR(512) COMMENT '二维码图片URL',
  `landing_page`    VARCHAR(512) COMMENT '落地页URL',
  `utm_source`      VARCHAR(64)  COMMENT 'UTM来源',
  `utm_medium`      VARCHAR(64)  COMMENT 'UTM媒介',
  `utm_campaign`    VARCHAR(64)  COMMENT 'UTM活动',
  `attribution_window` INT       DEFAULT 30 COMMENT '归因窗口期(天)',
  `click_count`     BIGINT       DEFAULT 0 COMMENT '点击次数',
  `register_count`  INT          DEFAULT 0 COMMENT '注册转化数',
  `open_account_count` INT       DEFAULT 0 COMMENT '开户转化数',
  `valid_from`      DATETIME     COMMENT '生效时间',
  `valid_to`        DATETIME     COMMENT '失效时间',
  `status`          TINYINT      DEFAULT 1,
  `create_time`     DATETIME     DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`channel_id`),
  UNIQUE KEY `uk_channel_code` (`channel_code`),
  KEY `idx_agent_id` (`agent_id`)
) ENGINE=InnoDB COMMENT='推广渠道表';


-- ⑥ 渠道点击日志表 (ClickHouse存储)
CREATE TABLE agent_channel_click_log (
  `log_id`          String    COMMENT '日志ID',
  `channel_code`    String    COMMENT '渠道码',
  `agent_id`        Int64     COMMENT '代理商ID',
  `device_id`       String    COMMENT '设备ID',
  `ip_address`      String    COMMENT 'IP地址',
  `user_agent`      String    COMMENT 'UA',
  `referer`         String    COMMENT '来源页',
  `click_time`      DateTime  COMMENT '点击时间',
  `click_date`      Date      COMMENT '点击日期',
  `is_registered`   UInt8     DEFAULT 0 COMMENT '是否转化注册',
  `register_time`   Nullable(DateTime),
  `customer_id`     Nullable(Int64) COMMENT '转化客户ID'
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(click_date)
ORDER BY (channel_code, click_date, click_time)
COMMENT '渠道点击日志(ClickHouse)';


-- ⑦ 客户-代理归因关系表 (核心表)
CREATE TABLE `agent_customer_relation` (
  `id`                  BIGINT       NOT NULL AUTO_INCREMENT,
  `customer_id`         BIGINT       NOT NULL COMMENT '客户ID',
  `agent_id`            BIGINT       NOT NULL COMMENT '直接归因代理商ID(最末级)',
  `l1_agent_id`         BIGINT       COMMENT '一级代理商ID',
  `l2_agent_id`         BIGINT       COMMENT '二级代理商ID',
```

```
多级分润计算逻辑伪代码：
─────────────────────────────────────────────────────────
function calcMultiLevelCommission(tradeEvent):

  // 1. 查询归因链路
  relation = getAgentCustomerRelation(tradeEvent.customerId)
  if relation == null: return  // 无代理归因

  // 2. 查询平台佣金
  platformCommission = tradeEvent.commission

  // 3. 计算各层分润
  results = []

  // L1代理分润
  if relation.l1AgentId:
    l1Rule = matchRule(relation.l1AgentId, tradeEvent, "L1")
    l1Commission = calcByRule(platformCommission, l1Rule)
    results.add({agentId: l1AgentId, amount: l1Commission, level: 1})

  // L2代理分润 (从L1中分出，不额外增加平台成本)
  if relation.l2AgentId:
    l2Rule = matchRule(relation.l2AgentId, tradeEvent, "L2")
    l2Commission = calcByRule(l1Commission, l2Rule)  // 从L1中计算
    results.add({agentId: l2AgentId, amount: l2Commission, level: 2})
    // L1实得 = l1Commission - l2Commission
    results[0].amount -= l2Commission

  // L3代理分润 (从L2中分出)
  if relation.l3AgentId:
    l3Rule = matchRule(relation.l3AgentId, tradeEvent, "L3")
    l3Commission = calcByRule(l2Commission, l3Rule)
    results.add({agentId: l3AgentId, amount: l3Commission, level: 3})
    results[1].amount -= l3Commission

  return results
─────────────────────────────────────────────────────────
```

### 5.2 关键技术设计

```
幂等设计 (防止重复计算分润)
──────────────────────────────────────────────────
幂等Key = MD5(source_type + source_id + agent_id)

Redis SETNX idempotent:{key} 1 EX 86400
  → 返回1: 首次处理，继续计算
  → 返回0: 已处理，直接返回

数据库唯一索引兜底:
  UNIQUE KEY uk_source_agent (source_type, source_id, agent_id)

──────────────────────────────────────────────────
账户余额并发控制 (防止超提)
──────────────────────────────────────────────────
方案: 乐观锁 + 数据库约束

UPDATE agent_account
SET available_balance = available_balance - #{amount},
    version = version + 1
WHERE agent_id = #{agentId}
  AND version = #{currentVersion}
  AND available_balance >= #{amount}

影响行数=0 → 余额不足或并发冲突 → 重试/拒绝

──────────────────────────────────────────────────
分布式事务 (分润计算+账户更新)
──────────────────────────────────────────────────
采用 Saga 模式：

Step1: 写入commission_detail (status=pending)
Step2: 更新agent_account余额
Step3: 写入account_flow流水
Step4: 更新commission_detail (status=normal)

任一步失败 → 触发补偿事务回滚
```
