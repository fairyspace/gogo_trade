# 证券行业消息中心架构设计文档（终版 V5.0）

> **设计说明**：本方案融合两份内部设计稿的精华，并参考高盛（Goldman Sachs）、摩根士丹利（Morgan Stanley）、Interactive Brokers、Robinhood、Stripe、Twilio 等全球头部金融机构与通知平台的工程实践，形成可直接落地的完整架构设计。

---

## 一、系统定位与边界

### 1.1 系统定位

消息中心是证券系统的**统一通知编排与送达中心**，职责边界如下：

**负责：**
- 接收上游业务事件，统一幂等校验与 Schema 校验
- 按策略编排通知任务（优先级、合规约束、用户偏好、频控、静默时段）
- 多语言模板渲染与内容快照留痕
- 多通道选择、发送、降级与重试
- 回执收集（送达、已读、确认）
- 站内信收件箱管理
- 审计日志、WORM 合规归档
- 用户通知偏好与同意授权管理
- 监控、告警、死信处理

**不负责：**
- 风控阈值计算与触发判定
- 交易撮合结果生成
- 资金账务记账
- 行情 tick 流处理
- 业务规则推导与决策
- **营销活动规则 / 赠金资格**（生产方为《营销与推荐系统设计》；本域只做投递）
- **资讯/公告 CMS 正文权威**（《内容与社区系统设计》；本域只按事件触发推送）
- **社群 IM 会话与消息存储**（内容与社区 IM；本域不承担群聊）
- **跟单订阅与排行榜**（社交交易域）

### 1.2 核心设计原则

| 序号 | 原则 | 说明 |
|------|------|------|
| 1 | **防重与幂等** | 以 `idempotency_key` 为幂等键，Redis SET NX + DB 唯一索引双重兜底 |
| 2 | **事件驱动** | 业务系统通过 Outbox/CDC 或内部 API 发布标准事件，消息中心订阅消费 |
| 3 | **策略与内容分离** | 通道路由、降级、频控、静默时段通过策略表配置，与模板内容解耦 |
| 4 | **合规与审计** | P0/P1 消息写入 WORM 不可篡改存储，全操作留痕，满足飞检与司法取证 |
| 5 | **降噪与防疲劳** | 时间窗口聚合 + 全局限频 + 用户偏好 + 静默时段四层防护 |
| 6 | **梯次降级** | P0/P1 关键消息具备跨通道自动重试与补偿，单通道故障不丢消息 |
| 7 | **有序性保障** | 同一业务实体的消息路由至同一 MQ Partition，保证状态流转严格有序 |
| 8 | **用户中心化** | 用户可自定义通知偏好，系统在合规约束内最大化尊重用户意愿 |
| 9 | **多实体与多辖区** | 支持同一集团下多法律实体(多牌照)、多司法辖区（CN/HK/SG/US/EU）差异化合规策略 |
| 10 | **供应商抽象与熔断** | 通道适配器模式 + 多供应商路由 + 熔断自动切换 |

## 二、国际大型金融机构通用实践参考

| 通用做法 | 为什么重要 | 本设计如何落地 |
|---|---|---|
| **统一通知平台** | 避免业务系统直连通道，减少耦合 | 所有业务统一走通知中心 API / 事件总线 |
| **Outbox / CDC 事件发布** | 防止业务成功但消息丢失 | 上游系统事务内写 outbox，CDC 发 MQ |
| **Transactional / Regulatory / Marketing 分流** | 避免营销消息影响关键通知 | 三类消息走不同策略、队列和频控 |
| **Preference + Consent Center** | 合规与用户体验双重要求 | 偏好中心 + 同意中心 + 退订中心独立建设 |
| **内容版本化 + 审批流** | 金融内容不能随意篡改 | 模板版本化管理、审批记录、发布记录、渲染快照留痕 |
| **多通道冗余但不盲目轰炸** | 关键消息要到达，营销不能打扰 | P0/P1 多通道降级，P3 严格限频 |
| **时区/静默时段感知** | 全球业务必须适配本地法规 | `timezone`、`quiet_hours`、`jurisdiction` 字段支持 |
| **送达证据与阅读证据分离** | `delivered` 不等于用户已知晓 | 通道回执、阅读回执、显式确认三层分离 |
| **Immutable Audit / WORM** | 法规、争议、飞检需要 | 审计快照 + SHA256哈希 + WORM/对象锁归档 |
| **Provider 抽象与熔断** | 单一供应商不可靠 | 通道适配器 + 多供应商路由 + 熔断自动切换 |
| **三层对象模型** | 事件→消息→通道任务，职责清晰 | Event / Message / ChannelTask 三表分离 |
| **Dead Letter Queue** | 失败消息不能丢失 | DLQ + 人工重放 + 告警 |

### 需要避免的常见"坑"

- 过度依赖"AI 自动文案"替代确定性模板
- 把"已送达"当作法律意义上的"已知悉"
- 营销与交易通知共用一套频控
- 全部消息都丢进 WORM，导致成本和检索灾难
- 业务系统直连短信/邮件供应商，难以治理
- 模板发布无审批流，导致合规风险

### 3.2 两条消息链路

#### A. 实时通知链路

适用于交易成交、资金变动、风控预警、安全告警、账户状态变化。

特点：低延迟、强幂等、P0/P1 多通道梯次降级、顺序保证。

#### B. 批量通知链路

适用于公告、投教、营销、服务类通知、活动运营。

特点：支持批量拆分、消息合并、严格限频、必须遵守退订与静默时段。

> 两条链路必须**逻辑隔离**，不得共用同一套队列和频控策略。

## 四、业务域与消息类型完整字典

### 4.1 一级分类（业务域）

| 业务域 | 核心场景 | 消息性质 | 监管依据 |
|--------|---------|---------|---------|
| **交易域 TRADE** | 委托/成交/撤单/废单、新股、可转债 | 事务型（幂等+顺序+WORM） | 交易所撮合规则 |
| **资金域 FUND** | 银证转账、冻结/解冻、手续费 | 事务型（幂等+顺序+WORM） | 证监会客户资金第三方存管 |
| **资产域 ASSET** | 持仓变动、分红、新股中签、质押 | 事务型（幂等+顺序+WORM） | 中登结算规则 |
| **风控域 RISK** | 保证金不足、强平、适当性 | 事务型（幂等+顺序+WORM+强制确认） | 《投资者适当性管理办法》 |
| **账户域 ACCOUNT** | 登录安全、KYC、权限变更、2FA | 事务型（幂等+WORM） | 网络安全等保2.0 |
| **合规域 COMPLIANCE** | 条款更新、风险揭示、法律公告 | 事务型（必须留痕，部分需签名确认） | 各辖区金融监管法规 |
| **IB域 IB** | 佣金结算、新客户引入、提现 | 事务型（幂等+WORM） | 经纪商合规要求 |
| **行情域 MARKET** | 股价预警、盘中异动 | 数据分发型（允许丢弃过期，不需WORM） | 无强制监管，独立降级 |
| **服务域 SERVICE** | 系统维护、客户服务提醒 | 通知型（限频，偏好控制） | 投资者权益保护 |
| **营销域 MARKETING** | 活动推荐、产品营销、投教 | 营销型（仅opt-in，严格退订+静默时段） | GDPR/个人信息保护法 |

## 五、核心数据模型（三层对象模型）

消息中心采用 **Event → Message → ChannelTask** 三层对象模型：

| 层级 | 对象 | 说明 |
|------|------|------|
| 第一层 | **Event（原始事件）** | 上游业务事实，做幂等、回放、审计的源头 |
| 第二层 | **Message（消息主对象）** | 消息中心生成的通知实体，一个事件可生成一或多条消息 |
| 第三层 | **ChannelTask（通道任务）** | 按通道拆分后的送达任务，每条消息可拆分为多个通道任务 |

## 六、消息与通道任务状态机

### 6.1 消息主状态（Message Status）

```
RECEIVED（事件已接收，待校验）
    │
    ▼
VALIDATED（校验通过，待策略编排）
    │
    ▼
RENDERED（模板渲染完成，快照已生成）
    │
    ▼
DISPATCHING（通道任务已拆分，正在投递）
    │                              │
    ▼                              ▼
PARTIALLY_SENT                  SENT
（部分通道成功，部分失败）        （所有通道任务已发出）
    │                              │
    └──────────┬───────────────────┘
               ▼
           COMPLETED（终态：所有通道达到最终状态）

FAILED（处理失败，已进入DLQ）
EXPIRED（超过expireAt，任意状态均可流转）
CANCELLED（人工取消，仅限未发送前）
```

### 6.2 通道任务状态（ChannelTask Status）

```
PENDING（待发送）
    │
    ▼
SENDING（发送中，已调用供应商API）
    │                        │
    ▼                        ▼
SENT（供应商已接收）         FAILED（供应商拒绝或超时）
    │                        │
    ▼                        ├──→ 可重试 → PENDING（指数退避）
DELIVERED                    └──→ 不可重试 → DEADLETTER
（平台确认送达）
    │
    ▼
READ（用户已读，客户端上报）
    │
    ▼
ACKED（用户主动确认/签名，ackPolicy要求时）

SUPPRESSED（被抑制：限频/退订/静默时段/黑名单）
DEADLETTER（死信，人工介入）
```

### 6.3 回执策略枚举（Ack Policy）

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| `NONE` | 无需回执 | P3 营销、低优通知 |
| `DELIVERED` | 平台确认送达即可 | P2 普通业务 |
| `READ` | 用户已读 | P1 重要业务 |
| `CONFIRM` | 用户主动点击确认 | P0 风控/强平 |
| `SIGNATURE` | 电子签名确认 | 合规类（条款、风险揭示书） |

```
P0 消息示例：MARGIN_CALL_WARNING（追保预警）

                │
                ▼
T+30s ─────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────┐

  │   否 → 触发降级通道：SMS 短信                     │

                │
                ▼

  ┌─────────────────────────────────────────────────┐
```

### 7.3 幂等防重流程

```
消费者收到 MQ 消息（idempotencyKey: "RISK_MARGIN_CALL_20250409_88001234_001"）
         │
         ▼
┌─────────────────────────────────────────┐
│  Redis GET Key: idempotent:{entityId}:{key}│
└──────────┬──────────────────────────────┘
           │
     ┌─────┴──────────────┬───────────────────┐
     │                    │                   │
   未命中              命中=SUCCESS         命中=PROCESSING
     │                    │                   │
     ▼                    ▼                   ▼
Redis SET NX          返回已处理结果       等待 500ms 重查
key=PROCESSING        幂等命中            仍 PROCESSING → ACK
TTL=24h               丢弃消息           （并发重复消费）
     │
     ▼
  执行业务逻辑
  （策略编排 → 模板渲染 → 通道任务拆分 → 投递）
     │
     ├── 成功 ──→ Redis 更新为 SUCCESS ──→ MQ ACK
     │
     └── 失败 ──→ 删除 Redis Key ──→ MQ NACK（允许重试）
                  DB 唯一索引(idempotency_key)作为最终兜底
```

## 十、优先级、SLA 与调度策略

| 优先级 | 典型场景 | 渠道策略 | 降级链 | 延迟 SLA | MQ 隔离 |
|--------|---------|---------|--------|---------|---------|
| **P0 紧急** | 强平、追保、异常登录、配股最后1日 | PUSH+INBOX 并行 | 30s→SMS → 5min→VOICE → 10min→人工 | INBOX落库<300ms；端到端<3s | 独占 VIP Topic |
| **P1 重要** | 成交回报、资金到账、KYC结果、废单 | PUSH+INBOX 并行 | 60s→SMS | INBOX落库<1s；端到端<5s | 交易专属 Topic |
| **P2 普通** | 委托状态、行情预警、分红、公告 | PUSH+INBOX | 无降级 | <10s | 业务混合 Topic |
| **P3 低优** | 营销、投教、关怀、费用明细 | INBOX为主 | 无降级 | 分钟级，错峰 | 低优离线 Topic |

> **SLA 基准说明**：Push 通道端到端延迟受 APNs/FCM/厂商通道影响，峰值可达 5-10s。**以站内信（INBOX）落库时间作为系统可控 SLA 的主要度量点**，Push 到达延迟单独监控，不纳入核心 SLA 考核。

## 十一、高并发与高可用关键机制

### 11.1 防疲劳策略

**时间窗口聚合（订单雨保护）：**

```
规则：同一用户，同一 msgType（ORDER_PARTIALLY_FILLED/ORDER_FILLED），
      5分钟窗口内触发次数 ≥ 5次 → 启动聚合模式

聚合消息格式："过去5分钟内 {N} 笔订单成交，累计成交金额 {amount} 元，点击查看详情"

例外：ORDER_REJECTED / FORCE_LIQUIDATION_EXEC / 所有P0消息 → 始终逐条实时推送
```

**全局限频（Redis 滑动窗口）：**

```
Key: rate_limit:{entityId}:{userId}:{templateCode}:{channel}:{window}

| 消息类型          | 渠道  | 每日上限 | 超出处理         |
|-------------------|-------|---------|-----------------|
| PRODUCT_MARKETING | PUSH  | 1       | 降级为 INBOX     |
| PRICE_ALERT       | PUSH  | 10      | 当日静默         |
| SERVICE 类        | PUSH  | 3       | 降级为 INBOX     |
| P0/P1             | 所有  | 不限     | 始终发送         |
```

**退订分级：**

```
全局退订（关闭所有Push）：24小时内生效，仅影响 P2/P3
分类退订（如仅关闭营销类）：立即生效
不可退订：P0/P1 风控类、账户安全类（allow_unsubscribe=0）
```

### 11.2 静默时段与时区

```
规则：
- P0/P1 紧急消息 → 突破静默时段，立即发送
- P2 普通消息 → 延迟至静默时段结束后发送
- P3/营销消息 → 必须遵守静默时段，违反则抑制
- 静默时段默认：22:00 - 08:00（用户本地时区）
- 支持按司法辖区配置法定静默时段
```

### 11.3 WORM 合规存储分层

| 消息类型 | 存储介质 | 保留期限 | 备注 |
|---------|---------|---------|------|
| P3 营销/服务类 | MySQL | 6个月 | 常规查询，到期归档 |
| P2 普通业务类 | MySQL + Elasticsearch | 12个月 | ES支持全文检索 |
| P1 重要业务类 | MySQL + ES + OSS归档 | 5年 | 含成交回报、资金变动 |
| P0 风控/强平类 | MySQL + ES + WORM冷存储 | **20年** | 含确认回执+时间戳+操作凭证 |

**WORM 写入内容规范（P0/P1）：**

```json
{
  "wormRecord": {
    "messageId": "msg_202504091029001",

```

## 十三、安全与合规设计

### 13.1 数据安全

- **传输加密**：全链路 TLS 1.3，内部服务间 mTLS
- **存储加密**：数据库透明加密（TDE），对象存储服务端加密
- **凭据托管**：供应商密钥通过 KMS/Vault 管理，`credential_ref` 引用路径，不落明文
- **敏感信息脱敏**：手机号、身份证号、卡号在日志和投递尝试快照中自动脱敏
- **权限隔离**：RBAC + 实体级数据隔离，模板/策略/审计操作均需鉴权

### 13.2 模板安全

- 模板变更必须走审批流（`approval_required`）
- 模板版本不可篡改，已发布版本只能 RETIRED 不能修改
- 渲染引擎限制为白名单（mustache/handlebars），不允许任意脚本执行
- 重要消息必须保留渲染快照（`rendered_snapshot_json`）

### 13.3 合规留痕

- 发送内容快照（渲染后不可变）
- 模板版本快照（版本号+校验和）
- 通道回执快照（每次供应商调用记录）
- 人工干预审计日志
- 用户确认/签名日志
- 退订/授权变更日志

### C.2 INBOX WebSocket 网关架构（补缺）

旧稿写"WS 推送"但无架构。v5.1 补：

```text
客户端 ──WSS──► [WS Gateway 集群]
                    │ 鉴权：Bearer Token（用户域签发）
                    │ 心跳：15s ping/pong
                    │ 连接注册：按 user_uid 路由到连接所在节点
                    ▼
              [连接注册中心 Redis]  user_uid → node_id
                    │
                    ▼
              [消息分发 Worker]
                    │ 订阅 notify.message.created.v1（按 user_uid partition）
                    │ 查连接注册中心 → 推送到对应 Gateway 节点
                    ▼
              [Gateway 节点] ──推送──► 客户端
```

- **断线重连**：客户端带 `last_msg_id` 重连，Gateway 查 `t_notify_inbox` 补发未读。
- **离线补偿**：超 24h 未读消息落 INBOX 表，客户端上线后拉取。
- **未读数推送**：消息送达后推 `unread_count` 增量。
- **连接数容量**：单节点 10 万连接，集群 100 万；超限拒绝新连接（返回 `NTF_RATE_LIMITED`）。
